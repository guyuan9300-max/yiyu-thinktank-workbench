## 核心定义

益语智库的数据中心**不是"资料库",而是"客户故事理解系统"**。它要把任务、文件、对话、外部情报、成长经验、组织计划、飞书资料等不同来源的信息,先分清性质,再进入不同的数据层,最后交叉验证,形成一个能说清楚"事实、判断、关系、时间线、来龙去脉、风险和下一步"的客户故事网。

---

## 一、核心目标:不是把数据汇进来,而是让系统真正"懂一个项目"

益语智库 2.0 的底层价值,不是把 Word、任务、聊天、会议纪要都存起来,而是把这些材料转化成一张可以被 AI 持续读懂的"客户档案网"。

下一阶段数据中心要解决的核心问题是 **5 个**:

1. **进来的每一条信息到底是什么性质**:事实、承诺、判断、计划、关系、风险、经验、偏好,还是系统日志?
2. **这条信息是谁说的**:客户官方、我方做的、外部媒体、用户口述判断、还是 AI 抽取?**"谁说的"决定了置信度**。
3. **这条信息能证明什么**:任务证明"我们做过什么",合同证明"客户正式确认过什么",对话证明"用户当时怎么理解",新闻证明"外部世界如何描述"。
4. **它和其他信息有什么关系**:支持、补充、冲突、更新、否定或解释。
5. **它能不能推动下一步行动**:真正高级的数据中心不只回答"发生过什么",还知道"现在该澄清什么、推进什么、提醒谁、更新哪份报告"。

---

## 二、整体设计原则:先分型,再计算,再进入故事网

**所有信息先进入"来源分型",再进入"事实/判断/关系/行动"的结构化层,最后进入"客户故事网"。**

不能一上来就把所有内容都抽成 atomic_facts。因为有些内容不是事实:
- 用户对客户性格的判断
- AI 的战略研判
- 成长中心的方法卡
- 组织计划里的目标
- 飞书里的提醒卡片

它们都重要,但**不能以同一种方式存储和计算**。

四项不要做的事:
- 不是所有数据都进入 atomic_facts
- 不是所有事实都拥有同等置信度
- 不是所有冲突都需要 AI 自动裁决
- 不是所有判断都能直接变成客户档案的 ground truth

## 三、工作台会进来哪些信息(8 板块)

### 1. 任务与日程:行为流、承诺流、推进流

**进入的信息**: 任务标题、描述、责任人、截止时间、完成状态、关联客户、关联项目、关联事件线、任务笔记、周复盘、会议 action items、决策记录、任务与计划项的挂接关系。

**数据性质**: 主要不是"客户事实",而是"我们做了什么、答应了什么、推进到哪一步"。

**进入**: events / commitments / tasks / decisions / task_plan_links / evidence_links

**计算价值**: 项目故事的"行动脊柱" — 推进节奏 / 谁负责 / 哪些承诺履行 / 哪些任务对应某个战略计划 / 客户关系如何推进。

**置信度**: 较高(用户主动)。但证明"我们记录了这件事",不一定证明"客户真实认可了"。

### 2. 客户工作台:客户资料、对话、提案、判断和工具生成内容

**进入的信息 5 类**:
1. 原始资料(Word/PDF/Excel/会议纪要/合同/方案)
2. 用户口述或贴入内容("我觉得客户现在真正担心的是预算风险")
3. AI 生成内容(总结、方案、回答)
4. **用户纠错**("这个预算不是 500 万,是 300 万")— ★ 高优先级信号
5. 工具生成结果(模板填充、链接转资料、粘贴生成文档)

**置信度判断**:
- 客户官方文件最高
- 用户纠错很高
- 用户即时判断中等偏高,但有主观性
- AI 生成内容不能直接当事实
- 链接抓取看来源级别

### 3. 智能文件导入:资料关系、文件角色、口述背景和结构化抽取

**进入的信息**: 用户讲述的背景 / 文件池里的文件 / 文件角色 / 关键人物 / 机构关系 / 时间锚点 / 承诺 / 风险 / 用户对 AI 解析结果的编辑 / 用户补充说明后的重新解析。

**计算价值**: "项目故事网"的起点 — 同时拥有三种信息:
- 用户讲述的故事背景
- 文件本身的正式内容
- 用户告诉 AI 这些文件之间是什么关系

**置信度按文件角色分**:
- 客户合同/章程/正式方案:高
- 合作方提交方案:中高(可能只是主张)
- 益语顾问产出:我方判断,不等同于客户事实
- 外部对标:参考材料,不应直接写入客户事实
- 政策文件:本身高,跟客户项目关系需要判断

### 4. 战略陪伴:AI 理解、用户裁决、事实澄清和战略研判反馈

**4 类数据**:
1. AI 研判(基于材料形成的判断,不是事实本身)
2. 事实澄清(AI 标记的不确定事实或冲突事实)
3. 用户裁决(采纳/修正/忽略 — 高价值反馈)
4. 客户画像叙事(6 段)

**置信度**: 用户在战略陪伴里做出的采纳和修正,**最高权重之一**。但保留原始证据,不简单删除旧事实,标注 superseded / 修正时间 / 修正理由 / 原始来源。

### 5. 资讯情报站:外部视角、公开信号、舆情和行业变化

**3 层来源**:
- L1:客户官网、政府官网、官方公告
- L2:主流媒体、行业媒体
- L3:一般网络、自媒体、论坛

**用户反馈动作**: 关注、忽略、转任务、写为提案

**计算价值**: 客户故事网的"外部镜子" — 外部对客户怎么看 / 舆情风险 / 行业政策变化 / 公开信息是否和内部资料冲突 / 哪些情报应该变成任务或提案。

### 6. 成长中心:个人经验、方法、成长证据和组织能力信号

**进入**: handbook_entries / growth_signal_events / growth_evidence_records / method_cards / learning_feedback / ability_profile

**性质**: 不是客户事实,而是"人的能力和方法"。

**对客户事实没有直接证明力**,但可以参与:
- 写作风格生成
- 方法推荐
- 相似项目经验调用
- 团队能力判断
- 组织方法沉淀

### 7. 组织计划工坊:战略目标、计划项和任务挂接关系

**进入**: org_plans / plan_items / plan_task_links / plan_progress / strategic_objectives / team_contribution_records

**计算价值**: 数据中心回答"某个客户任务为什么重要 / 这件事属于哪个组织目标 / 哪些客户项目支撑了组织战略"。

组织计划数据是"项目故事网"和"组织战略网"的连接点。

### 8. 飞书联动:外部协作触点,不是主记忆系统

**两个原则**:
1. 飞书只作为来源和触达层,不作为最终客户记忆库
2. 所有飞书数据必须显式绑定 org_id、user_id、client_id 或 project_id

## 四、数据中心应该如何分别计算这些信息

### 机制一:来源置信度计算

| 来源类型 | 代表内容 | 初始置信度 | 计算原则 |
|---|---|---|---|
| 客户官方文件 | 合同/章程/正式方案/官网公告 | 高 | 可作为事实基准,但要注意版本 |
| 用户确认事实 | 用户在澄清面板采纳或修正 | 最高 | 可作为当前系统权威值 |
| 任务与承诺 | 任务/action item/会议承诺 | 高 | 证明"我们记录/承诺/推进过" |
| 用户口述判断 | 工作台对话/智能导入讲述 | 中高 | 适合作为隐性判断,不宜直接当事实 |
| AI 抽取 | 从文件/对话中抽出的结构化内容 | 中 | 必须挂原文引用和抽取置信度 |
| 外部公开资料 L1 | 官网/政府/官方公告 | 高 | 可做外部确认 |
| 外部媒体 L2 | 主流媒体/行业媒体 | 中 | 可做外部观察 |
| 一般网络 L3 | 自媒体/论坛 | 低 | 只能做信号,不做事实 |
| 成长中心方法 | 个人手册/方法卡 | 高但限域 | 对方法有效,对客户事实无证明力 |
| 系统日志 | job/同步/投递/失败记录 | 高但技术性 | 证明系统发生过什么,不证明业务事实 |

**关键**: 置信度不是固定数字,而是不断变化的状态:
- 后被客户正式文件确认 → 提高
- 被新版本文件推翻 → 旧事实降权或被 supersede
- 用户修正 → 进入待复核或权威更新
- 外部资料冲突 → 不一定推翻内部事实,但触发澄清

### 机制二:事实聚类与版本计算

**事实簇** 例子:
```
主体:某项目
属性:预算
候选值:300 万 / 500 万
当前权威值:300 万(用户修正)
旧版本:500 万
冲突状态:已由用户修正,但仍建议保留外部口径差异
下一步:如对外材料涉及预算,需要确认"总投入"和"当前合作预算"是不是两个口径
```

不是简单选一个数字,而是把背后的口径差异讲清楚。

### 机制三:跨源印证与冲突检测

对每个事实簇做三件事:
1. **找支持** — 是否多个独立来源指向同一事实
2. **找冲突** — 不同来源给出不同时间/金额/角色/版本/状态
3. **找解释** — 冲突不一定是错误,可能是版本/口径/角色变化/外部滞后/口述不完整

**故事矛盾**(从事实矛盾扩展):
- 同一人物角色变化
- 同一项目目标变化
- 同一客户真实需求和公开表述不一致
- 任务推进和战略计划脱节
- AI 研判和用户判断冲突
- 外部舆情和客户内部说法冲突

### 机制四:故事网计算

**项目不是一堆事实,而是一条故事链**:

> 谁参与了这件事 → 为什么发生 → 什么时候发生 → 发生了什么 → 谁做了决定 → 谁承诺了什么 → 我们做了什么 → 客户如何反馈 → 产生了什么风险 → 下一步怎么走

**12 类节点 + 12 类关系** (见 § 五)

### 机制五:澄清优先级计算

```
澄清优先级 = 影响程度 × 冲突程度 × 当前行动依赖 × 时间紧迫度 × 置信度不足
```

**高优先级澄清**:
- 关键决策人不清
- 项目预算冲突
- 合同版本冲突
- 责任人和承诺不清
- 下一步任务依赖这个事实
- 对外材料即将引用这个事实
- 外部舆情和客户内部说法冲突
- AI 多次低把握度生成同一段叙事

## 五、什么叫"最高级别的澄清" — 4 层

### 第一层:事实澄清(回答"这个事实到底是什么")
- 项目预算到底是 300 万还是 500 万?
- 张真是不是法人代表兼理事长?
- 这份方案是哪一版?
- 这个会议是 5 月 19 日还是 5 月 20 日?

### 第二层:口径澄清(回答"为什么不同材料说法不一样")
- 300 万是本期合作预算,500 万是项目总预算
- 官网还没更新,但内部已经换届
- 客户对外说"项目进展顺利",但内部任务长期卡住
- 媒体说的是品牌宣传,客户真实关心的是执行风险

**比事实澄清高级 — 不是选对错,是解释冲突来源**

### 第三层:关系澄清(回答"谁真正影响这件事")
- 谁是名义负责人 / 实际推进人 / 最终拍板人
- 谁影响客户态度 / 信任关系 / 潜在阻力

**很多项目失败不是因为事实错,而是因为关系理解错。关系澄清是咨询系统的核心价值。**

### 第四层:战略澄清(回答"这件事对项目走向意味着什么")
- 客户是真的想推进,还是只是礼貌性沟通?
- 项目卡住是能力 / 资源 / 决策 / 内部关系问题?
- 外部舆情会不会改变资方判断?
- 这个客户是否进入深度陪伴阶段?
- 下一步应该补材料 / 做会议 / 出提案 / 还是暂停推进?

## 六、如何形成"最广阔的故事网" — 4 条故事线

### 线索一:时间线("事情怎么一步一步发生")
初次接触 → 背景资料进入 → 关键会议 → 客户提交资料 → 我方反馈 → 形成承诺 → 任务推进 → 方案迭代 → 风险暴露 → 决策变化 → 输出材料 → 后续行动

### 线索二:人物关系线("谁在推动、谁在决策、谁在影响")
不只姓名职位,还包括:正式角色 / 实际角色 / 决策权 / 沟通路径 / 信任关系 / 偏好 / 历史互动 / 对项目态度 / 与其他人的关系

### 线索三:材料证据线("我们凭什么这么判断")
每个判断都必须能回到材料:哪份文件 / 哪段原文 / 哪次会议 / 哪条任务 / 哪个用户纠错 / 哪条外部情报 / 哪个 AI 研判 / 哪次人工确认

### 线索四:行动结果线("这些理解最后推动了什么")
哪个判断生成了任务 / 哪个情报转成了提案 / 哪个风险触发了会议 / 哪个澄清修改了客户档案 / 哪个方法卡被复用 / 哪个计划项被任务推进

**没有行动结果线,故事网就会变成漂亮的知识图谱;有了,故事网才是咨询团队的工作底座。**

## 七、建议的数据架构 · 7 层

### 第 1 层:来源登记层 Source Registry

字段:
- source_id / source_type / source_channel / source_owner
- client_id / project_id / user_id
- source_time / capture_time
- visibility_scope / content_hash / version_id
- source_role / initial_confidence / raw_reference

**作用**: 先知道这条信息从哪里来、属于谁、能给谁看、它是什么性质。

### 第 2 层:原始内容层 Raw Material Layer
原始文件 / OCR 文本 / 文档 chunks / 聊天消息 / 用户口述 / 会议转写 / 任务笔记 / 情报原文 / 飞书妙记 / 飞书云文档 / 外部网页抓取

**原则**: 原始内容**不可被 AI 覆盖**,只能增加版本、引用和解析结果。

### 第 3 层:结构化抽取层 Extraction Layer
entities / events / relationships / commitments / decisions / risks / atomic_facts / quotes / opinions / methods / preferences / action_items

**原则**: 抽取结果必须带来源、置信度和原文引用。

### 第 4 层:事实簇与验证层 Verification Layer
fact_clusters / fact_assertions / supporting_evidence / conflicting_evidence / superseded_facts / user_confirmations / contradiction_records / clarification_questions

**原则**: 冲突不强行合并,先保留,再澄清。

### 第 5 层:故事图谱层 Story Graph Layer
project_story_graph / client_relationship_graph / timeline_graph / decision_chain / commitment_chain / risk_chain / evidence_chain / action_chain

**原则**: 每个项目都要能回答"来龙去脉"。

### 第 6 层:叙事与研判层 Narrative Layer
客户本质 / 合作历史 / 业务介绍 / 人物关系 / 时间线 / 下一步 / AI 主动研判 / 风险提示 / 低把握度段落 / 待补材料清单

### 第 7 层:行动与反馈层 Action Layer
任务 / 周复盘 / 提案 / 飞书提醒 / 会议问题清单 / 客户报告 / 成长方法卡 / 组织计划进度 / 数据中心重新理解任务 / 人工澄清任务

**原则**: 所有 AI 理解都要能进入行动闭环。

## 八、自动运转流程 · 8 步

1. **进入前先分型** — 判断数据类型/谁说的/性质/绑定/敏感权限
2. **保留原文** — 不丢任何原始形态
3. **抽取结构** — 不同数据类型抽不同结构
4. **归并同类项** — 谨慎合并:高置信同义自动 / 同音字待确认 / 金额冲突进簇 / 角色冲突进澄清 / 版本冲突进链
5. **交叉验证** — 客户官方 + 任务行为 + 用户口述 + 外部公开 + 后续行动 + 旧版本 + AI 研判 + 人工澄清 各维度
6. **生成澄清问题** — 不简单说"有冲突",生成可回答的问题
7. **用户确认后写回权威层** — confirmed_fact / corrected_fact / superseded_fact / dismissed_fact / clarification_record / confidence_update / narrative_refresh_trigger
8. **更新客户故事网和行动系统** — 自动更新客户画像 / 时间线 / 人物关系图 / 风险列表 / 下一步建议 / 周复盘 / 任务提醒 / 提案草稿 / 飞书卡片 / 成长方法沉淀

---

## 九、最重要的产品形态 · 项目故事卡

每张项目故事卡包含 **10 段**:
1. 项目背景
2. 当前阶段
3. 关键人物
4. 时间线
5. 核心事实
6. 关键判断
7. 冲突与待澄清
8. 风险
9. 下一步
10. 证据来源

**用户真正想要的东西**: 不用翻 100 个文件,也能知道一个项目到底发生了什么、现在怎么看、下一步怎么做。

## 十、现阶段最应该修正的几个数据断点(6 个)

### 断点 1 · 工作台对话没有充分进入数据中心
建议先分:用户原话 / 用户判断 / 用户纠错 / AI 回答 / AI 被纠正 / 可抽取事实 / 可沉淀经验 / 可转任务内容。
用户说的"我觉得客户真正关心的是……"应进入 **interpretive_claims**,再通过后续材料验证。

### 断点 2 · 资讯情报没有进入外部证据层
建议建立 **external_evidence_cards**(已有 schema 0 行):
- source_level / source_url / publish_time
- mentioned_entities / related_client / related_project
- sentiment / risk_level / relevance_score
- action_suggestion / support_or_conflict_with_existing_facts

### 断点 3 · 智能文件导入和工作台口述处理不一致
建立统一**用户口述处理器** **oral_or_freetext_claims**:
- 智能导入讲述 / 工作台贴入长文 / 语音转文字 / 手机 AI 聊天 / 飞书妙记摘录 / 用户补充说明 都先进入

### 断点 4 · 成长中心的方法论没有和项目场景连接
个人手册方法卡增加适用场景标签:客户类型 / 项目阶段 / 任务类型 / 写作类型 / 沟通对象 / 风险类型。

### 断点 5 · 组织计划与客户事实还没充分连接
增加:plan_item → client_project → task → evidence → outcome → review

### 断点 6 · 飞书不能成为旁路知识入口
**飞书只做**: 明确绑定客户/项目的资料导入 / 明确授权的妙记 / 明确用户范围的任务提醒 / 明确客户上下文的卡片入口 / 明确权限快照的文件引用

**飞书不能做**: 群内自由问全局客户知识 / 自动全组织爬取云盘 / 多维表格当客户全文知识库 / MCP 直接写客户事实 / 飞书权限替代益语权限

### 阶段 1 · 数据分型表 + 统一来源登记

**目标**: 任务、文件、外部情报、用户口述都能进入对应结构。

## 十二、最终判断

益语智库下一阶段的数据中心,不应该只是"让所有板块的数据进新数据库"。**更准确的目标**:

> 让每一种数据按照自己的性质进入正确的位置,再通过来源、时间、关系、证据和用户澄清进行交叉计算,最终形成一个能解释客户项目完整故事的理解系统。

**真正高级的地方,不是资料多,而是它能分清**:
- 哪些是事实 / 哪些是判断 / 哪些是承诺 / 哪些是外部观察 / 哪些是用户隐性知识
- 哪些已经被确认 / 哪些仍然冲突 / 哪些影响下一步行动

最后形成的不是"知识库",而是一张**持续生长的客户故事网**:有背景、有时间线、有人物关系、有决策逻辑、有证据来源、有行动结果、有风险判断、也有待澄清问题。

**这样数据中心才会真正成为益语智库的核心能力**:**不是帮用户保存信息,而是帮用户把复杂项目想清楚。**
