# Phase 0 · 功能反推工作稿（第二轮）

> 状态：**反推中**
> 日期：2026-05-15
> 上游文档：`DATA-CENTER-RELATIONSHIP-GRAPH-PLAN.md` v0.2
> 本稿目的：从用户在功能里要什么倒推关系网能力，验证 v0.2 的 4 层结构够不够，定 Phase 1 硬交付范围。
> 评分公式：**场景优化价值 = 频次（1-5）× 价值感知（1-5）**，只对 ≥12 分场景做硬交付。

---

## 0 · 反推工作方法

1. 按"看的视角"把 20 个 ≥12 分场景分 4 组（部门视角 / 客户视角 / 个人视角 / 时点机制）
2. 每组反推：用户要看到什么 → 关系网必须提供什么 → v0.2 有没有 → 缺什么
3. 汇总所有缺口 → 定 v0.3 结构修正
4. 倒推 Phase 1 硬交付范围

---

## 1 · 部门视角反推

涉及场景：
- #1 本周概览（25 分）
- #23 部门信号（25 分）
- #24 部门自动周复盘（20 分）
- #29 个人仪表盘（12 分）

### 1.1 用户在这一组场景里要看到什么（业务语言）

**本周概览**（最高频，每周一打开第一眼看的）：
- 这个部门本周到底发生了什么？（按时间排）
- 本周新增了哪些承诺？给谁？什么时候到期？
- 本周做出了什么决策？
- 本周出现了什么风险/卡点？
- 部门成员本周分别做了什么？
- 跟客户的所有互动一览？

**部门信号**（每周看一次，关键信号提醒）：
- 哪些主线快到期还没动？
- 哪些客户长期没接触？
- 哪些承诺没有承接者？
- 哪些人物状态需要留意（健康/离职/绩效）？
- 哪些资方有流失风险？
- 哪些主线的责任人最近变更了？

**部门周复盘自动草稿**（每周末看一次）：
- 本周做完了什么（完成的主线/任务/承诺）
- 本周新承诺什么
- 本周卡住什么
- 下周关键节点（哪条主线下周到期）
- 部门跟其他部门、跟客户的本周互动统计
- 部门成员本周表现汇总

### 1.2 关系网必须提供什么能力

| 能力 | v0.2 有没有 |
|---|---|
| 事实卡片按"部门"维度筛 | ❌ 缺 |
| 事实卡片按"本周/本月/季度"时间窗口筛 | 🟡 半有（有时间锚，没时间窗口接口） |
| 主线挂载部门 | ❌ 缺 |
| 人物挂载部门 | ❌ 缺 |
| 主线生命周期事件流（active → blocked → resolved 的时间序） | ❌ 缺 |
| 主线到期检测（expected_completion_at vs now） | 🟡 半有 |
| 客户冷却检测（最近互动 vs now） | ❌ 缺 |
| 承诺-承接对应链（这条承诺有没有人接) | ❌ 缺 |
| 人物状态变更监听（离职/休假触发信号） | ❌ 缺 |
| 跨客户聚合（部门下所有客户的本周状态合并看） | ❌ 缺 |
| 周快照对比（本周新增 vs 上周已存在的） | ❌ 缺（v0.2 把它推到 Phase 4） |

### 1.3 缺什么 → 暴露什么结构问题

**v0.2 文档完全没考虑"部门维度"**。我之前一直在想"客户维度 + 人物维度"，但**部门是组织内部维度** —— 任务与日程模块的核心 —— 我完全漏了。

**v0.2 把"周快照"推到 Phase 4 是错的**。本周概览/部门信号/周复盘**都需要时间窗口对比**，必须 Phase 1 就有，不能等 Phase 4。

**v0.2 缺"事件流"层**。我只设计了"主线当前状态"，没设计"主线状态变化的时间线"。但"本周新承诺/本周卡住"这类必须有事件流才能算。

## 2 · 客户视角反推

涉及场景：
- #6 客户工作台 AI 问答（25 分）
- #7 客户档案首页（20 分）
- #8 客户事件线总览（20 分）
- #10 客户战略陪伴主页（25 分）
- #11 客户关系书（20 分）
- #12 澄清面板（20 分）
- #16 资讯推荐（20 分）
- #27 客户冷却提醒（20 分）

### 2.1 用户在这一组场景里要看到什么

**客户战略陪伴主页**（每次切换客户进来都看）：
- 这个客户是谁、做什么的（治理结构 + 业务结构 + 服务对象）
- 当前在推进哪几条主线（按颗粒度分层）
- 哪些主线推进顺利、哪些卡住、哪些逾期
- 当前关键人物 + 状态
- 这周新增了什么变化
- 有什么矛盾点 / 待澄清

**客户关系书**（点开自动生成的 DNA 文档）：
- 8 段完整结构（机构信息 / 战略 / 业务 / 人物 / 合作 / 主线 / 风险 / 未澄清）
- 每一句话能钻取到原始证据
- 文档自动随事件更新

**澄清面板**（AI 主动提问）：
- AI 提出针对性问题（达到 10 问水平）
- 每个问题带引文支撑
- 一键打开原文档
- 答案自动沉淀

**客户工作台 AI 问答**：
- 用户问任何客户相关问题，答案带"关系网背景"
- 答案末尾列引用：3 次会议 / 5 份文档 / 2 条主线
- 如果问的是当前主线相关，自动接续主线上下文

**资讯推荐**（每天打开看）：
- 不是给所有人推所有新闻
- "您手上日慈续约这条主线，今天有这篇相关"
- "您客户洞察能力低于岗位中位，这 2 篇 case 跟您手上项目类型一致"

**客户冷却提醒**：
- 哪些客户超过 X 天没接触
- 冷却的原因是什么（没主线推进 / 关键人物离职 / 别的）

**事件线总览**：
- 客户下所有主线的时间线
- 哪些主线已结束、哪些仍在推进
- 时间线上的关键里程碑

### 2.2 关系网必须提供什么能力

| 能力 | v0.2 有没有 |
|---|---|
| 客户关系网快照（含人物/主线/事实卡片整合） | 🟡 部分（4 层结构有，但没整合接口） |
| 主线按颗粒度分层显示 | ✅ 有（thread_level） |
| 主线推进状态（顺利/卡住/逾期）量化 | 🟡 字段有，无统一判定标准 |
| 客户最近互动统计（互动频率 + 最近一次时间） | ❌ 缺 |
| 资讯-主线匹配（资讯按主线类型分类，匹配客户主线） | ❌ 缺 |
| AI 问答带关系网背景 | ❌ 缺（需"客户当前关系网快照"作为 prompt context） |
| 澄清问题自动生成（基于关系网缺口） | ❌ 缺 |
| 澄清问题质量评分（达到 80 分） | ❌ 缺 |
| 文档引文支撑（每个 AI 表述都带证据钻取） | ❌ 缺 |
| 关系书 8 段自动生成 | ❌ 缺 |
| 关系书段落自动更新（事件触发） | ❌ 缺 |

### 2.3 缺什么 → 暴露什么结构问题

**v0.2 设计了 4 层，但没设计"4 层如何被聚合成一份关系书"**。第 4 层"客户关系书"在 v0.2 里只是一个生成动作，**没明确每段读哪几层数据 / 怎么组合**。

**资讯推荐这种"主线-资讯匹配"能力，v0.2 完全没考虑**。这跟资讯情报站模块有协同需求，我之前没列。

**"AI 问答带关系网背景"**这个能力，v0.2 也没设计 prompt context 的接口。客户工作台 AI 问答现在是检索式的，要改成"先读关系网 → 答案锚定到关系网"。

## 3 · 个人视角反推

涉及场景：
- #19 个人能力快照（15 分）
- #21 个人产出证据钻取（15 分）
- #25 个人自动周复盘（20 分）
- #28 资方风险信号（15 分，归属客户视角也行）

### 3.1 用户在这要看到什么

**个人能力快照**（每月 1-2 次）：
- 我这个季度产出了什么
- 我的 6 维能力得分（基于真实证据）
- 跟我岗位 P80 比，哪些维度有 gap
- 我最近的代表性产出（5-10 条）

**个人产出证据钻取**（每月偶尔）：
- "客户洞察 78 分由这 5 件事得来"
- 每件事点开看原始 task / document / meeting

**个人自动周复盘**（每周末）：
- 我本周做了什么任务（按主线分组）
- 我本周新承诺了什么、卡在哪
- 我本周参加了哪些会议、产出了什么决策
- 下周我有什么节点

**资方风险信号**：
- 这家客户对接人状态异常（如章方）
- 资方可能流失的预警 + 原因

### 3.2 关系网必须提供什么能力

| 能力 | v0.2 有没有 |
|---|---|
| 任务 → operator 挂载 | ✅ 已有（owner_id） |
| 任务 → 主线挂载 | ✅ 已有（event_line_id） |
| operator 的能力维度证据汇总 | ❌ 缺（成长中心需求） |
| operator 周快照（本周 / 季度对比） | ❌ 缺 |
| 岗位锚点（P50 / P80） | ❌ 缺（v0.2 推到 Phase 3+） |
| 个人视角投影（按 operator 切关系网） | ❌ 缺 |
| 资方风险监听（关键人物状态变化 → 触发） | ❌ 缺 |

### 3.3 缺什么 → 暴露什么结构问题

**v0.2 把"成长中心"相关能力推得太晚**。个人周复盘是高频功能（20 分），但 v0.2 把"人物快照"放在 Phase 3。

**"按人物视角切关系网"是 v0.2 没考虑的第三种切面**（之前缺部门切，现在又缺个人切）。

## 4 · 时点机制反推

涉及场景：
- #2 任务智能简报（15 分）
- #4 任务相关文档自动关联（15 分）
- #5 客户拜访前自动备齐资料包（15 分）
- #26 承诺到期提醒（25 分）

### 4.1 用户在这要看到什么

**任务智能简报**（每次打开任务都看）：
- 这个任务挂在哪条主线
- 这个主线现在推进到哪
- 这个任务的客户最近发生了什么
- 这个任务的相关人物状态
- 这个任务相关的文档列表

**任务相关文档自动关联**：
- 任务标题/描述里提到的实体 → 自动关联相关文档
- 不需要用户手动绑定附件

**客户拜访前自动备齐资料包**：
- 即将拜访客户 X → 自动生成"拜访前简报"
- 含：客户当前关系网快照 + 最近 30 天事件 + 待澄清问题 + 关键风险

**承诺到期提醒**（每天看）：
- "今天到期的承诺：① 给詹瑶发更新价值观（5/1）② ..."
- "明天到期的承诺：..."
- "本周到期的承诺：..."

### 4.2 关系网必须提供什么能力

| 能力 | v0.2 有没有 |
|---|---|
| 任务 → 主线快照映射（任务详情自动加载主线背景） | ❌ 缺 |
| 任务 → 客户最近事件流映射 | ❌ 缺 |
| 实体识别 → 文档自动关联 | ❌ 缺（部分依赖工作台 FTS） |
| 拜访前简报生成 | ❌ 缺 |
| 承诺到期实时检测（每日 cron） | ❌ 缺 |
| 多种提醒（今日/明日/本周）的承诺归类 | ❌ 缺 |

### 4.3 缺什么 → 暴露什么结构问题

**v0.2 没考虑"被动提醒"机制**。承诺到期提醒（25 分）跟周快照一样必须有自动驱动器，不能等用户主动看。

**"任务智能简报"是另一种关系网投影方式**：以一个任务为中心，把它周围的关系网（主线/客户/人物/文档）拉出来。这是第四种切面。

## 5 · 反推暴露的所有能力缺口汇总

把上面 4 组反推暴露的所有缺口归类。

### 5.1 v0.2 现有 4 层就能补的（属于"加字段"）

| 缺什么 | 加在哪 |
|---|---|
| 人物 → 部门挂载 | 第一层 花名册：加 department_id |
| 主线 → 部门挂载 | 第三层 主线：加 department_id |
| 主线推进状态量化标准（顺利/卡住/逾期） | 第三层 主线：加一段判定规则 |
| 客户最近互动时间统计 | 客户战略画像表：加 last_interaction_at + 互动频率 |
| 资方-客户关系标记 | 客户战略画像表：funder_relations_json（v0.2 已设计） |

### 5.2 v0.2 现有 4 层装不下的（属于"新加层 / 新加能力"）

这一类是 **v0.2 完全没设计** 的：

#### 缺口 1 · 多维投影层

**v0.2 只能"按客户切关系网"，但功能需要"按部门切 / 按人物切 / 按任务切 / 按时间切"**。

要新加一个能力层：「多维投影引擎」—— 给定一个视角维度（部门/人物/任务/时间窗口），输出对应切面的关系网视图。

**4 种主要投影**：
- 部门投影：本周概览 / 部门信号 / 部门周复盘
- 个人投影：个人能力快照 / 个人周复盘
- 任务投影：任务智能简报
- 时间投影：本周新增 / 周快照对比

#### 缺口 2 · 事件流层

**v0.2 只记主线"当前状态"，没记"状态变化序列"**。但"本周新承诺/本周卡住"这类信号必须有事件流才能算。

要新加：**主线生命周期事件表**（每条状态变化是一条事件，带触发时间、触发原因、触发证据）。

#### 缺口 3 · 提醒/触发层

**v0.2 没考虑"被动驱动"**。承诺到期、客户冷却、人物状态变化、资方风险，都需要后台定期扫描 + 主动推送给用户。

要新加：**提醒/触发引擎**：
- 每日扫描：承诺到期、客户冷却、主线超期
- 事件触发：人物状态变化（离职/休假）、资方信号变化
- 输出：写入"待办提醒"表，工作台/任务与日程消费

#### 缺口 4 · 关系书生成引擎

**v0.2 设计了第 4 层"客户关系书"，但没设计"怎么生成"**。

要新加：**关系书生成引擎**：
- 8 段每段绑定一个"读什么数据"的规则
- 段落级自动更新触发条件
- 段落级可信度评分
- 段落级钻取链路

#### 缺口 5 · 澄清问题生成引擎

**v0.2 提了澄清面板，但没设计"问题怎么自动生成"**。

要新加：**澄清问题生成引擎**：
- 扫描关系网里的缺口（主线维度未澄清 / 人物归一未确认 / 矛盾点 / 信息缺失）
- 每个缺口生成一个针对性问题（达到 80 分质量评分）
- 答案回流自动 patch 关系网

### 5.3 上游工作台需要新增的协同需求

这次反推暴露**工作台线程要给我的字段** v0.2 之前没考虑：

| 工作台新增需求 | 用途 |
|---|---|
| 文档涉及部门识别（每份文档 → 哪些部门） | 部门视角投影 |
| 文档涉及主线识别（每份文档 → 哪些主线） | 任务智能简报 / 关系书 |
| 文档涉及人物状态变更识别（如"老高离职"） | 资方风险信号触发 |
| 文档关键时间承诺识别（如"5/1 前发"） | 承诺到期提醒 |
| 实体提及自动检测（任务描述里提到客户名） | 任务相关文档自动关联 |

我要把这些写进上下游交接合同，同步给工作台线程。

## 6 · v0.3 修正结构（基于反推结果）

v0.2 的 4 层升级为 **v0.3 的 4 层 + 5 个引擎层**：

```
┌────────────────────────────────────────────────────────────────────┐
│  服务下游（战略陪伴/工作台/任务/成长中心）                          │
└────────────────────────────────────────────────────────────────────┘
                          ▲
┌────────────────────────────────────────────────────────────────────┐
│  ⑤ 引擎层（新加）                                                   │
│  · 多维投影引擎（部门/人物/任务/时间四种切面）                       │
│  · 事件流引擎（主线状态变化事件序列）                                │
│  · 提醒/触发引擎（承诺到期/客户冷却/状态变化）                      │
│  · 关系书生成引擎（8 段自动生成 + 钻取）                            │
│  · 澄清问题生成引擎（缺口扫描 + 针对性问题）                        │
└────────────────────────────────────────────────────────────────────┘
                          ▲
┌────────────────────────────────────────────────────────────────────┐
│  ④ 关系书（v0.2 已有）                                              │
└────────────────────────────────────────────────────────────────────┘
                          ▲
┌────────────────────────────────────────────────────────────────────┐
│  ③ 项目台账 / 业务主线（v0.2 已有，需加 department_id + 事件流）    │
└────────────────────────────────────────────────────────────────────┘
                          ▲
┌────────────────────────────────────────────────────────────────────┐
│  ② 事实卡片（v0.2 已有，需加按部门/时间窗筛选接口）                  │
└────────────────────────────────────────────────────────────────────┘
                          ▲
┌────────────────────────────────────────────────────────────────────┐
│  ① 花名册（v0.2 已有，需加 department_id）                          │
└────────────────────────────────────────────────────────────────────┘
                          ▲
┌────────────────────────────────────────────────────────────────────┐
│  上游：工作台供应（含新增需求 5 项）                                │
└────────────────────────────────────────────────────────────────────┘
```

**v0.3 跟 v0.2 比，结构上的关键变化**：

1. ✅ 4 层不变（花名册 / 事实卡片 / 项目台账 / 关系书）
2. 🆕 新加**第 5 层「引擎层」**（v0.2 完全没有）：包含 5 个引擎，是关系网"主动驱动业务判断"的能力
3. 🆕 1-3 层都补了部门挂载（v0.2 漏的）
4. 🆕 3 层补了事件流（v0.2 只有当前状态，没事件序列）
