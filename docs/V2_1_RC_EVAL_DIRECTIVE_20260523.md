# V2.1 Release Candidate 评估口径 (顾源源 5/23 钦定)

> **核心**: V2.1 仓库 = 未来主仓库候选版 (Release Candidate)
> **作用**: 撤回旧"主仓库 vs lab 二分法", 评估对象切换为 V2.1 本身
> **触发**: 顾源源 5/23 09:30 钦定文档
> **B 落档**: 2026-05-23

---

## 1 · 新口径

**不再用** "旧主仓库 prod db 有没有变化" 来判断 A 工作有没有价值.

**改用**: V2.1 仓库作为未来主仓库候选版, 是否已经具备**可交付、可运行、可让用户感知价值**的能力.

---

## 2 · A 成果重新定位

A 在 V2.1 lab / V2.1 仓库里做出的能力 = **V2.1 候选主线能力建设**.

评估维度:
- V2.1 仓库代码是否完整
- V2.1 仓库服务是否能跑
- V2.1 仓库数据库 schema 是否完整
- V2.1 仓库里 R2 / R3 测试是否真实通过
- V2.1 仓库能否作为未来主仓库交付
- 用户迁移到 V2.1 后是否能感知到价值提升

→ **"主仓库没变" 不再是失败**.

真正的失败是:
- V2.1 自己跑不起来
- V2.1 的测试只在 snapshot 上跑, 不在完整运行环境里跑
- V2.1 的数据链路不能在真实客户数据上跑
- V2.1 没有完整迁移方案
- V2.1 无法替换主仓库

---

## 3 · B 后续停止做的事

1. **停止反复查旧主仓库 prod db**
   - 旧主仓库不再是判断重点
   - 旧主仓库 38 分只作历史 baseline

2. **停止要求"cherry-pick V2.1 → 主仓库"**
   - 不需要 cherry-pick, V2.1 替代主仓库

旧主仓库只保留 **2 个用途**:
- 历史 baseline (证明 V2.1 提升)
- 迁移风险检查 (旧数据迁到 V2.1)

---

## 4 · B 后续评估方式: V2.1 Release Candidate 评估 (4 层)

### 第一层 · 仓库完整性评估

V2.1 仓库是不是完整系统, 不是一堆实验脚本:
- 所有新增 service 是否在 V2.1 仓库内
- 所有新增表是否有 migration
- 所有 endpoint 是否在 V2.1 后端注册
- 所有测试脚本是否能从干净环境运行
- 所有服务是否能通过统一入口调用
- 不依赖临时 snapshot 或手工补数据

**通过标准**: 从一个干净的 V2.1 仓库启动系统, 可以跑通 R2 测试, 而不是只能在某个临时 snapshot 里跑通.

### 第二层 · 用户价值闭环评估

V2.1 必须证明用户能感知到:
- 会议纪要能被处理
- 客户事实能增加
- 时间线能更新
- 风险能出现
- 承诺能生成
- 澄清问题能出现
- 任务草稿能生成
- 故事卡能刷新
- 用户纠错后系统真改
- Agent Run Log 能看见 AI 做了什么

### 第三层 · 3.0 目标驱动能力评估

不能只测固定流程, 必须测 **AI 是否能根据目标和数据缺口主动调用不同模块补证据**.

**新测试: Data Gap 主动补证测试**

输入: "客户说预算还没定, 学校配合度也不确定, 但希望下周二看到一版轻量方案."

系统不仅要抽事实, 还要识别缺口:
- 预算未定
- 学校名单缺失
- 学校配合风险缺证据
- 是否有历史文件提到预算
- 是否要调用资讯情报站查客户官网或公开材料
- 是否要生成澄清问题问客户

测 AI 会不会**为了完成目标, 组合调用工具, 而不是只走硬编码流程**.

### 第四层 · 替代主仓库前的迁移评估

最后才做:
- 旧主仓库数据能否迁移到 V2.1
- 客户表、任务表、文档表、事实表是否兼容
- 旧客户数据迁移后是否能生成故事卡
- 旧任务是否能进入事件线
- 旧文件是否能被数据中心继续引用
- 用户打开后是否看不到断层

## 5 · B 报告结论修正

| 旧结论 (撤回) | 新结论 |
|---|---|
| "真 R2 必须先 cherry-pick 到主仓库, 否则不算" | "真 R2 不需要以旧主仓库为对象; 应直接在 V2.1 候选主线中运行. 但必须保证 V2.1 是完整运行环境, 不是单个 snapshot" |
| "主仓库一行没动, 用户感知 0" | "V2.1 候选主线是否完整 + R2 是否在 V2.1 完整环境里跑通" |

> 后续不再以当前旧主仓库 prod db 作为完成度判据.
> V2.1 仓库将作为未来主仓库候选版, 请把后续所有测试和交付都转为"V2.1 Release Candidate 评估".

> 2. 从干净 V2.1 环境可以启动并跑通 R2
> 3. R2 会议纪要处理闭环可以生成事实、时间线、风险、承诺、澄清问题、任务草稿、故事卡更新、Agent Run Log
> 4. 不依赖临时 snapshot
> 5. 所有写入带 test_run_id、actor_type、org_id、user_id、client_id、project_id
> 6. 不直接写数据库
> 7. 危险动作进入 Approval Queue
> 8. 跨客户隔离为 0 错误
> 9. 用户纠错后后续回答和故事卡同步更新
> 10. 能输出《V2.1 Release Candidate 客观评估报告》

> 1. V2.1 仓库是否包含 A 声称完成的 service / endpoint / migration / 测试脚本
> 2. 是否能从干净 V2.1 环境启动并跑通
> 3. R2 会议纪要处理闭环是否真实通过

> 6. 是否能输出用户可感知的"客户理解与行动包"
> 7. 是否达到 AI 客户工作闭环指数 R2 ≥70, R3 ≥80 的目标
> 8. **不再把"旧主仓库没有变化"作为扣分项**

## 8 · V2.1 RC 评估 5 步执行顺序

```
Step 1  V2.1 仓库完整性扫描          → 《V2.1 RC 仓库完整性报告》
Step 2  V2.1 R2 会议纪要闭环测试      → AI 客户工作闭环指数 ≥ 70, 7 硬门槛 ≥ 6/7
Step 3  V2.1 R3 Data Gap 主动补证测试 → AI 根据缺口组合工具
Step 4  V2.1 双驱动同题测试           → 内置 + 外置 Agent 一致性
Step 5  V2.1 替代主仓库准备度评估     → 迁移脚本 + 用户数据升级 + 回滚方案
```

---

## 9 · "真 R2" 新定义

**不是**: 在旧主仓库 prod db 里跑.

**是**: 在 V2.1 候选主线中, 从干净环境启动, 使用真实或准真实客户数据, 通过 API / CLI / 领域服务完成会议纪要闭环, 并输出用户可感知结果.

10 必满足条件:
1. 不依赖旧主仓库
2. 不依赖临时 snapshot
3. 不通过 UI 自动化
4. 不直接写数据库
5. 不只写 atomic_facts
6. 有 Agent Run Log
7. 有 Approval Queue
8. 有故事卡更新
9. 有任务草稿
10. 有用户可读执行摘要

---

## 10 · 旧主仓库以后什么时候再看?

**只在 2 个阶段**:

### 阶段一 · 历史 baseline
- 旧主仓库: 数据-功能闭环 38/100
- V2.1 目标: ≥70 / ≥80
- → 说明 V2.1 替代旧主仓库的价值

### 阶段二 · 迁移验收
- 客户不丢
- 任务不丢
- 文件不丢
- 原子事实不丢
- 权限不乱
- 故事卡可重建
- 工作台可继续问答

除此之外, 不再拿旧主仓库评估当前开发完成度.

---

## 11 · 一句话总结

既然 V2.1 会整体替代主仓库, 那现在真正要证明的不是"主仓库有没有同步", 而是 **"V2.1 有没有资格成为新主仓库"**.

评估口径切换:
- ❌ 旧主仓库有没有变化
- ✅ V2.1 是否能作为 Release Candidate 跑通用户可感知的 AI 客户工作闭环
