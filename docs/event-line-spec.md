# 事件线系统设计规格

## 1. 定位

事件线不是标签，也不是另一种任务列表。

它是：

`把任务、会议、支持请求、附件、复核、复盘和状态变化串起来的一条 AI 可读工作记忆线。`

对人类，它是：
- 一条可回看的工作证据线
- 一条可导出的过程线
- 一条能把重复任务串成一个整体的复盘线

对 AI，它是：
- 一条带时间顺序的背景记忆线
- 一条能持续吸收痕迹的上下文主线
- 一条能帮助判断“之前做了什么、现在卡在哪里、接下来要做什么”的工作线

## 2. 为什么不能做成标签

标签只能做分类，不能承载：
- 时间顺序
- 阶段状态
- 参与角色
- 关键决策
- 会议纪要
- 支持请求
- 证据附件
- AI 摘要与下一步

所以事件线必须是独立对象，而不是任务上的一个彩色标签。

## 3. 要解决的核心问题

### 3.1 复盘重复

同一周里，同一条业务线往往拆成多条任务。

例如：
- 云南连心会议准备
- 云南连心材料补充
- 云南连心会后跟进

如果按任务复盘，用户需要重复写三次。

事件线引入后，复盘可以改成：
- 这一周“云南连心”这条线推进了什么
- 这条线卡在哪里
- 下一步如何继续

### 3.2 AI 记忆断裂

现在 AI 主要读单条任务和本周摘要，缺少“这一条线此前发生过什么”的连续记忆。

事件线要把这些碎片串成一条连续线：
- 任务
- 会议
- 支持请求
- 附件
- 复核
- 复盘
- 时间安排痕迹

### 3.3 人类证据线缺失

需要对外说明“这个事情怎么发生的、怎么推进的、谁参与过、做过什么、有什么附件和结论”时，不能依赖聊天记录和记忆。

事件线应支持导出为 PDF 证据线。

## 4. 设计原则

1. 事件线优先服务 AI 上下文，不优先服务展示完整度
2. 不增加员工额外录入负担
3. 任务挂事件线应是轻量动作
4. 事件线自动吸收系统内已有痕迹
5. 复盘逐步从“按任务复盘”迁移到“按事件线复盘”
6. 项目 / 客户 / 部门 / 组织背景都可以作为事件线背景，但不强行混在一起
7. 事件线必须既能做 AI 记忆线，也能做人类证据线
8. 所有功能必须共用同一套底层数据与文件层，附件不是任务局部私有资产

## 5. 最小对象模型

## 5.1 EventLine

```ts
type EventLine = {
  id: string;
  name: string;
  kind: 'project_line' | 'issue_line' | 'coordination_line' | 'case_line' | 'custom';
  status: 'active' | 'blocked' | 'paused' | 'done' | 'archived';
  stage?: string;
  summary?: string;
  intent?: string;
  nextStep?: string;
  ownerId?: string;
  ownerName?: string;
  primaryClientId?: string;
  primaryClientName?: string;
  primaryDepartmentId?: string;
  primaryDepartmentName?: string;
  relatedProjectIds: string[];
  participantIds: string[];
  createdAt: string;
  updatedAt: string;
};
```

## 5.2 EventLineLink

用于把各类对象挂到事件线上。

```ts
type EventLineLink = {
  id: string;
  eventLineId: string;
  targetType: 'task' | 'meeting' | 'support_request' | 'attachment' | 'review' | 'note';
  targetId: string;
  relationType: 'main' | 'supporting' | 'evidence' | 'outcome' | 'reference';
  createdAt: string;
};
```

## 5.3 EventLineActivity

事件线自动吸收的时间序列活动。

```ts
type EventLineActivity = {
  id: string;
  eventLineId: string;
  sourceType: 'task_activity' | 'meeting' | 'support_request' | 'review' | 'attachment' | 'manual_note';
  sourceId: string;
  happenedAt: string;
  actorId?: string;
  actorName?: string;
  title: string;
  summary: string;
  metadata?: Record<string, unknown>;
};
```

## 6. 自动吸收哪些痕迹

当任务被挂到某条事件线后，系统自动把这些东西吸进去：

### 6.1 任务相关
- 创建任务
- 修改标题 / 描述
- 改时间 / 改时长 / 拖期
- 改负责人 / 协作者
- 状态变化
- 完成

### 6.2 协作与审批
- 发起复核
- 通过复核
- 退回复核
- 跨部门协作标记

### 6.3 支持
- 创建支持请求
- 接受 / 解决 / 驳回

### 6.4 会议
- 发起飞书会议
- 会议纪要回写
- action items 抽取

### 6.5 证据
- 附件上传
- 资料卡片 / 关联文档

### 6.6 复盘
- 周判断里对这条事件线补充的复盘说明

### 6.7 统一文件层规则

任务里上传的附件，默认应同时完成三件事：

1. 挂到当前任务
2. 进入当前客户 / 项目的总文件库
3. 如果任务已绑定事件线，则自动写入该事件线的证据引用

未来别的客户使用这套系统时，即使没有“外部客户”概念，也应遵守同一条规则：

- 业务 = 项目
- 项目下的任务附件仍进入统一项目文件库
- 同时为相关事件线提供可复用证据

## 7. 任务与日历里的交互

## 7.1 任务编辑器

新增一个轻量字段：
- `加入事件线`

交互要求：
- 可选已有事件线
- 支持 `从当前任务新建事件线`
- 这是轻量下拉，不是大表单

## 7.2 任务卡 / 日历卡

第一阶段只做轻量展示：
- 在任务卡或详情卡里显示事件线名称
- 允许点击进入事件线抽屉

## 7.3 顶部全局聚焦

后续增加：
- `当前聚焦事件线`

作用：
- AI 默认先读这条线
- 日历可优先筛相关任务
- 周判断可优先展示该事件线的上下文

## 8. 复盘与周判断如何变化

## 8.1 当前问题

当前周复盘仍以任务为底层颗粒。

这导致：
- 同一条业务线分散成多条任务
- 复盘重复
- 洞察容易断裂

## 8.2 目标形态

周判断需要改成：

`先按事件线分组，再在事件线内看任务、会议、支持和附件。`

每条事件线应形成一个复盘模块：
- 这条线本周发生了什么
- 这条线推进了哪些关键任务
- 卡在哪
- AI 判断的隐性问题是什么
- 下周可能会遇到什么困难或机会
- 应采取哪些动作

## 8.3 与单任务复盘的关系

不是立即删除单任务复盘，而是分两层：
- 单任务继续保留最轻量反思
- 周判断与深复盘逐步迁移到事件线层

## 9. 事件线详情页 / 抽屉

不建议一上来就做重页面，第一版可先做抽屉。

应显示：
- 事件线名称
- 当前状态 / 阶段
- 负责人
- 参与成员
- 关联项目 / 客户
- AI 摘要
- 下一步
- 时间序列活动
- 关联任务
- 会议与纪要
- 支持请求
- 附件证据

操作：
- 编辑事件线
- 加入当前任务
- 添加说明
- 导出 PDF

## 10. AI 如何使用事件线

## 10.1 默认阅读顺序

当 AI 在任务、周判断、支持请求、会议中遇到事件线时，优先读：
- 事件线摘要
- 最近活动
- 最近会议决策
- 当前阻塞点
- 下一步

而不是每次从零重新看任务列表。

## 10.2 AI 输出目标

AI 在事件线内主要回答：
- 这条线已经推进到哪里
- 目前主要阻碍是什么
- 哪些人/部门在参与
- 还缺什么
- 下一步应该先做什么

AI 在读取事件线时，不应只看任务标题和复盘，还应优先读取：

- 这条线关联项目的背景包
- 最近会议纪要
- 最近支持请求与处理结果
- 关键附件与证据文档
- 近几周连续任务痕迹

## 11. 人类证据线导出

事件线导出 PDF 时，应形成：
- 事件线基本信息
- 背景与目标
- 时间顺序过程
- 关键会议与决策
- 支持与卡点
- 关键附件索引
- 当前状态
- 下一步建议

用途：
- 对外说明
- 对内复盘
- 项目交接
- 管理佐证

## 12. 当前状态说明

截至 2026-03-21：
- 事件线概念已经明确
- 已进入产品设计阶段

`这项功能还没有正式实现。`
