# 客户计算中心 — 功能性差距清单

**审计日期：** 2026-05-12  
**审计视角：** 从用户实际使用场景（战略陪伴 / 内容生产 / 客户洞察 / AI 岗位）出发，看**需要的功能是否存在**——不是看"已设计的架构是否落地"。  
**与前一份报告的区别：**

| 维度 | 前一份"差距清单" | 本份"功能差距清单" |
|---|---|---|
| 视角 | 架构完整性 | 功能完整性 |
| 问题 | 设计有了实现不全 | 设计本身就缺 |
| 例子 | work_status 路由只有声明无实现 | 整个软件不能接收手机录音 |

---

## 一份话总结（如果只看这一行）

> **这套软件叫"客户计算中心"，但目前不能接收录音、不能识别图片里的实体、不能检测矛盾、不能定时主动推送、不能输出 PPT/Excel/PDF、没有"我"这一层记忆、不会做真正的统计/对比/模拟/预测——它实际上是一个"被动检索 + LLM 文本生成中心"，离"计算"两个字差得很远。**

---

## 总览：九大功能维度的功能性完成度

| 维度 | 功能完成度 | 最严重缺失 |
|---|---|---|
| 输入广度 | 30% | 移动端录音只能上传文本；邮件/IM 几乎为零 |
| 理解深度 | 20% | 实体只是关键词列表；无关系图谱；无矛盾检测 |
| 主动能力 | 15% | **完全没有定时/调度基础设施**——无 cron/scheduler |
| 生产能力 | 15% | 只剩 Word 一条腿——无 PPT/Excel/PDF/图表/TTS |
| 长期记忆 | 25% | **没有"我"这一层**；无晋升/衰减/学习闭环 |
| **真计算能力** | **5%** | **配不上"计算中心"四个字**——计算 ≈ COUNT(\*) + LLM.generate() |
| 协作能力 | 30% | 无评论/@/通知/版本锁/审计日志/工作交接 |
| 外部接入 | 25% | 飞书 + Bing RSS + yt-dlp 三件套；无专业搜索 API |
| 反思自知 | 20% | 只有数字 confidence；无可见的"盲区/推理链/答错记录" |

**最严重的两条：真计算能力（5%）和主动能力（15%）。**

## 维度一：输入广度（多模态多通道）

### 已实现
- **图片 OCR**：`knowledge_v2.py:1334-1345`（`.jpg/.png/.webp` 走 AI 视觉模型 `_read_image_markdown_with_ai_ocr`）
- **PDF/扫描件 OCR**：`knowledge_v2.py:933-1085`（文字层缺失自动 fallback 视觉 OCR）
- **网页与链接**：`internet_crawler.py` + `link_material_import.py`（B 站/小红书走 yt-dlp）
- **飞书"妙记→纪要"入站**：`feishu_sync.py:1-30`

### 雏形
- **录音转文字**：**只在 link_material（B 站/小红书）和飞书妙记里有**——直接上传 `.m4a/.mp3/.wav` 录音文件**没有入口**。`/api/v1/mobile/recordings/text-ingest` 这个名字误导——只收文本（`test_mobile_recording_text_ingest.py:78,120`）。
- **视频**：仅 `LinkMaterialPlatform = Literal["bilibili","xiaohongshu"]` 两个白名单。**YouTube / 抖音 / 本地 mp4 文件全部不支持**。
- **表格数据**：`.xlsx/.pptx` 走 `_archive_xml_text` **拆 XML 抽文字**，**不保留行列结构、无 Sheet 切分、无公式、不能跑数据问答**；`.csv` 当纯文本读。

### 缺失（关键）
- **邮件接入**：`imap / smtp / exchangelib / graph.microsoft / gmail_api / outlook_api` 全 0 命中。Gmail / Outlook / 企业邮箱**完全断流**。
- **IM**：飞书**只发不收**（无群消息历史拉取）；企业微信、Slack、Discord、钉钉**0 行代码**。
- **浏览器历史/收藏夹**：无导入路径。
- **Notion / 语雀 / 飞书文档**：无授权读取接口。
- **代码/技术文档**：无 `.py/.ts/.java/.go` 专项解析；`.md` 当普通文本。
- **手写笔记**：无 Apple Notes 接口、无手写体识别策略。
- **剪贴板截图直接粘贴**：无入口。
- **二维码识别**：无。

### 业务影响
战略陪伴场景里**最高频的两类信息**——客户对话录音、邮件往来——目前是黑洞。用户要么先在外部工具转写再粘文本（破坏工作流），要么这些信息根本进不来。

## 维度二：理解深度（语义级理解）

### 已实现
- **版本管理**：`db.py:164` `knowledge_document_versions` 表 + `supersedesId`（`models.py:6738,6759,6794`）+ `knowledge_base.py:2936/3197` 真实写入。具备基础 v1/v2/v3 替代链。

### 雏形
- **实体抽取**：`knowledge_v2.py:1949/2992` 有 `entities_json` 字段，但是 `entities: list[str]`（`knowledge_base.py:328`）——**只是关键词数组**，无类型（人/公司/项目/金额/日期）、无规范化、无 ID、无跨文档合并。
- **资料时效性**：`memory_foundation.py:257/270/621/658` 有 `freshness` 字段，但**写死常量** 0.92/0.9/0.8/0.7，**不随时间衰减、不关联 created_at**。

### 缺失（关键）
- **跨文档关系图谱**：全仓 `knowledge_graph` 只 1 命中（在 narrative 模块）。无 nodes/edges/三元组表，"项目 A 在 5 份资料里被提到、和哪些竞品同时出现"**问不出来**。
- **矛盾检测**：`conflicts` 字段都是**会议人际冲突/风险点**（`meeting_followup.py:53-65`/`data_center_prep.py:53/88/127`），**不是"事实 A 在文档 X 和 Y 说法不一致"**。无 contradiction detector。
- **重复观点识别**：无 chunk 去重，无"同一判断出现 5 次 → 权重叠加"。
- **资料鲜度评分**：见上，常量化。
- **隐含信息抽取**：无 fact/judgment/conclusion 三分类抽取字段。
- **客户特定术语库**：`glossary/terminology/术语` 0 个产品代码命中。**每个客户没有自己的产品名/黑话表**，同名歧义无法消解。

### 业务影响
- **半年前的客户判断和昨天的判断在系统里看起来一样新**——对战略陪伴致命。
- 决策场景里**"上次说预算 50 万、这次说 30 万"这种关键告警没有**。
- 客户 X 用某专有词意指 A，客户 Y 用同一个词意指 B——**会混淆**。

## 维度三：主动能力（不等用户问）

### 已实现
- 无（这一层基本是空白）

### 雏形
- **资料缺口识别**：`digital_asset_center.py` 有 9 个维度的 `gap / missing_template`，但**只在用户打开"数字资产中心"页面时计算并展示，不主动告知**。
- **风险信号**：`workspace_followups.py` 有 `risk_review` scenario，但**只在用户提问时生成**；无"扫描全部资料发现风险"的主动循环。
- **周/月摘要**：`weekly_review_material_pack.py` 等已存在，但**全部是用户点按钮时同步生成**——无定时调度。

### 缺失（最关键）
- **完全没有定时/调度基础设施**：`pyproject.toml` 无 APScheduler / Celery / schedule；`main.py:2779 @app.on_event("startup")` 只起 2 个 worker loop 处理用户提交的队列任务，**无 cron / 每天 / 每周触发器**。
- **没有通知中心**：前端 `unread / inbox` 全部是任务"待确认"语义，**没有面向 AI 主动洞察的 notification 模型**。即便后端检测出来也送不达。
- **新资料触发洞察**：入库链路只跑解析/索引，**无 `on_ingest_insight` 钩子**。
- **指标变化提醒**：无 delta 检测和"变化即推送"。
- **跟进提醒**：**完全缺失**——"3 周前问了 X，相关资料更新了"做不到。
- **跨客户模式发现**：无 `cross_client / pattern_discovery`——你的 3 个客户都遇到了同一个问题，系统看不见。
- **重要资料自动标签**：无"资料级重要度"持久化字段。

### 业务影响
**这是把当前系统从"被动工具"升级到"AI 主理人"的最大障碍**。没有 cron + 通知中心，所有"AI 岗位"都只能等用户来按按钮——这违背了你"AI 同事 / AI 公司"的核心愿景。

## 维度四：生产能力（多格式输出）

### 已实现
- **Word 报告**：`pyproject.toml` 含 `python-docx`，`template_fill.py` 有 `extract_docx_template_fields / apply_docx_template_values` 完整模板填充。**这是唯一完整的生产能力**。

### 缺失（几乎全军覆没）
- **PowerPoint**：依赖里无 `python-pptx`，零生成代码。
- **Excel**：依赖里无 `openpyxl / xlsxwriter`，零生成代码。
- **PDF 导出**：`pypdf / pymupdf` 仅用于读 PDF；**无 reportlab / weasyprint**。
- **公众号文章**：仅在关键词里有"公众号"三个字，**无样式转换器、无 publish 链路**。
- **网站文章**：无 HTML 文章生成模块。
- **图表**：无 matplotlib / plotly / echarts / mermaid / kroki。前端 `src/` 也无图表 lib。
- **思维导图**：全代码库无 mermaid / markmap / mindmap 关键字。
- **邮件/IM 格式输出**：无 SMTP、无"给客户发邮件版式"模板。
- **可分享卡片（小红书/朋友圈）**：仅"输入抓取"，**无"输出卡片"**。
- **TTS / 朗读**：无 pyttsx / edge-tts / piper。

### 业务影响
对"**内容生产**"业务这是致命缺口——客户汇报场景几乎只能输出 Word。公众号草稿、PPT 提案、Excel 客户清单、PDF 正式文档、图表分析——**全部不能直接生成**。

## 维度五：长期记忆与个性化

### 已实现
- **客户记忆层**：`client_profile.py` + `organization_dna_v2.py` 用 AI 按维度生成 profile blocks，扎根在客户 scope。**这是这一层最完整的部分**。

### 雏形
- **团队/组织记忆层**：`local_memory.py` 的 `org_memory.md` 单文件 + `learning_presets.py` 硬编码 19 条方法卡——**不是真"组织在用中学到的"**。
- **噪音遗忘**：`is_polluted_memory_text / is_generic_memory_placeholder` 是**写入前过滤垃圾**，不是"已存记忆随时间衰减"。`dream_cycle` 的 trim 只按文件字节数裁剪，**不按重要性遗忘**。
- **错误纠正学习（窄）**：`evidence_quality_feedback.py` 支持人工标注"useful/noise"，**仅限单条证据**，不会沉淀成"该用户认为 X 类信息是噪音"或修正回答风格。

### 缺失（关键）
- **没有"我"这一层**：`MemoryScopeType` 仅定义 `client / person / product / event_line / task`（`models.py:49`）——**无 `user / self / personal_dna`**。整个 `memory_facts` 表里"**我自己**"不是一个 scope。`local_memory.py` 目录只有 `org_memory / projects / weekly`，**没有 `me/` 或 `user_<id>/`**。
- **重要信息晋升机制**：`upsert_memory_fact` 只做 upsert，**无"高频出现 → 提升 confidence"**。`promote_*` 函数都是"候选转任务"，不是短期 → 长期记忆通道。
- **个性化检索权重**：`evidence_selector / rerank_provider` 按通用 quality/freshness 排序，**未读取"该用户偏好哪类材料"**。
- **用户偏好学习**：全代码库无 `user_preference / user_style / prefers_concise` 字段。
- **个人化知识图谱**：知识图按客户切分，**不按用户视角**；没有"我看过的资料"或"我打过标的关系"。
- **使用模式识别**：`badge_engine.py` 输出是徽章，**不是行为画像**。

### 业务影响
- 用户的偏好、口头禅、风格、习惯**全部不存在结构化承载**。
- 你纠正过一次的错误，**下次照样犯**。
- "AI 主理人"必须有用户画像才能定制化——**这一层不补，AI 永远像一个新来的实习生**。

## 维度六：真"计算"能力（**最严重的名实不符**）

### 已实现
- 无（一个都没有）

### 雏形
- **数据计算（统计/汇总/分组）**：散落在专用聚合函数：`_event_line_rollup` / `_count_answer_action_artifacts` / `_count_tasks` / `digital_asset_center._global_count_for_clients`。**全部是硬编码 SQL，不暴露为对话里的通用查询**。问"这个客户 3 个月内有几次会议"**无路径到达这些聚合**。
- **预测计算（标签级）**：`_prediction_readiness` 只是按完整度给"是否敢预测"打个标签，**真正预测交给 LLM 文字生成**，无定量预测、无置信区间。

### 缺失（几乎全部）
- **趋势分析**：`_trend_overview_lines` 是把 `trendSignals.title+statement` 拼成文本行——**不是真正的时间序列计算**。无周/月/季节性序列、无同比环比函数。`trendSignals` 由 LLM 生成。
- **对比计算**：无 `compare_clients / compare_options / compare_periods`。`review_simulation.py` 的"模拟"是 4 个硬编码 `DEPARTMENT_BLUEPRINTS`，**不是真对比**。
- **模拟计算（what-if）**：`MiroFishSimulation` 返回的是 LLM 文本研判，**不是参数化场景**。
- **决策计算（多方案排序）**：`evidence_selector._build_decision_trace` 只是"为答案选证据"，**不是给用户多方案打分排序**。
- **优化计算**：`local_model_optimizer.py` 是**"本地模型自身的调参"**，**不是业务方案优化**。无约束求解、无线性规划。
- **路径规划**：无依赖图遍历或目标分解算法。
- **风险/收益计算**：`_infer_risk_type` 只做**关键词分类**（协作/决策/流程/容量），**不打分**。无 ROI / 期望值 / 风险敞口。
- **资源分配计算**：无 capacity / workload / scheduling 算法。
- **结构化数据查询（NL→SQL）**：`text_to_sql / nl_to_sql / sql_generation` **全无命中**。客户/任务/会议都在 SQLite 里，但用户问"3 个月内这个客户有几次会议、卡在哪一步"**没有路径触达**——只能等 LLM 看检索片段。

### 业务影响

**"计算中心"这个名字目前严重名不副实**。当前所有"计算"类输出（趋势/风险/预测/模拟/对比/决策）都是 **LLM 文本生成 + 关键词分类 + 完整度打标**，没有一个真正的定量计算引擎。

> **当前"计算"在代码里基本等同于 `COUNT(*)` 和 `LLM.generate()`。**
>
> 它能记录、能检索、能生成判断文本，但不会做加减乘除以外的真计算——没有定量模拟、没有时间序列、没有方案排序、没有 NL→SQL。叫"计算中心"的预期（输入数据 → 输出量化结果与可比较的方案）至今由 LLM 自然语言糊在中间，缺乏可验证、可重现的计算骨架。

**如果只保留一个改进建议，就是补上 NL→SQL + 基础统计/对比/趋势计算引擎**——否则"客户计算中心"这个名字一直会被质疑。

## 维度七：协作能力

### 已实现
- **任务分配与追踪**：`task_collaborators` 表 + `inbox_status (pending/accepted/returned)` + `_task_collaboration_state`（`main.py:8537-9070`）——**这是协作维度里唯一完整的一块**。

### 雏形
- **知识共享/访问申请**：`data_center_access.py` 有可见性过滤（CEO/总监/员工分层），**但无"申请查看"工作流**——用户看不到的资料就是看不到。
- **跨团队协作**：`data_center_ingest.py` 有 `collaborationDepartmentIds` 字段，但**无跨部门工作流**。

### 缺失（几乎全部）
- **多人同时编辑/查看**：无 WebSocket / CRDT / OT / Yjs 等实时协同。单机 SQLite，**两人同改一条记录后写覆盖前写**。
- **资料锁/版本冲突防护**：全仓 `lock / version_conflict / optimistic_lock` **0 命中**。
- **评论与讨论沉淀**：**基本缺失**——数据库无 `comments / discussions / annotations` 表。在某份资料/某条判断下讨论做不到。
- **@ 提及与通知**：**完全缺失**——`notify_user / push_notification / mention` 0 实现。仅有飞书机器人单向卡片。
- **工作交接**：`handover / transfer / 交接` **0 命中**。员工离职后无一键转交。
- **代理身份/委托**：`delegate / proxy` **0 命中**。无"以总监视角查看"。
- **变更日志/审计追溯**：**无 `audit_log / change_log` 表**——三层组织却没有操作日志，"谁改了什么、什么时候"无法回答。**对 CEO/总监审计是硬伤**。

### 业务影响
你的目标是 CEO + 总监 + 普通员工三层角色，**但目前的协作能力几乎只支持单人 + 任务分配**。所有需要"在某份资料下讨论""@ 同事看一下""有人改了通知我""谁改过这条"的场景都做不到——这会限制公司化使用。

## 维度八：外部世界接入

### 已实现
- **RSS 订阅**：`topic_source_fetcher.py` 完整支持 RSSHub 路由、RSS/Atom 自动发现。
- **飞书 API**：`feishu.py + feishu_sync.py` 妙记纪要、任务双向、日历、消息卡片。

### 雏形
- **网络搜索**：`internet_crawler.py` 用 `BING_WEB_RSS_TEMPLATE` 拼 Bing RSS——**不是面向 AI 回答的专业搜索**。
- **浏览器/爬虫**：`internet_crawler.py + topic_source_fetcher.py` 用 httpx + 正则爬 HTML——**无 Playwright/Selenium，JS 渲染站点拿不到内容**。
- **社交媒体**（消费形态）：`link_material_import` 一次性导入 B 站/小红书单条 URL——**不是持续监控**。
- **客户公开信息追踪**：`external_evidence_cards` 表可手动绑客户 scope，**但无定时去抓客户公司公开动态**。

### 缺失
- **专业 Web 搜索 API**：无 Tavily / Serper / Google / Brave / Perplexity API。
- **行业数据库**：无招股书/年报/研报库、无 Wind/同花顺/天眼查。
- **公开数据集**：无政府开放数据/统计局 API。
- **实时信息**：`stock / 股价 / 汇率 / weather / holiday` 业务代码 **0 命中**。
- **CRM/ERP/营销 SaaS**：无 Salesforce / Hubspot / 钉钉 OA / 企微 CRM。
- **RSS 持续监控**：技术上可以，但**无调度器**（见维度三）。
- **竞品监控**：无竞品对象、无定期抓取。

### 业务影响
**AI 的"世界知识"完全停留在用户上传的资料层面**。问"客户公司昨天发了什么公告""现在汇率多少""竞品官网更新了什么"——目前都做不到。

## 维度九：反思与自知能力

### 雏形
- **可信度声明**：`query_router.py` 输出 `confidence` 数值，**但是路由元数据**，未必明确以"我只有 X 把握"形式呈现给用户。
- **盲区主动声明**：`answer_layer.py` 有 `missingContext` 字段，`digital_asset_center.py` 有"还缺这份资料"模板——**仅当次回答的缺料提示**，无系统级盲区图谱。

### 缺失
- **推理过程展示**：无 `reasoning / chain_of_thought / 推理链` 展示结构。
- **错误事后分析**：无"系统知道自己答错了"的机制。`evidence_quality_feedback` 让人对证据打分，**未反向归因到答案错误根因**。
- **元认知 "我擅长 X 不擅长 Y"**：无能力清单/边界声明。

### 业务影响
AI 没有可见的"自知之明"。用户没法看到"模型为什么这样答"、"它觉得自己哪里没把握"、"它过去在哪类问题上答错过"——**信任建立靠盲信，不可持续**。

## 跨维度的根因（为什么这些功能都缺）

排查下来，缺失功能集中暴露了**三个根因**：

### 根因 A：缺一个"事件总线 + 调度器"基础设施
没有 cron / scheduler / event bus，导致：
- 主动能力（维度三）几乎全空
- 周/月摘要、跟进提醒、风险扫描、新资料触发洞察都做不了
- AI 岗位无法定时跑

### 根因 B：缺一个"通知中心"
没有 notification 模型，导致：
- AI 即使发现了洞察也送不达用户
- 协作维度的 @mention、评论通知做不了
- 跨设备/跨成员的信息流动断裂

### 根因 C：缺一个"定量计算引擎"
没有 NL→SQL、无时间序列计算、无 what-if、无方案打分，导致：
- 整个"计算中心"名不副实
- 一切定量问题都被 LLM 自然语言糊过去
- 没有可验证、可重现的"计算骨架"

**这三个根因如果各补一个最小可用版本（Cron + Notification + ComputeEngine），主动能力 / 协作能力 / 计算能力三个维度的完成度会同时跃升。**

## 优先级建议

### 🔴 P0 必补（业务硬伤，不补战略陪伴和内容生产做不下去）

1. **录音上传转写入口**（输入广度）——手机录会议/客户对话是最高频场景
2. **邮件接入**（输入广度）——战略陪伴客户大量沟通在邮件里
3. **定时/调度基础设施**（主动能力根因）——补 APScheduler 一个最小版本
4. **通知中心**（协作 + 主动能力共享根因）——补 notifications 表 + 推送通道
5. **PPT / Excel / PDF 生成**（生产能力）——内容生产场景的基本盘
6. **资料鲜度真实衰减**（理解深度）——`freshness` 与 `created_at` 关联

### 🟠 P1 应该补（AI 时代要素，不补 AI 价值打折）

7. **NL→SQL 基础**（计算能力）——让"客户计算中心"名副其实的第一步
8. **基础统计/对比/趋势计算引擎**（计算能力）——Pandas 一层最小封装
9. **跨文档实体抽取与关系图谱**（理解深度）——为"客户洞察"提供结构化骨架
10. **矛盾检测**（理解深度）——决策场景的关键告警
11. **"我"这一层记忆**（长期记忆）——`MemoryScopeType` 加 `user`
12. **用户偏好学习**（长期记忆）——基础风格偏好就够
13. **新资料触发洞察 webhook**（主动能力）——`on_ingest_insight` 钩子
14. **审计日志**（协作）——CEO/总监审计硬需求

### 🟡 P2 可选补（未来扩展）

16. **专业 Web 搜索 API**（外部接入）
