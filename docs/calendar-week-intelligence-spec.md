# 日历模块周判断系统设计规格

> 对齐说明（2026-03-21）：
> 本文档继续作为 `P2` 的核心规格，但其上位原则已明确为：
> 总结与周判断优先利用背景母盘、计划层和执行痕迹，不应额外增加员工填报负担。
> 后续实现以 [AI 背景优先总规划](./ai-context-first-rollout-plan.md) 和
> [全局模块改造清单与阶段路线图](./global-module-refactor-roadmap.md) 为主。

## 1. 定位

这不是普通的“周总结”功能，而是：

`任务与日历模块中的周判断系统 / AI 背景驱动的任务痕迹分析器`

它的目标不是生成一篇周报，而是基于：
- 任务事实
- 日历排布痕迹
- 轻量复盘
- 组织 / 部门 / 岗位 / 项目背景

输出：
- 总结
- 判断
- 预测
- 动作建议

一句话定义：

`周判断系统 = 任务痕迹 + 背景母盘 + AI 推理 + 管理动作`

## 2. 设计原则

1. 员工输入保持轻量，不能做成申请表
2. AI 背景要重，判断要建立在组织与项目上下文之上
3. 同一批底层证据，服务个人、部门、机构三层视角
4. 页面优先呈现“判断和动作”，不是优先堆砌信息
5. 字段优先服务 AI 思考上下文，不优先服务展示完整度
6. 日历行为痕迹本身就是证据，不只是排期 UI
7. 总结的核心目标是“洞察”，不是“罗列”
8. 避免为了总结再要求人工补更多材料，预制背景与过程痕迹应承担主要证据职责

## 3. 模块新定位

当前 `周复盘` 应升级为 `周判断`。

它不再只是：
- 本周做了什么
- 填几句复盘

而应回答：
1. 这周真实发生了什么
2. 这些任务在推进哪些计划/项目/组织目标
3. 哪些问题已经暴露为管理问题
4. 未来 1-3 周哪些风险会升级
5. 个人、部门、机构分别最该做什么

## 4. 三层输出对象

### 4.1 我的周总结

面向成员，重点是：
- 我本周推进了什么
- 哪些任务真正有价值
- 我卡在哪里
- 我下周最该怎么调整
- 哪些地方是成长机会

### 4.2 部门周判断

面向部门负责人，重点是：
- 部门本周是否在推进部门计划
- 团队哪些任务是高价值推进，哪些是低价值忙碌
- 谁处在过载、等待、方向不清、流程卡点里
- 哪些问题需要负责人介入

### 4.3 机构周洞察

面向 CEO / 管理层，重点是：
- 哪些部门和项目在真实推进机构季度目标
- 哪些组织信号说明管理机制在失效
- 哪些风险会在未来 1-3 周放大
- 哪些经验值得沉淀成机制 / 模板 / SOP

## 5. AI 背景母盘接入层

周判断系统必须从以下 4 层读取上下文。

### 5.1 组织层背景

- 组织 DNA
- 年度战略
- 当前季度重点
- 本阶段不做清单
- 机构级关键风险

### 5.2 部门层背景

- 部门使命
- 部门季度重点
- 部门月度计划
- 部门本周计划
- 关键岗位与职责
- 岗位流程模板

### 5.3 项目层背景

对益语智库前台可显示为“客户/项目”。

- 项目 / 客户介绍
- 当前阶段
- 当前目标
- 当前阻力
- 风险摘要
- 会议纪要 / 决策 / action items
- 关联文档 / 历史任务

### 5.4 执行层背景

- 任务标题 / 描述 / 状态 / 优先级
- 任务具体时间 / 时长 / 是否进入日历
- 是否改期 / 改期次数 / 改期方向
- 是否长期停留在“未设时间池”
- 是否待复核 / 是否跨部门
- 支持请求 / 退回 / 审批链状态
- 轻量复盘与卡点标签

## 6. 日历痕迹作为证据

以下行为必须被正式当作“周判断证据”，而不是纯交互行为：

1. 任务是否进入时间轴
2. 一周内是否多次改时间
3. 是否长时间停留在“未设时间任务”
4. 是否在最后时刻才被塞入日程
5. 是否经常跨天、跨周拖期
6. 是否出现高优先级任务集中挤压
7. 是否出现明显的工作过度饱和
8. 是否存在长时间未被安排却一直高优先级的任务
9. 是否长期卡在待复核 / 跨部门协作节点

这些信号会帮助 AI 判断：
- 是执行问题
- 还是目标不清
- 还是流程问题
- 还是管理响应问题

## 7. 页面信息架构

建议把当前 `周复盘` 页面重构成四段式：

### 7.1 本周事实

自动生成，不要求员工填写。

显示：
- 本周任务总量
- 完成 / 延期 / 改期 / 待复核 / 支持请求数量
- 时间轴进入率
- 未设时间任务池情况
- 高频改期任务
- 高频卡点任务

作用：
- 给 AI 判断提供清晰事实面板

### 7.2 AI 判断

结合组织背景、部门计划、项目背景解释事实。

输出示例：
- 本周主要推进的是哪条部门 / 项目主线
- 哪些任务虽然忙，但没有贴着当前重点
- 哪些任务已经偏离岗位职责
- 哪些行为说明流程或管理设计正在拖慢推进

### 7.3 可能性分析

这是预测层。

输出示例：
- 哪些风险未来 1 周可能升级
- 哪些成员存在持续过载
- 哪些项目会因为上游不稳继续拖动任务重排
- 哪些任务可能在下周演变成管理瓶颈

### 7.4 建议动作

必须区分角色给动作。

个人动作：
- 我下周最该聚焦什么
- 我该补什么资料 / 发起什么支持

部门动作：
- 部门负责人要优先干预哪几个问题
- 哪些岗位需要重新分配任务

机构动作：
- 哪些问题应升级到 CEO / 管理层
- 哪些项目需要协调会议 / 机制调整

## 8. 个人、部门、机构三层展示差异

### 8.1 成员视角

重点显示：
- 本周推进摘要
- 卡点解释
- 成长建议
- 下周聚焦建议

弱化：
- 过重的组织结构语言
- CEO 级别判断

### 8.2 部门负责人视角

重点显示：
- 团队推进是否对齐计划
- 哪些人 / 岗位过载
- 哪些任务反复改期
- 哪些任务缺支持、缺决策、缺协作
- 哪些问题要做流程或分工调整

### 8.3 机构管理者视角

重点显示：
- 哪些部门在推进机构目标
- 哪些项目是高价值推进
- 哪些问题已从执行问题升级为组织问题
- 哪些风险值得预判与提前干预

## 9. 输入设计

### 9.1 保留给员工的输入

只保留轻量输入：
- 已完成：`完成了，有什么心得？`
- 未完成：`需要什么支持，或者有什么思考？`
- 轻量标签：
  - `资料不足`
  - `等待他人`
  - `方向不清`
  - `资源不够`
  - `工作过度饱和`

### 9.2 不让员工填的内容

以下内容改成 AI 后台自动推断：
- 是否对齐部门计划
- 是否对齐组织目标
- 是否偏离岗位职责
- 是否属于流程卡点
- 是否属于协作失配
- 是否属于工作过度饱和

### 9.3 明确不做的事

为了避免周总结反过来拖累执行，这套系统不应要求员工为了“写总结”再补以下内容：
- 额外项目背景说明
- 手工对齐部门计划
- 手工解释组织目标映射
- 逐条填写复杂原因分类
- 为了 AI 可读而重复录入已经存在于系统中的资料

这些信息应优先来自：
- 组织预制背景
- 部门计划与岗位流程
- 项目 / 客户背景
- 会议纪要与行动项
- 任务与日历行为痕迹

## 10. 数据结构建议

### 10.1 任务执行信号 `taskExecutionSignals`

- `rescheduleCount`
- `enteredTimeline`
- `daysInUnscheduledPool`
- `isLateScheduled`
- `reviewBlocked`
- `supportRequestOpen`
- `crossDepartment`
- `loadRiskLevel`

### 10.2 周判断上下文 `weeklyIntelligenceContext`

- `organizationFocusIds`
- `departmentPlanItemIds`
- `projectContextId`
- `roleTemplateId`
- `workflowTemplateId`
- `supportSignalIds`

### 10.3 周判断输出 `weeklyIntelligenceReport`

- `factSummary`
- `analysisSummary`
- `possibilitySummary`
- `actionSummary`
- `managementSignals`
- `growthSignals`
- `predictedRisks`

## 11. AI 输入输出结构

### 11.1 输入层

AI 输入应由四部分拼装：

1. `taskFacts`
2. `calendarSignals`
3. `backgroundContext`
4. `lightReflection`

其中 `backgroundContext` 必须优先吸收已存在的预制资料，而不是等待员工补充：
- 机构背景信息
- 部门周 / 月计划
- 项目 / 客户计划与介绍
- 会议纪要 / action items
- 岗位职责与流程模板

### 11.2 输出层

输出统一生成：
- `事实摘要`
- `关键判断`
- `可能性分析`
- `建议动作`
- `需要补充的上下文`

其中“需要补充的上下文”可以反向告诉组织：
- 哪些背景还不够细
- 哪些计划还不够具体

但这类“补充上下文”应首先指向：
- 管理层补根背景
- 部门负责人补计划或流程

而不是把负担再次压回一线成员。

## 12. 关键判断能力

这套系统至少应支持以下判断：

1. 高价值推进 vs 低价值忙碌
2. 岗位职责偏离
3. 计划对齐度
4. 流程卡点
5. 协作失配
6. 管理负荷
7. 工作过度饱和
8. 项目阶段不匹配
9. 未来 1-3 周风险预测

## 13. 实施分期

### P1：周总结升级为周判断页

- 保留轻量复盘输入
- 新增 `本周事实 / AI 判断 / 可能性分析 / 建议动作`
- 把日历排布信号接入分析器

### P2：接入背景母盘

- 组织战略
- 部门周 / 月计划
- 项目背景
- 会议纪要 / action items
- 岗位流程模板

### P3：接入预测和动作闭环

- 风险预测
- 过载预警
- 任务建议转动作
- 一键发起支持 / 复核 / 会议

## 14. 成功标准

用户打开这个页面后，不应该感觉：

`这是我填的一篇周报。`

而应该感觉：

`这是系统基于任务痕迹、日历排布、项目背景和组织计划，对我 / 部门 / 机构做出的周判断。`

这页真正的价值是：
- 帮个人提升
- 帮部门管理
- 帮机构前瞻判断
- 让 AI 在任务与日历里始终带着上下文工作
