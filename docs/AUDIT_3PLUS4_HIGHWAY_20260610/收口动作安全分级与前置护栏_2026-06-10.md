# 收口动作安全分级与前置护栏（动手前的闸）· 2026-06-10

> 缘起：顾源源担心——我们标为「问题」的结构（重复表/多写入路径/硬限额/吞异常/源登记分裂），很多当年有功能性基础、或在挡某个 bug，直接收口会破坏数据或暗中支撑的功能。
> 方法：对 30/90 计划每个收口/合并/删除动作做 git 考古 + 依赖追踪 + 独立反驳（默认「收口不安全」去找会崩的地方）。12 目标 × 考证+反驳，24 agent。**每条反驳都成立，且反驳者普遍补抓到考古员漏掉的活依赖。**

## 0. 一句话结论

**12 个收口动作里，只有 1 个能直接做，1 个加护栏能做，2 个根本不该做，8 个必须「先迁读边再收口」。** 你的担心被全面坐实：naive 收口几乎没一个安全。本表把每个动作钉死成「先做什么才不崩」。

| 裁决 | 数量 | 含义 |
|---|---|---|
| `truly_dead` 可安全删 | 1 | T10 |
| `safe_consolidate` 加护栏可做 | 1 | T8 |
| `migrate_first` 必须先迁读边再收 | 8 | T1/T2/T3/T4/T5/T9/T11/T12 |
| `keep_live_function` 不该收口 | 2 | T6/T7 |

详表见 `收口动作安全分级表.csv`。

## 1. 对我自己上一版 30/90 计划的纠正（最重要）

考古推翻了我上一版计划里两个会闯祸的动作——**如果你照我之前写的去做，会出事**：

- **❌「31 张死表接线启用或 DROP」是错的（T6 → keep_live_function）**：所谓「死表」`clarification_records`/`ambiguities`/`execution_tickets`/`goal_records`/`key_decisions`/`data_center_proposal_drafts` 等**绝大多数不是死表，是「建好了、有完整全栈接线（前端 api.ts→端点→service→表）、有 UI 组件真在调、只是数据为空」的功能**。DROP 它们 → 战略澄清视图 / 会议详情 / 驾驶舱 / 周复盘主线 `review_narrative` / 提案执行 / 成长总览 `/growth/overview` 运行时直接 `no such table` **500 崩页**。而且 DROP 本身**结构性无效**——`db.py:_init_schema()` 用 `CREATE TABLE IF NOT EXISTS` 每次启动重建。**正确动作：不是清死表，是给它们补数据/接 worker 让它们活起来，或确认产品要不要这功能。**
- **❌「job_stage_runs 正名/改回状态表」要降级（T7 → keep_live_function）**：那张「每状态一行」的表是**刻意的事件溯源（event-sourcing）设计**——`running` id 含 attemptCount、每次失败/完成各入一行，是为**保留每个分析任务的完整重试取证时间线**。改回「就地 update 状态表」会把实测样本 9 行（4 轮重试）**坍缩成 1 行，永久销毁重试取证**。**正确动作：只加注释正名 +（可选）加视图，绝不改数据形态。**

## 2. 可直接做 / 加护栏可做

### ✅ T10 · document_deep_read_states 孤儿深读壳 → `truly_dead` 可安全删
- **为何立**：commit 91fac96「文档深读地基统一服务（M1/M2/M3）」的愿景设计，想做「导入即登记深读→worker 消费→自愈」。
- **为何能删**：M3/M4（让导入入口、常驻 worker 改调它）**从未实施**；真正的文档深读走另一张表 `local_model_tasks`。14 行是同客户同毫秒的开发者自测种子，对应文档的真深读结果已由 `local_model_tasks` 链产出（`knowledge_surrogates` 已建）。
- **前置护栏（仅清洁性）**：一并废弃 `scripts/` 下 3 个 import 该服务的验证脚本；注意 `claim_next/coverage_for_client` 命名别误删活路径的同名函数。

### ⚠️ T8 · 硬限额（1800字/top-N/module_budget）→ `safe_consolidate` 但必须分预算（**这条直接回答你最大的担心**）
- **你怕「限额是挡某个 bug」——考古结论：不是。** 全仓 git log 用「超长/崩溃/超时/OOM/narrative crash/context 超」筛，**零条「fix:叙事超长崩溃」类提交**。限额唯一来源是 commit 3d24ea2「M5 硬上限改 token 预算驱动（facts 60→200…）」，是为**本地小模型 qwen2.5:7b 的 prompt 体量量身定的**，并用真模型 before/after 验证过（CFFC business_intro 41→234 字）。即它是「为本地小模型塑形」，不是「补崩溃」。
- **所以放大是安全的，但崩点是「回答降级」不是「app 崩溃」**：本地调用 `ai.py:4969` 只有 `max_tokens` 输出上限、**无 `num_ctx` 输入护栏**，超本地模型 context 会左截断/HTTP 错→重试→回退基础版（不崩，但答案变差 + 420s 超时占住 worker）。
- **前置护栏**：① 放大必须**按 provider 分两套预算**（云端大模型放大 / 本地小模型保守），尤其 `atomic_fact_limit` 对 883/988 facts 的大客户；② 给 narrative 路径补输入字符上限或 num_ctx 探测，别只靠 420s 超时；③ **注意邻近有一条真 bug-patch（commit edb7d44 fix(ai)），别误改**；④ 放大前对当前在用 provider 重跑 3d24ea2 的 before/after 实测。

## 4. 安全收口顺序（动手时按此序，每步带前置闸）

1. **先做仪表盘 + CI（原 30 天主干道一不变）**——这步无收口风险，且是后续所有收口的回归网。
2. **T10 删孤儿深读壳** —— 唯一可直接做的清洁动作，零风险热身。
3. **T8 硬限额放大** —— 加 provider 分预算 + num_ctx 护栏后做；避开 edb7d44 bug-patch。先 before/after 实测。
4. **T6 / T7 改为「不收口」** —— 从计划里移除「清死表 / 改状态表」；T6 改成「给空功能补数据或确认产品意图」，T7 只加注释正名。
5. **T9 growth 解冻** —— schema 先加 attempt_count + user-ownership 过滤 + 保留吞异常（只加日志）。**这是你最担心的「挡 bug」类，护栏最重。**
6. **T2 写入收口** —— 用「绕去重闸门的适配器」而非删 writer；先验证 normalizer 语义等价。
7. **T1 / T3 / T4 / T5 迁读边** —— 各自先把活读方（端点/视图/前端按钮/worker）改指新源或下线，再动源表/老表。
8. **T11 chat 激活 / T12 清理** —— 最后做，激活前加 confidence 过滤，except:pass 只加日志。

## 5. 给非技术负责人的总结

- 你的直觉对了：**这些「问题」绝大多数不是垃圾，是「建好了在撑功能、或在挡故障」的东西。** 直接收口会崩页或丢数据。
- 但也不是不能动——**只有 2 个真不该碰（T6 空功能其实活着、T7 重试取证），1 个可直接删（T10 自测残留），其余都是「先把靠着它的人挪开，再拆篱笆」。**
- 最让你安心的一条：**你最担心的硬限额（怕是挡崩溃）——考古证明它只是为本地小模型塑形，不是补崩溃，放大是安全的，只要云端/本地分两套预算。**
- 最该警惕的一条：**growth 同步那三处「吞异常」真的在挡 bug**——直接移除会让积分/徽章功能整体崩，必须保留、只升级成记日志。

> 这张表就是「动手前的闸」。按 §4 顺序、带 §3 前置护栏推进，就不会出现「收口反而破坏数据/功能」。准备好后进 `执行+1` 收敛具体某一步的最稳实施路径。
