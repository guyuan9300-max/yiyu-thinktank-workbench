# V2.2 北极星 · 每个 session 第一件事必读

> **这是 v2.2 架构升级唯一的判分标尺。**
> 任何 session（包括 AI / 人 / 顾源源本人）开始工作前必读完这份。
> 写完代码 / 收尾时回到这份对照，看你做的事是否服务了下面三个目标。
> 偏离目标 = 错误工作，不论代码多漂亮。
>
> **2026-05-22 修订**：从 2 目标扩为 3 北极星 + 完成定义 + 自主边界 + 4 主路径 + 朋友式澄清 + 3.0 预留。

## 0 · 项目背景一句话

益语智库当前数据中心是「169 张表 + 1063 处散落 SQL + 6 大 view 各自直接 fetch」的多池并列结构。v2.1 走了 3 周「代码模块化」方向，跑偏。**v2.2 不要再以"代码漂亮"为目标，只以下面三个客户价值北极星为目标。**

---

## 1 · 完成的标准（顾源源 2026-05-22 定义）

**v2.2 做完 = 现有软件功能真实接通验证 + 顺畅运行**。

具体含义：

- 接通不是补丁接通，**是通过完善整个功能架构来接通**
- 所有功能在一个干净的架构上跑
- 用最高效的方式 —— **能复用统一通道就复用统一通道**（不是每个功能单独写信息通道、单独写计算通道）
- 所有功能的顺畅实现 = 最终标准

**反例（不算做完）**：
- 通过 hack / monkey patch 让 6 个 view 都能跑 → 这是补丁
- 每个功能各自走自己的数据通道 → 这是多池并列
- 测试通过但用户用起来卡顿/数据不一致 → 这是表面完成

## 2 · 自主执行的边界（顾源源 2026-05-22 定义）

**v2.2 进入自动执行模式：除非把握 < 70%，否则不停下来问，一直做到整个系统做完。**

| 情况 | 处理 |
|---|---|
| 整体计划方向把握 ≥ 70% + 当前 feature 把握 ≥ 70% | 自动执行，不询问 |
| 整体把握 < 70% **且** 不澄清就完不成北极星 | 停下来报告，等顾源源决策 |
| 命名/技术选型抉择，影响后续多个模块 | 自己拍板继续，**在 milestone 报告里 highlight 这个决策让顾源源知道** |
| 跑出来的验证结果偏离北极星 | 停下来报告，重新校准方向 |

**判断"把握 ≥ 70%"的标准**：
- 我能说清楚这件事服务哪个北极星
- 我能预测做完后的客户感知变化
- 我有具体的实现路径，不是"试试看"

## 3 · 三个北极星

### N1 · v2.2 做完后所有功能顺畅运行

> 整个系统要支持现在软件每一个板块的功能，完整运作起来。

**包含但不限于**（v2.2 启动盘点时拉出来的 9 大模块）：

2. **资讯情报站**（IntelligenceStationView，55+ 处 API 调用）
3. **客户工作台**（chat / 文档 / 知识 / glossary / entity）
4. **任务/日程**（TaskListView / TaskCalendarView / EventLineReportPanel）
5. **成长中心**（GrowthCenterView / 徽章 / 周复盘）
6. **计划工作坊**（PlanWorkshopView / 部门计划编辑）
7. **数据中心**（SmartFileImport / RichTextDocumentEditor / 工作台资料库）
8. **手机端**（4 tab：任务 / 日历 / 咨询 / 我的）
9. **设置**（多账号 / 同步 / 飞书集成 / 本地 ASR / 组织管理）

完整清单见 `V2.2_FEATURE_INVENTORY.md`。

**判断标准**：每个里程碑完成后，**逐项跑过这份清单**——任何一项跑不通 = 里程碑 FAIL，必须先修复才能进下一里程碑。

### N2 · 手机 AI 自由对话成为数据中心 4 主路径之一，承载软件逻辑灵魂

> 顾源源原话（2026-05-22）："手机 AI 通过聊天对谈的澄清信息，是 4 主路径之一，让整个软件有逻辑的灵魂。"

**4 条信息主路径**（数据中心的全部输入来源）：

| 路径 | 来源 | 现有载体 | 信息源置信度 |
|---|---|---|---|
| **路径 1** | 工作台进入的文件 | SmartFileImportModal / v2_documents / 手填 dna_documents | 高（用户主动上传 + 官方文档） |
| **路径 2** | 任务/计划/复盘/协作信息 | tasks / weekly_reviews / collaboration messages | 高（经验性，可指导下一次行动） |
| **路径 3** | 互联网爬虫补全 | internet_crawler / intelligence_* services | **分层**：官网高 / 第三方公众号低（可能编造） |
| **路径 4** | 手机 AI 聊天澄清 | mobile chat / Phase 3 朋友式对话引擎 | 高（用户口述记忆，但需结构化） |

**数据中心不是简单存这 4 类信息，是要理解每类信息的来源意味着什么**：

每份输入要带 **5 维元数据**（从真实工作场景反推抽象，详细规范见 [V2.2_INFORMATION_SOURCE_METADATA.md](./V2.2_INFORMATION_SOURCE_METADATA.md)）：

1. **来源类型**（source_type，14 类细分枚举）→ 决定基础置信度
2. **业务角色**（content_role：fact/decision/risk/observation/quote/...）→ 决定 AI 怎么聚合引用
3. **关系语境**（client_id/event_line_id/person_id/time_anchor/...）→ 决定 AI 怎么聚合到客户
4. **作者/受众**（actor_type/speaker_person_id/recipient_person_id）→ 决定 AI 怎么追问、权威度排序
5. **生命周期**（verification_status/validity_status/superseded_by_id）→ 决定 AI 引用时是否还有效

**5 维 + 21 字段都是从真实工作场景反推出来的**，不是抽象设计。举例：
- 顾源源问"日慈最近什么风险" → 需要 source_type + content_role 分层呈现（客户已确认 vs 用户观察 vs 网络传言）
- 信息冲突时（同一事实两个数字） → 需要 time_anchor + validity_status 仲裁
- AI 主动友好澄清 → 需要 actor + content_role + verification_status 决定该不该问
- 3.0 AI 自己写的事实 → 需要 actor_type 区分人写 vs AI 写

完整场景反推 + 14 类 source_type 枚举 + 5 条 AI 行为规则（R1-R5）见配套文档。

**手机 AI 聊天澄清的设计原则**（顾源源 2026-05-22）：
- **不能生硬地问问题**（"这个是 300 还是 3000？"会让用户懵）
- **应该像朋友聊天一样**
- 把生硬的关键数据澄清融入闲聊
- 场景：用户不是对着屏幕看信息，而是把记忆中的事情娓娓道来跟 AI 聊
- AI 通过聊天补偿，把信息逻辑串清楚
- 根据已有信息空白主动决定下一步聊什么

**判断标准**：每个里程碑完成后，**用 5/19 张真那场会做验收**——
- 这场会的会议纪要在工作台资料库里
- 问机器人："5/19 张真那场会有什么决定？"
- 答案应包含：张真接任法人 / 强哥任秘书长 / 兴盛+心理魔法学院合并 / 安心妈妈新项目 / 6 件待办（顾源源 3 件 / 张真 3 件 / 严斌 1 件）/ 兼职儿童数据红线
- 答案应带出处（指向那份 docx 的具体段落）
- 4 路径都能通过 IngestPipeline 进数据中心，且数据中心知道每条信息的语义+置信度
- **答出几条 = N2 的真实刻度**（不是"切了多少 SQL"）

### N3 · 为 3.0 埋好接入基础，3.0 升级时 2.0 不需要返工任何板块

> 顾源源原话（2026-05-22）："让 v3.0 不需要把 2.0 任何的板块推翻重构，只需要接入马上就可以做 3.0 的持续升级。"

**3.0 是什么**：给大模型配一间共享办公室。AI 跟人共享同一个工作空间（同一份战略 / 同一个收件箱 / 同一份历史 / 同一种澄清节奏），AI 作为 actor 在软件里建任务、发协作信息、引用文件、自动校验、跨任务规划，并且**越用越聪明（具备长期记忆）**。

**v2.2 要为 3.0 做的接口预留**（只加字段、只建表、不加逻辑）：

| ID | 内容 | 时机 | 工作量 |
|---|---|---|---|
| **A1** | `actor_type` 字段加进所有写入路径（enum: human / ai_agent / system）+ `actor_id` | Phase 1 收尾前 | 1 天 |
| **A2** | `event_log` 总线表持久化（升级现有 broadcast，加 reversed_at 字段） | Phase 1 收尾前 | 1 天 |
| **A3** | `provenance` 加 `reasoning_trace_id` + `derived_from_ids` 字段 | Phase 2 启动前 | 0.5 天 |
| **A4** | `verification_status` enum 加进 atomic_facts（unverified / user_confirmed / auto_verified / contradicted / rejected） | Phase 1 收尾前 | 0.5 天 |
| **A5** | AI Memory 5 张表占位（ai_episode_log / ai_learned_rules / user_ai_preferences / project_procedures / ai_feedback_signals）—— 只 CREATE + 单向写入，无读取逻辑 | Phase 2 启动前 | 1 天 |
| **A6** | `idempotency_key` 全写入路径支持（防 AI retry 时重复创建） | Phase 2 任意时段 | 1 天 |
| **B** | 命名锁定（content/body 统一 / source_type 枚举锁死 / confidence_source 字段） | Phase 2 任意时段 | 0.5 天 |

**总成本 5-7 天，分摊到 Phase 1/2 sprint 边角**，对 v2.2 主线推进影响 < 2 天。

**判断标准**：
- 3.0 启动时，**不需要做任何 schema migration**（用现有字段即可表达 AI actor 行为）
- 3.0 启动时，**已有 N 个月的 event_log + episode_log 真实数据**（不是空白账户）
- 3.0 启动时，**所有 2.0 模块只需扩展，不需重写**

**坚决不在 v2.2 做的事**（留 3.0）：
- ❌ agent framework（LangChain/AutoGen 等）
- ❌ AI memory 读取逻辑（只写不读）
- ❌ prompt template store
- ❌ vector store 重做
- ❌ AI actor 权限边界设计

## 4 · 三重校验门（每个里程碑结束必跑）

不允许"我觉得这个里程碑做完了"。必须跑下面三道门，三道全过才算完成。

### 校验门 A：现有功能回归（对应 N1）

| 步骤 | 做什么 | 通过标准 |
|---|---|---|
| A1 | 跑 `scripts/run_w1_acceptance.py` + `run_w2_acceptance.py` + 新建的 v2.2 acceptance | 所有维度 PASS（SKIPPED 需说明原因） |
| A2 | 手动启动桌面应用，过 `V2.2_FEATURE_INVENTORY.md` 里所有 9 大模块功能 | 每个核心动作都能完成 |
| A3 | 启动 mobile 应用，过 4 tab | 任务 / 日历 / 咨询 / 我的都能正常用 |
| A4 | 多账号回归：用 2 个账号登录，做一次跨账号同步 | 数据不被互相覆盖（v1.0 frozen 同步 bug 不复现） |

**A 门 FAIL 处理**：当前里程碑回滚到上一个 commit，不许进下一里程碑。

### 校验门 B：机器人问答能力进阶 + 4 主路径接通（对应 N2）

每个里程碑应当让机器人**多答对一组问题** **且 4 主路径通畅度增进**。验收用 5/19 张真会议作为金标准案例：

| 里程碑 | 机器人新增能答的问题 | 4 主路径状态 |
|---|---|---|
| M1（L2 ClientFactView） | "日慈基金会现在有哪些事件线？任务进度？" — 能跨表合并出客户全貌 | 路径 1+2 读取统一走 ClientFactBundle |
| M2（L3 ExtractionRunner） | "5/19 张真那场会有什么决定？" — 能从 docx 抽出张真接任法人等关键事实 | 路径 1+2+3 写入统一走 IngestPipeline |
| M3（L3 + clarification 闭环） | "刚才那 6 件待办里顾源源要做的有几件？" — 抽出来的事实能被结构化追问 | 路径 4 雏形：低 confidence → 主动发起澄清 |
| M4（L4 NarrativeKernel 事件驱动） | "兴盛+心理魔法学院合并后客户战略状态变了吗？" — narrative 跟着事实重新生成 | 4 路径任一变更 → 触发 event_log → narrative 自动刷新 |
| M5（手机端 L2 接通 + 朋友式聊天） | 在手机上问同样问题，得到一致答案 + 朋友式对话能澄清空白 | 路径 4 完整：手机 AI 朋友式聊天反向回写数据中心 |

**B 门 FAIL 处理**：里程碑代码可保留，但**记一笔"目标 B 进度未推进"**，下一里程碑必须补这条。连续 2 个里程碑 B 门不动 = 重新评估方向。

### 校验门 C：3.0 接口预留（对应 N3）

每个里程碑必须确认对应阶段的预留任务已落地：

| 里程碑 | 必须完成的 3.0 预留 |
|---|---|
| Phase 1 收尾 | A1 actor_type / A2 event_log / A4 verification_status 字段已在 schema |
| Phase 2 启动前 | A3 reasoning_trace / A5 AI Memory 5 表占位 已建好（只 CREATE）|
| Phase 2 任意 | A6 idempotency_key / B 命名锁定 已落地 |
| Phase 3 收尾 | 跑 3.0 接入烟测：模拟 AI actor 写一条 task + event_log + 学习一条 rule → 全链路通 |

**C 门 FAIL 处理**：3.0 预留缺失 → 当前里程碑不算完成（即使 A/B 都 PASS）。

## 8 · 失败模式预警（看到这些立即停手对齐）

| 失败模式 | 症状 | 应对 |
|---|---|---|
| **代码漂亮陷阱** | 在算"切了多少 SQL"、"模块边界 0 违规" | 停。回到 N1/N2/N3 校验门，看客户能不能感知 |
| **完美主义陷阱** | "要先把所有功能都迁完才能发 release" | 停。每里程碑 4-6 周必发 release，不全完也发 |
| **业务并行污染** | 在 v2.2 分支上写业务功能（如 brand_strategy） | 停。业务回 main，架构留 v2.2 |
| **AI session 漂移** | 新 session 没读 NORTH_STAR，自作主张定方向 | 强制读这份文件，否则不许动代码 |
| **B 门连续不动** | 2 个里程碑过去，机器人答不出新问题 | 重新评估，可能要换方向 |
| **N3 预留被跳过** | 想"先把功能做完，3.0 预留留到最后" | 停。Phase 2 启动后再加 actor_type 要 migrate 几万行 |
| **补丁接通** | "先 monkey patch 让这个 view 能跑" | 停。补丁不算完成，必须走干净架构 |

## 9 · v2.2 想清楚的事（写给未来 session 看）

- **v2.1 失败的根本原因**：上一线程把"v2.1 = 代码重构"当主线，没接住顾源源画的 5 层架构里 ★★★ L2 客户共识层这个真正重点。3 周做了 30% 地基，跟客户价值没直接关系。
- **v2.2 跟 v2.1 的关系**：v2.1 已做的 30% 地基保留复用（Repository 模式 + 模块边界 linter + W1 organization sync 引擎），但**重心转回客户价值目标**。不再追求"代码漂亮"。
- **v2.2 跟 3.0 的关系**：v2.2 不做 3.0 功能，但为 3.0 留接口（A1-A6 + B 命名锁定）。3.0 启动时 2.0 不返工 = 北极星 N3。
- **手机端的角色**：手机端已经走 cloud_backend HTTP API（28 个 endpoint 干净 facade），这是 L2 ClientFactView 的现成参考样板。桌面端 v2.2 改造可以照搬 mobile 的 endpoint shape。同时手机 AI 是 4 主路径之一，承载 N2 的"软件逻辑灵魂"。
