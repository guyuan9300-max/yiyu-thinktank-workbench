# V2.2 北极星 · 每个 session 第一件事必读

> **这是 v2.2 架构升级唯一的判分标尺。**
> 任何 session（包括 AI / 人 / 顾源源本人）开始工作前必读完这份。
> 写完代码 / 收尾时回到这份对照，看你做的事是否服务了这两个目标。
> 偏离目标 = 错误工作，不论代码多漂亮。

---

## 0 · 项目背景一句话

益语智库当前数据中心是「169 张表 + 1063 处散落 SQL + 6 大 view 各自直接 fetch」的多池并列结构。v2.1 走了 3 周「代码模块化」方向，跑偏。**v2.2 不要再以"代码漂亮"为目标，只以下面两个客户价值目标为目标。**

---

## 1 · 两个核心目标

### 目标 A：**现有功能不掉链**

> 整个系统要支持现在软件每一个板块的功能，完整运作起来。

**包含但不限于**（v2.2 启动盘点时拉出来的 6 大模块）：

1. **战略陪伴**（StrategicBrainView / StrategicClarificationView / StrategicDnaCard）
2. **资讯情报站**（IntelligenceStationView，55+ 处 API 调用）
3. **客户工作台**（chat / 文档 / 知识 / glossary / entity）
4. **任务/日程**（TaskListView / TaskCalendarView / EventLineReportPanel）
5. **成长中心**（GrowthCenterView / 徽章 / 周复盘）
6. **计划工作坊**（PlanWorkshopView / 部门计划编辑）
7. **数据中心**（SmartFileImport / RichTextDocumentEditor / 工作台资料库）
8. **手机端**（4 tab：任务 / 日历 / 咨询 / 我的）
9. **设置**（多账号 / 同步 / 飞书集成 / 本地 ASR / 组织管理）

完整清单见 `V2.2_FEATURE_INVENTORY.md`。

**判断标准**：每个里程碑完成后，**逐项跑过这份清单**——任何一项跑不通 = 里程碑 FAIL，必须先修复才能进下一里程碑。

### 目标 B：**机器人问答能拿到全部数据，流畅回答**

> 支持机器人问答，它可以访问到所有的数据，流畅的回答。

**具体含义**：

- 所有 docx / 会议纪要 / 任务 / 决策 / 客户档案 / 关系网 / 战略文档 → **机器人都能在对话里调用**
- 不需要用户手动告诉 AI "去查 X 文档"
- AI 答案带出处（evidence trail，可点回原文）
- AI 知道得越多越深，**用户能感知到这是"懂这个项目的人"**
- 跨多个 session 也记得（不是每次冷启动）

**判断标准**：每个里程碑完成后，**用 5/19 张真那场会做验收**——
- 这场会的会议纪要在工作台资料库里
- 问机器人："5/19 张真那场会有什么决定？"
- 答案应包含：张真接任法人 / 强哥任秘书长 / 兴盛+心理魔法学院合并 / 安心妈妈新项目 / 6 件待办（顾源源 3 件 / 张真 3 件 / 严斌 1 件）/ 兼职儿童数据红线
- 答案应带出处（指向那份 docx 的具体段落）
- **答出几条 = 进度的真实刻度**（不是"切了多少 SQL"）

## 2 · 双重校验门（每个里程碑结束必跑）

不允许"我觉得这个里程碑做完了"。必须跑下面两道门，两道全过才算完成。

### 校验门 A：现有功能回归

| 步骤 | 做什么 | 通过标准 |
|---|---|---|
| A1 | 跑 `scripts/run_w1_acceptance.py` + `run_w2_acceptance.py` + 新建的 v2.2 acceptance | 所有维度 PASS（SKIPPED 需说明原因） |
| A2 | 手动启动桌面应用，过 `V2.2_FEATURE_INVENTORY.md` 里所有 9 大模块功能 | 每个核心动作都能完成 |
| A3 | 启动 mobile 应用，过 4 tab | 任务 / 日历 / 咨询 / 我的都能正常用 |
| A4 | 多账号回归：用 2 个账号登录，做一次跨账号同步 | 数据不被互相覆盖（v1.0 frozen 同步 bug 不复现） |

**A 门 FAIL 处理**：当前里程碑回滚到上一个 commit，不许进下一里程碑。

### 校验门 B：机器人问答能力进阶

每个里程碑应当让机器人**多答对一组问题**。验收用 5/19 张真会议作为金标准案例：

| 里程碑 | 机器人新增能答的问题 |
|---|---|
| M1（L2 ClientFactView） | "日慈基金会现在有哪些事件线？任务进度？" — 能跨表合并出客户全貌 |
| M2（L3 ExtractionRunner） | "5/19 张真那场会有什么决定？" — 能从 docx 抽出张真接任法人等关键事实 |
| M3（L3 + clarification 闭环） | "刚才那 6 件待办里顾源源要做的有几件？" — 抽出来的事实能被结构化追问 |
| M4（L4 NarrativeKernel 事件驱动） | "兴盛+心理魔法学院合并后客户战略状态变了吗？" — narrative 跟着事实重新生成 |
| M5（手机端 L2 接通） | 在手机上问同样问题，得到一致答案 + 跨设备同步 |

**B 门 FAIL 处理**：里程碑代码可保留，但**记一笔"目标 B 进度未推进"**，下一里程碑必须补这条。连续 2 个里程碑 B 门不动 = 重新评估方向。

## 3 · 不在 v2.2 范围内的事

为了让 v2.2 在 10-12 周内完成，**这些事明确推到 v2.3 或永远不做**：

| 不做的事 | 原因 |
|---|---|
| App.tsx 28758 行单文件拆分 | 客户价值 = 0，对目标 A/B 都无直接贡献 |
| main.py 1063 处 SQL 全部清零 | 数字指标，不是客户价值 |
| 全部 195 个 services 都进 modules | 治标不治本，应该按目标 B 需要的去切 |
| 5 层架构每层都拆完整 | 5 层是设计骨架，不是 KPI |

**判断规则**：当你想做某件事时，问自己——"这件事直接让目标 A 更稳 OR 目标 B 更强吗？" 不是的话 = 不在 v2.2 范围。

## 4 · 每个 session 开局必做（Anthropic 长任务 protocol）

```
1. pwd                       # 确认在哪个目录
2. cat V2.2_NORTH_STAR.md    # 重读本文件
3. cat V2.2_PROGRESS.md      # 看上次 session 留了什么
4. git log -5                # 看最近 5 个 commit
5. cat V2.2_FEATURE_LIST.json # 看当前里程碑剩什么 feature 未完成
6. 选 1 个 feature 开干（不要贪多）
```

详细 protocol 见 `V2.2_SESSION_PROTOCOL.md`。

## 5 · 每个 session 收尾必做

```
1. 跑 校验门 A1（acceptance runner）
2. git commit（commit message 必须写"对应 NORTH_STAR 哪个目标"）
3. 更新 V2.2_PROGRESS.md（这次做了什么 + 下次应该做什么）
4. 更新 V2.2_FEATURE_LIST.json（标记已完成的 feature）
```

## 6 · 失败模式预警（看到这些立即停手对齐）

| 失败模式 | 症状 | 应对 |
|---|---|---|
| **代码漂亮陷阱** | 在算"切了多少 SQL"、"模块边界 0 违规" | 停。回到目标 A/B 校验门，看客户能不能感知 |
| **完美主义陷阱** | "要先把所有功能都迁完才能发 release" | 停。每里程碑 4-6 周必发 release，不全完也发 |
| **业务并行污染** | 在 v2.2 分支上写业务功能（如 brand_strategy） | 停。业务回 main，架构留 v2.2 |
| **AI session 漂移** | 新 session 没读 NORTH_STAR，自作主张定方向 | 强制读这份文件，否则不许动代码 |
| **B 门连续不动** | 2 个里程碑过去，机器人答不出新问题 | 重新评估，可能要换方向 |

## 7 · v2.2 想清楚的事（写给未来 session 看）

- **v2.1 失败的根本原因**：上一线程把"v2.1 = 代码重构"当主线，没接住顾源源画的 5 层架构里 ★★★ L2 客户共识层这个真正重点。3 周做了 30% 地基（organization SSOT + 模块骨架 + Repository），跟客户价值没直接关系。
- **v2.2 跟 v2.1 的关系**：v2.1 已做的 30% 地基保留复用（Repository 模式 + 模块边界 linter + W1 organization sync 引擎），但**重心转回客户价值目标**。不再追求"代码漂亮"。
- **v2.2 跟 B 方案的关系**：B 方案 8 个底层问题 = v2.2 的症状清单。5 层架构 = v2.2 的骨架。两者结合 = 这份 NORTH_STAR。
- **手机端的角色**：手机端已经走 cloud_backend HTTP API（28 个 endpoint 干净 facade），这是 L2 ClientFactView 的现成参考样板。桌面端 v2.2 改造可以照搬 mobile 的 endpoint shape。

---

**最后一句话**：v2.2 不是为了让代码看着漂亮，**是为了让 5/19 张真那场会议的结论能在战略陪伴、客户档案、机器人问答、关系网 4 个地方自动出现**。其它一切都是手段。
