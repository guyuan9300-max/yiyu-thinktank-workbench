## 第一部分 · 工作模式（我自己的操作手册）

### 1.1 单次迭代的六步结构

每个迭代严格按以下六步走，不跳步、不省步：

```
①  目标锁定
    ↓ 写下精确的验收标准（量化）+ UI 验证场景
②  方案确认
    ↓ 设计完成后先和用户确认，再动手
③  实现（可派 Agent）
    ↓ Builder Agent 写代码 / 我直接 Edit
④a  代码验证（必派 Verifier Agent）
    ↓ 跑测试 + 读 diff + 烟雾测试 + 回归检查
④b  UI 验证（鼠标控制 + 截图比对）★ 新增
    ↓ 重启应用 → 操作界面 → 看实际效果 → 截图比对
⑤  集成与提交
    ↓ git commit + 更新 TodoList + 给用户简报
```

**强制规则：**
- 第 ① 步没有量化验收标准 → 不进入第 ②
- 第 ① 步没有 UI 验证场景 → 不进入第 ②
- 第 ② 步没有用户确认 → 不进入第 ③
- 第 ④a/④b 任何一项失败 → 回到第 ③，不强行进入 ⑤
- **UI 验证必须用真实操作**——不能只看截图静态状态，要点击、输入、等响应
- 每个迭代结束后**强制 checkpoint**——总结、待办、风险、before/after 截图对比，等用户回应

### 1.2 Agent 协作模式

我会用 **两个固定角色** + **临时调研** 的方式：

**Builder Agent**（按需启动）
- 职责：写新代码（service、migration、route、test）
- 输入：明确的目标 + schema + 现有代码集成点
- 输出：代码 diff + 自测结果
- 何时用：单次实现量超过 200 行代码、跨多个文件、需要大段重复 pattern

**Verifier Agent**（每个迭代末必跑）
- 职责：审查 Builder 的产出
- 输入：刚改过的文件清单 + 验收标准
- 输出：通过/失败 + 具体问题列表
- 何时用：每个迭代第 ④ 步

**临时调研 Agent**（按需）
- 职责：搞清楚某段历史代码的实际行为
- 何时用：遇到不确定的现有逻辑

**我自己（Claude 主）的职责：**
- 总体节奏控制
- 用户沟通
- 跨迭代的决策（哪个先做、风险如何处置）
- 简单的 Edit（< 100 行）
- 数据库 schema 设计（不外包）

### 1.4 风险止损机制

遇到以下情况**立即停下**，回 checkpoint 找用户：

1. **范围漂移**：发现要改的范围超过预估 30%
2. **架构冲突**：发现现有代码和我的方案严重不兼容
3. **数据风险**：要做不可逆的数据库改动（DROP / 删除字段 / 数据迁移失败可能丢资料）
4. **测试大面积红**：现有测试因为我的改动崩了 > 5 个
5. **依赖缺失**：发现需要装新依赖（python-pptx 这种）
6. **超时**：单个迭代第 ③ 步工具调用超过 30 次还没完成

### 1.5 验证基础设施

每个迭代必跑的**四项**验证（前 3 项是代码级，第 4 项是 UI 级）：

**A. 单元测试**
```bash
cd backend && python -m pytest tests/ -x -k "<相关测试名>"
```

**B. 烟雾测试（重点接口）**
- 启动 backend
- 调用受影响的接口
- 比对 before/after 输出

**C. 回归检查**
- 跑全量已有测试，看是否有红
- 读 git diff，确认没有意外改动

**D. UI 验证（鼠标控制）★**
- 切到副屏 F34G4Q
- 截图存档（before 已经在 ① 步存了，这里是 after）
- 按目标锁定时定义的"UI 验证场景"实际操作
- 对比 before/after 截图，验证：
  - 期望的新元素出现
  - 期望的数值/文案改变
  - 原有功能没有 regression
- 若发现 UI 没正确反映后端改动：
  - 第一步：检查应用是否重连 backend（可能需要重启 dev 进程）
  - 第二步：检查前端是否需要同步修改
  - 第三步：若是前端缺集成，列入"补 UI"待办，但**不算迭代失败**（除非该改动本来就需要 UI）

**关键决策：**
- 精确匹配规则：`normalized_name == lower(trim(name))` 且 `entity_type` 一致 且 `organization_id` 一致
- 模糊匹配规则：编辑距离 ≤ 2 且类型一致 → 进 pending_review
- 自动合并阈值：同样的两个实体被关联到 5+ 个相同提及上下文 → 自动合并

## 第四部分 · 跨迭代的工程规范

### 4.1 数据库迁移规范
- **只加不删**：新字段、新表用 `IF NOT EXISTS` / `ALTER TABLE ... ADD COLUMN`
- **不改类型**：现有字段类型不动
- **幂等性**：所有 migration 重复跑不报错
- **回滚预案**：每次 schema 改动配一份"如何回滚"备忘

### 4.2 LLM 调用规范
- **复用现有 `ai.py` 接口**，不引入新 LLM 库
- **失败兜底**：LLM 失败不让 ingest 整体崩，记录失败 + 重试队列
- **成本控制**：抽取类任务用本地小模型，关键裁决可以升级
- **可观测**：每次调用记录 input_tokens / output_tokens / latency

### 4.3 集成现有 ingest 的位置
所有迭代都在 `ingest_document_knowledge`（`knowledge_v2.py:1696`）的同一个位置插入：

```python
# 解析完成、写入 v2_documents 之后
# 顺序执行（任一失败不影响其他）：

if feature_flag("entity_extraction"):
    try:
        EntityExtractor.extract_and_persist(doc, chunks)
    except Exception as e:
        log.warning(f"entity extraction failed: {e}")

if feature_flag("semantic_type_classification"):
    try:
        SemanticTypeClassifier.classify_chunks(chunks)
    except Exception as e:
        log.warning(f"semantic classification failed: {e}")

if feature_flag("relation_extraction") and entities_ready:
    try:
        RelationExtractor.extract_and_persist(doc, chunks, entities)
    except Exception as e:
        log.warning(f"relation extraction failed: {e}")

if feature_flag("fact_extraction") and entities_ready:
    try:
        facts = FactExtractor.extract(doc, chunks, entities)
        FactExtractor.persist(facts)
        ContradictionDetector.detect_for_new_facts(facts)
    except Exception as e:
        log.warning(f"fact extraction failed: {e}")
```

**Feature flag 开关**：放在 settings 表里，运维面板可控。**默认关，等迭代验证完才开。**

## 第五部分 · 风险与止损

### 5.1 全局风险

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| LLM 抽取质量差 | 高 | 高 | 设置 confidence 阈值，低分进 pending；用户能修正 |
| 历史资料 backfill 量大跑不完 | 中 | 中 | 分批 + 后台进程 + 可中断恢复 |
| schema 改动影响现有功能 | 低 | 高 | 严格只加不改；每次迁移后跑全量测试 |
| ingest 流程因新步骤变慢 | 高 | 中 | 抽取异步化，不阻塞主 ingest |
| LLM 成本意外飙升 | 中 | 中 | 用本地小模型为主；加日预算上限 |
| 实体爆炸 | 中 | 低 | 自动归档长期无引用实体 |
| 矛盾检测误报多 | 高 | 中 | 初版宽松，dismiss 进反馈 |

### 5.2 单次会话止损

如果在一次会话内：
- 超过预算工具调用数 → 停下做 checkpoint
- 测试大面积红 → 立刻回滚到上一个 commit
- 用户没回应超过 N 轮主动 checkpoint → 不进入下一个迭代

### 5.3 跨迭代止损

如果某个迭代验收不通过：
- 不进入下一个迭代
- 把 feature flag 关掉
- 分析根因后回炉
