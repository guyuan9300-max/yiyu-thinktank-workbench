# 益语智库 v2.1 架构升级 · 执行方案

> 这是 v2.1 重构落地的工程文档。
> 决策"为什么"已在前期讨论中确定,本文档只回答"做什么"和"怎么做"。
> 基线:tag `v1.0-pre-architecture-rewrite`(commit `400c125`)

---

## 1. 目标(一行)

**Pre-launch big-bang 重构,把 144k 行单体代码改造成
Modular Monolith + Repository + SQL Views + LLM Context 一等公民的稳定架构,
根治 v1.0 已遇到的 4 类客户类 bug,并为多人协作 + 云同步 + 大文件场景打好地基。**

---

## 2. 核心原则

| # | 原则 | 落地表现 |
|---|---|---|
| 1 | **统一性** | Big-bang 替换,无新老并存,no Strangler Fig |
| 2 | **验证原则** | 主体用 30 年 boring tech(SQL Views / Repository / DDL constraints) |
| 3 | **超级经验** | LLM Context 一等公民(Karpathy / Cursor 验证) |
| 4 | **贴合功能** | 拒绝 Bounded Context / Event Sourcing / CQRS 重概念,只做必需 |
| 5 | **大文件 + 高并发** | streaming readers + LLM semaphore + 保留 WAL/RLock |

## 3. 最终架构(目标态)

```
┌─────────────────────────────────────────────────────────────┐
│ 1. 视图层  src/renderer/views/ + backend/app/api/           │
│            (薄,只做 IO + 路由 + auth + 简单 React 展示)   │
└─────────────────────────────────────────────────────────────┘
                          ↑ HTTP
┌─────────────────────────────────────────────────────────────┐
│ 2. LLM 上下文层  backend/app/llm_context/   ★ Karpathy      │
│    - ContextComposer:所有给 LLM 的 prompt 都走它           │
│    - ContextInspector:debug 时看上次 LLM 看到了什么        │
│    - PromptLogger:持久化 prompt + 输出 + 评分(数据驱动)  │
└─────────────────────────────────────────────────────────────┘
                          ↑ 读
┌─────────────────────────────────────────────────────────────┐
│ 3. 业务模块层  backend/app/modules/                          │
│   ├ client/      ClientRepository                            │
│   ├ glossary/    GlossaryRepository(含多用户主观字段)      │
│   ├ knowledge/   KnowledgeRepository(实体/事实/向量)        │
│   ├ task/        TaskRepository                              │
│   ├ commitment/  CommitmentRepository                        │
│   ├ intelligence/IntelligenceRepository                      │
│   └ narrative/   NarrativeRepository                         │
│                                                              │
│   铁律:模块间只通过 Repository 接口对话                    │
│   linter 强制:跨模块 import 内部文件直接报错               │
└─────────────────────────────────────────────────────────────┘
                          ↑ 读/写
┌─────────────────────────────────────────────────────────────┐
│ 4. 数据层  backend/app/modules/*/schema.sql                  │
│   - SQL Views(lifecycle / scope / user 守卫)               │
│   - DB Constraints(FK/CHECK/UNIQUE)                         │
│   - Streaming readers(pypdf incremental,docx 段落迭代)    │
│   - 保留:WAL + RLock + busy_timeout + per-client Lock      │
└─────────────────────────────────────────────────────────────┘
                          ↕
┌─────────────────────────────────────────────────────────────┐
│ 5. 同步层  backend/app/cloud_sync/                           │
│   - 显式 per-table sync contract                             │
│   - Version vector / lamport clock(不再 last-write-wins)   │
│   - cloud_backend yiyu_cloud.db 初始化                      │
└─────────────────────────────────────────────────────────────┘
```

**关键 SQL Views**(最少这 6 个):

```sql
CREATE VIEW v_active_clients AS
  SELECT * FROM clients
  WHERE frozen_at IS NULL AND archived_at IS NULL;

CREATE VIEW v_user_visible_clients AS
  SELECT c.*, u.user_id AS viewer_id, u.access_level
  FROM v_active_clients c
  JOIN client_user_access u ON u.client_id = c.id;

CREATE VIEW v_client_facts AS
  SELECT c.id, c.name, c.alias,
    COALESCE(json_group_array(DISTINCT e.id) FILTER (WHERE e.id IS NOT NULL), '[]') AS entity_ids,
    (SELECT COUNT(*) FROM commitments cm WHERE cm.client_id = c.id AND cm.status = 'open') AS open_commitments
  FROM v_active_clients c
  LEFT JOIN entities e ON e.client_id = c.id
  GROUP BY c.id;

CREATE VIEW v_active_event_lines AS
  SELECT el.* FROM event_lines el
  JOIN v_active_clients c ON c.id = el.primary_client_id
  WHERE el.status IN ('active', 'blocked');

CREATE VIEW v_user_pending_tasks AS
  SELECT t.* FROM tasks t
  WHERE (t.assigned_to_user_id IS NULL OR t.assigned_to_user_id = :viewer_user_id)
    AND t.status IN ('todo', 'in_progress');

CREATE VIEW v_searchable_knowledge AS
  SELECT k.* FROM knowledge_master_index k
  JOIN v_active_clients c ON c.id = k.client_id
  WHERE k.is_searchable = 1;
```

**Schema 补丁**:

```sql
ALTER TABLE glossary_attributes ADD COLUMN opinion_by_user_id TEXT
  REFERENCES users(id) ON DELETE SET NULL;

CREATE TABLE personal_client_notes (
  client_id TEXT NOT NULL,
  user_id TEXT NOT NULL,
  body TEXT NOT NULL DEFAULT '',
  updated_at TEXT NOT NULL,
  PRIMARY KEY (client_id, user_id),
  FOREIGN KEY (client_id) REFERENCES clients(id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

ALTER TABLE commitments ADD COLUMN assigned_to_user_id TEXT
  REFERENCES users(id) ON DELETE SET NULL;

CREATE TRIGGER trg_client_lifecycle_xor
BEFORE UPDATE ON clients
FOR EACH ROW
WHEN NEW.frozen_at IS NOT NULL AND NEW.archived_at IS NOT NULL
BEGIN
  SELECT RAISE(ABORT, 'Client cannot be both frozen and archived');
END;
```
