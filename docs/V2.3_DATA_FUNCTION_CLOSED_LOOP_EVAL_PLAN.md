# 益语智库 2.0 数据-功能闭环测试评估方案

> **来源**: 顾源源 2026-05-23 钦定
> **本质**: 把 V2.3 验收从"工程层 schema/SQL"升级到"产品层数据-功能闭环"
> **核心立场**: 不测数据库有没有数据, 测"越用系统越懂客户 + 越懂客户功能越启发用户 + 用户启发后继续记录 → 数据中心更聪明"的循环
> **B 落档**: 2026-05-23 (顾源源选方式 A · 完整接收)

---

## 一、这次评估要验证的核心假设

### 假设一: 真实客户数据能支撑功能板块调用

用户打开某客户的工作台/战略陪伴/任务复盘/资讯/成长中心时, 系统不是空页面也不是泛泛生成, 而是能基于真实客户数据给出:
- 客户现在是谁
- 项目推进到哪里
- 最近发生了什么
- 谁承诺了什么
- 哪些风险正在出现
- 哪些事实需要澄清
- 下一步应该做什么

产品手册已明确数据中心是益语智库地基, 所有板块建在它上面.

### 假设二: 功能板块产生的数据能反向进入数据中心

用户不是为"喂数据"而工作. 用户正常创建任务/写复盘/上传文件/问 AI/纠正错误/关注情报/写个人手册, 系统要把这些动作自然拆解成:
- 新事实 / 新事件 / 新承诺 / 新风险 / 新判断 / 新人物关系 / 新证据 / 新方法经验

### 假设三: 数据中心能把低层事实升级成高层理解

系统不能只停留在 atomic_facts. 要继续派生出:
- 时间线 / 风险线 / 承诺线 / 关系网 / 关键判断 / 事实冲突 / 澄清问题 / 下一步行动

(V2.3 青禾测试已经暴露: atomic_facts 有数据, event_line_activities / strategic_thought_insights / risk_signals / tasks/commitments 都是 0, 故事卡时间线/关键判断/风险/下一步 4 段为空. → 这是 A V2.4 P0-1 AtomicFactSemanticDeriver 解决的事)

### 假设四: 用户使用后, 数据中心会变得更聪明

每次使用都会更新系统理解:
- 用户纠错 → 提高权威值
- 周复盘 → 补充项目进展
- 新任务 → 更新承诺状态
- 新文件 → 改变客户画像
- 外部情报 → 触发风险判断
- 成长手册 → 沉淀方法经验
- 战略陪伴的采纳/否决 → 校准 AI 的判断

## 二、评估总目标 — 回答 4 个问题

**问题一**: 数据中心现在懂不懂真实客户?
**问题二**: 各功能板块能不能用上数据中心?
**问题三**: 各功能板块能不能反向喂养数据中心?
**问题四**: 这个循环能不能让客户理解升级?

---

## 三、测试对象: 用真实客户, 而不是虚拟客户

3 类真实客户, 每类 1 个:

### 客户 A · 资料丰富型
- 特点: 已有较多合同/会议纪要/方案/报告/任务记录/复盘资料
- 用途: 测试数据中心是否能从大量真实资料中形成完整客户故事
- 适合测: 工作台问答 / 战略陪伴 6 段叙事 / 时间线 / 人物关系 / 风险识别 / 事实冲突

### 客户 B · 正在推进型
- 特点: 最近 2-4 周还在产生任务/会议/沟通和交付物
- 用途: 测试日常使用是否能持续喂养数据中心
- 适合测: 任务与日程 / 周复盘 / 承诺跟踪 / 下一步建议 / 项目阶段变化

### 客户 C · 信息稀疏型
- 特点: 资料不多, 更多依靠用户口述/零散文件/外部公开资料
- 用途: 测试系统是否能识别"我不知道什么", 而不是乱编
- 适合测: 智能文件导入 / 用户讲述 / 事实澄清 / 外部情报补充 / 低把握度标记 / 待补资料建议

---

## 四、测试结构: 做两轮, 而不是只测一次

### 第 0 轮 · 基线测试

不新增任何数据, 看系统现在凭现有数据能做到什么.

测:
- 每个客户的现有数据量 / 现有事实数 / 现有事件数 / 现有风险数 / 现有承诺数 / 现有战略判断数 / 现有澄清问题数 / 现有故事卡质量 / 当前功能板块调用质量

### 第 1 轮 · 真实使用测试

用户按正常工作流操作 (不是为了测试而填字段).

每个客户至少完成:
- 新建 2 条协作任务
- 写 1 次周复盘
- 在工作台问 5 个真实问题
- 上传或导入 3-5 份真实资料
- 在工作台纠正 1 条错误理解
- 在战略陪伴处理 3 条澄清
- 在资讯情报站关注或转任务 2 条情报
- 在成长中心沉淀 1 条方法卡
- 在组织计划工坊挂接 1 条任务到计划项

### 第 2 轮 · 升级后复测

使用后再次让系统重新理解客户, 比较前后变化:
- 故事卡是否更完整 / 时间线是否新增事件 / 风险是否更具体 / 承诺是否可追踪 / 人物关系是否更清楚 / 战略陪伴是否提出更准的问题 / 工作台回答是否引用了新数据 / 周复盘是否使用了工作台和任务中新产生的数据 / 用户纠正是否变成权威值 / 下一步建议是否更贴近真实推进

## 五、评估维度一 · 数据中心能不能支撑各功能板块 (数据中心 → 功能板块)

### 1. 任务与日程测试

任务详情页必须展示:
- 关联客户 / 关联项目 / 关联事件线 / 最近 3 条客户记忆 / 上一次会议摘要 / 相关承诺 / 相关风险 / 是否超过 N 天没有动作

通过标准: 任务不能只是标题+截止日期+负责人, 至少能展示:
- 任务为什么重要 / 和哪个客户/项目有关 / 上一次相关动作是什么 / 这条任务是否回应了某个承诺 / 是否存在延期风险 / 完成后会沉淀到哪些客户资料里

失败表现: 任务没有客户上下文 / 任务完成后没有进入事件线 / 周复盘只罗列任务没有客户判断 / 承诺没有进入 commitments / 任务里的会议纪要没有被抽成事实

### 2. 工作台测试

对每个真实客户问 5 类问题:
1. 客户基本情况
2. 项目进展
3. 关键人物
4. 风险判断
5. 版本口径 (某个预算/方案/时间是否有新旧版本差异)

通过标准: 回答必须 有明确来源 / 能区分事实和判断 / 能标记不确定 / 能识别旧版本 / 用户纠错后能更新权威值 / 同一问题二次询问时使用纠正后的答案

失败表现: AI 只根据单份文件回答 / 没有引用源 / 把用户判断说成客户事实 / 被用户纠正后下一次仍然答错 / 不能说明"为什么现在用这个版本"

### 3. 战略陪伴测试

对每个客户点击"让 AI 重新理解", 检查:
- 6 段客户叙事是否完整 / 思考研判是否有引用源 / 事实澄清是否列出真实冲突 / 低把握度段落是否明确标注 / 下一步建议是否可执行 / 用户采纳/否决是否写回系统

通过标准: 每个客户至少应生成 3 条有证据的主动研判 + 3 条待澄清问题 + 1 条人物关系判断 + 1 条风险判断 + 1 条下一步建议 + 6 段叙事都不为空

### 4. 资讯情报站测试

给真实客户设置 2-3 条焦点指令, 检查情报卡能否:
- 分级来源 / 判断相关性 / 标注情绪或风险 / 提出建议动作 / 转成任务 / 进入数据中心的外部证据层 / 与内部事实做对照

通过标准: 每个客户至少产生 3 条有效外部情报, 每条有来源等级, 至少 1 条能转任务, 至少 1 条能触发风险或待确认, 外部情报不会直接覆盖客户内部事实

### 5. 成长中心测试

完成一组真实工作: 写方案 / 推进任务 / 处理卡点 / 修正 AI 错误 / 沉淀方法卡 / 领取或确认成长信号

通过标准: 每个用户至少形成 2 条成长信号 + 1 条可追溯来源 + 1 条方法卡 + 1 次 AI 生成时调用方法卡的证据 + 成长内容权限正确

### 6. 智能文件导入测试

每个客户 3-5 份真实文件, 用户讲背景 (从哪里来 / 客户给的 / 我们写的 / 对标 / 旧版/新版 / 文件之间关系)

通过标准: 导入后系统应写入 文件角色 / 关键人物 / 关系 / 时间锚点 / 承诺 / 风险 / 文件版本 / 文档 chunk / 原子事实 / 后续战略陪伴刷新任务

### 7. 组织计划工坊测试

输入真实部门计划 → AI 拆成计划项 → 真实客户任务挂接到计划项

通过标准: 系统能回答 "这个任务服务哪个计划项 / 这个计划项对应哪些客户项目 / 哪些计划项有动作哪些没有 / 哪些计划项的进展来自真实客户资料 / 计划执行是否产生新的任务或复盘"

## 六、评估维度二 · 功能板块能不能反向喂养数据中心 (功能板块 → 数据中心)

每个板块测 5 件事:

### 1. 是否进入来源登记
每条数据应有: source_type / source_id / client_id / project_id / user_id / created_at / visibility_scope / confidence_prior / raw_reference

### 2. 是否进入原始材料层
原始内容不能被 AI 直接覆盖. 任务/复盘/对话/文件/情报/方法卡都要能保留原文.

### 3. 是否抽成结构化对象
按板块抽取: 人物 / 机构 / 项目 / 时间 / 金额 / 行动 / 承诺 / 风险 / 判断 / 文件角色 / 方法经验 / 计划项

### 4. 是否进入正确语义表
不能所有东西都只进 atomic_facts. 应进:
- event_line_activities / commitments / risk_signals / strategic_thought_insights / fact_contradictions / clarification_records / entities / client_glossary / growth_signal_events / handbook_entries / plan_items

### 5. 是否触发重新理解
每个重要写入都要触发或标记:
- narrative stale / portrait rebuild / self verify / contradiction scan / story card refresh / strategic insight refresh / workbench answer update

## 七、核心测试场景 · 8 个真实业务场景

### 场景一 · 真实客户接手测试
新同事 10 分钟内能基本讲清客户, 每个关键判断能点回来源.

### 场景二 · 真实任务进入测试
"下周二前给某客户发送新版项目建议书, 客户上次反馈希望执行周期从 6 个月压缩到 4 个月"
→ 任务/责任人/截止/关联客户/事件线/客户反馈事实/预算说明待办/项目周期变化/风险

### 场景三 · 真实周复盘进入测试
"本周完成方案沟通, 客户对预算敏感, 下一版方案降低复杂度"
→ 工作台/战略陪伴/风险清单/后续任务全部联动

### 场景四 · 真实工作台纠错测试
预算 50 万 → 用户纠 30 万 → 系统降权旧值 + 重新生成故事卡 + 后续问答使用 30 万

### 场景五 · 真实资料导入测试
旧版方案 / 新版方案 / 我方反馈 / 对标案例 / 政策资料 → 系统识别角色 + 不混淆

### 场景六 · 真实外部情报测试
至少 1 条外部信息能触发真实行动 (发简报/更新风险/确认官网滞后/调整下次会议问题)

### 场景七 · 真实战略陪伴测试
至少 2 条研判对用户有启发 + 1 条澄清问题值得问客户 + 1 条下一步建议能变成任务 + AI 不把不确定说成确定

### 场景八 · 真实成长中心测试
方法卡: "公益客户第一次看方案要先讲降低执行风险" → AI 生成方案时风格受影响, 但不当客户事实

## 八、评分体系 · 6 维度

### 维度一 · 真实数据支撑能力 (20 分)
0-5: 只能查到文件 / 6-10: 能回答基础事实但缺时间线人物 / 11-15: 工作台和部分战略陪伴 / 16-20: 5 板块都能调用一致客户理解

### 维度二 · 功能数据反向入库能力 (20 分)
0-5: 只存功能表不进数据中心 / 6-10: 进 atomic_facts 但不派生 / 11-15: 进部分语义表 / 16-20: 完整入来源+事实+事件+承诺+风险+判断+澄清链路

### 维度三 · 高级澄清能力 (20 分)
0-5: 没有矛盾检测 / 6-10: 能发现数值冲突但不解释 / 11-15: 生成澄清问题 / 16-20: 区分版本/口径/角色/外部滞后/用户判断缺证 5 类 + 用户裁决闭环

### 维度四 · 项目故事网能力 (20 分)
0-5: 只列事实 / 6-10: 回答部分问题 / 11-15: 故事卡但不完整 / 16-20: 讲清背景+时间线+人物+证据+风险+冲突+下一步

### 维度五 · 用户启发质量 (10 分)
澄清问题是否有用 / 主动研判是否有证据 / 下一步是否可执行 / 用户是否愿意采纳 / 是否帮用户发现原本没注意到的问题

### 维度六 · 正向循环能力 (10 分)
使用前后故事卡是否更完整 / 用户纠错是否被记住 / 新任务是否进入时间线 / 新复盘是否改变下一步 / 新情报是否触发风险 / 新方法卡是否影响生成

**总分 100 分**

## 九、必须设置的 6 项硬门槛

任意一项失败 = 数据-功能闭环不成立.

| # | 硬门槛 |
|---|---|
| 1 | 真实客户数据必须接入 (不再用虚拟青禾/临时 DB/预标数据) |
| 2 | 不能只测 atomic_facts (必须验证 event_line_activities/strategic_thought_insights/risk_signals/tasks/commitments/fact_contradictions/clarification_records 都有数据) |
| 3 | 必须测试真实 LLM 抽取 (不能用预标 fact 喂系统绕过 LLM 抽取层) |
| 4 | 必须测试自然语言回答 (不能只用关键词反查) |
| 5 | 必须测试用户纠错回写 (更新权威值/降权旧事实/关闭澄清/后续问答使用新值/故事卡重新生成) |
| 6 | 必须测试跨客户隔离 (同名人物/相似项目/相同金额不能跨客户混淆) |

## 十二、最终评估结论应该怎么写

不要只写"数据中心得分多少", 应写 4 个判断:

**判断一**: 数据中心当前是否能支撑功能板块
**判断二**: 功能板块是否能反向喂养数据中心
**判断三**: 是否形成越用越聪明的循环
**判断四**: 下一步最该修什么

---

## 十三、一句话概括

这次评估不是测益语智库有没有功能, 而是测这些功能能不能共同养出一个越来越聪明的数据中心; 也不是测数据中心有没有数据, 而是测它能不能反过来让用户在任务/复盘/工作台/战略陪伴/资讯情报中更快看清客户正在发生什么/什么最重要/下一步该怎么做.
