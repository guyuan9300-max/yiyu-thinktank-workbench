### 1.2 测试流程(代码顺序)

```python
1. setup_db()                  # 建临时 db (V2.1 完整 schema)
                               # INSERT clients (青禾) + event_lines (项目)
2. ingest_all_12_data()        # 12 条 datum × 多个 fact 进 IngestPipeline.ingest()

                               #   按 must_contain 关键词反查 atomic_facts

```

### 1.3 关键说明

**问答机制**: 50 问的回答不调 LLM,直接用关键词反查 `atomic_facts` 然后拼 markdown。这是为了**只测数据中心是否抽得对**,排除 LLM 表达能力的干扰。

**评分判定**:
- hit_rate = (must_contain 关键词在答案中的命中数) / (must_contain 长度)
- correct 判定 = hit_rate ≥ 60% AND must_not_contain 全部不命中 AND 有证据

具体说:
- `cross_source_check` 服务找到 **0 对**(因为它只看 subject_text 同音字)
- `contradiction_detector` 服务**没被 ingest 路径调用**(0 条 fact_contradictions)
- `_detect_value_clusters_for_project_subject` 和 `_write_value_conflict_clarifications`
  这两个函数是测试 runner 里手写的,用来"补"数据中心当前没做的事

所以 D3 满分的真正含义是:**"用测试代码后补的检测逻辑能识别冲突",而不是"V2.3 数据中心本身能自动识别冲突"。**

### 7.3 为什么是这样

**6 道错题分两类:**

**类一:answerer 工程问题(Q02 / Q14 / Q17 / Q44)**
关键词反查机制太粗,部分关键词(例如 "首次沟通"、"待确认"、"外部口径")在
must_contain 里用了短语,但反查 atomic_facts 时只命中部分子串,所以 hit_rate < 60%。
**数据中心里有相关数据,但被 answerer 漏掉。**

**类二:时间线/外部时间问题(Q36 / Q39),facts 数 = 0**
这两题问的是"按时间线复述"和"哪几天有外部/客户文件",
反查关键词("5 月 3 日"/"5 月 6 日"/"5 月 18 日"/"5 月 20 日"等中文格式)
跟 atomic_facts 里的 time_anchor 列 ISO 时间格式(`2026-05-03T00:00:00+00:00`)
**字符串完全不重叠**(中文 vs ISO)。

LIKE '%5 月 6 日%' 在 ISO 时间字段里找不到,LIKE '%2026-05-03%' 在中文叙述
字段里也找不到。所以 facts 数 = 0,答案空。

**这两题暴露了数据中心的 time_anchor 不能用人类语言直接检索 — 需要时间标准化层。**

## 8 · 总结(只列事实,不做判断)

### 8.1 测试到底测了什么

| 维度 | 测的是什么 | 不测的是什么 |
|---|---|---|
| D1 source_type | 测试 runner 调对了 normalizer | 不测 LLM 是否能自动识别 source_type |
| D2 实体召回 | 预标 fact 是否进 atomic_facts | 不测 LLM 是否能从原文抽实体 |
| D3 冲突 | 测试 runner 后补的 4 种检测逻辑找冲突 | 不测 V2.3 数据中心本身能识别多少冲突 |
| D4 故事卡 | story_card_generator 拉 5 张表 + atomic_facts | 不测 atomic_facts → 派生表的链路 |
| D5 50 问 | 关键词反查 atomic_facts | 不测 LLM 是否能用 facts 写出连贯答案 |

### 8.2 真正客观的发现(数字)

```
✓ 12/12 datum 通过 IngestPipeline 写进 atomic_facts (33 条)
✓ 33/33 atomic_facts 写进 source_registry (V2.3 阶段 1 接通)
✓ 5/5 核心冲突可被检测识别 (但只有 1/5 写进 clarification_records 表)
✓ 6/10 故事卡段有内容
✓ 44/50 测试问题能从 atomic_facts 反查到答案

✗ 0 条 event_line_activities (时间线表)
✗ 0 条 strategic_thought_insights (判断表)
✗ 0 条 risk_signals (风险表)
✗ 0 条 tasks / 0 条 commitments (待办表)
✗ 0 条 fact_contradictions (V2.2 矛盾表 — contradiction_detector 没接 ingest 路径)
✗ 4/5 检测出来的冲突没有持久化到 clarification_records
✗ time_anchor 列 ISO 格式与中文日期查询完全无法重叠
```

### 8.3 测试本身的局限性

```
1. dataset 已预标 fact 三元组,直接喂 IngestPipeline,
   绕过了文本 → fact 的 LLM 抽取层。
   真实生产场景里 LLM 抽取错误是大头,本测试没测。

2. answerer 用关键词反查,不调 LLM,
   所以"问答正确率" 只反映"数据是否在 db",不反映"AI 能否用人话回答"。

3. D3 有 4/5 冲突是测试 runner 自己用代码后补的检测逻辑找出来的,
   不是 V2.3 数据中心既有服务找出来的。
   把这部分剥离后,V2.3 真实冲突识别 = cross_source 0 + fact_contradictions 0 = 0。

4. 测试用临时 db,V2.1 完整 schema,但没接入真实主仓库 prod 数据。
```
