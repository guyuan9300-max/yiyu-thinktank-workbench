# 主动能力板块 · 第二线程优化方案

**起草日期：** 2026-05-12  
**当前完成度：** 15%  
**目标：** 一阶段（3 个月）50% / 二阶段（6 个月）80%  
**与理解深度的关系：** 完全正交并行，二者合流后才能撑起 2.0 → 2.5 → 3.0 路线

---

## 一、什么叫"主动能力"

主动能力不是单一功能，它是系统从"被动响应"到"主动出击"的整条转型：

```
被动                                                       主动
├─ 用户问 → 答                                              ├─ 系统定时跑 → 输出
├─ 用户导入 → 解析                                          ├─ 资料一来 → 触发洞察
├─ 用户找 → 显示                                            ├─ 状态变化 → 主动告警
├─ 用户翻 → 翻到才看到                                       ├─ 风险出现 → 主动推送
当前位置:  ████████████░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░
           ↑ 几乎全部能力卡在这里
```

**核心问题：** 当前系统在 ingest / 检索 / 路由 / 检索 / 生成上都很完备，但**"系统自己启动一个工作"这件事完全做不到**——没有 cron、没有调度器、没有通知投递、没有事件总线、没有 webhook。

**为什么这层关键：**

1. **战略陪伴需要主动**——客户每周该有总结，新资料来了该有洞察，资料过期了该有提醒。否则系统就是个"打字机+检索框"。
2. **AI 岗位虚拟公司的硬前置**——你 2.0/3.0 路线里所有 AI 岗位都要"周期性跑"。没主动能力 = AI 岗位永远等着你按按钮。
3. **迭代 6 矛盾告警的硬前置**——理解深度做出来的告警没有通知中心就送不达，等于白做。

## 二、9 个子能力拆解

| # | 子能力 | 当前 | 目标 | 难度 | 业务价值 |
|---|---|---|---|---|---|
| 1 | 调度器（cron 基础设施） | 0% | 完整 APScheduler 接入 | 中 | ⭐⭐⭐⭐⭐ |
| 2 | 通知中心（站内 + 系统） | 0% | notifications 表 + 投递路由 | 中 | ⭐⭐⭐⭐⭐ |
| 3 | 事件总线（pub/sub） | 5% | 现有 outbox 升级为可消费 | 中 | ⭐⭐⭐⭐ |
| 4 | 新资料触发洞察 | 0% | `on_ingest_insight` 钩子 | 低 | ⭐⭐⭐⭐ |
| 5 | 周期摘要（日/周/月） | 0% | 周报 / 月报自动生成 | 中 | ⭐⭐⭐⭐⭐ |
| 6 | 风险扫描（定期全量） | 0% | 资料过期/缺口/矛盾扫描 | 中 | ⭐⭐⭐⭐ |
| 7 | 跟进提醒 | 0% | "你 3 周前问的 X，现在有新资料" | 中 | ⭐⭐⭐ |
| 8 | 指标变化告警（delta） | 0% | 客户关键指标变化推送 | 高 | ⭐⭐⭐⭐ |

**优先级强约束：** 子能力 1（调度器）+ 2（通知中心）+ 3（事件总线）是其他 6 个的基础。**这三个必须最先做。**

## 三、代码现状审计（沙箱内已确认）

### 已存在的（仅 5%）

- `data_center_ingest_events`（`db.py`）—— 业务对象 ingest 事件表，**只有写入没有 worker 消费**。可以作为事件总线的雏形。
- `data_center_sync_outbox`（`db.py`）—— 同步出站队列，**同样只有生产者无消费者**。
- `inbox_status`（`task_collaborators`）—— 任务"待确认/已接/已拒"状态枚举，**不是通用消息收件箱**。
- `readyForNotifications: bool`（`models.py:2380`）—— 只是一个布尔标记，**无投递实现**。
- 启动 worker：`knowledge_worker_loop` + `analysis_job_worker_loop` —— **都是消费队列，无定时触发器**。

### 完全空白的（95%）

- `pyproject.toml` **无 APScheduler / Celery / schedule** 任一调度库
- 全代码库 **0 个 `notification_*` 表 / 服务**
- 无 `subscribe / publish / event_bus`
- 无 `cron / interval / periodic`
- 无 `webhook / on_event / hook_`
- 无 `digest / weekly_report / monthly_report` 类型的定时任务
- 无 `delta_detection / metric_change`
- 无 `risk_scan_schedule`

**这是个 greenfield**——比理解深度的初始状态更干净（理解深度至少有 `evidence_quality.py` 的粗糙启发式可以重写，主动能力是真的从 0 起步）。

## 四、迭代设计（7 个迭代）

### 全局序列

```
迭代 A0 准备工作（依赖审计、APScheduler 选型、与理解深度协调）
   ↓
迭代 A1 调度器基础设施
   ↓
迭代 A2 通知中心
   ├─→ 迭代 A4 周期摘要
   ↓
迭代 A3 事件总线统一
   ├─→ 迭代 A5 新资料触发洞察
   ↓
迭代 A6 风险扫描（定期全量）
   ↓
迭代 A7 跟进提醒 + 指标变化告警
```

A1+A2+A3 是底座必须串行；A4-A7 可以基于 A1+A2+A3 并行展开。

### 迭代 A1 · 调度器基础设施（核心，先做）

**一句话目标：** 接入 APScheduler，建立可定义/可触发/可观测的定时任务系统。

**量化验收：**
1. [ ] `pyproject.toml` 新增 `APScheduler>=3.10`
2. [ ] 新建 `app/services/scheduler.py`：单例 `BackgroundScheduler`，启动时随 `@app.on_event("startup")` 一起起，进程退出时清理
3. [ ] 新建 `scheduled_tasks` 表：`id / task_name / cron_expr / status (enabled/disabled) / last_run_at / next_run_at / last_error / created_at`
4. [ ] 新建 `scheduled_task_runs` 表：每次跑都有一条日志（`task_id / started_at / finished_at / status / error_message`）
5. [ ] 一个示例任务："每天凌晨 3 点 echo 'hello'"，证明链路通
6. [ ] 运维面板加"定时任务"页签：列任务、看历史、手动触发、暂停/恢复
7. [ ] 至少 6 个测试
8. [ ] UI 验证：在运维面板能看到一个示例任务，手动 trigger 后历史里多一条记录

**预算：** 35-50 工具调用，派 Builder Agent

### 迭代 A2 · 通知中心

**一句话目标：** 系统能告诉用户"有事发生"——站内、Mac 通知、可选邮件。

**量化验收：**
1. [ ] 新建 `notifications` 表：`id / user_id / type / severity (info/warning/critical) / title / body_md / source_event_id / read_at / dismissed_at / actions_json / created_at`
2. [ ] 新建 `notification_delivery_log`：每个 channel 一条记录（`in_app / mac_native / email`），跟踪投递状态
3. [ ] 新建 `app/services/notifications.py`：`emit(user_id, type, ...)` 是公开 API
4. [ ] `in_app` 投递：写入 notifications 表 + IPC 推送给 Electron renderer
5. [ ] `mac_native` 投递：`osascript display notification`（可选）
6. [ ] Electron renderer 加一个"消息中心"图标 + 红点 + 弹层
7. [ ] 至少 8 个测试
8. [ ] UI 验证：调用 `notifications.emit(...)` 接口后，电脑标题栏图标出现红点 + 点开看到消息

**预算：** 40-55 工具调用，派 Builder Agent

### 迭代 A3 · 事件总线统一

**一句话目标：** 把现有 `data_center_ingest_events` + `data_center_sync_outbox` 统一为通用 pub/sub，新增订阅机制。

**量化验收：**
1. [ ] 新建 `event_bus` 表：`event_id / event_type / payload_json / source / created_at`
2. [ ] 新建 `event_subscriptions` 表：`subscriber_name / event_type_pattern / handler_ref / enabled / last_fired_at`
3. [ ] `app/services/event_bus.py`：`publish(event_type, payload)` + `subscribe(pattern, handler)` 装饰器
4. [ ] 现有 `data_center_ingest_events` 写入位置改为 `event_bus.publish("ingest.task.created", ...)` 等
5. [ ] 一个示例订阅者：监听 `ingest.*` 事件，写日志（证明链路通）
6. [ ] 至少 6 个测试

**预算：** 35-50 工具调用

### 迭代 A4 · 周期摘要（日/周/月）

**依赖：** A1 + A2

**一句话目标：** 系统每周自动给客户生成一份摘要，推送通知。

**量化验收：**
1. [ ] 新建 `digest_runs` 表：客户/周期/状态/内容/created_at
2. [ ] 新建 `app/services/periodic_digest.py`：
   - `generate_weekly_digest(client_id)`：基于最近 7 天的任务/会议/资料/事件线，调 LLM 生成结构化摘要
   - `generate_monthly_digest(client_id)`：类似
3. [ ] 注册定时任务：每周一 9:00 跑全部客户的周报
4. [ ] 生成完毕投递通知（点击通知跳到摘要详情页）
5. [ ] 至少 4 个测试
6. [ ] UI 验证：手动 trigger 周报 → 通知中心收到 → 点击进摘要页

**预算：** 30-40 工具调用

### 迭代 A5 · 新资料触发洞察

**依赖：** A3

**一句话目标：** 资料一来，AI 主动判断"这是不是有趣的、要立即提醒？"

**量化验收：**
1. [ ] 订阅 `ingest.document.created` 事件
2. [ ] 处理器：用 LLM 评估资料的"新颖性 / 紧迫性 / 与客户战略相关性"
3. [ ] 评分超过阈值 → 投递通知
4. [ ] 至少 4 个测试
5. [ ] UI 验证：导入一份与客户战略强相关的资料，等几秒看通知中心是否弹出"建议关注"

**预算：** 25-35 工具调用

### 迭代 A6 · 风险扫描（定期全量）

**依赖：** A1 + A2 + 理解深度迭代 5/6（关系 + 矛盾）

**一句话目标：** 每天/每周扫一遍全部客户的资料，发现资料过期、缺口、矛盾、风险信号，主动告警。

**量化验收：**
1. [ ] 新建 `risk_scan_runs` 表
2. [ ] 注册定时任务：每天凌晨 4:00 跑
3. [ ] 扫描维度（最小集）：
   - 资料过期（鲜度 < 阈值的关键资料）
   - context_pack 缺口
   - （如理解深度迭代 6 已上线）未 dismiss 的矛盾
4. [ ] 发现风险 → 投递通知
5. [ ] 运维面板加"风险扫描"页签
6. [ ] 至少 5 个测试

**预算：** 35-45 工具调用

### 迭代 A7 · 跟进提醒 + 指标变化告警

**依赖：** A1 + A2 + A3

**一句话目标：** "你三周前问了 X，这周相关资料更新了" + "客户 A 的 KPI 跌了 20%"。

**量化验收：**
1. [ ] 新建 `followup_topics` 表：记录"用户问过但没结论"的话题
2. [ ] 新建 `metric_snapshots` 表：周期捕获每个客户的关键指标
3. [ ] 跟进提醒任务：每周扫 `followup_topics`，看是否有新资料覆盖
4. [ ] 指标变化任务：每周对比 snapshots，发现 >20% delta 主动推送
5. [ ] 至少 5 个测试

**预算：** 35-45 工具调用

## 五、与"理解深度"线程的并行策略

### 文件级正交（不会撞车）

| 板块 | 主要触碰文件 | 新增表 |
|---|---|---|
| 理解深度 | `freshness_decay.py` `entity_extractor.py` `evidence_quality.py` `knowledge_v2.py:1696` 注入点 | `entities` `entity_mentions` `relationship_triples` `atomic_facts` 等 |
| 主动能力 | `scheduler.py` `notifications.py` `event_bus.py` `periodic_digest.py` `main.py:startup` | `scheduled_tasks` `scheduled_task_runs` `notifications` `event_bus` 等 |

唯一可能冲突的点：

- **`main.py` 的 `@app.on_event("startup")` 区**——两个线程都要加启动 worker。**协调办法：** 任一线程先动这里时显式留一个 `# === SCHEDULERS / SUBSCRIBERS ===` 注释区，另一线程往里加。或者拆出一个 `app/startup_hooks.py` 统一管理。
- **`db.py`**——两个线程都加表。但只要加在末尾、不动现有 schema，幂等。
- **运维面板 (`DataCenterOpsPanel.tsx`)**——两个线程都加页签。**协调办法：** 各加一个独立 tab，互不影响。

### 提交节奏正交

- **理解深度** 在 `feature/understanding-depth` 分支
- **主动能力** 在 `feature/proactive-capability` 分支（新建）

两条分支独立开发，**完成后分别 merge 回 main**。如果有冲突（极少数情况）merge 时解决。

### 工作流并行

两条线程**可以由同一个 Cowork session 交替推进**：

- 每个迭代约束在六步走（目标锁定 → 方案确认 → 实现 → 代码验证 → UI 验证 → 集成）
- 一个线程在等用户确认（如方案审核）时，另一个线程可以推进
- 你看到的 checkpoint 会有两条流水线，但每个 checkpoint 都明确标注是哪条线程

### 合流时机

**A3 事件总线** 完成后，**理解深度迭代 6 的矛盾告警**可以接入 A2 通知中心——这是两条线程**最自然的合流点**。届时矛盾检测出告警 → 通过事件总线发布 → 通知中心投递。

## 七、风险与权衡

| 风险 | 概率 | 缓解 |
|---|---|---|
| APScheduler 与 Electron lifecycle 冲突 | 中 | dev/packaged 分别测试；进程退出时主动 shutdown |
| 通知投递权限被 macOS 拦 | 高 | 走 osascript 而非自定义 daemon；首次运行引导用户允许通知 |
| 定时任务跑挂数据库（并发） | 中 | scheduled_tasks 加 max_instances=1，单实例 |
| 周报生成 LLM 调用费用 | 中 | 按客户数 ×4 周/月 估算预算；可降级到模板 |
| 主动通知扰民 | 高 | 严格分级 info/warning/critical，用户能整体静音 |
| 与理解深度迭代冲突 main.py | 低 | 拆 startup_hooks.py |

## 九、一句话总结

> **主动能力是从"被动检索系统"到"AI 主理人"的硬转折。它跟理解深度完全正交（不同的文件、不同的表、不同的 UI 入口），可以真并行做。3 个月跟理解深度合流后，迭代 6 矛盾告警就能落地（告警 → 事件总线 → 通知中心 → 用户看到），AI 主理人雏形成立。**

**先动的两件事：**
- A1 调度器（让系统能自己启动工作）
- A2 通知中心（让系统能告诉用户"有事发生"）

这两个做完后，主动能力的"地基"就有了，其他 5 个迭代都基于此展开。
