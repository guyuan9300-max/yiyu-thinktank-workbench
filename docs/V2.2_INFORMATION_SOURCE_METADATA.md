# V2.2 信息源元数据规范

> **配套**：[V2.2_NORTH_STAR.md](./V2.2_NORTH_STAR.md) §3 N2 的具体落地依据
> **核心问题**：4 条主路径进数据中心的每份信息，必须带哪些元数据，数据中心才能"理解"它们？
> **写作方法（顾源源 2026-05-22 原则）**：从真实工作场景以终为始反推，不靠抽象推演。

---

## 0 · 一句话总结

**信息进数据中心不是"存哪份文档"，是"标好这份信息的 5 维身份证"**：来源、角色、关系、作者/受众、生命周期。

5 维都标好，AI 才能：
- 区分事实 vs 观察 vs 推测
- 按权威度排序引用
- 知道哪些字段还要追问
- 自动判断哪些信息过期了

---

## 1 · 从真实场景反推：为什么必须有这 5 维元数据

### 场景 A：顾源源问机器人"日慈最近有什么风险？"

机器人手里有的信息（散在数据中心）：
- 日慈官网说"心盛计划已启动 5 期"
- 某公众号说"日慈面临资金链断裂风险"
- 5/19 张真会议纪要说"高老师离职导致品牌升级停摆"
- 顾源源手机 AI 聊天里说"我觉得岩冰短期补不上高老师的位置"
- 5/12 周复盘说"日慈课程交付延期 2 次"
- 互联网爬虫抓到一篇 AI 写的分析文章说"日慈缺乏现金流"

**没有元数据时机器人怎么回答**：
> "日慈面临资金链断裂、品牌升级停摆、课程延期，缺乏现金流。"
>
> ❌ 全部混在一起。"资金链断裂"是公众号编的、"现金流"是 AI 推的，但 AI 当成了事实。
> ❌ 没区分"客户自己说的" vs "外部分析"。
> ❌ 没标"用户的主观观察"。

**有元数据时机器人应该怎么回答**：
> **客户自己已确认的风险**（来自 5/19 会议纪要）：
> - 高老师离职导致品牌升级停摆
>
> **从复盘观察到的运行性风险**（来自 5/12 周复盘）：
> - 课程交付延期 2 次
>
> **顾源源主观观察（待验证）**：
> - 岩冰短期补不上高老师位置
>
> **互联网未经确认的传言**（仅供参考）：
> - 某公众号提到资金链断裂传言
>
> ✅ 分层呈现，权威度可追溯。

**反推出的必要元数据**：
- **来源类型**（决定基础置信度）
- **业务角色**（事实/观察/推测/传言）
- **作者**（客户自己 / 顾源源 / 第三方）

### 场景 B：AI 自动生成战略陪伴 narrative

AI 要写"日慈基金会战略陪伴本月叙事"，候选事实有 50 条。

**没有元数据时**：AI 把 50 条平等灌进去，输出像信息汇总，不是叙事。

**有元数据时**：AI 应该：
1. 优先引用"客户自己说的 + 已 user_confirmed"的事实
2. 次之引用"我们和客户协作的记录（任务/承诺）"
3. 再次之"用户的观察/复盘"
4. 最后"互联网官方信息"作为背景
5. **完全不引用**"互联网 UGC / AI 推断 / contradicted"的信息

**反推出的必要元数据**：
- **来源类型**（权威度排序的依据）
- **验证状态**（user_confirmed / contradicted / pending）
- **时间锚**（这条事实的时效性，半年前的复盘 vs 上周的纪要）

### 场景 C：AI 主动发起朋友式澄清（路径 4）

AI 在后台扫描，发现一条信息：
> "日慈秘书长可能是强哥"（来自 5/19 张真口述，confidence=0.7）

AI 决定要不要主动问顾源源：
- 如果这条来自客户官方文档 → 不需要再问
- 如果来自互联网公众号 → 不能信，需要走客户官方渠道确认
- 如果来自客户自己口述 + 未验证 → **应该主动友好地问顾源源**："上次张真说强哥接秘书长，这事最终定了吗？"

**反推出的必要元数据**：
- **来源类型**（决定该不该问）
- **验证状态**（unverified → 触发澄清）
- **作者**（口述的是谁，决定追问方式）
- **关系语境**（关联到哪条事件线，避免突兀的问话）

### 场景 D：信息冲突时怎么仲裁

两条信息打架：
- 路径 1（工作台文档 5/15）："日慈现金账上 800 万"
- 路径 4（手机 AI 聊天 5/20）：顾源源说"日慈账上现金应该只剩 300 万左右"

**没有元数据时**：AI 不知道相信哪个，可能交替使用甚至同一段同时说两个数字。

**有元数据时**：
- 比较两条的"时间锚"（5/15 vs 5/20）→ 后者更新
- 比较两条的"作者"（书面文档可能是几个月前数据 / 顾源源是直接当事人）
- 比较两条的"业务角色"（前者是 fact / 后者是 observation）
- 决策：标记前者 as `superseded`，后者 as `current`，但提示顾源源去客户处确认

**反推出的必要元数据**：
- **时间锚**（哪个更新）
- **作者**（谁更接近事实源）
- **生命周期状态**（superseded / current）

### 场景 E：3.0 AI actor 自己写的事实，要不要相信？

3.0 时 AI 自己从一份合同里抽出一条事实写入数据中心。

**没有元数据时**：跟人写的同等对待 → 灾难（AI 抽错的事实污染了真相源）。

**有元数据时**：
- 标 `actor_type = ai_agent` + `verification_status = unverified`
- 进入 pending 队列，由人审过才升级为 user_confirmed
- 引用时降权（"AI 抽取的 / 待确认"前缀）

**反推出的必要元数据**：
- **actor_type**（谁写的，人 vs AI vs 系统）
- **验证状态**（AI 写的默认 unverified）

## 2 · 抽象出的 5 维元数据规律

把上面 5 个场景的"反推出的必要元数据"合并去重，得到 **5 维 + 21 字段**：

### 维度 1 · 来源类型（source_type）

**作用**：决定基础置信度 + 决定 AI 怎么引用

完整枚举（按实际工作场景穷举）：

| 类别 | source_type 值 | 基础置信度 | AI 引用方式 | 实际场景 |
|---|---|---|---|---|
| **客户自报 · 正式** | `client_official_doc` | 0.95 | 直接引用，作为事实 | 客户官网、章程、年报、白皮书 |
| **客户自报 · 内部** | `client_internal_doc` | 0.90 | 直接引用，事实 | 会议纪要、内部邮件、内部 PPT |
| **客户自报 · 当面** | `client_verbal_meeting` | 0.85 | 引用时标"客户口述" | 5/19 张真会议这种 |
| **协作记录** | `collaboration_task` | 0.90 | 直接引用 | tasks、commitments、协作信息 |
| **协作记录 · 复盘** | `collaboration_review` | 0.85 | 标"复盘经验" | weekly_reviews、项目复盘 |
| **用户观察** | `user_observation` | 0.65 | 标"用户主观观察，待确认" | 顾源源手机 AI 聊天里说的判断 |
| **用户口述事实** | `user_verbal_fact` | 0.75 | 标"用户口述，待客户确认" | 顾源源讲的客户那边的事 |
| **互联网 · 官方** | `internet_official` | 0.85 | 引用时标来源 | 政府公示、慈善中国、官网爬虫 |
| **互联网 · 主流媒体** | `internet_media` | 0.60 | 标"媒体报道，仅供参考" | 财经新闻、人民日报等 |
| **互联网 · UGC** | `internet_ugc` | 0.30 | 标"网络传言，可能不实" | 公众号、知乎、微博 |
| **互联网 · AI 分析** | `internet_ai_inferred` | 0.20 | **不引用**，仅作背景 | ChatGPT 给的分析、AI 写的稿 |
| **LLM 抽取** | `llm_extracted` | 0.50 | 标"AI 抽取，待审核" | ExtractionRunner 从 docx 抽的 |
| **系统推断** | `system_derived` | 0.70 | 标"自动推断" | join 推出来的关系 |
| **AI agent 主动写入**（3.0） | `ai_agent_authored` | 0.40 | 标"AI 待人审" | 3.0 AI 自己建的任务/事实 |

**关键设计**：基础置信度只是默认值，用户校正（user_confirmed）后会升到 1.0；contradicted 后降到 0。

### 维度 2 · 业务角色（content_role）

**作用**：决定 AI 怎么聚合、排序、引用

| content_role | 含义 | AI 用它做什么 |
|---|---|---|
| `fact` | 客观陈述 | 引用、引用、引用 |
| `decision` | 行动结论 | 触发下游任务，影响其他事实 |
| `risk` | 警示信号 | 提醒、进入风险队列 |
| `progress` | 状态变化 | 更新 event_line 状态 |
| `plan` | 未来动作 | 进入 task 候选 |
| `lesson` | 复盘经验 | 指导未来类似场景 |
| `observation` | 主观观察 | 引用时标"观察" |
| `speculation` | 未验证假设 | 引用时标"推测"，触发澄清 |
| `quote` | 当事人语录 | 直引，带说话者 |
| `commitment` | 承诺 | 进入承诺到期监控 |

### 维度 3 · 关系语境（context_link）

**作用**：决定 AI 怎么聚合到客户/事件/人/任务

| 字段 | 含义 |
|---|---|
| `client_id` | 这条信息属于哪个客户（必填，除非是组织级） |
| `event_line_id` | 关联到哪条主线（可选） |
| `person_id` | 关联到哪个人物（可选，多对多） |
| `task_id` | 关联到哪个任务（可选） |
| `parent_doc_id` | 来源文档 id（精确到段落 chunk_id） |
| `time_anchor` | 这件事发生的时间（≠ 信息录入时间） |
| `topic_tags` | 主题标签（如"心盛计划"/"品牌升级"，从 glossary 来） |

### 维度 4 · 作者/受众（actor metadata）

**作用**：决定 AI 怎么追问、怎么排序权威度

| 字段 | 含义 | 真实场景 |
|---|---|---|
| `actor_type` | human / ai_agent / system | 3.0 时 AI 自己写的事实要区分 |
| `actor_id` | user_id 或 ai_session_id | 谁写的 |
| `speaker_person_id` | 这话是谁说的（≠ 录入人） | 张真说的话，actor=顾源源（录入者），speaker=张真 |
| `recipient_person_id` | 给谁的（如果是协作信息） | 给岩冰发的协作信息 |
| `intended_scope` | private / team / org / public | 决定谁能看到 |
| `created_at` / `updated_at` | 时间戳 | 标准字段 |

### 维度 5 · 生命周期（lifecycle）

**作用**：决定 AI 引用时是否还有效

| 字段 | 取值 | 含义 |
|---|---|---|
| `verification_status` | unverified / user_confirmed / auto_verified / contradicted / rejected | 验证状态 |
| `confidence_source` | rule / llm / user / cross_validated | 置信度怎么算出来的 |
| `confidence_score` | 0.0 ~ 1.0 | 当前置信度（受 source_type + verification 影响） |
| `validity_status` | current / superseded / archived | 是否被后续信息推翻 |
| `superseded_by_id` | 谁推翻了它 | 链式追溯 |
| `valid_from` / `valid_until` | 时间窗 | 时效性事实用 |
| `last_verified_at` | 上次验证时间 | 触发"该再确认了吗" |

## 3 · 4 条主路径对应的元数据默认值

每条路径进 IngestPipeline 时，元数据应该自动填好，用户不需要每次手填：

### 路径 1：工作台进入的文件

| 元数据 | 默认填法 |
|---|---|
| source_type | 看文件来源：用户上传的 docx → `client_internal_doc`；爬虫拉的 → 见路径 3 |
| content_role | 默认 `fact`，等 LLM 抽出来后分细 |
| actor_type | `human` |
| actor_id | 上传者 user_id |
| client_id | 上传时绑定的 client |
| parent_doc_id | v2_document.id |
| verification_status | `unverified`（等 LLM 抽出 + 人审） |
| confidence_source | `rule` |
| confidence_score | 由 source_type 决定基础值 |
| time_anchor | 从文件内容 LLM 抽（fallback = uploaded_at） |

### 路径 2：任务/计划/复盘/协作信息

| 元数据 | 默认填法 |
|---|---|
| source_type | tasks → `collaboration_task`；weekly_reviews → `collaboration_review` |
| content_role | task → `plan`；review 抽出来分 `lesson`/`observation`/`fact` |
| actor_type | `human` |
| actor_id | 创建者 user_id |
| recipient_person_id | task.owner / collaboration target |
| verification_status | `user_confirmed`（用户主动写的，直接信） |
| confidence_score | 0.90 |
| time_anchor | task.due_date 或 review.week |

### 路径 3：互联网爬虫补全

| 元数据 | 默认填法 |
|---|---|
| source_type | **关键** — 必须按域名+内容形式分细：<br/>- 客户官网 → `internet_official`<br/>- gov.cn / cssn.gov.cn 等 → `internet_official`<br/>- 主流财经媒体 → `internet_media`<br/>- 公众号/知乎/微博 → `internet_ugc`<br/>- AI 写的分析稿 → `internet_ai_inferred` |
| content_role | 默认 `fact`，但低 confidence 的标 `observation` |
| actor_type | `system` |
| actor_id | crawler_run_id |
| verification_status | `unverified`（爬来的都要审） |
| confidence_score | 由 source_type 决定（30-85） |
| time_anchor | 文章发布日期（非爬取日期） |

### 路径 4：手机 AI 聊天澄清

| 元数据 | 默认填法 |
|---|---|
| source_type | 用户主动说事实 → `user_verbal_fact`；用户主观判断 → `user_observation`；用户引用第三方 → 按转述类型 |
| content_role | LLM 实时判断（fact/observation/speculation/decision） |
| actor_type | `human` |
| actor_id | 顾源源 user_id |
| speaker_person_id | 如果用户在转述某人的话，标这个 |
| verification_status | `user_confirmed`（用户主动说的，但部分要标 `unverified` 等待跟客户确认） |
| confidence_source | `user` |
| confidence_score | 0.65-0.85（视 content_role 而定） |
| time_anchor | LLM 从对话上下文抽（如"上周二张真说..."） |

## 4 · AI 怎么用这 5 维元数据（具体行为规则）

### 规则 R1：引用时分层呈现

```python
def cite_facts_for_narrative(facts: list[Fact]) -> str:
    # 按 source_type + verification_status 分 3 档
    tier_a = [f for f in facts if f.source_type in CLIENT_AUTHORITATIVE
              and f.verification_status in (CONFIRMED, AUTO_VERIFIED)]
    tier_b = [f for f in facts if f.source_type in COLLABORATION
              or f.source_type == USER_VERBAL_FACT]
    tier_c = [f for f in facts if f.source_type in INTERNET_UNVERIFIED
              or f.confidence_score < 0.5]
    # Tier C 不引用，仅在"背景信息"段提及
```

### 规则 R2：冲突仲裁(3 路语义,2026-05-22 修订)

**核心认知:信息冲突 ≠ 信息更新 ≠ 互补。3 种不同语义,不能用同一套权威度排序。**

顾源源 2026-05-22 原话举例:"合同金额本来应该是 800 万,但合同显示 300 万,所以我们现在需要重签 —— 这不是哪个对,是有一个被更新了。"

**3 种语义详解**:

| 语义 | 判定信号 | atomic_facts 怎么写 | UI 展示 |
|---|---|---|---|
| **路径 A · 信息更新** | LLM 识别用户对话有更新意图(应该改 / 重签 / 不对了 / 已经变成 / 改成) | 旧 `status='superseded'` + `superseded_by_id=新.id`,新 `status='active'` | 默认显示新版本,点"历史"看旧版 |
| **路径 B · 互补(时间序列)** | 时间锚差 > 30 天 + 无更新意图触发词 | 都 `status='active'`,`time_anchor` 不同 | 时间序列展示(3 月 vs 5 月并列) |
| **路径 C · 真冲突** | 时间锚接近 + 无更新意图 + 来源都权威 | 都标 `status='contradicted'`,进 `clarification_queue` | 黄色警示卡 + 朋友式澄清问句 |

**实现伪代码**:

```python
def resolve_conflict(fact_a: Fact, fact_b: Fact,
                     dialog: ChatContext | None) -> ConflictResolution:
    # 路径 A: 信息更新语义检测 (依赖 dialog, 这是关键创新)
    if dialog:
        intent = detect_update_intent_in_dialog(
            dialog.recent_messages,
            target=fact_a if fact_a.time_anchor < fact_b.time_anchor else fact_b,
        )
        if intent.confidence > 0.7:
            return ConflictResolution(
                kind="update",
                superseded_id=intent.old_id,
                current_id=intent.new_id,
                reason=intent.user_quote,  # 保留用户原话作 audit
            )

    # 路径 B: 互补 (时间序列, 30 天阈值)
    if abs(fact_a.time_anchor - fact_b.time_anchor).days > 30:
        return ConflictResolution(kind="complementary", sort_by="time_anchor")

    # 路径 C: 真冲突
    return ConflictResolution(
        kind="contradicted",
        push_to_clarification_queue=True,
    )
```

**N3 预留**: `superseded_by_id` 是 v2.2 必加的 atomic_facts 字段。3.0 AI actor 用同一字段记录自己的"更新动作"。

### 规则 R3：主动澄清触发

```python
def should_ask_user(fact: Fact) -> bool:
    return (
        fact.verification_status == UNVERIFIED
        and fact.content_role in (DECISION, RISK, COMMITMENT)
        and fact.confidence_score < 0.7
        and fact.source_type in (USER_VERBAL_FACT, LLM_EXTRACTED)
    )
```

### 规则 R4：朋友式澄清的对话生成(2026-05-22 修订)

**核心机制澄清(顾源源 2026-05-22 原话反推)**:

跟豆包对比的真正差异**不是问得有人情味**,是 **"AI 已经看过资料,把资料拼成连贯故事,发现哪里有缝隙,对着缝隙问"**。

```
                ┌──────────────────────────────────┐
                │ AI 已看过路径 1-3 全部资料         │
                │ → 在脑子里拼成"客户故事网"          │
                └──────────────┬───────────────────┘
                               ▼
              ┌──────────────────────────────────┐
              │ 扫描故事网完整度,找"缝隙"            │
              │ (逻辑断点 / 时间断点 / 关系断点)     │
              └──────────────┬───────────────────┘
                               ▼
              ┌──────────────────────────────────┐
              │ 缝隙队列(按优先级排序):              │
              │  - 高:影响决策的关键空白             │
              │  - 中:影响 narrative 的二级空白      │
              │  - 低:细节补全                       │
              └──────────────┬───────────────────┘
                               ▼
              ┌──────────────────────────────────┐
              │ 早晨通勤场景(顾源源原话):            │
              │ "公司老板在来公司路上打开手机跟 AI 聊"  │
              │ AI 趁用户聊到相关话题时,自然抛出缝隙   │
              │ 一个澄清完刷新下一个 (队列模式,非弹窗) │
              └──────────────────────────────────┘
```

**问法人情味是结果,不是设计的核心**。核心是"AI 真的看过资料,知道哪里不通"。

**实现伪代码**:

```python
def generate_friendly_clarification(
    client_fact_bundle: ClientFactBundle,
    user_recent_dialog_topic: str,
) -> ClarificationQuestion | None:
    # Step 1: AI 把客户全部 atomic_facts 拼成故事网
    story_network = build_story_network(client_fact_bundle)

    # Step 2: 扫描缝隙(三类断点)
    gaps = scan_gaps(story_network)
    #   logic_gap: 张真是法人?是理事长?数据互斥
    #   time_gap: 5/19 之后到现在,缺这段经历
    #   relation_gap: 笑雨老师跟张真什么关系?没有连接

    # Step 3: 找跟用户当前聊天话题相关的缝隙
    relevant_gap = match_gap_to_topic(gaps, user_recent_dialog_topic)
    if not relevant_gap:
        return None  # 没相关的就不问, 别打扰用户

    # Step 4: 用 LLM 生成融入对话流的问法
    # 关键: prompt 喂给 LLM 的是"用户刚说什么 + 这个缝隙的具体内容",
    # 让 LLM 自己织进对话, 不要套模板
    return llm_friendly_question(relevant_gap, user_dialog_history)
```

**对比例 (一定要保留这个例子)**:

| 模式 | 写法 | 用户感受 |
|---|---|---|
| ❌ 问卷式(豆包做不到的事) | "请确认张真的职位:A.法人代表 B.理事长" | 被审讯 |
| ❌ 表面"温和"但本质问卷 | "你能告诉我张真是法人还是理事长吗?😊" | 还是被审讯,只是加了 emoji |
| ✅ **朋友式(我们的核心创新)** | 用户聊到"5/19 跟张真定下来的事" → AI:「上次张真说要接的那个事,你们最后定下来是法人代表还是理事长?我看你 5/19 笔记里两个都提到了」 | AI 真的看过你的笔记,顺着你刚说的话题接住 |

差异:**前两种 AI 都不知道用户在聊什么,只是按字段缺失硬问**;后一种 AI 在故事网里发现缝隙,**等用户聊到相关话题才抛出**,并 **引用具体证据**(5/19 笔记)。

### 规则 R6：用户反馈用行为信号驱动,不加 👍/👎 按钮(2026-05-22 新增)

**核心认知(顾源源 2026-05-22 原话)**: AI 心理学告诉我们,经常称赞 AI 它就扩张,经常打击它就收缩。用户行为偏差导致大多数情况下不会主动点👍,只有不满才点👎。**给生成类 AI 加 👍/👎 按钮 = 误导 AI 觉得自己回答的多半不行**。

**替代方案:从用户自然行动里推断反馈,用户付出行动成本才产生信号**:

| 用户行为 | 信号 | 信号方向 | 信号质量 |
|---|---|---|---|
| 直接采纳 AI 抽取的事实,没编辑 | `accepted` | 隐式正向 | 中等 |
| 编辑了 AI 拟的草稿 | `edited` + `edit_distance` | 高质量负向 | 高(用户付出改动成本) |
| 撤销 AI 写入 / 删除 AI 建的任务 | `reverted` | 强负向 | 高 |
| 大量修改后才确认 | `heavy_edit` (edit_distance > 阈值) | 高质量负向 | 高 |
| 回答 AI 澄清问题后 → AI 抽出的事实跟用户回答一致 | `clarification_validated` | 高质量正向 | 极高 |
| 用户直接改写 AI 抽出的事实(前端编辑器) | `corrected` + `user_correction_text` | 极高质量负向 | 极高(带训练样本) |

**关键设计**:
- AI **不知道用户在评价它**,所以不会触发讨好/收缩
- 所有信号都需要用户**付出真实行动**(点按钮 0 成本会导致虚假信号)
- `corrected` 信号带具体改动文本 → 3.0 AI Memory 的最佳训练样本来源

**写入路径**: 前端 6 个埋点位置(编辑器 onChange / 撤销按钮 / 删除任务 / 采纳事实 / 澄清回答提交 / 草稿改动 diff 计算)统一写 `ai_feedback_signals` 表。

**N3 预留**: `ai_feedback_signals` 是 v2.2 必加表(只写不读),3.0 启动时已经有 N 个月的真实信号样本可训练。

### 规则 R7：AI 澄清是双层结构(2026-05-22 新增)

**核心认知(顾源源 2026-05-22 原话反推)**: AI 澄清不是单层,有两层语义:

| 层 | 类型 | 触发场景 | 写入表 |
|---|---|---|---|
| **第一层 · 具体事情澄清** | 对单条事实 | "张真到底是法人代表还是理事长?(我看 5/19 笔记两个都提到)" | `clarification_queue`(已有) |
| **第二层 · 流程改进建议** | 对工具本身 | "我经常遇到合同分类困难,后台能不能加个'服务类合同'标签?" | `ai_improvement_suggestions`(v2.2 新增占位) |

**两层有本质区别**:
- 第一层是**业务数据问题**,AI 通过问用户来填字段
- 第二层是**产品功能问题**,AI 通过反馈来推动产品演进

**实现差异**:

```python
def classify_clarification(gap: Gap) -> ClarificationKind:
    if gap.is_about_specific_fact():
        # 例: "张真职位不明" → 进 clarification_queue, 朋友式问用户
        return ClarificationKind.SPECIFIC_FACT
    elif gap.is_about_recurring_pattern():
        # 例: "多个客户都有'服务类合同'但没标签" → 进 ai_improvement_suggestions
        # 不主动 push 用户, 只在用户进入"设置 → AI 管理"模块时才呈现
        return ClarificationKind.PROCESS_IMPROVEMENT
```

**呈现策略**(顾源源 2026-05-22 决策):

- 第一层:朋友式聊天主动抛出(R4 机制)
- 第二层:**被动展示模式** — AI 不主动 push,只在用户进入"设置 → AI 管理"模块查看 list 时才呈现
- 理由:AI 不应过度自我表达,用户来问才说

**N3 预留**: `ai_improvement_suggestions` 表 v2.2 只 CREATE + 单向写入,3.0 时 AI actor 才会读取并主动跟人类同事协作执行改进。
