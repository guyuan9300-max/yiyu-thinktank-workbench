虽然 ClientFactView 能找到 5/19 张真会议任务, 但任务的 `description_preview` 是**空的**:

```
✓ 找到 5/19 任务: 和日慈张真进行5月份第一次战略对齐会
  status: done, progress: done
```

**含义**: 会议纪要 docx 在工作台资料库里、markdown 也提取了, 但**任务字段层完全不知道纪要里发生过什么**(张真接任法人 / 强哥任秘书长 / 兴盛+心理魔法学院合并 / 安心妈妈新项目 / 6 件待办)。

**这正印证了 v2.2 方向的正确性**:
- Phase 1 F1.2 解决"L2 共识层数据访问统一"(完成 ✅)
- Phase 2 F2.1 LLM ExtractionRunner 解决"docx 内容沉淀进字段"(待做)

Phase 1 完成后,机器人可以**告诉你**5/19 那场会议存在; Phase 2 完成后,机器人可以**复述**会议里发生了什么。

### 3.5 新发现的洞察 (校正北极星)

ClientFactView 完成后, 暴露出一个之前没看清楚的关键事实:

**"任务的 description 字段 ≠ 会议纪要内容"**:
- 5/19 张真对齐会任务: description 是空的, 字段不知道会议里发生过什么
- 但工作台资料库有 docx, _v2_meta/markdown 有提取
- atomic_facts 表有 197 条事实 (W1 规则正则抽出来的), 但很可能不包含会议关键决策

**这给 Phase 2 F2.1 LLM ExtractionRunner 提出了更明确的需求**:
- 输入: 工作台 docx → 提取的 markdown
- 输出: 写入 atomic_facts (复杂决策) + 更新 event_line.recent_decision 字段 + 关联到对应任务
- 验收: 5/19 那场会 docx 跑过 ExtractionRunner 后, atomic_facts 应该出现"张真任理事长"等关键事实

**北极星无需调整**, 但 Phase 2 的具体 spec (PHASE1_SPEC §4) 可以基于此发现进一步明确。
