## 一、核心变化: 从流程驱动到目标驱动

### 2.0 硬编码链路 (现状)
```
用户上传会议纪要 → 数据中心 → 固定抽取事实 → 固定触发表 → 页面固定展示 → 人类再判断下一步
```

→ **流程驱动**, 系统知道"该做哪些固定动作", 但不知道:
- 现在这个客户最缺什么资料
- 当前判断最不稳的地方在哪里
- 应该调用哪个模块补证据
- 哪些问题应该问用户

### 3.0 目标驱动链路 (新方向)
```
用户给目标 → AI 形成计划 → AI 调用模块 → 数据中心作为共享记忆 → 各模块作为工具
                                                                    ↓
                                                  人类处理关系/确认/授权/对外沟通
```

→ **目标驱动**, AI 团队在做一家公司的事情.

## 三、10 类必须预留的接口

### 1. 目标接口 (Goal API)

未来用户/外置 Agent 不发"操作指令", 而发"目标".

```json
{
  "goal": "帮我处理今天和日慈的会议纪要, 并判断下一步最该推进什么",
  "client_id": "rici",
  "project_id": "teacher_pilot",
  "constraints": {
    "do_not_send_external": true,
    "only_create_drafts": true,
    "require_human_approval": true
  },
  "success_criteria": [
    "形成会议摘要", "新增客户事实", "识别风险",
    "生成澄清问题", "生成任务草稿", "刷新故事卡"
  ]
}
```

### 2. 计划接口 (Plan API)

AI 收到目标后先生成操作计划 (不直接执行).

### 3. 工具注册接口 (Tool Registry)

所有功能模块注册成工具. 不让 AI 自己猜系统有什么能力.

每个工具登记: 名/模块/能做什么/输入/输出/只读否/写数据否/确认否/外置允许否/幂等否/回滚否/上下文.

### 4. ★ 数据缺口接口 (Data Gap API) — 最关键新增

未来不是"数据中心发现缺口 → 页面显示缺口"
而是"数据中心发现缺口 → AI 决定调用哪个工具补缺口".

```json
{
  "gap_type": "missing_external_evidence",
  "client_id": "xxx",
  "description": "客户内部资料显示项目预算已变更, 但外部公开口径仍可能沿用旧版本",
  "suggested_tools": ["intel.search", "workbench.ask_user", "clarification.create"],
  "priority": "high",
  "impact": "影响对外材料和客户沟通口径"
}
```

→ **数据中心从"展示层数据源" 变成"AI 调度的判断源"**.

### 5. 模块调用接口 (Domain Tool API)

每个模块暴露"业务级工具", 不是"按钮级接口".

| 模块 | 业务工具样例 |
|---|---|
| 任务与日程 | task.create_draft / commitment.create_candidate / weekly_review.generate / task.link_to_event_line |
| 工作台 | workbench.ingest_note / workbench.query_context / workbench.generate_material / workbench.correct_fact |
| 战略陪伴 | strategy.refresh / insight.generate / clarification.create / clarification.resolve |
| 资讯情报站 | intel.search / intel.score_relevance / external_evidence.write / intel.convert_to_task |
| 成长中心 | growth.capture_signal / growth.create_method_card / growth.apply_method_to_generation |
| 组织计划工坊 | plan.parse / plan.link_task / plan.detect_gap / plan.suggest_next_actions |

### 6. 角色接口 (Agent Role API)

不同 CEO skill **不是模仿语气**, 而是抽象成不同管理风格包.

| skill 类型 | 关注 | 默认追问 |
|---|---|---|
| 产品愿景型 CEO | 用户体验/产品叙事/极致简化/长期品牌/非连续创新 | "对用户真正什么价值? 有没有过度复杂? 是否形成长期壁垒?" |
| 运营效率型 CEO | 成本/速度/组织执行/指标拆解/复盘闭环 | "48 小时内闭环? 谁负责? 指标是什么?" |
| 生态协作型 CEO | 伙伴关系/资源配置/外部联盟/市场节奏 | "怎么撬动外部资源? 长期联盟?" |
| 风险控制型 COO | 权限/交付风险/流程稳定/客户敏感/承诺兑现 | "是否越权? 是否有客户证据? 是否需要人工确认?" |

→ **决策优先级、追问方式、风险偏好、组织节奏、输出标准的不同**, 而不是 prompt 套词.

### 7. AI 内部会议接口 (Agent Deliberation API)

未来战略陪伴不是一个 AI 给一个判断, 而是 **AI 团队内部讨论**:
- AI 数据员: 证据不足
- AI 战略员: 趋势成立
- AI 风险员: 这里有交付风险
- AI 撰稿员: 对外表达不能这么写
- AI CEO: 最后形成决策建议

预留: agent_discussions / agent_arguments / agent_votes / consensus_decisions / human_arbitration_required

### 8. 人类确认接口 (Approval API)

AI **可以做**: 候选事实/风险候选/任务草稿/澄清问题/刷新故事卡
AI **必须人确认**: 正式任务/覆盖权威事实/对外发送/飞书推送/客户阶段/删资料/关键澄清/跨客户引用

### 9. 审计接口 (Agent Run Log)

无论内置/外置, 都留完整轨迹: 谁发起 / driver / 目标 / 计划 / 工具调用 / 输入输出 / 写入 / 确认 / 失败 / 回滚 / 用户采纳

### 10. 评估接口 (Evaluation API)

不评估"AI 答得好不好", 评估"AI 是否推动公司工作往前走":
- 完成目标 / 用户可感知价值 / 减少用户工作量 / 生成有效澄清 / 形成可执行任务 / 提升故事卡完整度 / 让后续问答更准

## 四、4 大设计原则

### 原则一 · 每个功能必须同时支持人类入口 + AI 工具入口

```
人类按钮 → task service
AI 调用 → task service
飞书回调 → task service
CLI 指令 → task service
```

前端只是其中一个客户端.

### 原则二 · 不要硬编码流程, AI 通过目标选择模块组合

同样一段会议纪要:
- 目标=生成周复盘 → 调用任务/时间线/复盘生成器
- 目标=准备客户会议 → 调用战略陪伴/待澄清/人物关系
- 目标=检查风险 → 调用风险表/外部情报/历史承诺
- 目标=生成方案 → 调用工作台/方法卡/客户事实/撰稿员

预留: goal / context / constraints / plan / tool_calls / observations / result / next_actions

### 原则三 · 数据中心从"被动数据源"变"主动缺口发现器"

数据中心必须能输出"我知道什么 / 不知道什么 / 哪些冲突 / 哪些证据不够 / 哪些过期 / 哪些外部缺失 / 哪些问客户 / 哪些调外部 / 哪些让人确认". 这是 AI 团队工作的起点.

### 原则四 · CEO skill = 决策协议, 不是 prompt

skill 应该包含: 优先级 / 判断标准 / 提问习惯 / 任务分配方式 / 风险偏好 / 审批边界 / 汇报格式 / 复盘方式. 这才是 skill 的真正价值.

## 五、最小接口包 (倒推预留, 7 件)

1. **Agent Gateway** — 所有 AI 操作入口统一 (内置/外置/CLI/API/飞书机器人/MCP-like)
2. **Tool Registry** — 模块能力注册 + 11 字段元信息
3. **Goal / Plan / Run 三件套** (goal.create / plan.generate / run.execute / run.status / run.diff / run.rollback)
4. **Data Gap API** ★ — 数据中心主动告诉 AI 缺什么
5. **Approval Queue** — 危险动作进入确认
6. **Agent Run Log** — 所有 AI 行动可追溯
7. **Skill Manifest** — 每个 AI 角色/CEO 风格通过 skill manifest 定义, 不散落在 prompt

## 六、下一轮测试目标 (升级)

### 之前 (R1/R2 设计)
"会议纪要处理闭环" — AI 是否能不点界面跑完 6 段输出

### 新方向 (顾源源 5/23 进一步)
**"AI 是否能根据数据缺口主动调用其他模块补证据"** — 验证 AI 会不会**根据目的组合工具**.

测试场景:

用户输入:
> "客户说预算还没定, 学校配合度也不确定, 但希望下周二看到一版轻量方案."

AI 应做:
1. 写入会议纪要
2. 抽取事实
3. **发现缺口**: 预算未定 / 学校名单缺失 / 学校配合风险缺证据
4. **调用资讯情报站 / 文件检索**:
   - 查客户官网是否提到试点范围
   - 查历史文件是否已有预算口径
   - 查同类项目是否有学校配合风险案例
5. 仍不确定的 → 生成澄清问题
6. 生成任务草稿: 向客户确认预算/获取学校名单/起草轻量方案
7. 刷新客户故事卡
8. 给用户汇报: 我知道了什么 / 还缺什么 / 调用了哪些工具 / 哪些需要你确认

## 七、量化指标 (升级 100 分制 → "目标调度能力" 为主)

### AI 目标调度能力

| 指标 | 目标 |
|---|---|
| 生成 goal → plan → tool call → result 全链路 | 100% |
| 至少调用 4 个模块 | ≥ 4 |
| 根据数据缺口调用非固定模块 | ≥ 1 次 |
| 明确哪些信息仍缺失 | ≥ 3 条 |
| 生成可执行任务草稿 | ≥ 2 条 |
| 生成澄清问题 | ≥ 2 条 |
| 保留完整 Agent Run Log | 100% |

### 数据缺口补证能力

| 指标 | 目标 |
|---|---|
| 数据中心识别缺口 | ≥ 3 条 |
| 调用资讯情报站或文件检索补证 | ≥ 1 次 |
| 外部证据写入 external_evidence_cards | ≥ 1 条 |
| 仍无法确认信息进入 clarification_records | 100% |
| 不把外部低可信信息直接写成客户权威事实 | 100% |

### 人类可感知价值

| 指标 | 目标 |
|---|---|
| 用户能看懂"这次会议改变了什么" | ≥ 1 段摘要 |
| 用户能看到"下一步该做什么" | ≥ 2 条任务草稿 |
| 用户能看到"哪些要问客户" | ≥ 2 条澄清问题 |
| 用户能看到"AI 调用了哪些工具" | 100% |
| 用户能决定"采纳/修正/忽略" | 100% |

## 八、最终判断 (顾源源 5/23 钦定)

益语 3.0 **不是**: AI 帮人用软件
益语 3.0 **是**: AI 团队用软件做人类团队过去大量后台工作, 人类负责关系/价值判断/最终确认

这套思路真正打开的是益语平台的长期空间:
- **把功能模块变成 AI 团队的工具**
- **把数据中心变成 AI 团队的共享记忆**
- **把人类团队变成关系、判断和确认的高价值角色**

要倒推预留的, 不是某一个功能, 而是整套底层协议 (10 类接口).

只要这些接口从现在开始预留, 未来无论是内置模型、外置 Codex/Claude, 还是不同 CEO skill, 都能通过同一套底座调度益语平台.
