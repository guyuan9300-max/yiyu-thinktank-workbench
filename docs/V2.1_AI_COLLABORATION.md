## 1 · 三个北极星 (N1 / N2 / N3) — 唯一判分标尺

**任何工作做之前问一遍**:"这件事直接服务 N1 / N2 / N3 哪个?如果都不是 → 不要做"。

### N1 · v2.2 做完后所有功能顺畅运行

**完成定义**:整个系统支持现在软件每一个板块的功能完整运作,**不是补丁接通,是通过完善整个功能架构来接通**。
- 能复用统一通道就复用统一通道(不是每功能各自写信息通道/计算通道)
- 所有功能在一个干净的架构上跑

**反例(不算做完)**:
- 通过 monkey patch 让 view 能跑 → 补丁
- 各功能走各自数据通道 → 多池并列
- 测试过但用户卡顿 / 数据不一致 → 表面完成

### N2 · 手机 AI 自由对话成为数据中心 4 主路径之一,承载软件逻辑灵魂

**4 主路径**:
| 路径 | 来源 | 信息源置信度 |
|---|---|---|
| 1 | 工作台进入的文件 | 高(用户主动上传 + 官方文档) |
| 2 | 任务/计划/复盘/协作信息 | 高(经验性,可指导下一次行动) |
| 3 | 互联网爬虫补全 | 分层(官网高 / 公众号低) |
| 4 | 手机 AI 朋友式聊天澄清 | 高(用户口述记忆,需结构化) |

**数据中心不是简单存这 4 类信息,是要理解每类信息的来源意味着什么**(详见 `docs/V2.2_INFORMATION_SOURCE_METADATA.md` 5 维元数据规范)。

**手机 AI 朋友式聊天设计原则**:
- 不能生硬地问 "这是 300 还是 3000?" — 让用户懵
- 应该像朋友聊天一样,把生硬数据澄清融入闲聊
- AI **已经看过资料**,才反问 — 跟豆包"先要用户教背景"的本质差异

### N3 · 为 3.0 埋好接入基础,3.0 升级时 2.0 不需要返工任何板块

**3.0 是什么**:给大模型配共享办公室,AI 跟人共享同一工作空间,AI 作为 actor 在软件里建任务、发协作信息、引用文件、自动校验、跨任务规划,**越用越聪明(具备长期记忆)**。

**v2.2 阶段必须落地**(7 项预留):
- A1: `actor_type` + `actor_id` 字段 (区分 human/ai_agent/system)
- A2: `event_log` 总线表持久化
- A3: `reasoning_traces` provenance 链
- A4: `verification_status` enum
- A5: AI Memory 6 张表占位 (`ai_episode_log` 真写入 + 其他 5 张占位)
- A6: `idempotency_keys` 表 + Store 工具
- A7: `superseded_by_id` 信息更新链路
- B: 命名锁定 (content/body 统一,source_type 枚举锁死)

**坚决不在 v2.2 做的事 (留 3.0)**:
- ❌ agent framework (LangChain/AutoGen)
- ❌ AI memory 读取逻辑 (只写不读)
- ❌ prompt template store
- ❌ AI actor 权限边界设计

## 2 · 三道校验门 (A / B / C) — 每个里程碑结束必跑

**强制规则**:每个里程碑做完,**必须反过来用三道门测**,**全过才能进下一里程碑**。任何一道 FAIL = 当前里程碑未完成。

### 校验门 A:现有功能回归 (对应 N1)

| 步骤 | 做什么 | 通过标准 |
|---|---|---|
| A1 | 跑 `scripts/run_w*_acceptance.py` + v2.2 新建的 acceptance | 所有维度 PASS (SKIPPED 需说明原因) |
| A2 | 手动启动桌面,过 `V2.2_FEATURE_INVENTORY.md` 9 大模块 | 每个核心动作都能完成 |
| A3 | 启动 mobile,过 4 tab | 任务/日历/咨询/我的都能正常用 |
| A4 | 多账号回归 | 数据不被互相覆盖(v1.0 frozen bug 不复现) |

**A 门 FAIL**:回滚到上一个 commit,不许进下一里程碑。

### 校验门 B:机器人问答能力进阶 + 4 主路径接通 (对应 N2)

每个里程碑应让机器人**多答对一组问题** + **4 路径通畅度增进**。用 5/19 张真会议作为金标准:

| 里程碑 | 机器人新增能答 | 4 路径状态 |
|---|---|---|
| M1 (ClientFactView) | "日慈基金会现在有哪些事件线?任务进度?" | 路径 1+2 读取走 ClientFactBundle |
| M2 (ExtractionRunner) | "5/19 张真那场会有什么决定?" | 路径 1+2+3 写入走 IngestPipeline |
| M3 (clarification 闭环) | "刚才 6 件待办里顾源源要做的有几件?" | 路径 4 雏形 |
| M4 (NarrativeKernel 事件驱动) | "兴盛+心理魔法学院合并后客户战略状态变了吗?" | 4 路径变更触发 narrative 自动刷新 |
| M5 (手机端 L2 接通 + 朋友式) | 在手机上问同样问题得到一致答案 | 路径 4 完整 |

**B 门 FAIL**:代码可保留,但记一笔"目标 B 进度未推进",下一里程碑必须补。**连续 2 个里程碑 B 门不动 = 重新评估方向**。

### 校验门 C:3.0 接入预留 (对应 N3)

| 阶段 | 必须完成的 3.0 预留 |
|---|---|
| Phase 1 收尾 | A1 actor_type / A2 event_log / A4 verification_status / A7 superseded_by_id |
| Phase 2 启动前 | A3 reasoning_trace / A5 AI Memory 6 表 |
| Phase 2 任意 | A6 idempotency_keys / B 命名锁定 |
| Phase 3 收尾 | 3.0 接入烟测:模拟 AI actor 写 task + event_log + 学习 rule,全链路通 |

**C 门 FAIL**:3.0 预留缺失 → 当前里程碑不算完成(即使 A/B 都 PASS)。

## 3 · 五大原则 (顾源源 5/22 定义,违反 = 错误工作)

### 原则 1:自主边界 70%

| 情况 | 处理 |
|---|---|
| 整体方向把握 ≥ 70% + 当前 feature 把握 ≥ 70% | 自动执行,不询问 |
| 整体把握 < 70% **且** 不澄清完不成北极星 | 停下来报告,等顾源源决策 |
| 命名/技术选型抉择,影响后续多个模块 | 自己拍板继续,**在 milestone 报告里 highlight** |
| 跑出来的验证结果偏离北极星 | 停下来报告,重新校准 |

**判断 70%**:
- 能说清楚这件事服务哪个北极星
- 能预测做完后客户感知变化
- 有具体实现路径,不是"试试看"

### 原则 2:给提案要含**问题/方式/成本/风险**

禁止 "A 还是 B" / "宽还是窄" 这种菜单式甩锅。**先讲判断,再求确认**。
- 要说:这件事是什么问题 / 怎么做 / 成本多少 / 风险在哪 / 我建议 X 因为 Y
- 不要说:你想 A 还是 B?

### 原则 3:**通识不抽,客户特定才抽**

LLM 自带通识 (常识/行业知识/公共认知),不需要再存。
- ❌ "公益基金会需要捐赠" → 通识,不抽
- ✅ "日慈的主要捐赠来自腾讯基金会" → 客户特定,抽

判断标准:**对同类客户都成立 → 不抽 / 只对这家客户成立 → 抽**。

### 原则 4:**信息商 (顾源源 5/22 洞察)**

不是"哪个信息对",是"信息有没有被更新"。AI 必须能分辨:

| 语义 | 判定 | AI 行为 |
|---|---|---|
| 真冲突 (A 说 X, B 说 Y, 无法判断谁对) | conflict | 进澄清队列问用户 |
| **信息更新** (用户说"重签/改成/不对了") | supersedes | 旧版标 `superseded_by_id`,新版 current |
| 互补 (不同时间锚 / 不同切面) | complement | 都保留,按 time_anchor 排序 |
| 重复值 | none | 不重复写 |

**关键词触发"更新"**:重签/改为/改成/不对/弄错/搞错/更新/应该是/其实是/纠正/修正/替换/失效/作废

### 原则 5:**渠道驱动 content_role 判断**(顾源源 5/22 渠道分层)

| 来源 | 默认 content_role | 默认置信度 |
|---|---|---|
| 已签合同 / 章程 | `fact` | 0.95 |
| 会议纪要决议段 | `decision` | 0.90 |
| 个人复盘/描述 | `lesson` / `observation` | 0.85 |
| 当事人原话 | `quote` (必带 speaker_person_id) | — |
| 用户主观判断 | `observation` | 0.65 |
| 互联网公众号 | `speculation` | 0.30 |

LLM 抽取时**优先用 source_type 兜底,语义判断作为二次精化**。

## 4 · 职责分工 (基于 5/22 自然形成的协作模式)

| 责任区 | 主线 AI (代号 A) | 跟进 AI (代号 B) |
|---|---|---|
| backend schema 新增 | ★ A 主导 | B 补丁修 bug |
| 工具服务层 (Repository / Store) | ★ A 主导 | B 加单测 |
| 新功能 (F2.1 LLM extractor 等) | ★ A 主导 | — |
| 跟顾源源 prompt 设计 + 业务对齐 | ★ A 主导 | — |
| frontend view 接入 | — | ★ B 主导 |
| endpoint patch (POST/PATCH 接入 idempotency 等) | — | ★ B 主导 |
| 文档完善 (从 commit message 提炼成文档) | — | ★ B 主导 |
| 测试补充 (覆盖率 + 边界场景) | A 写主测试 | ★ B 加额外测试 |
| 修对方引入的 bug | 互相 | 互相 |
| 工程基础设施 (linter / 构建脚本) | — | ★ B 主导 |

- `9415c1f` B 给 A 的 IdempotencyStore 加了 17 额外测试

## 7 · 每个里程碑结束必做 (强制 — 顾源源 5/22 原则)

里程碑做完后,**反过来用三道门测**,**通过了才能继续下一个**:

```
1. 跑校验门 A (acceptance + 手动 9 模块)
   ├─ FAIL → 回滚 + 修复, 不进下一里程碑
   └─ PASS → 继续

2. 跑校验门 B (机器人能答的新问题 + 4 路径状态)
   ├─ FAIL → 记笔"未推进", 下一里程碑必须补
   ├─ 连续 2 个 FAIL → 重新评估方向, 暂停推进
   └─ PASS → 继续

3. 跑校验门 C (N3 接入预留)
   └─ 当前阶段对应预留缺失 → 不算完成 (即使 A/B PASS)
```

**输出 milestone audit 文档** (`docs/MILESTONE_*_NORTH_STAR_AUDIT.md`),对照 3 个北极星 + 整体规划位置评估,不只看代码。

## 9 · 失败模式预警 — 看到立即停手对齐

| 失败模式 | 症状 | 应对 |
|---|---|---|
| **代码漂亮陷阱** | 在算"切了多少 SQL"、"边界 0 违规" | 停。回到 N1/N2/N3,看客户能不能感知 |
| **完美主义陷阱** | "全做完才能发 release" | 停。里程碑必须可独立 release |
| **业务并行污染** | 在 V2.1 写业务功能 (例如 brand_strategy) | 停。业务回 main,架构留 V2.1 |
| **AI session 漂移** | 新 session 没读本文件 + NORTH_STAR | 强制重读,否则不许动代码 |
| **B 门连续不动** | 2 个里程碑机器人答题没进步 | 重新评估,可能换方向 |
| **N3 预留被跳过** | 想"先做功能,3.0 留最后" | 停。Phase 2 启动后再加 actor_type 要 migrate 几万行 |
| **补丁接通** | "先 monkey patch 让 view 能跑" | 停。补丁不算完成,必须走干净架构 |
| **没查并行 AI commit** | 不知道对方做了什么直接干 | 强制跑 §6 第 5 步 |

## 10 · 不要污染原仓库 — 顾源源 5/22 红线

原仓库 (`yiyu-thinktank-workbench/` 或 rename 后的 `main/`) 是顾源源的真实生产仓库,**任何 V2.1 工作禁止流入原仓库**。

**禁止的事**:
- ❌ 在原仓库目录 `cd`
- ❌ 给本仓库加 remote 指向原仓库
- ❌ `git push` 到任何指向原仓库的 ref
- ❌ 在原仓库 clone 出来的 worktree 里干 V2.1 的事

**允许的事**:
- ✅ 把原仓库的代码作为参考 read (不写回)
- ✅ 跟顾源源讨论原仓库的事(但代码改动他自己同步)
