# 日历与任务判断系统落地方案 V3.1 修订清单

## 0. 文档定位

这份 V3.1 不重写 V3 架构，也不另起一套系统。

它只做一件事：

> 在 V3 的骨架上，把“发布规则、个人边界、证据范围、回填入口、交叉分析链、管理视角”这些工程边界钉死，让系统真正能产出可指导管理的判断。

V3.1 的目标不是“更会写”，而是让任务、事件线、周判断真的能回答：

- 这周到底推进了什么
- 当前真正卡在哪里
- 为什么会卡
- 下周最该压哪条线
- 哪些风险、机会已经形成连续信号

---

## 1. 本次新增拍板结论

### 1.1 共享判断发布机制

共享判断采用：

- **人工发布为主**
- **指定机器人可发布**

一期不做任何“所有判断默认自动共享”的逻辑。

允许自动发布的机器人必须满足：

- 拥有明确的发布权限
- 只对已达到阈值的判断执行发布
- 发布结果带完整版本和来源元数据

### 1.2 个人任务边界

个人任务采用：

- `PERSONAL_ONLY`

并且与组织任务采用两个完全隔离的沙箱容器。

明确规则：

- 个人任务不能挂客户
- 个人任务不能挂项目
- 个人任务不能挂事件线
- 个人任务不能挂模块/流程
- 个人任务不进入组织、部门、机构复盘
- 个人任务附件不进入客户/项目总文件库
- 个人任务只参与个人周判断和个人趋势

一期不允许例外迁移。

### 1.3 附件摘要范围

附件/文档/会议证据在一期只做到：

- `title`
- `summary`

不做全文深推理，不做长段落语义抽取，不做复杂知识图谱。

### 1.4 旧任务回填入口

旧任务归属回填的人工确认入口优先放在：

- **任务详情页**

不先做独立批量回填中心，也不先把事件线页做成主确认入口。

---

## 2. V3.1 仍然沿用的 V3 主骨架

以下内容保持不变：

- Task 主表上的正式归属字段继续作为唯一已确认真源
- local / cloud 的语义分工不变
- 共享判断必须带版本、指纹、失效状态
- 低完整度时必须进入正式降级模式
- 周判断第一页固定为四层 dashboard
- 角色化 dashboard 继续沿 CEO / 部门负责人 / 成员 三套视角推进

V3.1 不改变骨架，只强化执行规则和链路优先级。

---

## 3. V3.1 最核心的调整：把系统做成“交叉分析链”

### 3.1 不是任务摘要链，而是交叉分析链

后续所有判断都要围绕这条链：

`任务事实 / 会议 / 附件摘要 / 澄清池 / 客户业务笔记 / 事件线记忆 -> EventLineContextBundle -> EventLineJudgmentRecord -> Dashboard 卡片 / 任务详情判断 / 动作卡`

重点不是多加字段，而是统一判断入口。

### 3.2 统一背景包 `EventLineContextBundle`

后续任务详情 AI 面板、周判断、dashboard 卡片，统一使用同一个背景包。

背景包至少包含：

- 事件线基础信息
  - `id`
  - `name`
  - `businessCategory`
  - `stage`
  - `summary`
  - `intent`
- 当前推进信息
  - `currentWork`
  - `currentBlocker`
  - `nextStep`
  - `recentDecision`
  - `recentProgress`
- 项目与组织笔记
  - `organizationIntro`
  - `collaborationRelationship`
  - `currentChallengeSummary`
  - `currentGoalSummary`
  - `recentFacts`
- 关键对象信息
  - `keyPeople`
  - `keyProducts`
  - `relationshipSummary`
- 本周任务事实
  - 任务列表
  - 状态变化
  - 改期
  - 待复核
  - 支持请求
- 会议摘要
  - 最近会议标题
  - 最近会议摘要
  - 决策摘要
  - 行动项摘要
- 附件摘要
  - 附件标题
  - 附件摘要
  - 关联目标
- 澄清池
  - 已澄清问题
  - 已确认答案
  - 最近使用时间
- 趋势信号
  - 连续改期
  - 连续待复核
  - 连续支持请求
  - 事件线无收束动作
  - 证据不足但持续产出判断

### 3.3 统一判断对象 `EventLineJudgmentRecord`

所有输出，先生成结构化判断对象，再投影到不同页面。

统一字段：

- `what_happened`
- `why_it_matters`
- `core_blocker`
- `blocker_type`
- `evidence_summary`
- `manager_implication`
- `next_week_focus`
- `minimum_action`
- `risk_if_ignored`
- `opportunity_if_amplified`
- `evidence_refs`
- `target`
- `coverageScore`
- `confidenceScore`
- `safeOutputMode`

这一步很关键：

- 不允许前端再自己拼一套启发式判断
- 不允许任务详情和周判断吃不同链

## 4. V3.1 的 5 条硬规则

### 4.1 风险只能写“业务风险”，不能写“系统结构风险”

面对用户输出时，只允许这些阻碍类型：

- `business_blocker`
- `coordination_blocker`
- `decision_blocker`
- `resource_blocker`
- `evidence_blocker`

以下内容不允许直接写成业务阻碍：

- 模块没挂
- 流程没挂
- 结构不完整
- 任务没归属

这些内容只能进入：

- `missingReasons`
- `coverageScore` 扣分项
- 内部调试提示

### 4.2 共享项目背景永远不能覆盖任务和事件线事实

优先级固定：

1. 手工澄清 / 已确认输入
2. 事件线维护背景
3. 会议摘要 / 附件摘要 / 支持请求证据
4. 任务自身语义
5. 项目共享摘要兜底

也就是说：

- `ProjectBriefSnapshot` 只能兜底
- 不能再把同一项目下所有任务洗成一套话术

### 4.3 证据不足必须显式降级，不允许硬写

当判断证据不够时，系统只能输出：

- 当前已确认事实
- 当前缺什么
- 为什么现在不能下更强判断
- 需要补什么后才能继续判断

不允许输出：

- 资料不足但继续写业务风险
- 方向不清但继续写管理建议
- 空泛的“系统化/协同效率”判断

### 4.4 旧任务回填要保守，不自动补脏

自动回填只允许在高置信度时自动落：

- `clientId`
- 必要时 `sourceType/sourceId`

以下字段在没有确定性证据时，只生成建议，不自动写死：

- `eventLineId`
- `projectModuleId`
- `projectFlowId`

无把握时直接标记：

- `contextStatus = unresolved`

### 4.5 共享判断与本地预览必须清楚区分

一期界面上必须明确区分：

- `本地预览`
- `待发布`
- `已人工发布`
- `已机器人发布`
- `已过期`

不允许让用户把本地 preview 误认为组织共享判断。

## 5. Dashboard 第一屏怎么真正变成管理驾驶舱

### 5.1 固定四层，不再回到摘要页

第一页固定：

- `本周关键事件线`
- `本周最值得关注的风险`
- `本周最值得放大的机会`
- `本周建议动作`

### 5.2 每张卡必须是结构化卡片

统一结构：

- `title`
- `statement`
- `chips`
- `targetType`
- `targetId`
- `targetFilters`
- `evidenceRefs`
- `coverageScore`
- `confidenceScore`
- `staleState`

### 5.3 每张卡必须能 drill-down

默认 drill-down 目标：

- 关键事件线卡 -> 事件线详情
- 风险卡 -> 该线的阻碍证据 + 近两周趋势
- 机会卡 -> 该线的推进证据 + 可放大条件
- 动作卡 -> 对应任务 / 会议 / 支持请求 + 回写结果

### 5.4 角色排序规则

#### CEO

排序固定：

1. 关键事件线
2. 机构级风险
3. 值得加码的机会
4. 管理动作

只允许出现：

- 战略对齐
- 关键项目推进
- 组织级资源 / 决策 / 协同阻碍
- 机构级动作

禁止出现：

- 个人-部门对齐
- 纯执行层动作罗列

#### 部门负责人

排序固定：

1. 本部门关键推进线
2. 协同/容量/流程风险
3. 可复制势能
4. 干预动作

#### 成员

排序固定：

1. 本周推进线
2. 当前最卡点
3. 可延续势能
4. 下周最小动作

## 6. 个人任务双沙箱的具体落地

### 6.1 任务对象新增或明确字段

- `scopeMode = COLLAB_SHARED | PERSONAL_ONLY`

### 6.2 PERSONAL_ONLY 的强规则

当 `scopeMode = PERSONAL_ONLY` 时：

- UI 隐藏客户/项目/事件线挂接
- API 拒绝写入 `clientId / eventLineId / module / flow`
- 周判断只进入个人视图
- 不进入部门/机构 dashboard
- 不进入项目总文件库
- 不进入组织业务笔记
- 不参与组织趋势信号

### 6.3 不做例外迁移

一期明确：

- 个人任务就是个人任务
- 不允许“先私人，后组织”的隐式升格
- 如需进入组织链路，必须新建正式工作任务

---

## 7. 附件摘要一期的最小实现

### 7.1 一期只做

- `title`
- `summary`

### 7.2 附件摘要的产品状态必须可见

必须有：

- `digest_pending`
- `digest_ready`
- `digest_failed`
- `unsupported`

### 7.3 附件摘要的系统作用

附件摘要进入：

- 任务详情 context preview
- 事件线背景包
- dashboard drill-down
- 证据引用计数

但不直接做全文深推理。

---

## 8. 旧任务回填在任务详情页怎么呈现

### 8.1 保留并升级“待补归属建议”

在任务详情页里继续保留这组能力：

- 建议项目
- 建议事件线
- 建议模块
- 建议流程

### 8.2 交互目标

这块不是让用户补长背景，而是让用户确认：

> 这条任务到底属于哪里

### 8.3 UI 文案原则

避免系统味太重的表述，比如：

- `待补归属建议`

改成更自然的：

- `请确认这条任务属于哪里`
- `补上归属后，后面的判断才会更准确`

## 9. V3.1 的真正管理价值目标

不是让 AI 多写一点，而是让系统能交叉分析并回答：

### 机构层

- 本周真正推进了哪些关键线
- 哪些风险已经形成组织级趋势
- 哪些机会值得继续加码
- 下周最该投入资源和注意力的是哪几条线

### 部门层

- 本部门的重点是不是在推进
- 哪条线在占带宽
- 卡点是业务、协同还是审批链
- 该停什么、压什么、收什么

### 个人层

- 本周自己真实推进了什么
- 当前最关键阻碍是什么
- 下周最小动作是什么
- 需要谁支持

这三层输出都必须来自同一套：

- 背景包
- 判断对象
- 卡片结构

只调整排序和权重，不造三套平行系统。

## 10. 实施顺序（在 V3 基础上的 V3.1 细化）

### 第一阶段：先把边界钉死

- 共享判断发布规则
- 指定机器人发布规则
- 个人任务双沙箱规则
- 附件摘要范围
- 旧任务回填规则
- 风险类型规则
- 结构问题不冒充业务问题

### 第二阶段：统一背景包

- `EventLineContextBundle`
- 接入客户业务笔记
- 接入附件摘要
- 接入会议摘要
- 接入澄清池
- 接入趋势信号

### 第三阶段：统一判断对象

- `EventLineJudgmentRecord`
- 先落在任务详情页
- 再接周判断第一页

### 第四阶段：dashboard 驾驶舱

- 四类卡片
- drill-down
- CEO / 部门负责人 / 成员三套排序逻辑

### 第五阶段：旧任务回填与业务笔记生长

- 任务详情页人工确认
- 客户工作台业务笔记持续长出来
- 让后续判断越来越准

---

## 11. 本期明确不做

为了避免范围失控，一期明确不做：

- 全文级附件/文档深推理
- 复杂知识图谱
- 个人任务向组织任务例外迁移
- 独立重型事件线中心
- 中长期泛预测
- 三套完全不同的 dashboard 数据结构

---

## 12. V3.1 的一句话结论

V3.1 不是发明新架构，而是在 V3 上把下面这些关键工程边界写死：

- 谁发布共享判断
- 个人任务怎么彻底隔离
- 附件证据这期做到哪
- 旧任务归属怎么保守回填
- 风险如何只写业务风险
- 背景包和判断对象如何统一

这样最后得到的不是“更长的总结”，而是：

> 一套能够交叉分析任务、事件线、证据和背景，并真正指导管理动作的日历与任务判断系统。
