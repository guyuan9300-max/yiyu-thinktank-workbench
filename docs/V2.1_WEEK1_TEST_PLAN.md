## 维度详解

### 1-8. 已有测试覆盖

这 8 个维度都对应已经写好的测试文件 / 脚本,Week 1 commit `de1bfb8` 里:

- W1-1 → 11 tests
- W1-2 → 8 tests(含 sync 失败降级、deleted scope)
- W1-3 → 17 tests(directory getter 容错 / frozen dataclass)
- W1-6 → 12 tests(6 个 view 过滤逻辑)
- W1-4a → 3 tests(API endpoint)
- W1-5 → 6 tests + lint 真实跑
- 不破坏 v1.0 → golden diff 跑(已经 100%)
- 真实 e2e → 启动 backend + 真火山云 sync + profile 验证

共 **58 个 pytest** + **1 lint 实跑** + **1 golden diff 实跑** + **1 真实生产 e2e**。

### 9. 模块边界(新增)

**为什么需要**:v2.1 铁律之一是 "mirror_* 表只能被 organization 模块写,业务代码必须走 Repository"。如果有人偷偷直接 `SELECT * FROM mirror_users` 写在业务代码里,SSOT 就被破坏。

**检查**:grep 整个仓库,看哪些非 organization 模块文件直接引用了 `mirror_organizations` / `mirror_departments` / `mirror_users` / `mirror_client_related_users`。

**允许的位置**:
- `backend/app/modules/organization/*`(模块自己,合法)
- `tests/test_organization_*` / `backend/tests/test_local_organization_*`(测试)
- `scripts/run_w1_acceptance.py`(本验收脚本)
- 注释 / 文档

**期望**:0 个非法引用。

### 10. 启动健康(新增)

**为什么需要**:这次升级了 `BACKEND_SCHEMA_VERSION 20260520 → 20260522`,需要验证:
- 空 db 启动:能直接建出 4 张 mirror 表 + 6 个 view
- **已有 v1.0 db 启动**(关键):跑 migration 时不会破坏现有 schema

**检查**:
- 用 `create_app(empty_dir)` 启动一次 → 跑 query 看 mirror 表 + views 都在
- 拷贝真实生产 db 到 tmp,用 `create_app(tmp)` 启动 → 验证现有数据没掉 + mirror 表新建出来

**期望**:两个都不报错,且现有业务数据完好。

### 11. 性能基线(新增)

**为什么需要**:sync 是用户会触发的操作,要确保不卡顿。

**检查**:
- 真实生产 sync 一次 → 计时(应该 < 5s)
- `/api/v1/local/organization/profile` 调 100 次 → 平均响应时间(应该 < 100ms)

**期望**:在合理范围,作为后续优化的基线。

### 12. 数据真实性比对(新增)

**为什么需要**:前面 e2e 只验证了"sync 后 3 部门 4 用户在",没验证"sync 后的数据跟用户实际看到的(cloud_session_user)是否一致"。

**检查**:
- 从 prod db 读 `cloud_session_user` JSON
- sync 后从 mirror_users 读同一个 user_guyuan
- 比对:`departmentId`/`departmentName`(应该一致)/`isDepartmentLead`(应该一致)/`fullName`(应该一致)

**期望**:0 不一致(本地缓存跟权威源对得上)。
