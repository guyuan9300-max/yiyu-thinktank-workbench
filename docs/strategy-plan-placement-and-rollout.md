# 战略计划放置层级与分阶段落地方案

> 对齐说明（2026-03-21）：
> 本文档的放置判断继续有效，但已被纳入更完整的 AI 背景优先框架。
> 现在统一按：
> `P1 组织启动中心承接低频根语境`
> `P2 任务与日历承接高频浅入口`
> `P3 洞察、成长与沉淀联动`
> 来执行，详见 [AI 背景优先总规划](./ai-context-first-rollout-plan.md)
> 与 [全局模块改造清单与阶段路线图](./global-module-refactor-roadmap.md)。

## 这次要解决的问题

当前已经明确：

- 年度战略
- 机构季度计划
- 部门季度计划
- 部门月度计划

这四层都需要存在。

但新的问题不是“要不要有”，而是：

1. 它们应该放在哪个模块里才不显得过深
2. 用户第一次进入任务与日历时，怎样被自然引导去补这些前置信息
3. P1 先落哪些非任务模块
4. P2 再怎样让任务与日历真正吃进这套结构

## 一、核心判断

### 1. 年度战略 / 机构季度计划

**不应该放在客户工作台。**

原因：

- 这是组织内的经营与执行结构，不是单一客户项目语境
- 客户工作台是项目/客户上下文，不是组织经营母盘

**也不能只埋在系统设置深处。**

原因：

- 如果它只是深设置项，用户会认为那是“配置”，不是“工作中的长期导航”
- 一旦用户感知不到这层结构，任务和成长系统就不会被理解为与它相关

### 2. 部门季度计划

**应该以“组织搭建中心 / 组织经营面板”的方式承载。**

理由：

- 它跟部门职责、负责人、岗位模板、流程模板是同一层母结构
- 这层是“组织运行框架”，不是具体任务页里的临时内容

### 3. 部门月度计划

**最终应该放在任务与日历里。**

理由：

- 月度计划已经接近执行层
- 它要直接转化成任务、会议、日程、周计划
- 它和时间、节奏、排期天然有关

但：

**P1 不应该先把它硬塞进任务页。**

原因：

- 任务与日历现在仍是活跃施工区
- 如果现在就把月度计划主入口压进任务页，容易让任务页承担过多上游结构
- 更稳的是先把“上游计划骨架”落好，再让任务页在 P2 正式接入

## 二、推荐放置方案

## P1：先落“组织经营层”

建议新增一个明确的概念：

**组织经营面板**

它不是新的独立大模块，可以先作为：

- `系统设置 > 组织搭建中心` 的升级态
- 或者更准确一点：`组织搭建中心` 里新增“战略与计划”分区

P1 在这里放：

1. 年度战略
2. 机构季度计划
3. 部门季度计划

也就是说：

- **设置页负责结构定义**
- **任务页负责执行承接**

### P1 页面分区建议

在组织搭建中心里新增 3 个板块：

1. **机构年度战略**
   - 年度目标
   - 年度叙事
   - 成功定义
   - 年度关键指标

2. **当前季度机构计划**
   - 季度主题
   - 季度目标
   - 成功标准
   - 关键押注
   - 主要风险

3. **部门季度计划**
   - 每个部门的季度目标
   - 关键产出
   - 成功标准
   - 关键动作
   - 主要依赖和风险

这样做的好处：

- 不会把战略结构藏到任务页里
- 也不会把它做成“只会配一次的后台设置”
- 用户能在组织中心里看见“年度 -> 季度 -> 部门”的完整分解链

## P2：任务与日历正式承接“部门月度计划”

任务与日历要承担的是：

1. 部门月度计划
2. 周计划
3. 任务拆解
4. 日程排布

所以 P2 时放进任务页的是：

- **部门月度计划**
- 不是年度战略，也不是部门季度计划主体

## 三、任务与日历里应该放在哪一层

我建议不要把“部门月度计划”塞进“总结”里。

### 不能只放在总结里的原因

- 总结是事后动作
- 月度计划是事前和事中动作
- 如果等用户写总结时才提示“你前面最好先有月度计划”，太晚了

所以更合适的做法是：

### 在任务与日历首页增加一个“计划层入口”

位置建议：

- 在任务主看板顶部
- 在日历主视图顶部
- 作为一个稳定存在的横向计划条

建议结构：

1. 当前月部门计划状态
2. 当前周是否有承接
3. 是否已经生成任务包

入口文案可以是：

- “本月计划未建立”
- “本月计划已建立，待拆成任务”
- “本月计划已承接到本周”

这样它就是执行层入口，而不是总结层附属物。

## 四、第一次进入任务与日历时怎么引导

你这个判断是对的：

**不能等用户写总结时才说“前面有这个会更好”。**

所以任务页第一次进入时，需要一个明确的前置引导。

### 触发条件

当用户第一次进入任务与日历，并且：

- 已有组织战略 / 季度计划
- 但当前部门没有月度计划

就弹一个轻提示卡，不是报错，不是模态强拦。

### 提示文案建议

第一句：

- “这个月还没有部门计划，任务建议会偏泛。”

第二句：

- “先补本月重点后，系统才能更准确地帮你排任务、排会议和做成长判断。”

按钮：

- `去补本月计划`
- `稍后再说`

### 如果连上游季度计划都没有

那就不要直接让他填月度计划，而是提示：

- “当前部门还没有季度计划，请先去组织搭建中心补季度目标。”

按钮：

- `去补部门季度计划`

也就是说：

任务页不直接承担所有上游创建，但要承担：

- 发现缺口
- 给出原因
- 一键深链到正确上游位置

## 五、P1 和 P2 的职责边界

## P1 先做这些

放在组织搭建中心 / 组织经营层：

1. 年度战略字段
2. 当前季度机构计划字段
3. 部门季度计划字段
4. 组织树和部门卡片显示这些信息
5. AI 和成长系统先能读取到这三层

### P1 不做这些

1. 不把月度计划主入口塞进任务页
2. 不在任务页做完整拆解流程
3. 不在日历页做自动任务包生成

## P2 再做这些

放在任务与日历：

1. 部门月度计划编辑入口
2. 从月度计划拆任务
3. 从月度计划排会议和日历
4. 周计划承接月度计划
5. 总结/复盘读取月度计划作为判断依据

## 六、页面信息架构建议

### 组织搭建中心（P1）

新增一个明显分区：

**战略与计划**

下面分成：

- 年度战略
- 当前季度机构计划
- 部门季度计划

### 任务与日历（P2）

新增一个靠前入口：

**本月计划**

在任务页顶部，比总结更靠前。

因为顺序应该是：

1. 月度计划
2. 周计划 / 任务 / 日历
3. 总结 / 复盘

不是反过来。

## 七、建议系统如何接这套入口

在 P1 阶段，建议系统就可以先做一部分：

### 在组织搭建中心

提示缺口：

- 年度战略未补
- 当前季度计划未补
- 某部门季度计划未补

### 在任务与日历

只做引导，不做主编辑：

- “当前部门还没有季度计划，请先去组织搭建中心补齐”
- “当前部门还没有本月计划，后续建议会偏泛”

也就是：

P1 时任务页先当“发现缺口的地方”

P2 时任务页再变成“真正承接月度计划的地方”

## 八、最终推荐方案

一句话收口：

### 年度战略 / 机构季度计划 / 部门季度计划

**放在组织搭建中心的“战略与计划”层**

### 部门月度计划

**放在任务与日历的前置执行层，不放在总结层**

### 第一次进入任务与日历

**先提示是否缺月度计划或缺季度计划，并给深链入口**

### 当前阶段安排

**P1 先做组织经营层；P2 再做任务与日历承接层**

这样既不会把战略计划放得太深，也不会让任务页过早承担整个上游结构。
