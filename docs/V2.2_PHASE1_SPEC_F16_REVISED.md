# F1.6 修订 Spec · 前端 useClientFact 接入

> **修订原因**（2026-05-22 自主判断）：原 spec §7 假设 6 大 view 各自核心 fetch 都能被 useClientFact 替代，但实地探查发现：
> - EventLineReportPanel 入口是 `eventLineId` 不是 `clientId`，不匹配
> - GrowthCenterView 入口是 `userId`，跟 client_fact 关系小
> - IntelligenceStationView / StrategicBrainView 核心数据是 view 专属 API（intelligence_items / strategic_thoughts），不在 ClientFactBundle 范围
> - **ClientFactBundle 真正能替代的是各 view 的"客户基础信息 + 跨表汇总"那部分，不是核心 fetch**
>
> 把握 80% → 按自主执行原则修订 spec，记录为契约

---

## 0 · 修订后的 F1.6 真实目标

**不是**：把 5 个 view 的核心 fetch 全部重写为 useClientFact

**而是**：建立 `ClientFactContext` 全局共享，让 currentClientId 切换时所有 view 自动拿到一致的 client bundle，**消除 B 方案 #9 "多池并列"在前端的表现**。

---

## 1 · 修订后的 4 件事

### F1.6a · `ClientFactContext` Provider + `useClientContext` hook

**新建文件**：`src/renderer/contexts/ClientFactContext.tsx`

```typescript
interface ClientFactContextValue {
  currentClientId: string | null;
  bundle: ClientFactBundle | null;
  isLoading: boolean;
  error: Error | null;
  refresh: () => Promise<void>;
}

export const ClientFactProvider: React.FC<{
  currentClientId: string | null;
  children: React.ReactNode;
}>;

export function useClientContext(): ClientFactContextValue;
```

**作用**：App.tsx 包一层 Provider，把 currentClientId 注入。各 view 通过 useClientContext() 拿到当前 bundle，**不再重复 fetch**。

### F1.6b · App.tsx 顶部包 ClientFactProvider

```tsx
<ClientFactProvider currentClientId={currentClientId}>
  <ExistingAppContent />
</ClientFactProvider>
```

**不破坏现有 currentClient props 派生模式**（避免大改 App.tsx 28758 行）。Provider 只是新增一条数据通道，老通道并存。

### F1.6c · StrategicBrainView 接入 useClientContext

**改造范围**：把 view 里"显示当前客户基础信息 + 概览（事件线/任务数）"那一小块切到 useClientContext。

**不改造**：strategic_thoughts / contradictions / duplicate_documents 等专项 API（这些不在 ClientFactBundle 范围）

### F1.6d · StrategicClarificationView 接入 useClientContext

**改造范围**：客户档案 tab 里展示 dna_documents + atomic_facts 那部分切到 useClientContext。

**不改造**：clarification 流程本身（这是它的核心专属功能）
