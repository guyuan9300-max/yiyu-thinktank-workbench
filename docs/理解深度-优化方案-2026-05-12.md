# 理解深度板块 — 优化方案

**起草日期：** 2026-05-12  
**当前完成度：** 20%（功能差距清单的评估）  
**目标完成度：** 第一阶段 60%（3 个月内）/ 第二阶段 85%（6 个月内）

---

## 一、先弄清楚问题：什么叫"理解深度"

"理解深度"不是单一能力，它是**系统从"看到文字"到"理解含义"之间的一连串能力台阶**。

```
文字级    →    结构级    →    语义级    →    时间级    →    认知级
(字符/token  (章节/表格   (实体/关系   (何时/变化   (矛盾/确信
 /关键词)    /列表)       /事件/事实)  /过期)       /推理)

当前位置:   ████░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░
            ↑ 大部分能力卡在这里
```

**核心问题：** 当前系统主要在"文字级 + 结构级"工作。把文档切成 chunk、向量化、检索回来——这是搜索引擎的基本盘，不是"理解"。真正的理解从"识别实体、抽取关系、判断时间、检测矛盾、评估可信度"开始。

**为什么这层很重要：**
- 战略陪伴需要矛盾检测和鲜度衰减（否则会基于过期信息建议）
- 内容生产需要隐含信息分类（事实/判断/假设要分开处理）
- 客户洞察需要跨文档关系图谱（否则只能就单条问题答）
- AI 岗位全部都需要（虚拟公司必须有语义共识）

---

## 二、把"理解深度"拆成 7 个可独立优化的能力

| # | 能力 | 当前 | 目标 | 难度 | 业务价值 |
|---|---|---|---|---|---|
| 1 | 类型化实体抽取 | `list[str]` 关键词 | 类型化 + 规范化 + ID | 中 | ⭐⭐⭐⭐⭐ |
| 2 | 跨文档实体合并 | 无 | 同名归一 + 别名管理 | 中 | ⭐⭐⭐⭐⭐ |
| 3 | 关系三元组抽取 | 无 | (主体, 关系, 客体) 表 | 高 | ⭐⭐⭐⭐ |
| 4 | 原子事实抽取与矛盾检测 | 无 | 事实表 + 矛盾告警 | 高 | ⭐⭐⭐⭐⭐ |
| 5 | 鲜度真实衰减 | 写死常量 | 与 created_at 关联 | 低 | ⭐⭐⭐⭐ |
| 6 | chunk 语义分类 | 无 | 事实/判断/假设/行动四档 | 中 | ⭐⭐⭐ |
| 7 | 客户术语库 | 无 | 每客户黑话/产品名表 | 中 | ⭐⭐⭐ |

**优先级原则：** 实体（能力 1+2）是其他能力的基础——关系、事实、矛盾全部依赖实体规范化。所以**能力 1+2 必须最先做**，做完之后 3/4/5/6/7 才能并行展开。

## 三、逐一展开优化方案

### 能力 1：类型化实体抽取

#### 当前状态
- `knowledge_base.py:328` 定义 `entities: list[str]`
- `knowledge_v2.py:1949/2992` 写入 `entities_json`，但本质是关键词数组
- 无类型、无规范化、无 ID、无属性

#### 目标状态
每个文档抽取出来的实体都带：
- **类型**：人 / 公司 / 项目 / 产品 / 竞品 / 金额 / 日期 / 地点 / 事件 / 指标
- **规范名**：把"小米"、"小米公司"、"小米科技"统一为同一个实体
- **别名**：保留所有出现过的写法
- **属性**：例如人物的"职位"、公司的"所属行业"
- **客户绑定**：某些实体是客户私有的（如客户内部的项目名）

#### 实现路径

**方案 A：纯 LLM 抽取（不推荐）**
- 优点：灵活、可处理复杂语境
- 缺点：贵、慢、不稳定（同一文档两次抽取结果可能不一致）

**方案 B：spaCy/HanLP 本地 NER + 规则后处理（不推荐）**
- 优点：快、确定性
- 缺点：中文领域专有名词识别差、难定制

**方案 C：LLM + 本地规则混合（推荐）**
- 抽取阶段：LLM 一次性抽出类型化实体（带 prompt 模板）
- 规范化阶段：本地规则 + 历史实体库做 dedup
- 增量更新：新文档与已有实体库比对，命中已知实体则复用 ID，否则新建
- **关键技巧：** 对每个客户维护一个"实体白名单"，新实体先放入"待审核"，避免噪音污染

#### 数据模型草图

```sql
-- 实体主表（跨客户/跨组织）
CREATE TABLE entities (
    id TEXT PRIMARY KEY,
    entity_type TEXT NOT NULL,  -- person|company|project|product|competitor|...
    canonical_name TEXT NOT NULL,
    aliases_json TEXT,  -- ["小米","小米公司","小米科技"]
    client_id TEXT,  -- NULL 表示通用实体；非 NULL 表示客户私有
    organization_id TEXT,
    attributes_json TEXT,  -- {"industry": "消费电子", "title": "CEO"}
    first_seen_doc_id TEXT,
    confidence REAL DEFAULT 0.8,
    status TEXT DEFAULT 'active',  -- active|pending_review|merged|archived
    merged_into TEXT,  -- 如果与其他实体合并，指向新 ID
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
CREATE INDEX idx_entities_type_client ON entities(entity_type, client_id);
CREATE INDEX idx_entities_canonical ON entities(canonical_name);

-- 实体提及表（每次出现都记录）
CREATE TABLE entity_mentions (
    id TEXT PRIMARY KEY,
    entity_id TEXT NOT NULL REFERENCES entities(id),
    doc_id TEXT NOT NULL,
    chunk_id TEXT,
    mention_text TEXT NOT NULL,  -- 实际出现的字面文本
    char_offset INTEGER,
    context_snippet TEXT,  -- 前后各 50 字
    confidence REAL,
    extracted_by TEXT,  -- llm|rule|manual
    created_at TIMESTAMP
);
CREATE INDEX idx_mentions_doc ON entity_mentions(doc_id);
CREATE INDEX idx_mentions_entity ON entity_mentions(entity_id);
```

#### 与现有系统对接
- 在 `ingest_document_knowledge`（`knowledge_v2.py:1696`）的解析阶段之后、写入 `v2_documents` 之前插入抽取步骤
- 在 `v2_chunks` 加一个 `entity_ids_json` 字段记录该 chunk 涉及的实体 ID
- 检索时按实体过滤："只看涉及实体 X 的 chunk"
- 现有 `entities: list[str]` 字段保留，作为快速展示用的"扁平视图"

#### 关键决策
- **Q: 用什么 LLM 做抽取？** 推荐用本地 7B 模型（成本可控）+ 重要文档可选 GPT-4 复核
- **Q: 抽取频率？** 入库时一次 + 用户编辑时增量 + 月度 batch 重跑（修正历史）
- **Q: 通用实体 vs 客户私有？** 默认通用，但客户私有项目/产品自动归入该客户

### 能力 2：跨文档实体合并

#### 当前状态
- 每份文档独立抽取，**不与其他文档比对**
- 同一个客户在 50 份资料里出现可能产生 50 个"实体"

#### 目标状态
- 自动识别同名/相似实体并合并
- 别名管理（"小米"、"MI"、"Xiaomi" 指向同一实体）
- 跨文档"同一实体"查询能立即拉出所有提及

#### 实现路径
- **入库时**：抽取实体后立刻和现有实体库比对（按 canonical_name + entity_type + organization_id）
- **匹配规则**：
  - 完全匹配 → 复用 ID
  - 模糊匹配（编辑距离 + 类型一致）→ 进"待合并队列"
  - 无匹配 → 新建实体
- **人工合并入口**：运维面板加一个"实体管理"页面，看待合并实体并一键合并
- **自动合并**：超过 N 次共现且类型一致的可自动合并（带审计日志）

#### 数据模型（在能力 1 基础上扩展）

```sql
-- 实体合并历史
CREATE TABLE entity_merge_log (
    id TEXT PRIMARY KEY,
    source_entity_id TEXT NOT NULL,
    target_entity_id TEXT NOT NULL,
    merge_reason TEXT,  -- auto_match|user_merge|alias_added
    merged_by TEXT,  -- user_id 或 'auto'
    merged_at TIMESTAMP
);

-- 实体别名表（可选独立表，也可用 entities.aliases_json）
-- 独立表的好处是别名也可被检索
CREATE TABLE entity_aliases (
    entity_id TEXT NOT NULL REFERENCES entities(id),
    alias TEXT NOT NULL,
    source TEXT,  -- llm_extracted|user_added|merged_from
    confidence REAL,
    PRIMARY KEY (entity_id, alias)
);
```

#### 业务价值
解锁这些之前问不出来的问题：
- "这个客户在所有资料里被提到过多少次"
- "项目 A 涉及的所有人"
- "和这个竞品对比过的所有方案"
- "客户 X 提过的所有目标"

### 能力 3：关系三元组抽取

#### 当前状态
- 缺失。全仓 `knowledge_graph` 只 1 命中（`digital_asset_narrative.py`）
- 无 nodes/edges/三元组表

#### 目标状态
从每份文档抽取 (subject, predicate, object) 三元组，例如：
- (客户A, 战略目标, 进入华东市场)
- (项目B, 负责人, 张三)
- (产品C, 上线时间, 2026-08)
- (竞品D, 优势, 价格低)
- (决策E, 依据, 会议纪要-20260315)

#### 实现路径

**第一步：定义关系词典**

不要让 LLM 自由发挥，**预定义关系类型**：

| 关系类别 | 关系 | 主体类型 | 客体类型 |
|---|---|---|---|
| 客户 | 战略目标 | 客户 | 文本 |
| 客户 | 当前痛点 | 客户 | 文本 |
| 客户 | 负责人 | 客户 | 人 |
| 项目 | 负责人 | 项目 | 人 |
| 项目 | 客户 | 项目 | 客户 |
| 项目 | 状态 | 项目 | 枚举 |
| 项目 | 截止时间 | 项目 | 日期 |
| 产品 | 竞品 | 产品 | 竞品 |
| 决策 | 依据 | 决策 | 文档 |
| 会议 | 参与者 | 会议 | 人 |
| 会议 | 客户 | 会议 | 客户 |

预定义约束严格，但**质量比覆盖率重要**。

**第二步：LLM 抽取**

给 LLM 一份文档 chunk，配上：
- 已识别的实体列表（来自能力 1）
- 关系词典
- 要求只输出落在词典内的三元组

**第三步：写入三元组表**

每个三元组带 provenance（来源 chunk）和 confidence。

#### 数据模型

```sql
CREATE TABLE relationship_triples (
    id TEXT PRIMARY KEY,
    subject_entity_id TEXT NOT NULL,
    predicate TEXT NOT NULL,  -- 来自关系词典
    object_entity_id TEXT,  -- 客体是实体
    object_value TEXT,  -- 客体是字面值（日期/数字/枚举）
    object_type TEXT,  -- entity|text|number|date|enum
    doc_id TEXT NOT NULL,
    chunk_id TEXT,
    confidence REAL,
    valid_from TIMESTAMP,  -- 关系生效时间
    valid_until TIMESTAMP,  -- 关系失效时间（如 NULL 表示仍有效）
    extracted_at TIMESTAMP,
    extracted_by TEXT
);
CREATE INDEX idx_triples_subject ON relationship_triples(subject_entity_id, predicate);
CREATE INDEX idx_triples_object ON relationship_triples(object_entity_id);
CREATE INDEX idx_triples_predicate ON relationship_triples(predicate);
```

#### 查询样例

```sql
-- "客户 A 的所有战略目标"
SELECT object_value, doc_id, valid_from
FROM relationship_triples
WHERE subject_entity_id = '客户A_id'
  AND predicate = '战略目标'
ORDER BY valid_from DESC;

-- "项目 B 涉及的所有人"
SELECT DISTINCT e.canonical_name
FROM relationship_triples t
JOIN entities e ON t.object_entity_id = e.id
WHERE t.subject_entity_id = '项目B_id'
  AND t.predicate IN ('负责人', '协作者');
```

#### 关键决策
- **Q: 用图数据库（Neo4j）还是 SQL？** 桌面端用 SQLite + 三元组表足够，避免引入额外运行时
- **Q: 关系词典如何维护？** 初版硬编码，后续做成可配置
- **Q: 抽取出错怎么办？** 三元组都带 confidence，置信度低的进"待审核"

### 能力 4：原子事实抽取与矛盾检测（**最关键的一个**）

#### 当前状态
- 缺失。`conflicts` 字段是会议人际冲突，不是事实矛盾。

#### 目标状态
- 从文档抽取原子事实：(subject, attribute, value, time)
- 检测同一 (subject, attribute) 在不同文档中出现不同 value
- 主动告警："上次客户 A 说预算 50 万、本周会议说预算 30 万"

#### 实现路径

**第一步：原子事实抽取**

事实模板：(实体, 属性, 值, 时间, 来源置信度)

例如从一份会议纪要里抽出：
- (客户A, 预算上限, 50万, 2026-03-15, high)
- (客户A, 决策人, 王总, 2026-03-15, high)
- (客户A, 上线目标, Q3, 2026-03-15, medium)

**第二步：矛盾检测器**

每次有新事实写入时：
1. 按 (subject_entity_id, attribute) 查现有事实
2. 比较 value 是否相同
3. 若不同：
   - 数值类：判断是否在容差范围内
   - 文本类：用 LLM 判断是否实质性矛盾（同义不算）
   - 时间类：判断是否冲突
4. 矛盾入库，触发告警

**第三步：矛盾解决策略**

- **新事实优先**（默认）：最新事实有效，但保留历史
- **可信度优先**：高 confidence 来源覆盖低 confidence
- **人工裁决**：标记为"待裁决"，等用户决定

#### 数据模型

```sql
-- 原子事实表
CREATE TABLE atomic_facts (
    id TEXT PRIMARY KEY,
    subject_entity_id TEXT NOT NULL,
    attribute TEXT NOT NULL,  -- 来自属性词典
    value_text TEXT,
    value_number REAL,
    value_date TIMESTAMP,
    value_type TEXT NOT NULL,  -- text|number|date|enum
    fact_time TIMESTAMP,  -- 这个事实何时成立
    doc_id TEXT NOT NULL,
    chunk_id TEXT,
    confidence REAL,
    superseded_by TEXT,  -- 被哪个更新的事实替代
    status TEXT DEFAULT 'active',  -- active|superseded|disputed
    created_at TIMESTAMP
);
CREATE INDEX idx_facts_subject_attr ON atomic_facts(subject_entity_id, attribute);

-- 矛盾记录表
CREATE TABLE fact_contradictions (
    id TEXT PRIMARY KEY,
    subject_entity_id TEXT NOT NULL,
    attribute TEXT NOT NULL,
    fact_ids_json TEXT NOT NULL,  -- 涉及的 atomic_facts.id 列表
    contradiction_type TEXT,  -- value_diff|time_overlap|source_conflict
    severity TEXT,  -- info|warning|critical
    detected_at TIMESTAMP,
    resolution_status TEXT DEFAULT 'open',  -- open|resolved|dismissed|escalated
    resolution_note TEXT,
    resolved_by TEXT,
    resolved_at TIMESTAMP
);
```

#### 业务价值（关键）

战略陪伴场景里**矛盾告警是核心价值之一**：
- "客户 A 的预算在三份资料里写法不一致"
- "项目 B 的截止时间会议纪要和邮件不一致"
- "产品 C 的定价在不同渠道资料里不同"

这是 AI 真正"帮你看到你自己看不到的东西"——这种价值是单纯检索做不到的。

#### 关键决策
- **Q: 何时跑矛盾检测？** 入库时即跑（新事实进来立即与历史比对）+ 周末 batch 全量复查
- **Q: 多严格？** 初版宽松（只检测明显矛盾），积累误报修正记录后再收紧
- **Q: 怎么呈现给用户？** 在客户工作台首页加"矛盾告警"模块，进入资料时也内联显示

### 能力 5：鲜度真实衰减（**最低成本最高回报**）

#### 当前状态
- `memory_foundation.py:257/270/621/658/677/1541` 的 `freshness` 是写死的 0.92 / 0.9 / 0.8 / 0.7
- 不随时间衰减

#### 目标状态
- 鲜度 = f(资料类型, 资料时间, 当前时间)
- 半衰期按类型差异化设置

#### 实现路径

非常简单的一处改动：

```python
from datetime import datetime, timedelta

# 按资料类型定义半衰期（天）
HALF_LIFE_BY_TYPE = {
    "news": 30,           # 新闻动态
    "client_judgment": 90,  # 客户判断
    "meeting_minutes": 90,  # 会议纪要
    "strategy_doc": 180,  # 战略文档
    "policy_doc": 365,  # 政策文档
    "background": 9999,  # 历史背景（不衰减）
    "evidence_artifact": 180,  # 证据材料
    "default": 90,
}

def compute_freshness(
    doc_type: str,
    created_at: datetime,
    now: datetime = None,
    user_pinned: bool = False
) -> float:
    """
    返回 0-1 之间的鲜度分数。
    - 1.0 = 完全新鲜
    - 0.5 = 已过半衰期
    - 0.0 = 接近无效
    """
    if user_pinned:
        return 1.0  # 用户手动 pin 的资料永远新鲜
    
    now = now or datetime.now()
    age_days = (now - created_at).days
    half_life = HALF_LIFE_BY_TYPE.get(doc_type, HALF_LIFE_BY_TYPE["default"])
    
    # 指数衰减
    return 0.5 ** (age_days / half_life)
```

#### 与现有系统对接
- 替换 `memory_foundation.py` 里所有静态 freshness 赋值
- 在 `evidence_selector` 重排时，把鲜度作为权重之一
- 在 UI 里展示资料的"鲜度评分"和"距今多久"

#### 关键决策
- **Q: 半衰期怎么定？** 第一版按上表，后续可以让用户自定义
- **Q: 用户能 pin 资料吗？** 强烈建议——某些"圣经"级文档不该衰减
- **Q: 衰减到 0 怎么办？** 不删除，但检索时降权；超过阈值进入"归档"提示

#### 业务价值
- 解决"半年前的客户判断和昨天的判断在系统里看起来一样新"
- 是战略陪伴的硬需求
- **实现成本最低（1-2 天工作量），收益最高之一**

### 能力 6：chunk 语义分类

#### 当前状态
- 缺失
- chunk 只有 `material_profile = background | evidence` 两档

#### 目标状态
每个 chunk 打标语义类型：
- **fact**（事实）：可被验证的陈述
- **judgment**（判断）：基于事实的推断或评价
- **opinion**（观点）：主观态度
- **conclusion**（结论）：从一组事实得出
- **hypothesis**（假设）：未验证的推测
- **action**（行动）：要做的事
- **question**（问题）：未解答
- **definition**（定义）：术语解释

#### 实现路径
- LLM 分类器，prompt 给定义和示例
- 在 `ingest_document_knowledge` 流程里加入分类步骤
- 每个 chunk 加 `semantic_type` 字段

#### 数据模型

```sql
ALTER TABLE v2_chunks ADD COLUMN semantic_type TEXT;
ALTER TABLE v2_chunks ADD COLUMN semantic_confidence REAL;
```

#### 业务价值
- 综合分析时按类型加权：事实高权重，假设低权重
- 内容生产时按类型组织：先列事实，再呈判断，最后给行动
- 矛盾检测时只比对事实，不比对观点
- 让 AI 回答时知道"这是事实"还是"这是某人的看法"

### 能力 7：客户特定术语库

#### 当前状态
- 缺失。全仓 `glossary / terminology / 术语` 0 个产品代码命中

#### 目标状态
每个客户有一份自己的术语表：
- 产品名（客户内部叫法）
- 项目代号
- 行业黑话
- 客户特定的人物缩写

#### 实现路径
- **自动发现**：能力 1 抽取实体时，标记"客户私有"实体
- **人工维护**：客户工作台加一个"术语表"标签页
- **使用**：
  - 提问时优先在客户术语库里解析名词
  - 解析"小李"时，先看这个客户的术语库里有没有定义
  - LLM 回答时，prompt 注入术语表作为上下文

#### 数据模型

```sql
CREATE TABLE client_glossary (
    id TEXT PRIMARY KEY,
    client_id TEXT NOT NULL,
    term TEXT NOT NULL,
    definition TEXT NOT NULL,
    category TEXT,  -- product|project|person|industry|internal_jargon
    aliases_json TEXT,
    source TEXT,  -- user_added|auto_discovered|llm_suggested
    confidence REAL,
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    UNIQUE(client_id, term)
);
```

#### 业务价值
- 消除歧义：客户 X 用"平台"指 A，客户 Y 用"平台"指 B，不会混
- 跨客户隔离：避免把"客户 X 内部黑话"用到客户 Y 的回答里

## 六、风险与权衡

### 风险 1：LLM 抽取成本
**问题：** 每份新文档要跑 1-2 次 LLM 抽取，量大成本可观
**缓解：**
- 用本地 7B 模型（成本极低）跑批量
- 重要文档（用户 pin）才用 GPT-4 复核
- 异步处理，不阻塞用户

### 风险 2：抽取质量不稳定
**问题：** 同一文档两次抽取结果可能不一致
**缓解：**
- 用 temperature=0 + structured output
- 全部抽取结果带 confidence，低置信进"待审核"
- 用户可手动修正，修正进入反馈训练

### 风险 3：实体爆炸
**问题：** 抽多了导致实体库爆炸、合并困难
**缓解：**
- 类型白名单严格控制
- 自动合并阈值
- 定期 archive 长期未被提及的实体

### 风险 4：矛盾误报
**问题：** "上次说 50 万、这次说 30 万"可能是不同语境（一个是总预算、一个是首期）
**缓解：**
- 初版严格匹配 (subject, attribute, time_range)
- 矛盾告警分级（info/warning/critical）
- 用户能一键 dismiss 误报，dismiss 进反馈

### 风险 5：本地资源消耗
**问题：** 桌面端要存储更多结构化数据，跑更多 LLM 任务
**缓解：**
- 实体/关系/事实表都是小数据（相比原始文档极小）
- LLM 任务进 knowledge_jobs 队列，CPU 空闲时跑
- 用户可关闭实时抽取，改批量

### 权衡决策点

| 决策 | 选项 A | 选项 B | 建议 |
|---|---|---|---|
| 关系词典 | 严格预定义 | LLM 自由发挥 | A（质量优先） |
| 实体存储 | SQL 三元组 | 图数据库 | A（桌面端简洁） |
| 矛盾检测时机 | 实时 | 批量 | 混合（事实写入时实时 + 周末批量复查） |
| 抽取模型 | 云端 GPT-4 | 本地 7B | 本地为主 + 关键 case 升级云 |
| 用户参与 | 全自动 | 半自动 | 半自动（关键决策保留人工） |

## 八、一句话总结

> **理解深度的本质是从"文字处理"升级到"语义处理"。优先级是：先建实体基础设施（能力 1+2，4 周），再做低成本高回报的鲜度衰减和语义分类（能力 5+6，2 周），然后建立关系图谱和矛盾检测（能力 3+4，6 周），最后是客户术语库（能力 7，2 周）。3 个月把理解深度从 20% 提升到 60%。**

最值得先做的两件事：
1. **能力 5（鲜度衰减）** —— 1-2 天就能完成，对战略陪伴价值巨大
2. **能力 1+2（实体基础设施）** —— 是后面所有能力的基础，绕不开

最该谨慎的事：
- 不要让 LLM 在关系词典外自由发挥
- 不要追求 100% 准确率而推迟上线，60% 准确率 + 用户反馈循环更重要
- 不要把所有抽取做成实时阻塞，进队列异步跑
