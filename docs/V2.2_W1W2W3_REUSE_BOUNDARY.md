# V2.2 跟 W1/W2/W3 的复用边界

> **写作动机**: 顾源源 2026-05-22 严肃问 "V2.2 工作是不是被 W1/W2/W3 那条方向有问题的实验线污染了"。
> **本文档作用**: 把"哪些 W 产物保留 / 哪些放弃"逐项说清, 后续 session 接手时**一眼能查**。
> **判断标准 (NORTH_STAR §5)**: 任何 W1/W2/W3 产物, 问"它服务 N1/N2/N3 哪个北极星?", 不服务任何一个 → 放弃; 服务至少一个 → 保留。

---

## 0 · 一句话

**V2.2 不是 W1/W2/W3 的延续, 也不是从 0 开始。** V2.2 是**站在 W1/W2/W3 留下的工程基础设施上, 重新校准了目标方向**:

| 维度 | W1/W2/W3 时代 | V2.2 时代 |
|---|---|---|
| 北极星 | 代码模块化 / 1700 SQL 清零 / App.tsx 拆 | **N1 功能顺畅 / N2 4 主路径理解信息源 / N3 3.0 接入预留** |
| 验收门 | "main.py 0 裸 SQL" / "边界 0 违规" | **5/19 张真会议机器人答出 / 客户感知到改变 / 3.0 直接接入** |
| 判断标准 | 数字漂亮 | 客户价值 |

V2.2 NORTH_STAR §5 明确放弃了 W3-W4 "main.py 1047 SQL 清零" / "App.tsx 28758 行拆 view" / "全部 195 个 services 进 modules" 这些**数字目标**。

## 1 · W1/W2/W3 简史 (5/20 之前那条实验线)

| 周 | 目标 | 实际做了什么 (落到代码) |
|---|---|---|
| W1 | "数据层底座" | organization 模块 SSOT (4 mirror 表 + sync 引擎 + 7 view + 11 Repository 方法) + 12 维度 acceptance runner + golden fixtures dump |

W1/W2/W3 留下的代码不是问题, 留下的**目标方向是问题** — "数字漂亮" 不等于"客户价值"。NORTH_STAR 校准时, 我们重新定了目标但**保留了可用基础设施**。

## 2 · V2.2 复用 vs 放弃 — 逐项判断

### 2.1 ✅ 保留 (服务至少一个北极星)

| W 产物 | 出处 | 服务哪个北极星 | V2.2 怎么用 |
|---|---|---|---|
| `backend/app/modules/organization/` (mirror 表 + sync 引擎 + 7 view + 11 方法) | W1 | **N3 cloud_sync 预留** + N1 多账号 frozen 守门 | F1.7 `client_stage_audit` + `apply_cloud_stage_change` 直接基于此 |
| `backend/app/modules/{client,commitment,glossary,task,knowledge,intelligence,narrative}/` 8 个模块目录 | W2 | N1 工程基础设施 (新 service 住进去) | V2.2 所有新 service 都进这个目录 (`ingest_pipeline.py` 进 services/, schema 进 db.py) |
| `scripts/lint_module_boundaries.py` 模块边界 linter | W2 | N1 防回滚 | V2.2 acceptance runner A2 用它防"代码漂亮陷阱" 反向 |
| `app/modules/llm_context/` (composer/inspector/prompt_logger + `prompt_log` 表) | W2 | N2 LLM 推理可追溯 | F2.7 `reasoning_traces` 用 `prompt_log_id` 关联 |
| W3 Repository pattern (`GlossaryRepository`, `CommitmentRepository` + `_ConnAdapter`) | W3 | N2 4 主路径统一通道 + N3 actor 接入 | F2.4 `IngestPipeline` 通过 Repository 写入 atomic_facts; F2.1 `DocumentLLMExtractor` 全程走 Repository |
| W3 services 切线 (glossary_store / todo_aggregator / clarification_context / narrative_collector) | W3 | N1 不破坏现有 (历史代码继续跑) | V2.2 不动这 4 个文件, 走 W3 切好的 Repository 路径 |
| `tests/test_w3_services_migration.py` 26 测试 | W3 | N1 回归基线 | V2.2 acceptance A3 跑它确保 W3 切线没被 V2.2 工作破坏 |
| `tests/test_lint_module_boundaries.py` 9 测试 | W2 | N1 linter 自检 | V2.2 acceptance 间接依赖 |
| `tests/fixtures/golden/{日慈,为爱黔行,CFFC}.json` | Day 0 | N1 跨版本 diff 验证 | V2.2 后续 release 前会用 (Phase 3) |
| `dump_golden_fixtures.py` | Day 0 | N1 同上 | 同上 |

**判断逻辑**: 这些都是**中性工程基础设施** — 不绑定"代码漂亮"那条错误方向, 服务"功能顺畅 + 数据中心理解信息源 + 3.0 接入"这三个 V2.2 北极星。**保留是有理由的, 不是路径依赖**。

### 2.2 ❌ 放弃 (不服务任何北极星, 或绑定到"代码漂亮"目标)

| W3-W4 原计划 | 放弃理由 | 替代方案 |
|---|---|---|
| "main.py 1047 处裸 SQL 清零" | NORTH_STAR §5 明确放弃 (数字 ≠ 客户价值) | 只清"4 主路径需要的 SQL", 按需切, 不强求清零 |
| "App.tsx 28758 行拆 7 个 view + stores" | NORTH_STAR §5 明确放弃 (前端拆分对 N1/N2/N3 都没直接贡献) | App.tsx 内嵌一个 `useClientContext` provider (F1.6), 内部不强求拆分 |
| "全部 195 个 services 都进 modules" | NORTH_STAR §5 明确放弃 | 只按 N2 4 主路径 + F2.1 抽取需要的 service 切迁 |
| "5 层架构每层都拆完整" | NORTH_STAR §5 (5 层是设计骨架, 不是 KPI) | 关心"客户能感知" 而不是"层是否漂亮" |
| W1 完成的 organization SSOT 目标本身 | **目标继承了, 实现保留** — 即 organization 模块产物保留, 但叙事换成"N3 cloud_sync 预留", 不再叙事为"代码模块化里程碑" | 复用代码, 叙事换 |
| 把"数字进度 (XX% 清零)" 作为里程碑标准 | 顾源源 5/22 多次强调"客户价值优先, 不是数字" | 改用"5/19 张真会议机器人答出几条 + 4 路径状态" |

**判断逻辑**: 这些都是**绑定"代码漂亮"目标**的事。即使做完, 客户感知 = 0; 不做, 不影响任何北极星。所以放弃。

## 4 · 判断标准 (后续 session 一眼能查)

**任何 W1/W2/W3 留下的代码 / 工具 / 测试 / 文档, 都按这 3 个问题判断**:

1. **它服务 N1/N2/N3 中至少一个吗?**
   - 是 → 保留, 在 §2.1 表里标注怎么用
   - 否 → 放弃, 在 §2.2 表里标注理由

2. **它是中性工程基础设施 (linter / 模块骨架 / Repository pattern), 还是绑定到"代码漂亮"目标?**
   - 中性 → 保留
   - 绑定 → 即使能跑通也放弃 (例: "main.py 0 SQL" 这种 KPI)

3. **不复用它, V2.2 工作量增加多少?**
   - 增加大 + 服务北极星 → 强烈保留
   - 增加小 + 不服务 → 放弃

## 5 · 哪些"看起来污染但实际中性"的事

容易被误判的:

| 看起来 | 实际 |
|---|---|
| W2 模块骨架 = W2 方向 | 模块骨架是**中性容器**, V2.2 把新代码住进去, 跟 W2 "代码漂亮"目标无关 |
| W3 切线 = W3 方向 | 切线是**让 SQL 走 Repository**, 这正是 V2.2 N2 "4 主路径统一通道" 需要的, 不矛盾 |
| `lint_module_boundaries.py` = W2 工具 | linter 是**中性检查工具**, V2.2 用它防回滚, 跟 W2 目标无关 |
| `tests/test_w3_services_migration.py` = W3 测试 | 这测试验证 Repository pattern 跑得通, V2.2 工作也建立在 Repository pattern 上, 测试还要继续跑 |

容易被忽略的真污染:

| 看起来安全 | 实际有问题 |
|---|---|
| commit message 前缀 `feat(v2.1 F1.X)` | 这是 W1/W2/W3 实验线的命名习惯, V2.2 应该用 `[A]/[B]` 前缀 (V2.1_AI_COLLABORATION.md §5) |
| 文档名 `V2.2_WEEK1_*` | W 是 W1/W2/W3 时代的概念, V2.2 用 Phase 1/2/3 (RFC 标准) |
| 把"进度 = 切了几处 SQL" 作为里程碑成果 | 错的, 进度 = "客户能感知到什么改变" + "三道门哪几个 PASS" |

## 6 · 这份文档的更新触发条件

**只在以下场景更新**:
1. V2.2 工作发现新的 W1/W2/W3 产物可复用 → 加到 §2.1
2. V2.2 工作放弃了某条原 W 计划 → 加到 §2.2
3. 顾源源校准 NORTH_STAR → 全面 review 边界

**不要随便加 W 产物到 §2.1**, 加之前必须能回答"它服务哪个北极星 + 怎么用"。

---

**底线**: V2.2 跟 W1/W2/W3 的关系是 "复用基础设施, 校准目标方向, 物理隔离不混"。任何模糊地带, 用 §4 三个问题判断。
