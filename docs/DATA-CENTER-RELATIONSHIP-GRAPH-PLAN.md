> 状态：**v0.3 — 功能反推后修正版**
> 日期：2026-05-15
> 变更说明（相对 v0.2）：① 4 层结构升级为 4 层 + 引擎层（共 5 层）/ ② 全层补"部门"挂载维度 / ③ 主线层补"事件流" / ④ 上游协同合同新增 5 项工作台需求 / ⑤ Phase 1-3 硬交付范围按反推结果重新定义 / ⑥ 时间线重新评估
> 反推依据：`PHASE-0-FUNCTION-DRIVEN-REVERSE-PLAN.md`（20 个高/中价值场景反推得出）
> 用户原话定义的目标："让数据中心能读到每一份文档、完整理解每一份文档、检索时不遗漏内容，并把每个细节中的关键事实提炼出来，串成完整的逻辑描述系统 —— 发生了什么事、每个人扮演什么角色、每件事的价值和核心意义、它发生的时间段和关键里程碑、交付物是什么。"

---

## 0 · 一句话总结

**把数据中心从一座"文档仓库"升级成一张"客户关系网"** —— 不只是存了什么文档，而是能讲清楚每个客户在做什么、谁在做、做到哪一步了、为什么这件事重要 —— **作为所有上层业务模块（战略陪伴/成长中心/任务与日程）所有判断的背景知识**。

## 1 · 现在的问题（保留 v0.2）

数据中心目前知道**每片树叶长什么样**（每份文档都存着），但**不知道这片森林是怎么长起来的**（这些文档说的是同一件事吗？谁负责？什么时候开始的？卡在哪儿？）。

| 问题 | 现在的回答（功能型） | 我们想要的回答（业务型） |
|---|---|---|
| 日慈这个客户现在卡在哪？ | 知识完整度 47%，建议补充文档 | 高老师上周离职导致品牌升级停摆；张真新加坡出差未谈成实质合作 |
| 詹瑶是谁？ | 一个名字 | 黔行秘书长 + 共同创始人 + 吴建林配偶 + 4 月刚从精神状态恢复回归 |
| 黔行 6 月工作坊要做什么？ | 找不到答案 / 散落在多份文档里 | 顾老师创新方案：用自研软件做工作坊，从绩效矛盾切入 |
| 你部门本周做了什么？ | 列任务，没有汇总 | 部门新增 3 条承诺、1 条卡住（品牌升级，原因=责任人离职）、5 条状态变化 |
| 哪些承诺到期了？ | 看不到 | 今天到期：给詹瑶发更新价值观；明天到期：开展实地走访 |

第一行是当前数据中心给的；第二行才是**超级员工**应该给的。第二行需要的不是更多文档，而是**关系网 + 引擎驱动的判断**。

## 3 · 我的产品定位 + 评估标准

### 3.1 产品定位（用户原话）

> **"你做的东西就是所有判断的背景"**

- 工作台模块 = **"问什么找什么"**（用户主动搜文档）
- 我 = **"AI 心里清楚整个客户的来龙去脉"**（背景知识引擎）
- 战略陪伴 / 周复盘 / 任务智能简报 等模块的判断 → **它们是从我这里读的背景**

### 3.2 评估标准

❌ 不是："关系网有多完整 / 我建了多少表 / 我抽了多少字段"
✅ 是：**"下游模块拿到我的背景之后，能否给出更好的业务判断"**

### 3.3 优化方向公式（用户原话）

> **场景优化价值 = 场景需求频次 × 用户对价值的感知**

| 频次档 | 分数 | 价值感知档 | 分数 |
|---|---|---|---|
| 每天 | 5 | 强（无可替代） | 5 |
| 每周 1-2 次 | 4 | 中（"有用，比之前好"） | 3 |
| 每月 1-2 次 | 3 | 弱 | 1 |
| 季度 | 2 | | |
| 偶尔 | 1 | | |

**硬约束**：只有总分 ≥ 12 的场景，才进 Phase 1-3 的硬交付范围。避免造三个轮子的车。

### 5.1 第一层 · 「花名册」（人物档案）

**保留 v0.2 内容**：人物 + 别名 + 状态 + 关系 + 健康信号 + 工作风格 + AI 模拟标记

**v0.3 新增**：
- `department_id`（人物挂在哪个部门 —— 部门切面必需）
- `last_activity_at`（最后一次出现在事实卡片上的时间 —— 客户冷却检测必需）

### 5.2 第二层 · 「事实卡片」（事实证据）

**保留 v0.2 内容**：业务语义（承诺/决策/风险/事实/教训/进展）+ 影响（推进/中性/阻塞）+ 时间锚 + 置信度

**v0.3 新增**：
- 按 **部门** 维度筛卡片的接口
- 按 **时间窗口**（本周/本月/季度）筛卡片的接口
- 按 **人物** 维度筛卡片的接口
- 卡片之间的**承诺-承接对应链**（这条承诺有没有人接）

### 5.3 第三层 · 「项目台账」（业务主线）

**保留 v0.2 内容**：5 种颗粒度 + 起止 + 交付物 + 责任人 + 父子嵌套 + 当前状态 + 澄清状态

**v0.3 新增**：
- `department_id`（主线归哪个部门 —— 部门切面必需）
- **事件流子表**：每次状态变化产生一条事件（active→blocked / blocked→resolved / responsible_person 变更等），带触发时间、原因、证据
- 主线推进健康度量化标准（顺利/卡住/逾期 三档统一判定规则）

### 5.4 第四层 · 「客户关系书」

**保留 v0.2 内容**：8 段结构（机构信息/战略/业务/人物/合作/主线/风险/未澄清）

**v0.3 新增明确**：
- 每段绑定明确的"读什么数据"规则（不再是抽象描述）
- 每段绑定"自动更新触发条件"
- 每段绑定"钻取链路"

### 5.5 第五层 · 「引擎层」（v0.3 全新）

5 个引擎，让关系网"主动"驱动业务判断而不是死的：

#### 引擎 A · 多维投影引擎

**作用**：给定一个视角，输出对应切面的关系网视图。

**4 种主要投影**：
- **部门投影** → 本周概览 / 部门信号 / 部门周复盘
- **个人投影** → 个人能力快照 / 个人周复盘
- **任务投影** → 任务智能简报
- **时间投影** → 本周新增 / 周快照对比

**接口形态**：
```
GET /api/v1/relationship-graph/projection?
    by=department&id=dept_xxx&time_window=this_week
→ 返回该部门本周全部事件 + 主线变化 + 人物活动
```

#### 引擎 B · 事件流引擎

**作用**：把主线状态变化序列化成可消费的事件流。

**用例**：
- 本周概览读"本周事件" → 拿这条引擎的输出
- 周复盘读"本周状态变化" → 同上
- 资方风险信号读"关键人物状态变更" → 同上

#### 引擎 C · 提醒 / 触发引擎

**作用**：后台定期扫描 + 事件触发，主动推送提醒。

**触发种类**：
- **日扫描**：承诺到期、客户冷却、主线超期
- **事件触发**：人物状态变化（离职/休假）、资方信号变化、主线责任人变更

**输出**：写入"提醒队列"表，任务与日程消费。

#### 引擎 D · 关系书生成引擎

**作用**：自动生成 8 段客户关系书 + 段落级钻取。

**关键设计**：
- 不是"AI 一次性写一篇" → 是"每段独立生成 + 独立更新"
- 每段读固定的关系网数据 + 用模板组织语言（不全靠 LLM 编）
- 段落级版本控制（"这段是 5/15 生成的"）

#### 引擎 E · 澄清问题生成引擎

**作用**：扫描关系网缺口 → 自动生成针对性问题。

**生成规则**：
- 主线维度未澄清（交付物模糊、责任人未定）→ 生成主线澄清问题
- 人物归一未确认（多别名未合并）→ 生成人物归一问题
- 矛盾点（同一事实两种说法）→ 生成矛盾澄清问题
- 关系书段落质量低（证据稀薄）→ 生成补料问题

**质量标准**：每个问题自评分必须 ≥ 80（达到我今天问日慈 10 问的水平）。
