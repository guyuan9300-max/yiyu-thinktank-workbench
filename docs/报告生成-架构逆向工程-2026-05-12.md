# 战略陪伴报告生成 · 架构逆向工程

**起草日期：** 2026-05-12  
**基于产物：** `docs/日慈战略陪伴报告-Q1-2026-MOCK.md`  
**目的：** 把模拟报告倒推为数据流水线，回答三个问题：
1. 每段内容从计算中心哪里取数？
2. 自动化程度多高（纯模板 / 模板+LLM / 纯 LLM）？
3. 要不要配生图模型？

---

## Part 1 · 总体架构

```
┌──────────────────────────────────────────────────────────────────┐
│                       报告生成 Pipeline                          │
│                                                                  │
│  ① 收集层 → ② 加工层 → ③ 写作层 → ④ 排版层 → ⑤ 审阅层 → ⑥ 交付层  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                                  │
        ┌─────────────────────────┼─────────────────────────┐
        ↓                         ↓                         ↓
   计算中心数据             LLM 主理人写作           图表 / 配图
   (结构化为主)             (synthesis 模式)         (代码生成为主)
```

每一步的职责：

| 步骤 | 输入 | 输出 | 自动化程度 |
|---|---|---|---|
| ① 收集 | 报告期 + 客户 ID + 报告类型 | DataPack（结构化包） | 100% 全自动 |
| ② 加工 | DataPack | 摘要 / 趋势 / 主题候选 | 90% 全自动 |
| ③ 写作 | 加工产物 + 模板 | 报告 Markdown 初稿 | 70% LLM + 30% 模板 |
| ④ 排版 | Markdown + 图表 | docx/pdf/html 渲染 | 100% 全自动 |
| ⑤ 审阅 | 初稿 | 终稿 | 主理人介入（关键判断） |
| ⑥ 交付 | 终稿 + 元数据 | 客户工作台 + 邮件 + PDF | 100% 全自动 |

**结论先行：** 系统能做到"主理人花 20-30 分钟审一份完整报告"的程度，本身就是从零到 80 分。**完全脱手 100 分的报告不现实也不应该**——战略陪伴的核心价值之一就是主理人判断。

## Part 2 · 数据源映射表（每章节 ← 数据库表 / 服务）

按模拟报告章节顺序：

### 封面 + 元信息

| 内容 | 数据源 | 取数方式 | 备注 |
|---|---|---|---|
| 客户名 / 编号 | `clients` 表 | SELECT by client_id | - |
| 报告期范围 | 参数输入 | 用户指定 | - |
| 编制日期 | now() | 系统时间 | - |
| 编制单位 | 配置 / 当前 user | - | - |
| 报告类型 | 参数 | enum (季度/月度/项目) | - |

**自动化：** 100% 模板填充

---

### 一、本期摘要

| 内容块 | 数据源 | 取数方式 | 自动化 |
|---|---|---|---|
| 核心判断（本期沉淀） | `judgments` 表 (status=official OR confidence>0.9) WHERE created_at IN 报告期 | ORDER BY confidence DESC LIMIT 5-8 | 模板拼接 |
| 关键决定 | `proposal_records` (status=approved) + `judgments` | filter by 报告期 | 模板拼接 |
| 待解决问题 | `openQuestions` （线程记忆）+ `missingContext` 累计 | 去重 + 排序 | 模板拼接 |

**自动化：** 90%。"核心判断"段的过渡文字由 LLM 写。

---

### 二、战略目标进展

| 内容块 | 数据源 | 取数方式 | 自动化 |
|---|---|---|---|
| 战略核心（不变量） | `organization_dna_v2` + `client_profile.strategy_core` | 直接读 profile block | 100% 模板 |
| 双飞轮架构图 | `entities` (type=project) + `relationship_triples` | 关系图算法 | 图表生成（见 Part 4） |
| 四大核心项目矩阵 | `entities` (type=project, client_id=X) + 每个项目的 `event_lines` 最新状态 + `tasks` 未结案 | JOIN + GROUP BY | 模板表格 |
| 关键指标快照 | `metric_snapshots` (待 主动能力 A7 完成) | 期间 SELECT + 对比上期 | 模板表格 |

**关键缺口：** `metric_snapshots` 表当前不存在 — 这是 主动能力 迭代 A7 才会建的。**这意味着报告生成功能本身必须等主动能力 A1+A7 一定程度完成后才能 100% 实现**。但当前可以用定性描述代替。

**自动化：** 80%。每个项目的"状态判断（🟢/🟡/🔴）"需要规则引擎或 LLM。

### 三、关键事件回顾

| 内容块 | 数据源 | 取数方式 | 自动化 |
|---|---|---|---|
| 重要事件时间线 | `event_lines` + `event_line_entries` WHERE 报告期 | 时间排序 | 100% 模板 |
| 重要决策事项 | `proposal_records` (status=approved) + `judgments` (type=official) | 报告期 filter | 模板表格 |
| 风险与卡点 | `meeting_followup.risks` + `tasks` (risk_type) + `data_center_quality.gaps` | 严重度排序 | 模板表格 |

**自动化：** 95%。事件描述如果太短，由 LLM 补全上下文。

---

### 四、战略洞察与判断

| 内容块 | 数据源 | 自动化 |
|---|---|---|
| 当前阶段定位 | `ThreadContextPack.threadSummary` + `confirmedJudgments` 综合 | LLM synthesis |
| 关键不变量 / 关键变量 | `organization_dna_v2.invariants` + `recent_judgments` 对比 | 模板 + LLM |
| 核心判断（如"为何 X 项目战略潜力最高"）| `lastSelectedObject` + `evidence_quality_annotations` | LLM synthesis |
| 判断置信度（⭐⭐⭐⭐⭐） | `evidence_quality_annotations` + `confirmedJudgments[].confidence` 加权 | 算法计算 |
| 待澄清的开放问题 | `openQuestions` + `missingContext` | 100% 取数 |

**自动化：** 60%。**这章是 LLM 主写**，模板只搭骨架，内容靠 synthesis。

---

### 五、核心议题深度分析

| 内容块 | 数据源 | 自动化 |
|---|---|---|
| 议题选择 | `themes` + `topic_candidates` 按优先级 + 用户标记的"本期重点议题" | 算法 + LLM |
| 每个议题内容 | 该议题关联的所有 documents/chunks/judgments | LLM synthesis（长 prompt）|
| 建议结论 | `action_suggestion_service` | LLM + 模板 |

**自动化：** 40%。议题分析是报告里**最难自动化、最有价值**的部分。

**这章必须主理人参与：** 议题选择主理人确认 + 议题分析 LLM 初稿 + 主理人润色结论。

---

### 六、行动建议与下阶段计划

| 内容块 | 数据源 | 自动化 |
|---|---|---|
| 短期行动（带优先级） | `tasks` (open) + `action_suggestion_service` | 模板表格 |
| 中期 Q3 计划 | `event_lines.upcoming_milestones` + 议题分析衍生 | LLM 合成 |
| 长期 H2 计划 | LLM synthesis（基于战略目标 + 当前进展）| LLM 合成 |

**自动化：** 70%。短期行动 95% 自动，中长期 LLM 主写。

### 七、附录

| 内容块 | 数据源 | 自动化 |
|---|---|---|
| 资料来源清单 | `document_card` 表 GROUP BY type | 100% SQL |

| 信任级别标注 | 每段的 metadata + evidence_quality | 算法 |

**自动化：** 95%。

## Part 3 · LLM 角色分工

报告生成需要至少 **3 类 LLM 调用**，扮演不同角色：

### 角色 A：**结构化抽取员**（小模型，本地可跑）

- 输入：原始文档 / 会议纪要
- 输出：结构化 JSON（判断/决策/事件/风险）
- 何时调用：日常 ingest 时（已存在）+ 报告生成前补抽
- 模型：本地 Qwen 7B 或 GPT-4o-mini

### 角色 B：**章节起草员**（中等模型）

- 输入：DataPack（某章节的结构化数据）+ 章节模板
- 输出：该章节的 Markdown 初稿
- 何时调用：每个章节一次
- 模型：豆包 Seed 2.0 Pro 或 Claude Sonnet
- 提示词风格：**给定数据 + 模板 → 用主理人口吻写**

### 角色 C：**全局润色员**（大模型）

- 输入：所有章节初稿 + 报告基调说明
- 输出：润色后的全文 + 段间过渡
- 何时调用：所有章节初稿完成后一次
- 模型：Claude Opus 或 GPT-4
- 提示词风格：**保持事实不变，统一文风，添加段间衔接**

**为什么三层分工？** 单一 LLM 一次性写完一份长报告会出现：
1. 事实漂移（细节记不住）
2. 内部矛盾（前后逻辑打架）
3. 重复啰嗦
4. 风格不稳定

分工让每个角色专注一件事，质量可控。

## Part 4 · 图表生成策略

### 4.1 报告里出现的图表类型

回看模拟报告，需要的图表有：

| 图表 | 类型 | 生成方式 |
|---|---|---|
| 双飞轮架构图（封面/二章）| 概念图 | mermaid graph 或 SVG 模板 |
| 项目进展矩阵 | 表格 | Markdown 表格（无图） |
| 关键指标快照 | 表格 | Markdown 表格 |
| 事件时间线 | 时间线 | mermaid timeline 或 Python matplotlib |
| 风险矩阵 | 散点 / 热力图 | matplotlib / plotly |
| 项目依赖关系图 | 图谱 | mermaid graph |
| 资料来源饼图（附录）| 饼图 | matplotlib |

### 4.2 选型决策

**主要采用：**
- **mermaid**（轻量、文本配置、可在 Markdown 直接嵌）—— 用于：时间线、概念图、关系图
- **matplotlib / plotly**（数据驱动）—— 用于：定量图表（柱状、折线、饼）
- **SVG 模板填空**（高定制）—— 用于：双飞轮架构图、品牌封面背景

**不采用：**
- AI 生图（DALL-E / Midjourney）—— 见 Part 5 详细论证

新建 `backend/app/services/report_chart_generator.py`：

## Part 5 · 生图模型必要性讨论（**关键决策**）

### 5.1 哪些场景"看起来"需要 AI 生图

候选场景：

1. **封面图** — 报告封面要不要做一张精美的 AI 生成图？
2. **章节配图** — 每章开头一张配图？
3. **概念插图** — "双飞轮"这种战略概念能不能让 AI 画一张？
4. **客户 logo / 主理人头像** — 个性化定制？

### 5.2 我的判断：**首期不需要 AI 生图**

**理由 1：核心价值不在装饰**

战略陪伴报告的核心价值是**判断、洞察、行动建议**——客户付费买的是这些。装饰性配图带来的边际价值远低于内容本身。

**理由 2：AI 生图对结构化内容不靠谱**

战略概念图（"双飞轮"）让 DALL-E 画 → 出来的图大概率是不知所云的抽象艺术，反而不如手画 SVG 模板靠谱。

**理由 3：成本与可控性**

- AI 生图 API 单次调用 0.04-0.1 美元，一份报告 10 张图 = 1 美元
- 100 份报告 = 100 美元，规模化后是真金白银
- 而且每次生成不可预测，需要人工筛选，反而拖慢流水线

**理由 4：风格一致性**

AI 生图很难保证多份报告之间风格一致。品牌识别度反而下降。

**理由 5：监管 / 隐私顾虑**

把客户数据描述喂给 DALL-E 等云端 API 有数据外发风险，对战略陪伴这种高敏感场景不合适。

### 5.3 真正需要的是：**一套高质量 SVG 模板库**

替代方案是建立一套品牌化的 SVG 模板：

- 封面模板（含 logo 位、客户名位、报告期位）
- 章节起始页模板
- 双飞轮架构图模板（可填入项目名）
- 关系图模板
- 进度条 / 状态卡片模板

这些一次性设计完毕（让人类设计师做或买现成的），后续每份报告**模板填空**即可。

**成本：** 设计师 1-2 周的工作量，永久使用。  
**质量：** 一致、可控、专业。

### 5.4 何时考虑 AI 生图？

如果未来出现这些场景，可以引入：

- **教育内容生产**（不是报告）：例如生成给孩子的故事书插图
- **传播物料**：公众号封面、海报、营销卡片
- **客户高度定制版**：单次定制版报告做封面图（仅当客户主动要求）

**结论：** 当前**不建议**为战略陪伴报告生成系统配置 AI 生图模型；先把 SVG 模板 + matplotlib + mermaid 三板斧做到位。

## Part 6 · 实施路线

### 6.1 依赖

报告生成功能需要这些底层能力**部分就绪**才能跑：

| 依赖 | 来源线程 | 当前状态 | 影响 |
|---|---|---|---|
| `event_lines` + entries | 已有 | ✅ 可用 | 时间线 / 重要事件 |
| `judgments` 表 | 已有 | ✅ 可用 | 核心判断 |
| `proposal_records` | 已有 | ✅ 可用 | 关键决策 |
| `tasks` | 已有 | ✅ 可用 | 行动项 |
| `document_card` | 已有 | ✅ 可用 | 附录资料清单 |
| `organization_dna_v2` | 已有 | ✅ 可用 | 战略不变量 |
| `entities` | 理解深度 迭代 2 | ❌ 未实施 | 关键人物 + 项目实体 |
| `relationship_triples` | 理解深度 迭代 5 | ❌ 未实施 | 双飞轮关系图 / 项目依赖 |
| `metric_snapshots` | 主动能力 迭代 A7 | ❌ 未实施 | 定量指标快照 |
| `client_glossary` | 理解深度 迭代 7 | ❌ 未实施 | 术语精准 |
| 定时调度器 | 主动能力 迭代 A1 | ❌ 未实施 | 月报 / 季报自动触发 |

**判断：**
- **MVP 报告生成器**可以**现在就做**，不带量化指标 + 不带实体关系图 + 手动触发即可
- **完整版**等理解深度 + 主动能力推进到一定程度（约 2 个月后）

### 6.2 推荐分两期

#### 第一期 · MVP（2-3 周）

**功能：**
- 单客户、单期、手动触发
- 7 个章节都填上内容
- 图表用 mermaid + Markdown 表格
- 主理人审阅后导出 docx + pdf

**实施：**

```
迭代 R1 准备：报告框架定型 + 模板库
迭代 R2 数据收集层：DataPack 生成（按报告期聚合现有所有结构化数据）
迭代 R3 章节起草层：7 个章节的 LLM prompts
迭代 R4 排版导出：Markdown → docx + pdf
迭代 R5 主理人审阅 UI + 一键导出
```

**预算：** 250-350 工具调用

#### 第二期 · 完整版（依赖前置线程）

**新增功能：**
- 定时触发（每季度/每月自动生成）
- 量化指标快照与对比
- 实体关系图自动生成
- 跨客户对比报告
- AI 生图（如果届时确实需要）

**触发时机：** 理解深度 迭代 5 + 主动能力 迭代 A1+A7 完成后

### 6.3 与已规划线程的关系

| 关系 | 说明 |
|---|---|
| 不冲突理解深度线程 | 报告生成是消费者，不修改理解深度的表 |
| 不冲突主动能力线程 | A1 调度器完成后，第二期自动触发可挂载到调度器 |
| 不冲突生产能力线程 | docx 生成已有 python-docx；PDF 等待 P3 生产能力迭代；可优先用 pandoc |

**建议：** 作为**独立第三条线程**起手，分支 `feature/report-generation`。

## Part 7 · 一句话总结

> **报告生成的关键不是 LLM 写得多漂亮，而是把计算中心已经沉淀的结构化判断、事件、决策、风险，按主理人口吻整理成有引用、有边界、有行动建议的可交付物。**

**核心架构选择：**

- **三层 LLM 分工**：抽取员 / 起草员 / 润色员
- **图表用 mermaid + matplotlib + SVG 模板**，**不用 AI 生图**
- **第一期 MVP 不依赖未来线程**，现有数据已经够
- **完全脱手 100% 自动化不是目标**；目标是"主理人 20-30 分钟审一份完整报告"

**关于生图模型的最终建议：**

❌ 不建议在战略陪伴报告生成系统中引入 AI 生图。  
✅ 建议投资一套高质量 SVG 模板库 + matplotlib/mermaid 三板斧。  
✅ 未来如果做教育内容生产或营销物料，再单独考虑 AI 生图。
