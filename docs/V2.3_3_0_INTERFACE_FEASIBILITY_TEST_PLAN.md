# 10 · 益语智库 3.0 接口化可行性测试方案 (第一轮)

> **生成日期**: 2026-05-23
> **来源**: 顾源源 2026-05-23 钦定
> **作用**: 第一轮测试 — 验证 3.0 接口化底座现在有多少已经存在
> **核心目标**: AI 不点界面, 能不能调度软件工作

---

## 一、测试目标

这轮测试只验证一个核心问题:

**用户交给系统一段会议纪要或一批资料后, 大模型能不能通过后端服务/API 调动任务、工作台、智能文件导入、数据中心、战略陪伴等模块, 而不是模拟人去点页面.**

这轮测试要回答:
1. 哪些功能模块已经有可被 AI 调用的服务/API?
2. 哪些能力现在只能通过前端按钮触发?
3. 一个大模型能不能生成"操作计划", 并按计划调用多个模块?
4. 调用后的数据, 能不能进入数据中心并触发重新理解?
5. 每一步有没有上下文、权限、来源、审计和人工确认机制?

---

## 二、测试结论要服务的决策

测试结束后, 不急着让 AI 写代码改系统, 先得到 4 个判断:

### 判断一: 现在能不能做 AI 原生操作层
- 后端有足够多领域服务 → 可以开始做 Agent Gateway / Tool Registry
- 大部分功能绑在前端页面里 → 要先"服务化", 不能直接做 3.0

### 判断二: 第一批 AI 可调度模块选谁
初步判断优先测:
- 任务与日程
- 工作台
- 智能文件导入
- 数据中心
- 战略陪伴

### 判断三: 数据中心能不能支持 AI 真正工作
如果 AI 调了模块, 但只生成 atomic_facts, 没有生成时间线、风险、承诺、判断、澄清, 它还是不能真正接管工作.

### 判断四: 3.0 是否需要重新设计接口层
- 现有模块能被统一封装为工具 → 直接设计 AgentCommandGateway / ToolRegistry / AgentRunLog / ApprovalQueue / DomainServiceAdapter
- 不能 → 先重构模块服务层

---

## 三、测试总原则

### 原则一: 不测试 UI 自动化
不测: 点击/拖文件/操作 modal/浏览器自动化
要测: 后端服务能否直接调用 + API 能否直接调用 + 数据中心能否被写入 + 写入后是否触发 broadcast + 各模块是否能通过接口完成同样动作

### 原则二: 优先真实客户数据, 但不能污染真实业务
- 优先选择真实客户作测试对象
- 写入统一标 test_run_id
- 只允许写入草稿/候选事实/候选任务/候选澄清
- 不允许发送给客户/推送飞书/覆盖权威事实/删除真实资料
- 所有新增内容可回滚

### 原则三: AI 只能通过领域服务写入
不能让 AI 直接写数据库. 必须通过系统已有或临时封装的服务/API.

---

## 四、测试对象: 5 个核心模块

1. **任务与日程**: 创建任务/承诺, 挂客户/事件线, 生成周复盘草稿
2. **工作台**: 写入会议纪要, 抽取客户事实, 保存对话, 生成材料草稿, 提交用户纠错
3. **智能文件导入**: AI 发起导入会话, 增加讲述片段, 识别文件角色, 抽取人物/关系/时间/承诺/风险
4. **数据中心**: 作 AI 共享内存, 承接各模块写入 → 触发事实抽取/来源登记/置信度记录/broadcast/自校验/语义派生
5. **战略陪伴**: AI 触发重新理解, 生成 6 段叙事/主动研判/事实澄清问题

## 七、实际要执行的测试步骤

### 第一步: 静态排查接口
扫描代码, 列每模块现有可调用能力. 输出表格 (模块 / 已有服务/API / 是否能脱离 UI / 是否写数据 / 是否需确认 / 当前缺口)

### 第二步: 建立临时 Tool Registry
每个工具描述: tool_name / module / input_schema / output_schema / required_context / writes_to / approval_required / audit_required / idempotency_key / rollback_support / current_status

### 第三步: 执行 Headless 调用测试
不通过前端页面, 测试脚本直接调用后端服务/API, 完成 8 步会议纪要处理闭环.

### 第四步: 检查数据库变化
执行前后对比 11 张关键表 (chat_messages / source_registry / atomic_facts / event_line_activities / risk_signals / commitments / tasks / clarification_records / strategic_thought_insights / agent_runs / agent_tool_calls)

### 第五步: 检查用户可感知结果
生成执行摘要 (新增几条候选事实/风险/澄清/任务草稿 + AI 主动提醒)

## 八、通过标准 (100 分制)

| 维度 | 分值 | 满分标准 |
|---|---|---|
| 接口可调用性 | 20 | 5 模块至少 4 个能脱离 UI 调用 |
| 操作计划质量 | 15 | 拆 5+ 模块动作 + 区分自动/确认 |
| 多模块调度能力 | 20 | 至少调动 4 模块 + 每次有上下文 |
| 数据中心升级质量 | 20 | 不只写 atomic_facts, 派生事件/风险/承诺/澄清/判断 |
| 人工确认与安全边界 | 15 | 正式任务发布需确认 + 跨客户隔离 + audit log |
| 用户可感知价值 | 10 | 至少 1 条有价值澄清 + 1 真实风险 + 1 任务草稿 |

---

## 九、硬门槛 (任一失败 = 不能进下一阶段)

1. **AI 不能直接写数据库**
2. **没有上下文绑定就不能写入** (org_id/user_id/client_id/project_id/source_id/permission_scope 全必填)
3. **不能只写 atomic_facts** (必须派生事件/风险/承诺/澄清/战略判断)
4. **危险动作必须进入确认队列** (正式任务/权威更新/对外发送/飞书推送/覆盖旧版本)
5. **必须有 Agent Run Log** (AI 计划 + tool 输入输出 + 是否成功 + 写入数据 + 错误回滚)

## 十三、测试结束后的 4 种情况分支

- **情况一**: 大部分模块已有后端服务 → 下一步做 Agent Gateway/Tool Registry/Agent Run Log/Approval Queue/会议纪要处理闭环
- **情况二**: 模块有 API, 但写入只进 atomic_facts → 先做语义派生器 (风险/承诺/时间线/判断) + 澄清持久化 + 故事卡重建
- **情况三**: 很多能力只能前端触发 → 先做功能服务化 + 前端按钮改调用领域服务 + AI 与前端共用服务
- **情况四**: 安全边界不清 → 先做上下文绑定 + 权限检查 + 人工确认队列 + 审计日志 + 回滚机制

---

## 一句话总结

这轮测试不是测 AI 有多聪明, 而是测益语智库现在有没有资格被 AI 操控.

如果它能在不点界面的情况下, 把一段会议纪要转成客户事实/风险/承诺/澄清问题/任务草稿和战略刷新, 说明 3.0 的接口化方向成立.

如果做不到, 我们根据测试结果反推: 到底是模块没有服务化, 还是数据中心不能派生语义, 还是缺少权限/审计/人工确认.
