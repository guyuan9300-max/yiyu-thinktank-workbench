# AI 超级助理澄清与记忆系统设计规格

## 1. 页面与系统定位

这不是一个“给任务多加一个备注框”的功能，而是一套：

- AI 学习业务资料的主链路
- AI 在执行任务时主动补齐未知信息的澄清链路
- 澄清结果沉淀为可复用组织记忆的总池
- 后续事件线判断、周总结、风险预测的背景母盘

产品人格设定：

- 像一个名校毕业、非常能打的总裁助理
- 平时靠已有资料学习业务
- 在领导布置任务时，顺手问最关键的澄清问题
- 问过一次后写进自己的知识底盘
- 后面不再反复问同样的问题
- 随着任务、项目、事件线推进，逐渐更懂老板、更懂业务

一句话目标：

`让 AI 既能靠资料主动学习，也能在任务执行时补齐未知，并把这些补齐结果沉成长期可复用的组织记忆。`

## 2. 核心设计原则

### 2.1 不增加重复人工负担

- 不能要求用户每个任务都重新解释背景
- 不能让用户在多个地方重复回答同一个问题
- 不能为了让 AI 更聪明，反过来让人不断填表

### 2.2 先复用，后追问

AI 每次分析任务或事件线时，优先顺序必须是：

1. 先读已有资料
2. 先读历史澄清
3. 先读事件线和项目已有上下文
4. 只有关键槽位缺失时，才发起澄清

### 2.3 澄清结果不是任务私有备注

用户回答的内容，不应该只挂在当前任务里。

必须根据语义写回对应层级：

- 客户/项目层
- 联系人/关键人层
- 产品/解决方案层
- 事件线层
- 任务层

### 2.4 问题可以多条，回答只保留一个输入口

AI 可以列出 1-5 个短问题，但用户只需要：

- 在一个统一输入框里
- 用一段自然语言回答

系统负责把回答拆解成结构化记忆。

### 2.5 问过一次就进入总池

例如：

- 日慈是什么机构
- 张真是谁
- 张真与益语上一轮对话是哪条线
- 益语的软件具体是哪一个产品

这些一旦澄清过，后面同类任务应直接复用，不再重复询问。

### 2.6 每个组织都应该有一本“业务笔记”

可以把 AI 想成一个学霸型总裁助理。

它学习每一个客户/组织时，不应该只是临时读几份资料，而应该像每个学科都有一本长期积累的笔记一样，为每一个组织持续形成一本会不断生长的“业务笔记”。

这本笔记不是一篇固定文档，而是一套围绕该组织持续积累的结构化记忆，包含：

- 这个组织是什么
- 它主要做什么
- 它有哪些特点
- 它正在面临什么困境
- 它和益语是怎么合作的
- 当前有哪些事件线在共同推进
- 近期发生了哪些关键任务、会议、附件、决策和判断

原则上：

`只要任何功能模块捕捉到与这个组织有关的新事实，这些事实都应该自动回流到同一本业务笔记里。`

后续任务卡、事件线、周总结、风险判断、建议动作，都应该优先把这本组织业务笔记作为思考背景之一。

## 3. 这套系统要解决什么问题

### 3.1 当前问题

当前周总结、任务卡、事件线摘要虽然已经开始读：

- projectContext
- eventLineContext
- orgContext

但仍然缺少更深层的业务关系理解，例如：

- 日慈是什么组织
- 跟益语当前合作关系是什么
- 张真是谁
- 张真在这家机构里的角色是什么
- 这次任务接的是张真的哪一轮对话
- 益语的软件是哪一个产品
- 这个产品和当前客户组织的关系是什么

于是 AI 只能停留在：

- 罗列任务
- 泛化解释
- 宽泛判断

而无法进入真正的“业务助理式”洞察。

### 3.2 目标状态

当系统看到任务：

`给日慈张真看益语的系统`

它应该尽可能自动知道：

- 客户：日慈基金会
- 联系人：张真
- 联系人角色：项目负责人 / 决策参与者 / 使用方
- 关联事件线：张真看系统 / 合作推进线
- 上一轮互动：上次演示 / 上次会议 / 上次决策点
- 产品对象：益语某个系统 / 某个模块
- 当前目的：推进合作、验证需求、确认交付、收束判断

如果仍缺关键信息，再只问最必要的问题。

## 4. 系统总体架构

整套系统拆成四层：

### A. 资料学习层

AI 通过现有资料理解业务：

- 客户工作台资料
- 会议纪要
- 任务与附件
- 项目模块与流程
- 组织 DNA
- 周计划 / 季度重点

### B. 澄清提问层

当 AI 判断：

- 关键背景缺失
- 但缺失程度已经影响判断质量

则在任务详情里发起轻量澄清。

### C. 结构化记忆层

用户回答后，系统不只是存一句话，而是写入：

- ClientMemory
- PersonMemory
- ProductMemory
- EventLineMemory
- TaskClarification

### D. 复用生成层

后续所有地方优先读这层记忆：

- 任务卡
- 周总结
- 风险判断
- 建议动作
- 事件线详情
- PDF 证据线

在这四层之上，再补一个横向概念：

### 组织业务笔记层（Notebook Layer）

这不是一个单独页面，而是一层统一底盘。

它负责把同一个客户/组织相关的所有信息不断收束成一份长期可复用的背景系统：

- 客户工作台资料
- 客户画像
- 联系人画像
- 产品关系
- 事件线
- 会议与决策
- 任务与附件
- AI 澄清记录

它的作用类似于：

`某个组织的持续学习笔记 + 协作发展档案 + AI 思考底稿`

## 4.1 组织业务笔记的核心内容

建议围绕每个客户/组织形成一个统一的 `Organization Notebook` / `Client Notebook`，最少包含：

- `组织介绍`
  - 这是一家什么样的组织
  - 它主要做什么
- `组织特点`
  - 决策风格
  - 工作风格
  - 合作节奏
- `组织困境`
  - 当前主要困难
  - 当前最急需解决的问题
- `合作关系`
  - 和益语的合作方式
  - 当前合作阶段
  - 当前合作目标
- `关键人`
  - 谁是谁
  - 各自扮演什么角色
- `正在推进的事件线`
  - 当前有哪些主要连续事项在推进
- `关键证据`
  - 会议
  - 附件
  - 决策
  - 周判断
- `AI 学到的澄清结果`
  - 问过什么
  - 答过什么
  - 哪些内容已经稳定可复用

## 4.2 组织业务笔记如何自动生长

这本笔记不能靠人工维护整页长文，而是通过系统自动回流增量生长。

只要以下模块里发生与某个组织相关的事实，就应该自动写回：

- 任务创建/更新
- 任务附件上传
- 事件线更新
- 会议发起与纪要
- 支持请求
- 周总结与周判断
- 手工澄清
- 产品关系补充

这意味着：

- 任务不是孤立的
- 会议不是孤立的
- 附件不是孤立的
- 周总结也不是孤立的

它们都只是“组织业务笔记”的不同输入来源。

## 5. 新增的核心对象

### 5.1 ClarificationRecord

用于记录 AI 问了什么、用户答了什么、最终写回哪里。

建议字段：

- `id`
- `sourceTaskId`
- `sourceEventLineId`
- `sourceClientId`
- `scopeType`
  - `client`
  - `person`
  - `product`
  - `event_line`
  - `task`
- `scopeId`
- `fieldKey`
  - `organization_intro`
  - `relationship_status`
  - `person_role`
  - `previous_interaction_line`
  - `product_mapping`
  - `current_goal`
  - `current_blocker`
  - `next_action`
- `question`
- `answer`
- `normalizedAnswer`
- `confidence`
- `status`
  - `open`
  - `answered`
  - `applied`
  - `stale`
- `createdAt`
- `updatedAt`
- `lastUsedAt`

### 5.2 Client Memory

补客户/项目层长期背景。

建议字段：

- `organizationType`
- `organizationIntro`
- `cooperationRelationship`
- `currentCooperationStage`
- `currentNeeds`
- `decisionStyle`
- `keyContacts`
- `relatedProducts`
- `lastImportantInteraction`

这层对象本质上就是组织业务笔记的核心骨架之一。

### 5.3 Person Memory

补联系人/关键人层背景。

建议字段：

- `name`
- `clientId`
- `roleTitle`
- `relationshipSummary`
- `influenceLevel`
- `lastInteractionSummary`
- `linkedEventLineIds`
- `linkedMeetingIds`
- `notes`

### 5.4 Product Memory

补产品/解决方案层背景。

建议字段：

- `productName`
- `moduleName`
- `useCase`
- `fitClients`
- `clientRelationSummary`
- `currentRelevance`
- `linkedTaskIds`
- `linkedEventLineIds`

### 5.5 Event Line Memory

补连续工作线层的记忆。

建议字段：

- `summary`
- `intent`
- `stage`
- `currentFocus`
- `currentBlocker`
- `nextStep`
- `recentDecision`
- `linkedPeople`
- `linkedProducts`
- `linkedMeetings`
- `linkedAttachments`

这层对象负责把“组织笔记”中的连续推进事项串起来，避免所有内容只堆在客户总背景里。

## 6. AI 什么时候发起澄清

不是所有任务都问。

只有满足这两个条件才问：

### 条件一：关键槽位缺失

例如当前任务需要判断，但以下槽位缺失：

- 客户组织身份
- 合作关系
- 联系人角色
- 对应产品/系统
- 上一轮互动线索
- 当前这次任务的业务目的

### 条件二：缺失已经影响判断质量

例如 AI 发现：

- 只能说“看系统”
- 不能说清“为什么看、给谁看、接哪条线、有什么管理意义”

这时才值得问。

## 7. 问题如何生成

### 7.1 问题生成原则

问题必须：

- 具体
- 简短
- 针对缺口
- 一次不超过 5 个

不能问：

- 宏大空泛问题
- 用户已经答过的问题
- 通过现有资料能推断出来的问题

### 7.2 问题样例

针对：

`给日慈张真看益语的系统`

可生成：

- 日慈目前是什么类型的机构？
- 张真在日慈里负责什么？
- 这次看系统接的是上一轮哪条推进线？
- 这次看的具体是哪一个益语产品或模块？
- 这次看的目标是判断合作、推进交付，还是验证需求？

## 8. 前台交互设计

### 8.1 任务详情中的区块

新增一个统一区块：

`待澄清背景`

展示方式：

- 上面列出 1-5 个短问题
- 中间显示“AI 为什么问这些问题”的一句话
- 下面只保留一个统一输入框
- 用户可直接自然语言回答

例如：

> AI 还缺少这条任务对应的客户关系、关键人角色和上一轮互动背景。补齐后，后续任务卡、事件线摘要和周总结会更准确。

输入框标题：

`用一段话补充这些背景`

### 8.2 回答后的行为

用户提交后：

1. 当前任务立即能读到这段回答
2. 系统后台自动拆解
3. 写回对应对象
4. 这条澄清标记为 `applied`
5. 后续同类任务默认复用

## 9. 回答如何自动拆解

系统不能把整段回答原样塞进一个备注字段。

必须做最小结构化拆解：

### 拆解目标

- 哪部分属于客户背景
- 哪部分属于联系人画像
- 哪部分属于产品关系
- 哪部分属于事件线背景
- 哪部分只属于当前任务

### 例子

回答：

“日慈是做儿童心理健康的公益机构，张真是他们项目负责人。上次跟她主要在聊把益语的任务系统和陪伴式项目管理结合起来，这次看的是益语内部的任务与周判断系统，主要是想看看是否适合做他们项目推进的底盘。”

应拆成：

- `client.organizationIntro`
  - 日慈是做儿童心理健康的公益机构
- `person.roleTitle`
  - 张真是项目负责人
- `eventLine.previousInteraction`
  - 上次主要在聊把益语的任务系统和陪伴式项目管理结合起来
- `product.productMapping`
  - 这次看的是益语内部的任务与周判断系统
- `eventLine.currentGoal`
  - 看是否适合做他们项目推进的底盘

## 10. 总池去重与复用规则

### 10.1 不重复问

发问前必须先查：

- 同一客户下是否已有相同字段
- 同一联系人是否已有角色描述
- 同一事件线是否已有上一轮互动
- 同一产品是否已有关系说明

如果已有，则直接复用，不问。

### 10.2 近似问题归并

例如：

- “张真是谁？”
- “张真在日慈里负责什么？”
- “张真是什么角色？”

应归到同一个澄清槽位：

- `person_role`

### 10.3 过期重问

以下情况允许重问：

- 超过 90 天未更新
- 合作阶段变化
- 联系人角色变化
- AI 对当前答案置信度低

重问方式应是：

- `已有信息可能变化，是否需要更新？`

而不是从头重复问。

## 11. AI 如何像总裁助理一样成长

这套系统的长期目标不是“更会写总结”，而是：

### 第一阶段

会识别自己不知道什么。

### 第二阶段

会在合适时机只问最必要的问题。

### 第三阶段

会把问题答案沉进长期可复用记忆。

### 第四阶段

以后看到类似任务时，不再重复问，而是直接带着背景工作。

### 第五阶段

逐渐形成：

- 客户画像
- 关键人画像
- 产品关系画像
- 事件线连续上下文
- 管理判断历史

这时候它才真正像一个“老板越用越顺手的超级助理”。

## 12. 与现有系统的关系

这套方案不是另起炉灶，而是站在现有底座上扩展：

- `ClientWorkspace`
  - 作为客户/项目背景总入口
- `EventLine`
  - 作为连续上下文主线
- `Task`
  - 作为澄清触发点
- `Weekly Review`
  - 作为复用这些背景做高质量洞察的输出层

同时新增一个统一理解：

- `ClientWorkspace`
  不只是资料库页面，而应该逐步升级成该组织的“业务笔记壳层”
- `EventLine`
  不是标签，而是组织业务笔记里的连续事项主线
- `ClarificationRecord`
  不是聊天记录，而是给这本业务笔记持续补知识的问答链路

所以未来所有模块都要遵守一个共同原则：

`底层数据、底层文件、底层记忆完全打通，共用、复用，并持续反哺同一本组织业务笔记。`

当前系统已经有：

- `manual_clarification` 作为证据来源概念
- `event_line` 对象
- `projectContext / eventLineContext`

这次要补的是：

- 真正的澄清记录对象
- 联系人与产品关系对象
- 澄清答案写回总池
- 去重与不重复发问机制

## 13. 分期实施

### P1

- 新增 `ClarificationRecord`
- 任务详情增加 `待澄清背景` 区块
- 一组问题 + 一个统一输入框
- 回答先写回：
  - `client`
  - `event_line`
  - `task`

### P2

- 新增 `Person / Contact`
- 新增 `Product / Solution`
- 回答可自动拆到联系人和产品关系
- 周总结开始优先读取这些对象

### P3

- 做总池去重
- 做不重复发问
- 做过期重问
- 做跨任务 / 跨事件线 / 跨项目复用

## 14. 最终判断标准

这套功能是否成功，不看“问了多少问题”，而看：

- AI 是否更少重复发问
- AI 是否开始知道客户、关键人、产品和关系链
- 任务卡是否更像真正的判断，而不是摘要
- 周总结是否开始出现“这条关系线 / 这条合作线 / 这条推进线”的深层洞察
- 老板是否感觉这个系统越来越像“真正懂业务的总裁助理”
- 系统是否真的开始为每一个客户/组织形成一本会持续生长、跨模块复用的业务笔记

一句话收口：

`不是让 AI 变成一个总爱追问的系统，而是让它像一个会学习、会补齐未知、会长期记住答案的总裁助理。`
