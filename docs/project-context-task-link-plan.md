# 项目背景源接入任务系统实施方案

> 对齐说明（2026-03-21）：
> 本文档继续作为“项目上下文链”的专项设计使用，但现在应放在
> [全局模块改造清单与阶段路线图](./global-module-refactor-roadmap.md) 的框架下理解。
> 它的核心职责已收口为：
> `项目上下文 -> 任务识别 -> 周判断 -> 成长解释`。
> 后续实现时，应优先减少人工补录，改为从既有项目资料、会议、任务痕迹中自动生成候选上下文。

## 目标

把当前益语智库里的“客户工作台”升级成任务系统与周总结系统可消费的“项目背景源”。

对益语智库：
- 一个客户，通常就是一个项目
- 任务创建时应自动识别对应客户/项目
- 一旦识别成功，任务应自动继承该项目的背景、阶段、目标与风险语境

对通用产品：
- 底层概念不写死为 `客户`
- 统一抽象为 `业务对象 / 项目对象`
- 益语前台可继续显示为“关联客户”

## 一、项目对象最小字段表

第一阶段不追求复杂 CRM，只做 AI 可消费的最小字段。

### 1. 基础标识
- `id`
- `name`
- `aliases`
- `displayType`
  - 益语当前取值：`client`
  - 通用版可取值：`project`
- `active`

### 2. 背景摘要
- `mission`
  - 一句话说明该项目为什么存在
- `backgroundSummary`
  - 3 到 5 句短摘要，供 AI 读取
- `annualIntent`
  - 年度合作/年度目标
- `quarterFocus`
  - 当前季度重点，最多 3 条

### 3. 阶段与目标
- `currentStage`
  - 例如：诊断期 / 对齐期 / 方案期 / 实施期 / 交付期 / 复盘期
- `keyGoals`
  - 当前最关键的 3 条目标
- `keyRisks`
  - 当前主要阻力
- `informationCompleteness`
  - 资料完整度：`low / medium / high`

### 4. 协作上下文
- `relatedDepartments`
- `ownerUserId`
- `keyStakeholders`
- `workflowHints`
  - 本项目通常怎么推进，输出物是什么

### 5. AI 判断辅助
- `recognitionKeywords`
  - 客户全称、简称、别名、品牌词、项目词
- `stageKeywords`
  - 识别阶段的关键词
- `goalKeywords`
  - 目标相关关键词
- `riskKeywords`
  - 风险相关关键词

## 二、任务识别项目的判定规则

原则：
- 先规则识别，后续再升级为大模型加权判断
- 默认自动识别，始终允许手动调整
- 识别成功后，保存 `clientId / businessObjectId`

### 1. 第一优先级：强匹配
- 标题中出现客户全名
- 标题中出现客户简称
- 标题或说明中出现已配置别名
- 任务来源本身已带 `clientId`
  - 客户工作台里创建
  - 会议发布成任务
  - 目标发布成任务

### 2. 第二优先级：上下文匹配
- 当前用户正在某个客户工作台上下文里创建任务
- 当前任务由某个客户的会议纪要、action item 派生
- 当前任务由某个客户目标卡片派生

### 3. 第三优先级：关键词推断
- 说明中出现该客户独有业务词
- 说明中出现该客户项目名、品牌词、关键联系人
- 说明中出现该客户当前季度重点关键词

### 4. 第四优先级：历史相似任务回推
- 同一用户近期相似标题任务已绑定同一客户
- 同一部门近期大量同类任务挂在某个客户下
- 同一会议线程或目标线程衍生任务已挂同一客户

### 5. 置信度策略
- `high`
  - 全名/ID 命中
- `medium`
  - 别名 + 上下文命中
- `low`
  - 仅关键词弱匹配

规则：
- `high`：直接预填
- `medium`：预填，但显示“系统识别”
- `low`：只给建议，不直接覆盖

## 三、客户工作台到任务系统的背景注入链

### 1. 注入入口

当任务满足以下任一条件时，触发项目背景注入：
- 已明确绑定 `clientId`
- 在客户工作台页面创建
- 来源于某个客户会议
- 来源于某个客户目标
- 标题/说明识别出高置信度客户

### 2. 注入内容

后台生成一个轻量 `taskProjectContext`，不要求员工手填。

字段建议：
- `clientId`
- `clientName`
- `stage`
- `goalSummary`
- `riskSummary`
- `departmentHints`
- `backgroundSummary`
- `sourceEvidence`
  - 例如：`title_match / meeting_source / goal_source / workspace_context`

### 3. 使用位置

#### 任务编辑弹窗
- 自动填充 `关联客户`
- 优先级建议
- 负责人/协作者建议
- 提示“当前项目阶段”

#### 任务列表 / 我的月历
- 不大面积展示背景
- 仅在详情或 hover 里显示：
  - 客户/项目
  - 当前阶段
  - 项目重点

#### 组织上下文判断
- 结合 `orgContext`
- 形成：
  - 任务属于哪个部门重点
  - 是否跨部门
  - 是否需要复核
  - 是否偏离岗位职责

## 四、接入周总结引擎的方式

### 1. 个人总结

个人总结不再只看“做了什么任务”，而看：
- 本周围绕哪些项目推进
- 哪些项目推进到了下一阶段
- 哪些项目阻力来自资料、协作、方向、资源或过载
- 已完成任务沉淀了什么心得

输出改造：
- `项目推进摘要`
- `可能性分析`
- `下周关注重点`
- `建议动作`

### 2. 部门总结

部门总结新增一层“项目维度聚合”：
- 本周部门重点项目有哪些
- 哪些任务真正服务了这些项目
- 哪些项目推进符合部门计划
- 哪些项目偏离计划或卡在协作链

AI 判断重点：
- 项目推进是否和部门周计划一致
- 项目卡点是流程问题、权限问题还是资源问题
- 哪些项目具备复用/标杆潜力

### 3. 机构总结

机构层不再只看任务总数，而看：
- 哪些重点项目在推进机构季度重点
- 哪些部门围绕项目推进出现重复性问题
- 哪些项目需要 CEO 介入

AI 判断重点：
- 部门推进是否服务机构目标
- 当前最重要的项目级风险是什么
- 哪些项目值得高层继续押注

## 五、实施阶段

### P1：识别与绑定
- 任务新增稳定的 `clientId`
- 任务弹窗支持“自动识别客户 + 手动调整”
- 规则识别器接入：
  - 标题
  - 说明
  - 当前客户工作台上下文
  - 任务来源

验收：
- 能自动识别高频客户名
- 能保存手动调整结果

### P2：背景注入
- 本地代理从客户工作台拉取：
  - `goals`
  - `dnaModules`
  - `meetings`
  - `relatedTasks`
  - `recentMessages`
- 生成轻量 `taskProjectContext`
- 用于：
  - 优先级建议
  - 协作者建议
  - 风险提示

验收：
- 同一任务在编辑弹窗里能看到项目背景建议
- 不增加员工手填负担

### P3：总结引擎升级
- 周总结读入 `taskProjectContext`
- 部门总结按“项目聚合”输出
- 机构总结按“项目对机构目标贡献”输出

验收：
- 个人总结可按项目聚合
- 部门总结不再只是任务堆砌
- CEO 总结能指出重点项目与机构季度重点的关系

## 六、必须注意的设计原则

- 员工端永远轻，不让员工手动填客户背景
- 默认自动识别，永远允许人工校正
- 先做规则识别，再做大模型增强
- 底层用 `业务对象 / 项目对象` 建模，益语前台继续显示“客户”
- 任务系统、周总结系统和客户工作台必须共用同一条 ID 链，不要再做摘要孤岛

## 七、下一步开发清单

1. 给任务对象补稳定的项目背景上下文字段
2. 做本地代理的客户背景摘要生成器
3. 把客户工作台来源任务的 `clientId` 透传彻底统一
4. 在任务编辑弹窗里增加项目识别置信度提示
5. 在周总结引擎里新增“项目推进聚合器”
