## 二 · 测试找到的 4 个数据中心真漏洞

**核心发现**:数据进了 `atomic_facts`(第 3 层 extraction),但没派生到顶层语义表(第 5/6 层 story_graph/narrative)。

### 漏洞 1 · 时间线派生缺失

| 项目 | 现状 |
|---|---|
| `atomic_facts.time_anchor` | 33 条都有 ISO 时间 |
| `event_line_activities` | **0 条** ❌ |
| 故事卡段 4 时间线 | `_(无时间线活动)_` |

**根因**: ingest 写完 atomic_facts 没派生到 event_line_activities。
**注**: B AI 已在 `scripts/run_event_line_activities_extraction.py` 启动这条链路(并行)。

### 漏洞 2 · 关键判断接入缺失

| 项目 | 现状 |
|---|---|
| `atomic_facts` 含我方判断 | "不建议扩到 10 校" / "扩张过快导致服务质量失控" 等 |
| `strategic_thought_insights` | **0 条** ❌ |
| 故事卡段 6 关键判断 | `_(无关键判断记录)_` |

**根因**: 用户判断 / 我方判断只到 atomic_facts,没升级为顶层 insight。

### 漏洞 3 · 风险信号派生缺失

| 项目 | 现状 |
|---|---|
| `atomic_facts` attribute=风险 | 3 条(扩张过快/服务质量失控/宣传压力) |
| `risk_signals` 表 | **0 条** ❌ |
| 故事卡段 8 风险 | `_(无活跃风险信号)_` |

**根因**: ingest 不识别 attribute='风险' 自动派生到 risk_signals。

### 漏洞 4 · 待办/承诺派生缺失

| 项目 | 现状 |
|---|---|
| `atomic_facts` attribute=承诺 | "李明 5/10 前给 3 校名单" |
| `commitments` / `tasks` | **0 条** ❌ |
| 故事卡段 9 下一步 | `_(无待办)_` |

**根因**: atomic_facts attribute='承诺/下一步' 没自动写到 commitments。

## 三 · V2.4 P0 数据派生层(补 4 个漏洞)

蓝图 § 七 第 5 层 story_graph / 第 6 层 narrative 的真实施。新增 1 个服务:

```
backend/app/services/atomic_fact_derivation.py
  · derive_event_line_activities(client_id) — atomic_facts → event_line_activities
  · derive_strategic_insights(client_id)    — 判断/观察 → strategic_thought_insights
  · derive_risk_signals(client_id)          — attribute=风险 → risk_signals
  · derive_commitments(client_id)           — attribute=承诺/下一步 → commitments
  · derive_all(client_id)                   — 一次跑全部
```

**触发时机**: IngestPipeline.ingest() 写完 atomic_facts 后自动调 derive_all(broadcast 一样)。

**预期收益**: 跑完后再跑青禾测试 → D4 应从 60% → 90%+(10/10 段有内容),总分 → 92+。

**成本**: 1 个新文件 ~200 行,5 个 derive 函数,无外部依赖,~2 小时。
**风险**: 派生表可能有现存数据(主仓库),需先 dry_run。

## 四 · 主仓库 → V2.1 反向同步路线

顾源源 5/23 新方向:**V2.1 测试完替代主仓库**,不再 cherry-pick 回主仓库。

### 评估 3 条路线

| 路线 | 操作 | 成本 | 风险 |
|---|---|---|---|
| A. V2.1 全量 push 替代主仓库 | git remote set + force push | 5 min | 主仓库 5/19—5/23 改动丢 |
| B. 主仓库代码 pull 到 V2.1,V2.1 在上面跑 | git remote add main + merge | 30 min | merge conflict 多 |
| **C. 主仓库分支 cherry-pick 到 V2.1**(推荐) | 反向同步主仓库 commits | 1-2 小时 | 可控 |

**推荐 C**:
1. V2.1 加 `git remote add main_repo <主仓库>` (fetch only)
2. `git fetch main_repo main`
3. `git log V2.1...main_repo/main` 看主仓库 5/19 后哪些 commit V2.1 没有
4. 主仓库 commit 逐条 review + cherry-pick
5. V2.1 跑完整测试套件(青禾 + 现有 286+ tests)
6. 测试通过 → V2.1 push 到 main_repo 的新分支 `v2.1-promoted`
7. 顾源源拍板 → main_repo 切主分支为 v2.1-promoted

**关键约束**:
- 反向同步过程中,**V2.1 改动不再 cherry-pick 回主仓库**(永久红线翻转)
- 主仓库 5/19—5/23 期间另一 AI 改动需审过再吸收
