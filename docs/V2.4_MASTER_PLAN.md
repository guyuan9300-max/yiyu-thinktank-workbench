## 二 · V2.4 优化路径(6 阶段)

### 阶段 1 · 原子事实 → 语义表派生链路(P0,最高优先级)

完成"原始资料 → 原子事实 → **语义表** → 项目故事卡 → 澄清问题 → 下一步行动"中**第三步**。

```
atomic_facts 中的内容            →  应派生到                    →  用户最终看到
─────────────────────────────────────────────────────────────────
日期/会议/方案版本/客户动作       →  event_line_activities       →  项目时间线
风险/担忧/卡点/质量失控          →  risk_signals                →  风险清单
承诺/责任人/截止时间             →  commitments / tasks         →  下一步行动
我方判断/用户判断/AI 判断        →  strategic_thought_insights  →  关键判断
人物/角色/拍板人/执行人          →  entities / relationship     →  人物关系
金额/范围/版本冲突              →  fact_contradictions / clar  →  待澄清问题
```

**验收**: 青禾测试 4 张空表全部填充,故事卡 10 段全部有内容。

### 阶段 2 · 冲突识别从"测试后补"变成"数据中心自己会做"(P0)

正式冲突检测器接 Ingest/Broadcast 路径,**真实持久化**:

1. 同主体同属性不同值(300 万 vs 500 万)
2. 同项目不同范围(3 所 vs 10 所)
3. 同人物不同角色(负责人 vs 执行推进)
4. 外部口径滞后(媒体 vs 内部新版)
5. 用户口述但缺正式证据(陈老师拍板)
6. 版本链检测(v2 covers v1)

**验收**: `fact_contradictions ≥ 5`, `clarification_records ≥ 5`,每条带来源+建议澄清+影响。

### 阶段 3 · 时间标准化层(P0)

每个时间事实 3 套字段:
- `time_anchor_iso` → 排序(`2026-05-06T00:00:00+00:00`)
- `time_anchor_text` → 原文(`5 月 6 日`)
- `time_aliases` → 可匹配(`["2026-05-06","5月6日","5/6"]`)

支持: 中文/ISO/相对时间(上周/上次会议/5 月中旬)/同日多事件排序/发生 vs 导入时间分开。

**验收**: Q36(按时间线复述)/ Q39(哪几天有外部信息)能正确回答并排序。

### 阶段 4 · 故事卡升级为"项目理解报告"(P0)

10 段 每段都要有:
- 内容
- 来源
- 把握度(confidence)
- 是否待确认
- 是否来自用户口述
- 是否来自客户正式文件
- 是否存在冲突

**验收(业务向)**: 新同事不看原始资料,只看故事卡,5 分钟内能说清"项目怎么来的、现在在哪、谁关键、风险在哪、下一步问谁"。

### 阶段 5 · 用户纠错真正成为"权威值更新"(P1)

用户说"500 万是旧版,现在 300 万"后,系统应:

```
1. 新增 user_confirmed_fact = 300 万
2. 旧 500 万事实 → status = superseded
3. confidence_history 记录 user_correct 触发
4. broadcast_data_changed 触发下游
5. 重新生成故事卡
6. 重新生成战略陪伴 6 段叙事
7. 重新计算冲突
8. 后续问答 → 300 万
```

**验收**: 纠错后再问预算,系统明确说"当前 300 万 / 500 万为旧版+外部 / 对外引用需确认外部口径"。

### 阶段 6 · 飞书集成(P3,严格边界)

数据中心没稳定前,**不做**飞书自由问答。**只做**通知/提醒类:
- 周复盘提醒 / 逾期任务提醒 / 待澄清提醒
- 故事卡生成完成提醒
- 飞书妙记显式导入 / 资料导入复核卡片

**禁止**: 飞书群里自由问客户知识 / 自动全量同步飞书云盘 / 飞书直接写入客户事实 / 把故事卡完整发到群里。所有入口必须显式绑定 `org/user/client/project`。

## 四 · 测试方式优化(9 层分层测试)

不再一个总分。改成**门禁 + 分数**。

### 9 层测试

```
L1 数据入口测试      (source_type/role/scope 正确率)
L2 真实抽取测试      (LLM 从原文抽事实, 不喂预标)
L3 语义派生测试      (atomic_facts → 语义表持久化)
L4 冲突与澄清测试    (5 个核心冲突全持久化 + 高质量澄清问题)
L5 时间线测试        (中文/ISO 双向问答)
L6 故事卡测试        (10 段 + 每段 source/confidence)
L7 真实问答测试      (A 数据库确定性 + B LLM 受限)
L8 用户纠错回写测试  (supersede + broadcast + 重新生成)
L9 跨客户隔离测试    (同名人物不串, 飞书无 client_id 不能问)
```

### 门禁(任一失败 = 假设不成立)

```
① 关键事实没有来源
② 冲突没有持久化
③ 用户纠错不能更新权威值
④ 故事卡有 3 段以上为空
⑤ 时间线不能按中文日期回答
⑥ LLM 回答出现严重幻觉
⑦ 跨客户数据串线
⑧ 飞书入口无 client_id 仍能访问客户知识
```

### 分数(版本比较用,100 分)

```
数据分型     10
真实抽取     15
语义派生     20
冲突澄清     20
时间线       10
故事卡       15
受限问答     10
```

(剔除"预标 fact 喂入"的虚高分,真实抽取与语义派生权重最高)
