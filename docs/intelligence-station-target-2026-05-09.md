# 资讯情报站目标设计与底层核查记录

记录日期：2026-05-09

## 一、目标前端形态

资讯情报站作为工作台的外部信息入口，不直接展示“算法”，而展示已经加工后的内容。页面默认让用户看到两类内容：资料补全和时效情报；再提供一个稳定的用户反馈入口：我重点关注什么。

### 1. 顶部全局筛选区

顶部固定为全局筛选器：

- 工作对象：默认「全部工作对象」，可切换到具体客户或项目。客户和项目在这里统一称为工作对象。
- 排序：支持「最新发现优先」「发布时间由近到远」「发布时间由远到近」「相关度优先」。
- 时间范围：默认「近 30 天」，可切换「今天」「近 7 天」「近 90 天」「全部」。
- 搜索：可搜内容、来源、标签。
- 更新状态：显示最近更新时间和本次新增数量，例如「最近更新：今天 09:18，资料补全新增 12 条，时效情报新增 3 条」。
- 立即刷新：只刷新当前筛选范围内的工作对象。

工作对象下拉示例：

- 全部工作对象
- 广东省日慈公益基金会
- 云南儿童资助研究
- 益语市场
- 大山里的音乐课堂
- 益语智库平台

### 2. 我重点关注什么

该区域位于内容标签页上方，是用户直接告诉系统需求的入口，类似自定义指令。

默认展示为一条紧凑横幅：

```text
我重点关注什么
全局：优先找资助窗口、合作方动态、青少年心理健康政策；少看泛泛行业新闻。
日慈：重点关注儿童心理健康、家庭支持、广东省及珠三角公益合作机会。
[编辑关注重点]
```

点击「编辑关注重点」后打开右侧抽屉：

- 生效范围：
  - 全部工作对象
  - 指定工作对象，可多选
- 资料补全优先：
  - 例如：补齐机构登记信息、历史项目、合作方、公开负责人、项目成果。
- 时效情报优先：
  - 例如：资助申报、政策窗口、招标采购、监管变化、合作方动态、同类机构动作。
- 少看或不看：
  - 例如：无明确来源的转载、泛泛新闻稿、与对象地区无关的信息。
- 保存后影响下一轮抓取、筛选和排序。

### 3. 内容标签页

内容区采用标签页：

```text
[资料补全 128] [时效情报 7]
```

两个标签页各自拥有分页器，不混排。

### 4. 资料补全标签页

每页 10 条。页面说明语：

```text
这些内容已自动核验并进入数据中心。发现不实、重复、过期或无关时，可以点“不采纳”。
```

资料补全卡片是精简卡，用于快速核验，不要求用户逐条处理。

卡片结构：

- 工作对象
- 标签
- 提炼后的关键内容
- 来源与发布时间
- 自动核验提示
- 操作：查看来源、不采纳

模拟内容：

```text
广东省日慈公益基金会
标签：机构登记 / 基金会基础资料 / 广东
关键内容：公开登记信息显示，该基金会业务范围包含扶贫济困、助学、助医及公益慈善项目支持，可作为客户基础画像的稳定事实。
来源：广东省社会组织信息网，发布时间：2026-04-18
状态：已自动核验并写入数据中心
[查看来源] [不采纳]
```

```text
广东省日慈公益基金会
标签：历史项目 / 儿童心理健康 / 家庭支持
关键内容：公开项目介绍提到“心灵魔法学院”面向儿童及家庭开展心理支持服务，可补充客户服务对象、项目方向和案例素材。
来源：机构公众号，发布时间：2026-03-29
状态：已自动核验并写入数据中心
[查看来源] [不采纳]
```

```text
云南儿童资助研究
标签：区域案例 / 困境儿童 / 社会工作服务
关键内容：某县域儿童服务中心公开披露了留守儿童陪伴、家庭探访和学校协作服务流程，可作为项目方案中的相似案例参考。
来源：地方民政局官网，发布时间：2026-04-07
状态：已自动核验并写入数据中心
[查看来源] [不采纳]
```

不采纳原因：

- 不实
- 无关
- 重复
- 过期
- 来源不可靠

### 5. 时效情报标签页

每页 5 条。页面说明语：

```text
这些内容可能影响客户、项目或近期行动，适合阅读后追问、转任务或关注后续。
```

时效情报卡片是深度卡，用于让用户判断是否跟进。

卡片结构：

- 情报类型
- 关联工作对象
- 标题
- 顾问判断
- 为什么推给你
- 证据
- 还需要核验
- 建议动作
- 操作：追问、转任务、关注后续、查看来源

模拟内容：

```text
政策窗口
关联对象：广东省日慈公益基金会
标题：广东省拟征集社会组织参与未成年人心理支持服务试点
顾问判断：这条政策信号与日慈的儿童心理健康服务方向强相关，适合先核验试点地区、承接主体条件和申报时间。
为什么推给你：命中“儿童心理健康”“社会组织参与”“广东省”三个关注点。
证据：省级公开征求意见稿提到建立未成年人心理支持社会协同机制。
还需要核验：正式通知是否发布、是否允许基金会或合作机构参与、是否有区县试点名额。
建议动作：关注后续，并建立一个本周核验任务。
[追问] [转任务] [关注后续] [查看来源]
```

```text
资助机会
关联对象：云南儿童资助研究
标题：某公益基金会开放县域儿童心理韧性项目小额资助计划
顾问判断：该资助方向与县域儿童支持、心理韧性和家庭陪伴高度接近，但需要确认是否接受研究型或陪伴型项目申请。
为什么推给你：命中“儿童”“县域”“心理韧性”“资助机会”。
证据：基金会官网发布项目征集公告，资助周期为 6-12 个月。
还需要核验：地域限制、预算上限、是否要求本地注册主体。
建议动作：转为任务，先核验申报资格，再决定是否深度分析。
[追问] [转任务] [关注后续] [查看来源]
```

```text
行业风险
关联对象：益语智库平台
标题：多地开始加强公益项目数据采集与未成年人信息保护要求
顾问判断：这不是直接机会，但会影响儿童服务项目的数据采集、评估问卷和数字化工具设计。
为什么推给你：命中“未成年人信息保护”“公益项目数据”“数字化工具”。
证据：多个地方公开文件提到未成年人相关数据采集需最小必要、单独授权和留痕管理。
还需要核验：哪些要求已经落到广东、云南等重点区域。
建议动作：关注后续，并纳入产品合规待办。
[追问] [转任务] [关注后续] [查看来源]
```

### 6. 抓取频率与呈现数量

默认频率建议：

- 新建工作对象后：立即进行一轮资料补全抓取。
- 新对象前 7 天：资料补全每天 1 次。
- 稳定对象：资料补全每周 2 次，权威来源检测到变化时增量抓取。
- 时效情报：工作日默认每天 2 次，建议 09:00 和 16:00。
- 高关注对象或高关注议题：时效情报每天 3 次。
- 政策、招标、资助类来源：适合高频增量检测。
- 舆情、同类机构动态、合作方动态：默认每天 1 次。

呈现数量规则：

- 资料补全量大，每页 10 条；同一来源同类信息过多时自动合并摘要。
- 时效情报量小，每页 5 条；第一页只放最值得处理的信号。
- 若某日时效情报超过 5 条，优先按工作对象关注重点、截止时间、影响程度排序。

## 二、为达到目标前端所需的底层能力

### 1. 工作对象层

底层需要把客户、项目和其他可被情报绑定的对象统一抽象为「工作对象」。

每个工作对象至少应提供：

- 对象 ID
- 对象名称
- 对象类型：客户、项目、组织内部项目等
- 当前状态
- 关键词与别名
- 地域、服务对象、议题标签
- 与任务、事件线、客户画像、数据中心资料的关联关系

前端筛选器只关心工作对象，底层负责把不同表里的客户和项目合并成统一候选项。

### 2. 用户关注指令层

底层需要保存用户明确写下的关注重点，并区分生效范围：

- 全局关注指令
- 指定工作对象关注指令
- 资料补全优先项
- 时效情报优先项
- 少看或不看
- 更新时间与来源用户

这些指令不是普通备注，而应进入抓取、筛选、排序、生成分析和后续反馈的共同上下文。

### 3. 内容供给层

底层需要持续抓取外部公开信息，并把来源分层管理：

- 权威来源：政府、民政、招标采购、基金会官网、客户官网等。
- 机构来源：公众号、官网新闻、项目介绍、公开年报等。
- 媒体与行业来源：报道、行业平台、公益资讯平台等。
- 舆情与动态来源：搜索结果、社交平台、同类机构动态等。

抓取任务需要支持：

- 按工作对象触发
- 按来源类型触发
- 按关注指令触发
- 定时抓取
- 手动刷新
- 增量检测
- 失败记录和重试

### 4. 自动核验与入库层

资料补全不是普通候选内容。只要展示在资料补全页，就意味着已经被自动核验并写入数据中心。

底层需要具备：

- 来源可靠性判断
- 多来源或强来源核验
- 去重与过期判断
- 事实抽取
- 标签生成
- 写入数据中心
- 写入证据链
- 可被「不采纳」撤回或标记失效

资料补全卡片不再依赖固定「写入位置」，而是依赖标签供后续复用。

### 5. 时效情报加工层

时效情报需要从外部信号加工成顾问式判断。

每条情报至少应具备：

- 情报类型：政策、招标、资助、舆情、风险、合作方动态、竞品动作等
- 关联工作对象
- 标题
- 摘要
- 顾问判断
- 为什么推给你
- 证据
- 还需要核验
- 建议动作
- 截止时间或关注窗口
- 关注状态
- 来源列表
- 后续追踪规则

### 6. 推荐与排序层

前端不展示算法，但底层必须产生排序依据：

- 工作对象匹配
- 用户关注指令匹配
- 内容类型匹配
- 来源可靠性
- 发布时间与抓取时间
- 截止时间紧迫度
- 是否已被用户不采纳
- 是否已转任务或关注后续
- 同类内容是否过多

排序结果应支持资料补全和时效情报分别分页。

### 7. 反馈层

用户操作需要回流到底层：

- 资料补全：不采纳，并记录原因。
- 时效情报：追问、转任务、关注后续、查看来源。
- 关注指令：编辑后影响后续抓取和排序。

反馈不只是 UI 状态，应影响：

- 后续抓取关键词
- 来源权重
- 内容类型权重
- 对象关注重点
- 数据中心资料有效性
- 后续情报追踪任务

### 8. API 与前端视图层

前端至少需要这些接口能力：

- 获取工作对象筛选列表
- 获取关注指令
- 保存关注指令
- 获取资料补全分页列表
- 资料补全不采纳
- 获取时效情报分页列表
- 时效情报追问
- 时效情报转任务
- 时效情报关注后续
- 手动刷新当前筛选范围
- 获取抓取状态与最近更新时间

## 三、代码与数据现状核查

核查时间：2026-05-09

### 1. 已有且可直接复用的部分

- `clients` 与 `project_modules` 已能表达客户和项目对象，可作为统一「工作对象」列表的基础来源。
- `intelligence_profiles` 已经具备 scope_type、scope_id、client_id、project_module_id、priority_needs_json、material_gaps_json、grounding_facts_json 等字段，可承接对象级情报画像。
- `fetch_topic_candidates_from_web` 已支持搜索、首选来源、AI shortlist、时间过滤，并且函数签名里已有 search_intents 参数。
- `topic_source_fetcher.py` 的历史订阅和第三方抓取入口已在 2026-05-11 清理；后续内容供给层只保留普通公开网页与中文公开搜索适配。
- `AiService.build_topic_candidate_insight` 已能生成三段式顾问 memo：情报速览、情报研判、行动建议。这部分和时效情报深度卡片方向一致。
- `POST /api/v1/topics/candidates/{candidate_id}/chat` 已能支持围绕单条情报追问。
- `POST /api/v1/topics/candidates/{candidate_id}/task-plan` 与 `promote-tasks` 已能支持把情报转任务。
- `external_evidence_cards` 已有 source_url、source_tier、tags_json、related_scope_type、related_scope_id、status、review_note 等字段，适合改造成资料补全卡片的证据记录基础。
- `data_center_ingest_events` 已有 source_type、source_id、client_id、project_module_id、lifecycle_status、document_id、metadata_json 等字段，可承接「已入数据中心」和「不采纳后失效」。
- `task_understanding_cache` 已保存任务理解结果，可作为未来需求信号的一类输入。

### 2. 已有但明显有偏差、需要修改的部分

- 当前主接口仍是 `/api/v1/topics`，返回 radars、candidates、intelligenceProfiles 的旧组合；没有面向新前端的「工作对象筛选」「资料补全分页」「时效情报分页」「更新时间摘要」接口。
- 当前主前端是 `TopicsManagementView` 的「外部情报顾问简报」形态，最多取 3 条精选判断；没有资料补全 / 时效情报两个标签页，没有各自分页器，也没有全局工作对象筛选。
- 当前页面里的「顾问正在理解的对象」仍然是画像卡片，不是用户说的「我重点关注什么」指令区。
- `intelligence_profiles.admin_focus_json / admin_exclude_terms_json / admin_priority_urls_json` 可以复用，但现在是管理员画像配置，不是普通用户可编辑、可区分全局和指定工作对象的关注指令。
- `intelligence_profiles.search_intents_json` 和测试里的 search intents 设计还在，但当前源码没有 `AiService.build_intelligence_profile`，也没有 `/api/v1/intelligence/profiles/bootstrap`；画像生成链路断了。
- `fetch_topic_candidates_from_web` 能接收 search_intents，但当前 `capture_topic_radar_internal` 只传 radar title、prompt、time_range、preferred sources，没有把 profile search intents 传进去。
- 运行库里的 `topic_candidates` 已有 content_kind、scope_type、scope_id、client_id、why_recommended、match_strength、credibility_score、confidence_score、matched_intent_json、external_evidence_card_id、evidence_status、data_center_ingest_event_id 等字段；但当前源码的表初始化和 `create_topic_candidate_record` 不写这些字段，`build_topic_candidate` 也基本不返回这些字段。
- 运行库里有 `topic_candidate_items`、`topic_fetch_jobs`、`topic_source_configs`，但当前源码不创建、不管理这些表；这些更像历史链路残留或未合并完成的设计。
- `external_evidence_cards` 的 accept/reject 目前只改证据卡本身，当前源码没有同步更新 topic_candidates.evidence_status，也没有把 data_center_ingest_events lifecycle_status 改成 rejected。
- `data_center_ingest.py` 的 SUPPORTED_SOURCE_TYPES 不包含 `external_intelligence`，但运行库里已经有 external_intelligence 记录；这说明外部情报入数据中心的路径和通用 ingest 层没有完全对齐。
- `run_due_intelligence_profiles` 当前直接返回 0，画像刷新和定时抓取频率字段存在，但没有真实调度执行。
- `trial-run` 当前也只是返回「当前没有新的可精选公开情报」，没有真实抓取。
- 当前追问和转任务的上下文注入仍偏薄，`organization_context` 为空，和「顾问 memo prompt 强、上下文注入弱」的判断一致。
- 当前没有资料补全和时效情报的明确分流逻辑；content_kind 虽然在运行库存在，但源码没有稳定生成和消费。
- 当前没有「关注后续」这一操作的持久化模型，最多能转任务或删除情报。
- 当前没有支持时间排序、工作对象筛选、每页 10 / 5 条分页的后端查询。

### 3. 基本无用、冲突或会误导新设计的部分

- 旧版 `legacy_features/topics` 里的收藏、共享、状态筛选、画像管理、证据通过/不采用等 UI 很多，但它已经不是当前主入口，而且交互复杂度和新设计的轻量标签页不一致，只能局部借鉴，不宜整体搬回。
- 用户可见的「雷达」概念与新设计冲突。新前端不应让用户管理算法或雷达，而应让用户看资料补全、看时效情报、写关注重点。
- 当前 demo/sample 数据里存在孤儿客户 ID，例如日慈样本曾引用 `client_rici_foundation`，运行库真实 clients 里并没有该 ID；这会误导数据中心入库、证据卡和任务关联。
- 「南沙区公益创投」示例已经过度使用，且当前运行库中仍作为 candidate 存在但 client_id 不匹配，不适合作为后续前端预期样例。
- 当前测试文件里保留了较完整的新链路规格，例如 `/api/v1/intelligence/items`、`/api/v1/intelligence/radars/bootstrap`、source-config、fetch-jobs、candidate-items、external_intelligence 入库与证据状态回写；但当前源码缺少对应实现。它们更适合作为找回目标能力的参考规格，不能当作现有可运行能力。

### 4. 最小验证结果

本轮只跑了与画像链路相关的最小测试：

```text
uv run pytest tests/test_api_smoke.py -q -k "intelligence_profiles_bootstrap_system_radars_and_capture_with_search_intents or intelligence_profiles_admin_overrides_custom_profile_and_trial_run or intelligence_radars_bootstrap"
```

## 四、对新前端的底层补全优先级

### P0：先补结构，不急着做漂亮前端

- 新增统一工作对象 API。
- 新增用户关注指令模型，支持全局和指定工作对象。
- 新增资料补全 / 时效情报两类列表 API，支持工作对象、时间范围、排序、分页。
- 明确 content_kind：profile_completion 与 timely_intelligence。
- 让 topic_candidates 或新表稳定保存 scope、标签、推荐理由、证据状态、数据中心事件 ID。

### P1：补内容供给链路

- 恢复或重建 intelligence profile bootstrap。
- 让画像 search intents 进入抓取函数。
- 接入 source configs，并记录 fetch jobs。
- 把抓取结果先落入 candidate_items，再分流为资料补全或时效情报。
- 为资料补全建立自动核验和数据中心写入路径。

### P2：补反馈闭环

- 资料补全不采纳应同时更新证据卡、候选内容、数据中心生命周期。
- 时效情报关注后续应生成跟踪规则，而不只是 UI 状态。
- 追问、转任务、关注后续、不采纳、编辑关注重点都应回流到后续排序和抓取。

### P3：再做前端替换

- 用新 API 替换当前顾问简报页。
- 保留现有追问、转任务的可用部分。
- 删除或迁移用户不可见的雷达管理入口。
- 旧版复杂收藏/共享/状态筛选仅作为 legacy 保留，不进入新主流程。
