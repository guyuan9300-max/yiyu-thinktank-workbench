### 墓碑（被推翻，禁区，留死因）

- **T1（死因：纸面机制≠真隔离）**：原断言"外部内容/机器人产出会无感污染事实层（因消费点裸查、核心表无 tier 字段）"——被 R2 真实数据推翻。真相见 F7：靠入口闸隔离，今天无实际污染。**但**消费侧确无 tier 兜底（R1 属实），故残留**条件性风险**：若新建一条让 bot 产出**自动直写/升格进被消费事实表**的干道，就会污染。→ 结论降级为"隔离靠入口闸；机器人产出必须沿用草稿+人工升格闸，禁止新建直写干道"。
- **T2（死因：找错了指派入口）**：原断言"用户完全无法 UI 派单给 bot"——被 R2 推翻（F3：走 AICommandModal 而非普通指派下拉）。
- **T3（死因：把'没接活'与'不做业务写'混为一谈）**：原断言"MCP 只 dry-run 没接活后端"——被 R1 部分推翻（F8：真连活后端，但只读/审计）。"先走 HTTP 干道做写闭环"的实务建议仍成立。
- **T4（死因：'完全没手册'夸大）**：原断言"无机器人手册"——降级（F9：有 5 份契约+动态源，缺的是 app 内 surface + 一键落地）。

### 收敛后的真根因（连续 2 轮未翻）

**功能做不成的真根因不是"缺几个端点/缺手册"，而是：系统是 100% 内部-bot 单执行模式——bot 被派单后由本地 `plan_executor` 抢着执行，没有"外部 agent 自己在沙箱外干、轮询认领、回填结果"这第二种执行模式。** 具体四个断点（F4/F5/F6）：① execute_plan 不分内外、会本地执行外部 agent 的 plan 使其形同虚设；② 无 awaiting_external 挂起态；③ 无 submit-result 端点；④ 完成不回写人看任务（_auto_complete_task 未接）。其余（身份/token/任务看板/ingest 漏斗/手册/MCP 读干道）都已就绪可复用。

### 待验/暂信

- 「200-300 行可成」：暂信→需 执行+1 验证。读/身份/看板/ingest 干道可复用属实，但"外部执行模式"（origin/runner 语义+挂起态+submit-result+回写闭环+UI 外部预设+skill 落地）是净新增管道，非纯接线。
- 「F7 的'无污染'成色」：R2 数据取自 **测试 db**（`backend/output/org-dna-heavy-chain-test/app-copy.db`），非真 256MB userData 库。结论对本功能不依赖此（靠 code 级入口闸 + 草稿闸），但若日后判"线上是否已有外部数据"需在真库复核。
