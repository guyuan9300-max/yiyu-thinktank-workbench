## 三 · 客观结论

### ✅ 真兑现的能力

1. **LLM 真能从原文抽取** — 12/12 datum 都抽到了关键人物/数字/事件,无 timeout
2. **0 幻觉** — 所有数字都在原文里,无任何编造
3. **承诺/角色/日期/金额识别准确** — 例如 A1 抽到 '李明 · 承诺 = 5月10日前提供 3 所试点学校名单' 
4. **速度可用** — 平均 3.5s/datum,12 datum 全跑完 ~45s

### ⚠️ 暴露的真问题

**LLM 抽取角度跟 expected_facts 不一致**(语义对、字符不对):

| expected (我标的) | LLM 抽的 (qwen2.5:14b) | 评分 |
|---|---|---|
| `项目 · 对外公开日 = 2026-05-03` | `测试机构C · 发布日期 = 2026 年 5 月 3 日` | 不算召回 (subject 角度不同) |
| `项目 · 媒体口径学校数 = 10 所` | `测试机构C · 范围 = 10 所乡村学校` | 不算召回 |
| `项目 · 当前权威预算 = 300 万元` | `预算 · 我方判断 = 300万` | 不算召回 |

**这不是 LLM 抽得不好,是测试 expected 跟 LLM 抽取的主语粒度不一致**。LLM 倾向把 '预算/范围' 当独立 subject,我标的 expected 倾向把项目当 subject。

### 🎯 P1 阶段真发现

- LLM 抽取**无幻觉** (P1 第 1 硬指标 PASS)
- LLM 抽取**有速度** (45s 跑完 12 datum, 可在线)
- LLM 抽取**有覆盖** (12/12 datum 都抽到内容)
- 评分函数和 expected 标注**仍然依赖人工** — 严格自动化召回率不准

### 下一步建议

**不要继续调评分函数追召回率**(over-engineering, 过拟合 expected)。

应该做的:
1. **prompt 工程**: 让 LLM 标 subject 时优先用项目名(而非动词/属性)
2. **抽取后 normalization**: 把 LLM 输出的 'subject=预算' 归一化为 'subject=项目, attribute=预算'
3. **人工抽样 review**: 50% 随机 datum, 人判断 LLM 抽对了几条
4. **P1-6 LLM 受限问答**: 让 LLM 用 evidence 回答, 检查是否引用源 / 是否说不知道
