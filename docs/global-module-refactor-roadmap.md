# 全局模块改造清单与阶段路线图

## 1. 目标

基于新的统一原则，重新定义整个软件的改造重点：

- 后台数据首先服务 AI 思考背景，不首先服务表单完整度
- 系统优先使用已有资料、预置计划、上传原始材料和过程痕迹
- AI 先生成候选判断，人类只做确认、修改、澄清
- 总结的核心目标是洞察，不是罗列

一句话收口：

**把系统从“模块各自聪明”改造成“共享背景母盘 + 执行痕迹理解 + 洞察生成”的统一系统。**

## 2. 模块分级判断

## 2.1 必须重构的核心模块

### 组织启动中心 / 组织搭建中心

定位：

- AI 根背景母盘
- CEO 与部门负责人接力的低频母体层

必须承接：

- 组织 DNA
- 年度战略
- 机构季度计划
- 部门季度承接

必须避免：

- 继续做成普通设置页
- 让 CEO 反复补重表单

核心改造方向：

- 上传资料优先
- 自动结构化优先
- 人工确认优先于人工重写

### 任务与日历

定位：

- 高频执行入口
- 月度计划浅入口
- 过程痕迹采集器
- 周判断系统

必须承接：

- 部门月度计划
- 总结前缺失提醒
- 任务、会议、日历痕迹
- AI 判断、可能性分析、建议动作

必须避免：

- 变成第二个重型后台
- 为总结额外制造填写成本

核心改造方向：

- 先读上游背景，再读执行痕迹
- 输出隐形问题、未来风险、机会，不只输出周报

### 项目 / 客户上下文底座

定位：

- 各模块共享的项目语境层

必须承接：

- organization_profile
- project_profile
- growth_rubric
- evidence_map
- project_match

必须避免：

- 继续让任务、成长、情报只读模块内局部字段

核心改造方向：

- 任务自动识别项目
- AI 默认先读项目上下文，再做判断

## 2.2 明显完善，但不先推倒的模块

### 成长手册 / 成长系统

定位：

- 从“手填成长记录器”升级成“成长解释器”

必须增强：

- 读取任务痕迹
- 读取项目背景
- 读取部门 / 组织计划
- 读取成长标尺

必须避免：

- 继续靠员工额外写长成长总结驱动

### 战略陪伴 / 客户工作台

定位：

- 项目背景源
- 后续供任务、总结、成长共用

必须增强：

- 当前阶段
- 当前目标
- 当前阻力
- 风险摘要
- 关键会议与行动项
- 识别关键词

必须避免：

- 只服务客户问答，不向其他模块供背景

### 资讯情报站

定位：

- 信号发现器
- 背景增强器

必须增强：

- 情报服务哪个项目 / 部门 / 季度目标
- 是风险、机会还是判断补强
- 可转任务、转会议、转成长沉淀

必须避免：

- 只做抓取与候选堆积

## 2.3 需要收口边界的模块

### 系统设置

保留职责：

- 身份
- 权限
- AI 配置
- 飞书配置
- 更新配置

不再承担：

- 高频业务补录
- 月度计划
- 总结逻辑

### 测试工作台

保留职责：

- 模板验证
- 分析试验
- 规则测试

不建议承担：

- 正式业务入口
- 高频业务流

### 全局助理

定位：

- 第二阶段以后的流程纠偏器与最佳路径导航器

当前处理：

- 暂不进入主实现
- 等模块通信、动作入口、上下文层稳定后再接入

## 3. 全局上最该补的 4 条连接

### 3.1 组织背景连接

目标：

- 组织背景 -> 年度战略 -> 季度计划 -> 部门计划 -> 月度执行

如果没有这条链：

- AI 永远只有宏观介绍，没有执行语境

### 3.2 项目上下文连接

目标：

- 项目上下文 -> 任务识别 -> 周判断 -> 成长解释

如果没有这条链：

- 任务、总结、成长都会和真实业务对象脱节

### 3.3 痕迹理解连接

目标：

- 任务 / 会议 / 日历痕迹 -> 风险 / 机会 / 洞察

如果没有这条链：

- 总结永远只能罗列事实

### 3.4 自动候选连接

目标：

- 系统自动生成候选 -> 人类确认 / 修改 -> 反写背景层

如果没有这条链：

- 一线员工会继续背负高输入成本

## 4. 分阶段路线图

## P1：AI 背景母盘阶段

核心目标：

- 把长期、低频、根层背景立住

主模块：

- 组织启动中心
- 项目 / 客户上下文底座

重点事项：

- 组织 DNA 结构化
- 年度战略上传与四季草稿
- 部门季度承接
- 项目上下文字段标准化
- 项目与任务绑定设计定型

本阶段不做：

- 月度计划主入口
- 高频流程沉淀入口
- 全局助理

验收标准：

- AI 在组织层和项目层不再裸奔
- 人只做轻量确认，不再重填根背景

## P2：执行浅入口与周判断阶段

核心目标：

- 让高频执行入口变浅
- 让总结升级为周判断

主模块：

- 任务与日历

重点事项：

- 部门月度计划浅入口
- 顶部计划状态条
- 总结前缺失提醒
- 任务与日历痕迹信号化
- 周判断四段式页面
- 从计划和痕迹自动生成候选判断

本阶段不做：

- 让员工补更多总结材料
- 把任务页做成重型后台

验收标准：

- 任务与日历能基于背景和痕迹给出更具体的建议
- 周判断可以输出风险、机会和动作，而不是流水账

## P3：洞察、成长与沉淀联动阶段

核心目标：

- 让 AI 跨模块共享判断结果

主模块：

- 成长系统
- 资讯情报站
- 全局助理

重点事项：

- 成长信号自动解释
- 情报与项目 / 部门 / 季度目标联动
- 隐性风险与机会提示
- 流程、模板、经验自动候选沉淀
- 全局助理基于稳定动作入口做纠偏

验收标准：

- AI 开始真正输出跨模块洞察
- 人工补录成本继续下降

## 5. 各角色的入口分配

### CEO / 组织管理员

主入口：

- 组织启动中心

负责：

- 组织 DNA
- 年度战略
- 季度主线
- 部门负责人

### 部门负责人

主入口：

- 组织启动中心 + 任务与日历

负责：

- 部门季度承接
- 部门月度计划
- 团队执行调整

### 普通员工

主入口：

- 任务与日历

负责：

- 高频执行
- 轻量确认
- 流程 / 模板 / 经验沉淀

## 6. 后续设计的硬性判断问题

以后设计任何字段、页面或流程前，都先问：

1. 这是否显著增强 AI 的思考背景？
2. 这能否先从已有资料和过程痕迹自动生成？
3. 能否让人只做确认，不做重填？
4. 这次输出的目标是洞察，还是又回到了罗列？
5. 这个入口应该放深还是放浅？

如果不能回答这些问题，就不应该直接往软件里继续加表单和按钮。
