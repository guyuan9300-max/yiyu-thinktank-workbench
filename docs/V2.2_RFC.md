## 0 · 一页摘要

**v2.2 不是 v2.1 的延续，是方向调整后的新规划。**

| 维度 | v2.1（已搁置） | v2.2（本规划） |
|---|---|---|
| 北极星 | 代码模块化 / SQL 清零 | **目标 A 现有功能不掉链 + 目标 B 机器人能拿全部数据流畅回答** |
| 验收门 | "main.py 1063 → 0"、"模块边界 0 违规" | **5/19 张真会议 4 个面板自动一致 + 机器人能复述全部决定** |
| 时间承诺 | 6 周（被低估） | **10-12 周 + 接受局部完成** |
| 方法论 | 自创 acceptance runner | **Anthropic 长任务 harness + Karpathy 生成-验证循环** |
| 业务并行 | 混在 v2.1 分支（已被污染） | **物理隔离：业务回 main / 架构留 v2.2-arch-v2 新分支** |

**核心策略**：分 3 阶段交付，每阶段 4-6 周，每阶段结束跑双重校验门，**任何阶段做完都可独立发 release**。

### 1.3 三个最致命的发现

1. **前端债比后端还重**：6 大 view 各自直接调 150+ API，App.tsx 60+ useState 全局耦合。L2 建好后前端有同等工作量重新接入。
2. **L3 处理层占 70% 代码**：194 个 services 大多数是 L3，重构难度的真正来源不是 1063 处 SQL。
3. **手机端是现成的 L2 参考样板**：cloud_backend 28 个 endpoint 已经是 facade pattern，桌面端 v2.2 改造**直接照搬 mobile 的 endpoint shape** 是最快路径。

### 1.4 跨视图数据不一致已经在出 bug

- `EventLineReportPanel` 改 task → `App.tsx` 的 tasks 数组不同步
- `IntelligenceStationView` dismiss 一条 → 别处无感知
- `client_dna_documents` 模块文本 vs `atomic_facts` 行 → 同一份事实两个版本

**这就是 B 方案 #9"多池并列"在生产代码里的具体表现，是 v2.2 必修的核心病。**

## 2 · 用什么方法论（Anthropic + Karpathy）

### 2.1 Anthropic 长任务 harness 5 大 pattern（直接套用）

| Pattern | 在 v2.2 里怎么落 |
|---|---|
| **Initializer / Coding agent 分工** | 本 RFC = initializer 产出（feature_list + progress + protocol 文件）。之后每 session 都是 coding agent 推进具体 feature。 |
| **Planner / Generator / Evaluator 三角** | 每 session 必须明确扮演哪个角色。Planner 写 spec → Generator 实现 → Evaluator 跑双校验门。**不许同一 session 既写代码又自我审查。** |
| **Context Reset > Compaction** | session 长到 50% context 时强制 reset（不是 compact）。用文件传递状态。 |
| **状态用文件不用 context** | `V2.2_FEATURE_LIST.json`（机器可读）+ `V2.2_PROGRESS.md`（人可读）+ git log = 三层状态快照 |
| **强约束防过早完成** | NORTH_STAR.md 写死："双校验门未过 = 里程碑未完成，不许打 release tag" |

### 2.2 Karpathy Software 3.0 核心（思想指导）

- **Demo = works.any(), Product = works.all()**：每个 feature 必须用真实用户场景验收（不是单元测试 PASS 就行）
- **Generation-Verification Loop 加紧**：AI 生成代码 + 立即用 5/19 案例验证 + 反馈再生成。**不要一次写一周代码再验证**。
- **Autonomy Slider**：v2.2 早期保守（每个 commit 都人审），后期可放松到"跑通双校验门 = 自动接受"
- **llms.txt 思想**：项目根加 `LLMS.md` 文件，专给未来 AI session 看（不是给人看）

## 4 · 5 层架构在 v2.2 的落地路径

回应顾源源画的 5 层（L1 数据 / L2 共识 ★★★ / L3 处理 / L4 认知 / L5 呈现）：

| 层 | v2.2 计划 | Phase | 完成度承诺 |
|---|---|---|---|
| **L1 数据层** | 加 2-3 张新表（key_decisions / org_events / event_line_state_changes）+ **AI Memory 5 表占位**（N3 预留） | P2 | 90% |
| **L2 共识层 ★★★** | **建 ClientFactView + ClientScopeFilter + 状态字段唯一真相 + 5 维元数据 schema** | P1 | 80% |
| **L3 处理层** | 建 **IngestPipeline（4 主路径统一入口）** + ExtractionRunner 统一 ingest，其他 service 暂不动 | P2 | 40% |
| **L4 认知层** | NarrativeKernel 事件驱动 + 5 narrative 合并，引用按 5 维元数据规则 R1-R5 | P3 | 60% |
| **L5 呈现层** | 6 大 view 重写 fetch 层接 L2，**不强求 App.tsx 拆分** + 手机端朋友式聊天澄清 | P1+P3 | 50% |
| **L6 AI 记忆层**（3.0 预留） | 5 张表占位 + 单向写入（无读取逻辑） | P2 | 占位 100% / 功能 0% |

**未达 100% 是有意的**——v2.2 不追求"全层做完"，只追求"三校验门过"。剩余推 v2.3 / 3.0 慢慢补。

**4 主路径在 5 层中的位置**：

```
路径 1 (工作台文件) ─┐
路径 2 (任务/复盘)  ─┼──► L3 IngestPipeline ──► L2 ClientFactView ──► L4 NarrativeKernel ──► L5 view
路径 3 (互联网爬虫) ─┤                            (5 维元数据)            (R1-R5 规则)
路径 4 (手机 AI)    ─┘
                                  └──► L1 atomic_facts + event_log
                                                  └──► L6 ai_episode_log (3.0 预留)
```

## 5 · Planner / Generator / Evaluator 三角分工

每个 session 开始时**必须明确扮演哪个角色**，不允许同一 session 全包。

| 角色 | 职责 | 产出 | 不允许做的事 |
|---|---|---|---|
| **Planner** | 把里程碑拆成可执行 feature spec | `docs/V2.2_PHASE1_SPEC.md` 等 | 不写代码 |
| **Generator** | 按 spec 写代码 + 单测 | commit | 不审查自己 / 不跑双校验门 |
| **Evaluator** | 跑双校验门 + 给具体失败原因（不是"看起来不太对"） | `docs/V2.2_PHASE1_EVAL.md` | 不修代码 |

**为什么这样分**：Anthropic 实验证明，单一 agent 既写代码又审查 → 会"自我表扬"。v2.1 的 W3 commit 自称"PASS"但实际只切 1% = 这个错误的样本。

## 6 · 业务并行隔离（v2.1 失败的核心教训）

### 6.1 分支策略

```
main                         ← 业务功能继续在这（如 brand_strategy 后续优化）
  └─ v2.2-arch-v2            ← v2.2 架构改造唯一分支（新建，不复用 v2.1-architecture）
       └─ feature/*          ← 单个 feature 短分支（合并回 v2.2-arch-v2）
```

### 6.2 跨分支同步规则

- **每 7 天**：跑 `scripts/v2.2-cross-branch-check.sh`（待写）—— 检查 main 上新 commit 有没有踩到 v2.2 正在改造的文件
- **踩到了**：立即开会议（人参与，不让 AI 自己决定），决定 cherry-pick 还是改造方案调整
- **业务功能严禁直接在 v2.2-arch-v2 上做**：违反这条 = v2.1 的失败模式重演

### 6.3 v2.1-architecture 分支怎么处理

- **冻结**，不删除，保留作历史
- 已 commit 的 W1 organization SSOT 等成果**通过 cherry-pick 拉到 v2.2-arch-v2**
- 不再在 v2.1-architecture 上做任何工作

## 8 · 失败模式 + 早停条件

### 8.1 失败模式预警

| 模式 | 症状 | 兜底 |
|---|---|---|
| 业务速度太快 | brand_strategy 类 1 天接通的事在 v2.2-arch-v2 分支被多次踩到 | 暂停架构线，先合并业务 |
| 完美主义陷阱 | "L2 没全切完不能发 release" | NORTH_STAR 强制"4-6 周必发"，砍 scope |
| AI session 漂移 | 新 session 不读 NORTH_STAR，自作主张换方向 | 在 LLMS.md 写硬指令 + 第一句话就引用 |
| B 门连续不动 | 2 个里程碑机器人答题能力没进步 | 重新评估方向 |
| 多账号同步意外破坏 | dogfood 阶段发现新 v1.0 bug 复现 | 立刻回滚到上一里程碑 commit |

### 8.2 早停条件（meta-decision）

任何下列条件出现，**正式宣告 v2.2 暂停 + 写完整复盘**：

- 连续 3 个里程碑双门 FAIL
- 业务 main 分支 30 天内有 > 5 次跟 v2.2-arch-v2 冲突
- 顾源源主观感受"没看到客户价值进展"超过 4 周
- 重大业务方向变化（如转 SaaS / 转模板化）让 v2.2 假设失效
