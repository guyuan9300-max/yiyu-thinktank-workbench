## 0 · 接入模式(对所有 endpoint 通用)

每个 endpoint 函数体最小改动:
1. **签名加 3 个 Header 参数**(idempotency_key / actor_type / actor_id)
2. **函数体最前面加幂等检查**(~10 行)
3. **try/except 包原 endpoint body**(失败 mark_failed)
4. **return 前 complete**(~3 行)

不带 `Idempotency-Key` header 的旧客户端 100% 向后兼容。

## 1 · POST /api/v1/event-lines (main.py:30067)

### 1.1 函数签名改动

```python
# 改前
def create_event_line(payload: EventLineCreatePayload) -> EventLineRecord:

# 改后
def create_event_line(
    payload: EventLineCreatePayload,
    idempotency_key: str | None = Header(None, alias="Idempotency-Key"),
    actor_type: str = Header("human", alias="X-Actor-Type"),
    actor_id: str = Header("", alias="X-Actor-Id"),
) -> EventLineRecord:
```

注:`Header` 已经在 main.py 顶部从 fastapi import 了(grep 确认)。如果没,加进 `from fastapi import` 那一行。

### 1.2 函数体头部插入(就在 `timestamp = now_iso()` 前)

```python
    # v2.2 F2.8 (N3 A6): 幂等性检查 - 防 AI agent retry 重复创建
    _METHOD, _PATH = "POST", "/api/v1/event-lines"
    _idemp = None
    if idempotency_key:
        from app.services.idempotency_store import (
            IdempotencyKeyMismatchError,
            get_idempotency_store,
        )
        _idemp = get_idempotency_store(state.db)
        try:
            _cached = _idemp.find(
                idempotency_key, _METHOD, _PATH, payload=payload.model_dump()
            )
        except IdempotencyKeyMismatchError as _e:
            raise HTTPException(status_code=422, detail=str(_e))
        if _cached and _cached.status == "completed":
            return EventLineRecord.model_validate_json(_cached.response_body)
        if _cached and _cached.status == "in_progress":
            raise HTTPException(
                status_code=409,
                detail="Request still in progress, retry after a brief wait",
            )
        _idemp.start(
            idempotency_key, _METHOD, _PATH,
            payload=payload.model_dump(),
            actor_type=actor_type, actor_id=actor_id,
        )
```

### 1.3 函数体尾部修改(原 `return build_event_line(row)` 改成)

```python
    # v2.2 F2.8: 缓存响应供 retry 复用
    _result = build_event_line(row)
    if _idemp:
        _idemp.complete(
            idempotency_key, _METHOD, _PATH,
            status=200, response_body=_result.model_dump(mode="json"),
        )
    return _result
```

### 1.4 异常路径(可选,但推荐做)

为了避免 in_progress 卡住,理想是用 try/except 包整个 body:

```python
    try:
        # ... 原 endpoint body ...
    except Exception:
        if _idemp:
            _idemp.mark_failed(idempotency_key, _METHOD, _PATH)
        raise
```

如果这一步嫌侵入大,可以**先不加**,F2.8 v1 接受"卡 24 小时 TTL 后自动过期"作为兜底。

## 6 · 重要警告(给下个 session)

### 6.1 业务并行污染避免

main.py 经常有业务流(workspace_query_router 等)的脏改动:

```bash
# commit 前必须确认:
git diff --stat backend/app/main.py
# 应该 < 30 行变化 (3 个 endpoint × ~10 行/endpoint)
# 如果 > 80 行 = 一定混进了业务脏改动, 必须用 git add -p
```

**强制规则**:这次必须 `git add -p backend/app/main.py`,**不许** `git add backend/app/main.py`。F1.4 commit a628e09 已经踩过这个坑(80 行业务污染),NORTH_STAR §6 记录在案。

### 6.2 异常路径选择

§1.4 / §2 / §3 都标了"异常路径可选"。建议:**先做 happy path 接入 + 24h TTL 兜底**,验证模式 OK 后再扩展异常路径。

### 6.3 测试先行

写测试比改代码先做。如果 §4 的集成测试写完跑不通 → 说明 endpoint 接入有 bug,改代码即可。如果直接改代码,出问题不知道是 endpoint 改坏了还是 idempotency 逻辑错。
