## 1 · 按 7 维度逐项评估(只讲 Karpathy 视角新发现, 不重复技术 gap)

### §1 · Software 1.0/2.0/3.0 三层共存 ✅

**当前架构对照**:
| 层 | V2.2 实例 | 是否得当 |
|---|---|---|
| 1.0 | `detect_update_relation` 5 条规则(信息商) | ✓ 简单确定的判 conflict/supersedes,LLM 不应该做 |
| 1.0 | `metadata_for_workbench_file/task_review` normalizer(渠道驱动) | ✓ source_type → role 兜底,规则正确 |
| 2.0 | `entities` 表 4987 条(NER 抽人物/组织/日期) | ✓ 训练好的模型,LLM 不该重做 |
| 2.0 | embedding_provider.py(doubao embedding) | ✓ 向量化是 2.0 经典任务 |
| 3.0 | `DocumentLLMExtractor`(LLM 抽 atomic_facts) | ✓ "张真接任理事长" 这种规则盖不全的事 |
| 3.0 | `narrative_generator.py`(6 段叙事) | ✓ LLM 编排 + 推理 |

**判决**: ✅ 三层共存原则**已落地**,没看到错层错用。

**回顾教训**:5/22 早上的 V2.1 `NarrativeKernel 8 段 deterministic 列事实` 是把 **3.0 写成 1.0**(LLM 该做的事写成规则) → 5/22 砍 .DEPRECATED 是 Karpathy §1 视角的**正确决策**。

### §2 · LLM as OS · Context = RAM 🟡

**OS 类比检查**:
| OS 组件 | V2.2 对应 | 状态 |
|---|---|---|
| CPU | doubao seed pro / qwen3-vl:32b | ✅ |
| RAM (context window) | narrative_collector progressive disclosure | ✅ 拉 30 persons / 25 docs 等,不全塞 |
| IO (tool/function call) | IngestPipeline.ingest() / broadcast / endpoint | ✅ 设计在 |
| 持久化(硬盘) | atomic_facts 1998 / entities 4987 / v2_chunks 11998 | ✅ |
| **syslog (日志)** | **reasoning_traces 0 / ai_episode_log 0 / event_log 0** | **🔴 失败** |

**Karpathy 视角真信号**:

**V2.2 设计了完整 OS,但 syslog 没启用** → 用户/B AI 撞墙时**调试无据**:
- 5/22 B AI 写了 baseline runner 测 87% atomic_facts 单源命中,**自己以为对了**
- 直到 A AI dogfood 跑出 1/7 才发现 eval 对象错
- 如果 reasoning_traces 有流量,会记录"每条 atomic_fact 是从哪 doc/chunk 推出来的",**能立刻发现单源 == 没跨源印证**

**判决**: 🟡 RAM 管理 ✓,**syslog (reasoning_traces / ai_episode_log) 必须启用** — 已在 v2 计划 M-E。

### §3 · Eval-Driven Dev 🟡

**当前 eval 现状**:
| Eval 元素 | 状态 |
|---|---|
| Golden dataset | ✅ 5/19 张真会议 7 个关键事实 + 6 件待办 + 5 个人物 |
| Eval runner | 🟡 A 写过 (.DEPRECATED), B systematic 报告里没提替代品 |
| 命中率数字 | ✅ dogfood 1/7 (诚实) |
| 多 dataset | 🔴 只有 5/19 一个 |
| 跟北极星对齐 | 🟡 1 个北极星 (N2) 有 eval, N1/N3 缺 eval |

**Karpathy 视角真信号 1**:**eval 对象错了的教训**

B AI baseline runner 测 atomic_facts 单源 87% — 但用户在 app 里看到的是**经过 narrative_generator 6 段加工后的故事网**,命中只 1/7。

```
B 测的是: "atomic_facts 这一层 有没有 keyword"  (底层数据)
真正该测的是: "用户在战略陪伴页面看到的 6 段叙事 有没有 keyword"  (用户感知层)
```

这两者**差着 narrative_generator 一层**。B 之前 baseline 给了顾源源 87% 错觉,导致他差点决定"V2.2 N2 突破" → 实际打开 app 一看 1/7,**严重误导决策**。

**Karpathy 视角真信号 2**:**1 dataset 不够**

5/19 是好 eval,但只有 1 个客户 1 场会议。N2 命中 5/19 1/7 → 5/7+ 后,**不代表其他客户也通**。

但**这跟顾源源 V2.2 阶段定义对齐** — V2.2 就是"先突破一个客户证明方向",V2.3 才扩多客户。所以**单 dataset 在 V2.2 阶段是有意识的选择,不是 bug**。

**判决**: 🟡 eval 框架在,**eval 对象需要重新对齐用户感知**,新 baseline runner 应该测 **dogfood 6 段输出 + 5/19 keyword 命中**,不是 atomic_facts 单源命中。

### §4 · Planner / Generator / Evaluator 三角色 🟡

**当前角色齐**:
- Planner = 顾源源(NORTH_STAR / 决策点 / 接力指令)
- Generator A = A AI(backend schema / 内核 / 业务)
- Generator B = B AI(endpoint / 前端 / 测试 / systematic 盘点)
- Evaluator = baseline runner + dogfood + 三道门 audit

**Karpathy 视角真信号**:**Evaluator 是手工的,不是自动的**

```
当前 Evaluator 流程:
  A 改完 collector → 跑 dogfood 脚本 → 人读 markdown → 看 1/7 数字 → 判断
  
理想 Evaluator (Karpathy §4):
  A 改完 collector → 自动 trigger eval runner → 出综合 P% → CI 给红绿灯
  → 数字推进 = ship; 数字不动 = 回炉
```

**5/22 V2.1 Evaluator 工作不充分的 2 个证据**:
1. **5/22 早上 4 小时撞车** — A/B 都在做 8 段,**没自动 eval 告警**直到顾源源说"重新评估"
2. **B 的 87% 跟 dogfood 1/7 严重背离** — **没自动 eval 检查 eval 对象正确性**

**Planner 视角问题**:5/22 早上顾源源给的 "继续 Week 4" 是模糊指令 → 双 Generator 各自猜方向 → 撞车。Karpathy §4 说"Planner 必须强"。顾源源 5/22 之后给的 NORTH_STAR + 决策点 + 接力指令很清晰 → 5/22 下午开始效率高。

**判决**: 🟡 三角色齐,**Evaluator 自动化**需要补 — v2 计划没明确写,应该加 M-G。

### §6 · Unhobbled Iteration · 拆轮子 🔴(3 个过度约束)

**Karpathy 视角真信号**:V2.2 当前有 **3 个可能过度的约束**,值得审视拆掉。

#### 过度约束候选 1 · V2.1 lab 双 app 同跑

**当前**: V2.1 仓库物理隔离 + git remote fetch-only + LAB_MODE ENV-gated + 双端口 + 双 userData + 双 bundle id + dev:lab npm scripts

**真实必要性审视**:
- 顾源源红线: "不污染主仓库"
- 但主仓库本来就是 dev mode(`npm run dev` + uvicorn `--reload`),直接改 collector.py 就立即热重载
- 双 app 同跑的真实价值 = **可视化对比**(A/B 同比看效果)
- 但当前 V2.1 lab 跟主仓库代码同源(只 LAB_MODE 配置差),**A/B 看到的内容一样**,对比价值 = 0,要等阶段 2 改完 collector 才显现

**Karpathy 拆轮子建议**:
- 选项 a: **保留双 app**(顾源源喜欢可视化对比) — 当前
- 选项 b: **砍双 app**(直接在主仓库改,守红线 = "代码不污染" 而不是 "运行时隔离")— 简化
- 选项 c: 折中 — 不改 V2.1 配置,但**阶段 2 直接在主仓库改 collector**(顾源源 app 自动热重载看效果),V2.1 留作"长期实验场"不用于 A/B

**我建议**: c. 简化执行路径 + 保留 V2.1 lab 作为"未来激进设计实验" 选项。

#### 过度约束候选 2 · 4 主路径必须全接通 IngestPipeline

**当前**: NORTH_STAR N2 说"4 主路径(文件/任务复盘/爬虫/手机聊天)统一通道"

**真实必要性审视**:
- 路径 1 (文件) + 路径 2 (任务复盘) → V2.2 必须接(N2 北极星)
- 路径 3 (爬虫) → intelligence_* 16 services 已有,**但跟 V2.2 北极星 5/19 命中无关**(5/19 关键事实不在爬虫)
- 路径 4 (手机聊天) → 跟手机端 v2 架构强耦合,**手机端 v2 还没启动**

**Karpathy 拆轮子建议**:V2.2 阶段**强约束 4 路径** = 把"未来要做的事" 拉到当前里程碑,违反 §6。

**判决**: **路径 3/4 留 V2.3** (v2 计划已经这样,M-F 标 V2.3)— Karpathy 视角印证。

## 2 · Karpathy 视角带来的 3 个新行动项(补 v2 计划)

### 新行动 K-1 · Eval 对象重新对齐(高优先级)

**问题**: B AI baseline 测 atomic_facts 单源 87%,但用户感知 1/7。eval 对象错了。

**做法**:
- 复活 baseline runner,但 **eval 对象改为 dogfood 6 段输出**(不是 atomic_facts)
- 通过 `narrative_generator.generate(client_id) → 6 段 markdown → keyword 匹配 5/19 关键事实` 的链路
- 跟 dogfood 脚本结果一致才算对齐

**位置**: 跟 v2 计划 M-C(collector 补表)同步做。

### 新行动 K-2 · Evaluator 自动化(中优先级)

**问题**: 当前 eval 全靠手工跑 dogfood,贵。

**做法**:
- 改 baseline runner 加 auto-mode:跑完 6 段叙事 → 自动出综合 P% → 写 `tests/reports/auto_eval_*.json`
- 加 git hook(可选):commit narrative_collector / narrative_generator 改动后自动跑一次 baseline

**位置**: V2.2 完成后做(M-E 之后),不阻塞主路径。

### 新行动 K-3 · B AI 独立 review v2 计划(立刻)

**问题**: v2 计划是 A 单方写,缺模式 C 互检环节。

**做法**:
- 写 sync 指令给 B AI:**让 B 用 Karpathy 7 维度跑同样评估**
- A + B 视角对齐 = 真信号,异点 = 视角差或盲点
- 顾源源拿到 A + B 两份 Karpathy 评估再决策

**位置**: 立刻 commit 后做。
