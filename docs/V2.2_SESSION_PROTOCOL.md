# V2.2 Session 协议 · 开局/收尾标准流程

> **每个 v2.2 session 必须严格遵循此协议。**
> 不遵循 = 必然回到 v2.1 那种"切 SQL、自我表扬、双校验门不跑"的失败模式。
> 基于 Anthropic 长任务 harness 方法论。

---

## 0 · 角色判定（开局第一件事）

每个 session 开始时立即决定扮演哪个角色，**不允许中途切换**。

| 角色 | 何时选 | 输出 |
|---|---|---|
| **Planner** | 当前里程碑还没有 spec / 上一个里程碑的失败需要分析 | `docs/V2.2_PHASE{N}_SPEC.md` |
| **Generator** | 已有 spec，需要写代码 | feature 分支 commit |
| **Evaluator** | 一组 feature 写完，需要校验 | `docs/V2.2_PHASE{N}_EVAL.md` + 跑双校验门 |

判定规则：
```
if not exists("docs/V2.2_PHASE{current}_SPEC.md"):
    role = "Planner"
elif feature_list.json 还有 in_progress 的 feature:
    role = "Generator"
elif 上一个 milestone 边界刚到:
    role = "Evaluator"
else:
    role = "Generator"（继续推进）
```

## 1 · Session 开局流程（不可跳过任一步）

```bash
# Step 1: 定位
cd ~/openclaw/workspace/yiyu-thinktank-workbench

# Step 2: 必读 4 份契约文档（每次都读，不许"我记得"）
cat docs/V2.2_NORTH_STAR.md          # 2 核心目标 + 失败模式
cat docs/V2.2_RFC.md                 # 3 阶段规划 + 验收门
cat docs/V2.2_FEATURE_INVENTORY.md   # 现有功能清单（校验目标 A）
cat docs/V2.2_PROGRESS.md            # 上次 session 留了什么

# Step 3: 校准当前状态
git branch --show-current             # 应该在 v2.2-arch-v2 分支
git status --short                    # working tree 应该干净
git log -5 --oneline                  # 看最近 5 commit

# Step 4: 检查 feature_list
cat docs/V2.2_FEATURE_LIST.json       # 当前 phase 还有哪些 in_progress / pending

# Step 5: 自我宣告角色
echo "本 session 角色：[Planner / Generator / Evaluator]"
echo "本 session 目标：[具体到 feature ID，如 F1.2]"
echo "预计输出：[文件名 + commit message]"

# Step 6: 写入 progress
更新 docs/V2.2_PROGRESS.md：
  ## Session {timestamp}
  - 角色: ...
  - 目标: ...
  - 起点 commit: ...
```

**违反任何一步 = 立即停手，从 Step 1 重来**。

## 2 · Session 工作流（按角色不同）

### 2.1 Planner 角色工作流

```
1. 读 V2.2_RFC.md 当前 phase 的 feature 列表
2. 选 1-3 个 feature 写详细 spec：
   - 具体改哪些文件
   - 每个文件改前/改后对比
   - 双校验门怎么过（具体到执行命令 + 期望结果）
   - 风险点（哪些 feature 改了会破坏现有功能）
3. 写完 spec 提交 git
4. 更新 V2.2_FEATURE_LIST.json 标记这些 feature "spec_done"
5. 不写任何代码
6. 收尾（见 §3）
```

**Planner 禁止**：
- 写实际代码
- 跑 acceptance runner
- 给自己写的 spec 打分

### 2.2 Generator 角色工作流

```
1. 读 spec_done 状态的 feature spec
2. 选 1 个 feature 开干（不是多个！）
3. 写代码 + 单测
4. 跑单测确认 PASS
5. git commit（commit message 必须包含：
   - feature ID（如 [F1.2]）
   - 对应 NORTH_STAR 目标（"服务目标 A" 或 "服务目标 B"）
6. 更新 V2.2_FEATURE_LIST.json 标记 "impl_done"
7. 收尾（见 §3）
```

**Generator 禁止**：
- 一次做多个 feature
- 跳过单测
- 自我审查后宣告"PASS"（必须等 Evaluator）
- 修改 spec（如发现 spec 错，停手返回 Planner 角色重写）

### 2.3 Evaluator 角色工作流

```
1. 看 V2.2_FEATURE_LIST.json 中 "impl_done" 的 feature
2. 跑双校验门：
   门 A（功能不掉链）:
     - backend/.venv/bin/python3 scripts/run_v22_acceptance.py
     - 按 V2.2_FEATURE_INVENTORY.md 逐项手工跑（至少跑改动涉及的模块）
     - 多账号 e2e（如改了同步相关）
   门 B（机器人能力进阶）:
     - 用 5/19 张真会议作为金标准
     - 问当前里程碑应该能答的问题
     - 检查答案完整度 + 是否带 evidence
3. 写 docs/V2.2_PHASE{N}_EVAL.md 报告：
   - 门 A 结果（PASS 项 / FAIL 项 + 具体错误）
   - 门 B 结果（机器人答出几条 / 哪几条没答出来）
   - 总判定 (PASS / FAIL)
4. 更新 V2.2_FEATURE_LIST.json：
   - PASS → "verified"
   - FAIL → 退回 "impl_failed" + 详细原因
5. 收尾（见 §3）
```

**Evaluator 禁止**：
- 修代码（发现 bug 写报告，不动手）
- 略过 FAIL 项（"看起来差不多" 不许说）
- 跳过手工校验门 A 的清单

## 3 · Session 收尾流程（不可跳过任一步）

```bash
# Step 1: 最后状态校验
git status --short                    # 必须为空（所有改动都 commit）
git log -5 --oneline                  # 看本 session commit 数

# Step 2: 更新 progress（关键！下个 session 靠这个续接）
追加到 docs/V2.2_PROGRESS.md：
  ## Session {timestamp} 收尾
  - 角色: ...
  - 完成的 feature: F1.x, F1.y
  - 未完成的 feature: F1.z（原因 + 下次怎么续）
  - 双校验门状态: A=PASS B=N/A 或 A=FAIL B=FAIL（写明）
  - 下次 session 应该: [具体动作]

# Step 3: 更新 FEATURE_LIST.json（每个 feature 状态）

# Step 4: commit 这些文档变更（如果有）
git add docs/V2.2_PROGRESS.md docs/V2.2_FEATURE_LIST.json
git commit -m "session log: {timestamp}"

# Step 5: 如果跨里程碑边界，正式发 release tag
git tag v2.2.0-phase1-alpha  # 例
```

## 4 · Context 管理

按 Anthropic 方法论：**Context Reset > Compaction**。

| 信号 | 应对 |
|---|---|
| Context 用了 > 50% | 强制 reset：写完 progress + commit + 启动新 session |
| 想"再扛一会儿" | 不许。Context anxiety 会让 agent 过早 wrap up |
| 临近 wrap up 写"应该差不多了" | 立即停。reset 后让新 session 继续 |

---

## 5 · 跨 Session 状态传递（不靠 context）

状态完全通过文件传递：

| 文件 | 用途 | 谁写 | 谁读 |
|---|---|---|---|
| `V2.2_NORTH_STAR.md` | 目标契约 | 顾源源（不动） | 每 session 必读 |
| `V2.2_RFC.md` | 阶段规划 | 顾源源审批（修改需走 PR） | 每 session 必读 |
| `V2.2_FEATURE_INVENTORY.md` | 功能清单 | 新功能上线时追加 | Evaluator 校验门 A |
| `V2.2_FEATURE_LIST.json` | feature 状态机器可读 | 每 session 更新 | 每 session 读 |
| `V2.2_PROGRESS.md` | 跨 session 工作日志 | 每 session 收尾追加 | 每 session 开局读 |
| `V2.2_PHASE{N}_SPEC.md` | 当前 phase 详细 spec | Planner 写 | Generator 读 |
| `V2.2_PHASE{N}_EVAL.md` | 当前 phase 校验报告 | Evaluator 写 | 下一 phase Planner 读 |

## 6 · 失败模式自检（每 session 结束前问自己）

| 自检问 | 通过标准 |
|---|---|
| 本 session 是否只扮演了一个角色？ | 不是 → 重做 |
| 本 session 改的代码服务于 NORTH_STAR 哪个目标？ | 答不上来 → 工作可能跑偏 |
| 双校验门有没有跑？ | Evaluator 角色没跑 → 失败 |
| Progress / FeatureList 有没有更新？ | 没更新 → 下个 session 接不上 |
| commit message 有没有引用 feature ID？ | 没有 → 历史追溯困难 |
| 想"做完这个再做下一个"了吗？ | 是 → 立即停，违反"一 session 一 feature"原则 |

---

## 7 · 紧急情况

| 情况 | 应对 |
|---|---|
| 跑双校验门发现现有功能挂了（A 门 FAIL） | 立即 git revert 当前 commit，写 incident 到 PROGRESS.md |
| 发现 V2.2_RFC.md 的 spec 错了 | 停手，切到 Planner 角色，写 RFC 修订 PR |
| Context 不够用了 | reset 不 compact，新 session 接 |
| 顾源源主观感受"不对" | 暂停所有架构线工作，回 NORTH_STAR 重新对齐 |
| 业务 main 分支跟 v2.2-arch-v2 冲突 | 不许自作主张 merge，等顾源源决策 |

## 8 · 自动化机制

```bash
# 每周一次自动检查（建议设 cron）：
scripts/v22-weekly-check.sh
  - git log v2.2-arch-v2..main 看业务有没有踩到 v2.2 改造文件
  - 跑 acceptance runner 看现状是否还绿
  - 检查 PROGRESS.md 最后更新时间（超过 7 天报警）
```
