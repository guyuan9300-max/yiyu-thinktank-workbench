## Round 1 — Team Debate

### CEO (Product CEO)

看完审计结果，最让我难受的不是某个页面丑或者某个按钮不对。是**系统已经在记忆了，但用户完全感知不到它在记忆。**

understanding_builder 已经能输出"这是什么事/为什么重要/推进到哪/还缺什么"，但它在测试文件里躺了不知道多久，main.py 一行调用都没有。这意味着我们花了大量精力搭记忆管道，数据确实在写入，但用户从来没看到过系统说"我理解了 XX"。

战略陪伴页全是 mock 数据更是不能忍。用户上去看到的不是真实记忆，是我们编的假数据。

**但这一轮不能两个都做。**

本轮我只要一件事：**让 understanding_builder 接入任务详情页，让用户第一次看到系统"理解了什么"。**

### Product Lead

同意 CEO 的判断。从用户场景看：

用户每天打开软件最多的动作是看任务、改任务状态。如果他打开一个任务，看到的只是自己输入的标题和描述，那系统就只是一个记事本。

但如果他打开任务，看到系统说��
> "这是一次跟日慈的复盘跟进。你之前跟笑雨和高老师分别聊过 Q1 复盘，教师赋能那条线还有设计缺口。目前推进到了等待补完设计方案的阶段。我还不确定的是：补完的截止日期和负责人。"

**这才是"越用越懂你"的第一步感知。**

而且这个改动的价值不只是展示——用户看到理解不对，就会纠正；纠正了，理解就更准。这是一个正向循环的起点。

### Memory & Judgment Architect

从架构角度看，understanding_builder 接入的最佳位置是**任务详情展开时**（用户点开某个任务查看详情的那一刻）。

原因：
1. 这是最高频的查看场景
2. 此时系统已有：任务标题、描述、状态、关联客户、关联事件线、历史复盘笔记
3. understanding_builder 需要的输入都在手边，不需要新的数据管道
4. 输出的 4 层理解（是什么/为什么重要/推进到哪/还缺什么）跟任务场景完美对应

**风险提醒：** understanding_builder 内部调用了 AI 接口（_qwen_generate），如果每次打开任务都调一次，会有延迟和成本。建议：
- 第一次生成后缓存到 DB
- 只有当任务数据变化时才重新生成
- 前端显示时先展示缓存版本，标注"上次更新于 X"

### Tech Lead

可行。具体路径：

1. **后端**：新增 API `GET /api/v1/tasks/{task_id}/understanding`
   - 调用 understanding_builder.build_understanding_basic()
   - 传入：任务数据 + 关联的组织 DNA + 事件线记忆 + 历史复盘笔记
   - 返回：UnderstandingSnapshotV1Record
   - 第一版不做缓存（先验证价值），第二轮加缓存

2. **前端**：在任务编辑器面板中新增一个"系统理解"折叠区域
   - 展示 4 个段落：是什么/为什么重要/推进到哪/还缺什么
   - 如果理解为空或正在加载，显示骨架屏
   - 用户点开任务时触发请求

3. **涉及文件**：
   - backend/app/main.py — 新增 1 个端点
   - src/renderer/App.tsx — 任务编辑器区域新增理解面板
   - src/renderer/lib/api.ts — 新增 1 个 API 函数

**不做的事：**
- 不改 understanding_builder.py 本身
- 不改记忆写入链路
- 不改战略陪伴页
- 不加缓存（下一轮）

### QA & Interaction Auditor

验收标准：

1. 打开任一个任务 → 看到"系统理解"区域
2. 理解内容跟任务实际相关（不是泛泛的废话）
3. 加载中有骨架屏，不是空白
4. API 返回 500 时前端不崩溃，显示"暂无法生成理解"
5. 没有关联客户/事件线的简单任务也能给出基础理解
6. 不影响现有任务编辑的任何功能
