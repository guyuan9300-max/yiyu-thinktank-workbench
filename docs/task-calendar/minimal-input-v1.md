# 任务与日历模块：最小输入模型 v1

## 核心原则

任务与日历模块不是等资料补齐后才开始判断，而是先用最少资料把事情理解对，再用更多资料把理解做深、做准、做细。

---

## 一、最小输入（必须）

系统只需要以下 6 项就能产出第一层理解：

| # | 输入 | 说明 |
|---|------|------|
| 1 | 益语背景卡 | 益语是谁、靠什么服务客户、当前重点方向 |
| 2 | 客户/项目背景卡 | 客户是谁、行业位置、需求痛点 |
| 3 | 组织或部门季度主线卡 | 当前季度在推什么、对齐什么方向 |
| 4 | 任务标题 | 这条任务叫什么 |
| 5 | 任务说明 | 任务的描述、背景 |
| 6 | 任务复盘资料 | 一线复盘说明、结构化笔记、轻量标签 |

## 二、增强输入（可选，提升精度）

以下内容不是前置门槛，只是增强项。缺少任何一项都不影响系统产出 basic 结果：

- 月计划
- 会议纪要
- 录音摘要
- 事件线记忆
- 支持请求
- 日历时间块
- 附件摘要

## 三、系统主输出

### 第一层（必须，始终存在）

无论输入多少，系统必须优先回答这 4 个问题：

1. **这是什么事** — 用一段话说清楚这条任务/事件线在做什么
2. **为什么重要** — 结合益语背景和客户背景，解释这件事对双方意味着什么
3. **现在推进到哪** — 当前处于什么阶段，相比之前有什么进展
4. **还缺什么理解** — 系统目前对这件事还不够了解的地方

### 第二层（可选，仅在证据足够时出现）

只有当证据充分时，系统才进一步给出：

- **真正阻碍** — 不是泛泛的"沟通不畅"，而是具体的卡点
- **时间闸门** — 过了这个时间点，情况会发生质变
- **最小动作** — 当前最该做的 1 件事
- **supportAsk** — 需要谁提供什么支持

## 四、行为约束

1. **少资料先出结果** — basic 模式下，只靠最小输入就必须产出完整的第一层输出
2. **多资料提升精度** — 增强输入只提升置信度和精度，不改变输出结构
3. **不允许首页先报一堆缺失项** — 用户打开后第一眼看到的是理解，不是"你还缺 XX"
4. **不允许因为缺增强资料就不给 basic 结果** — 永远不能返回"无法判断"

## 五、模式定义

| 模式 | 触发条件 | 输出范围 |
|------|---------|---------|
| basic | 只有最小输入 | 第一层 4 项必须存在，第二层为空 |
| enhanced | 有增强输入 | 第一层 4 项 + 第二层按证据充分度填充 |
