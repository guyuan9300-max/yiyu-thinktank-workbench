# 客户计算中心 — 功能差距清单

**审计日期：** 2026-05-12  
**审计依据：** 已澄清的"客户计算中心"功能设想（九层架构）  
**审计方式：** 源代码静态审计（backend/app + src/renderer）  
**结论分档：** 已到位 / 雏形 / 缺失

---

## 总览：差距全貌

| 层 | 名称 | 总体完成度 | 最关键缺口 |
|---|---|---|---|
| 1 | 收集层（多入口） | 60% | 文件/文本入口缺业务元数据，与业务对象入口断层 |
| 2 | 入库与生命周期 | 55% | 单文档失败拖垮整批 job；无 hash 查重；无重试 |
| 3 | 结构化解析 | 75% | 失败兜底不完整；material_profile 仅 2 档 |
| 4 | 路由与判断 | 55% | work_status / diagnostic 路径只有声明无实现 |
| 5 | 多源检索与证据 | 50% | DNA / event_line / state_pool 是名义来源；证据四档分类不存在 |
| 6 | 回写与经验沉淀 | 30% | answer_experience 写了不读；几乎不形成下次输入 |
| 7 | 权限与同步 | 50% | sync outbox 有生产者无消费者；权限收口未推广 |
| 8 | 运维与修复 | 65% | 多个按钮空跑（refresh_context_pack/rebind） |
| 9 | **AI Gateway 与写回闭环** | **20%** | **/api/ai/\* 全部不存在；状态机过短；无 service token** |

**最大缺口在第 9 层**——这正好是 2.5 阶段（AI 工作流中台）要补的位置。

## 第 1 层 收集层（多入口）

### 已到位
- `POST /api/v1/imports` 文件/文件夹导入：`backend/app/main.py:35993`，支持 folder 模式递归扫描、扩展名过滤、按路径查重、`stage_import_copy` 复制进受管目录。
- `POST /api/v1/clients/{clientId}/documents/from-text` 手动文本：`main.py:38385`，落地"项目与业务/手动新增文档"，直接调用 `ingest_document_knowledge`。
- 客户业务对象 → `data_center_ingest`：task / note / attachment / meeting / weekly_review / event_line **六类全部接入**（`data_center_ingest.py:1306/1341/1386/1489/1617/1742`），`DataCenterIngestPayload` 携带组织/部门/负责人/可见范围/事件线/任务全套元数据（同文件 :89-101）。
- 对话临时上下文 `workingDocumentIds`：`App.tsx:18122-18130` + `main.py:35522/38205/38265` 跟随对话。

### 雏形 / 缺失（关键）

1. **文件导入入口元数据极其贫瘠**。`ImportPayload` 只有 `clientId / mode / paths / allowLegacy`（`models.py:919`），**没有 organizationId / ownerUserId / visibilityScope / eventLineId / taskId**。这与 `DataCenterIngestPayload` 形成断层——业务对象有完整 RBAC/可见性元数据，文件资料没有，后续路由层做权限过滤时只能凭客户 ID 拦截。

2. **from-text 入口同样裸字段**。`ClientTextDocumentPayload` 只有 `title / content`（`models.py:7069`），手动新增的资料天然丢失负责人 / 可见范围 / 事件线归属。

3. **import 与 ingest_event 两套表互不对齐**。经文件导入的资料只进 `documents` + `v2_documents`，**不会**进 `data_center_ingest_events`；只有业务对象才进。所以可见范围、所有权这套机制完全绕过了文件导入。

4. **link_material 是第三条不互通的入库管线**。`link_material_import_runs` 表 + 单独 executor，不进 imports、不进 knowledge_jobs、不进 ingest_events，监控/恢复/查重/状态机全是另一套。

## 第 2 层 入库与生命周期

### 已到位
- `imports` 表（`db.py:107`）+ `knowledge_jobs`（`db.py:318`）+ `knowledge_job_events`（`db.py:334`）。
- 单线程 worker（`main.py:3850`）+ 外层兜底 try/except。
- 启动恢复 `recover_stale_knowledge_jobs`（`main.py:3115`）+ `recover_stale_imports`（`main.py:3139`）把崩溃残留的 running 重置回 queued。
- 受管目录 `_imports/{importId}/`（`knowledge_v2.py:728`）。

### 雏形 / 缺失（关键）

1. **单文档失败会整批 fail**。`process_knowledge_job` 的 `ingest_import` 分支在 per-document 循环里**没有 try/except**（`main.py:3613-3674`）——一份 PDF 坏了或编码异常，整个 job 标 failed，同批后续文件不再处理，对应 imports 整批 failed。**这是我们澄清时明确要求的反例。**

2. **没有内容 hash 查重**。导入闸门只比路径（`main.py:36045`：`import_source_path OR current_human_path OR original_path`）。`content_hash` 在 v2 入库后才计算（`knowledge_v2.py:1764`），不作用于跨批查重；同一文件从两个位置拖入会重复入库。

3. **状态机声明与实际不一致**。`ImportRecord.status` 声明含 `scanned`（`models.py:911`），但 grep 全代码 0 命中；实际只有 `queued / processing / completed / failed`，**没有"扫描中"可观察阶段**。

4. **没有失败重试机制**。`knowledge_jobs` 表无 `retry_count / next_retry_at`，worker 也没有重试入口。`retry_parse` job_type 只重试解析，不重试整个 ingest。

5. **imports 表缺关键列**：无 `failed_count / error_summary / organization_id / owner_user_id / visibility_scope / event_line_id`。与 `data_center_ingest_events` 那套丰满 schema 形成对比，imports 这条线在权限和事件线上是"二等公民"。

## 第 9 层 AI Gateway 与写回闭环（**最大缺口**）

> 这一层基本是从 0 开始，与 2.5 阶段目标差距约 80%。

### 已到位（内部雏形）
- `proposal_records`（`db.py:657-679`）：有 status / risk_level / target_refs / source_refs / boundary_notes / payload / created_by / decided_by / rejected_reason / execution_ticket_id——**写回闭环最接近成型的载体**。
- `execution_tickets` + `execution_ticket_logs`：执行侧带重试和日志。
- `approval_records`（`db.py:643-655`）：审批流水（actor_id / actor_name / note）。
- `topic_candidates / topic_candidate_insights / external_evidence_cards`：选题池雏形。
- `card_review_queue`：knowledge_document 审核（**仅此用途**）。

### 缺失（按优先级）

1. **/api/ai/\* 路由全部不存在**。代码库 0 命中。`local_request_guard.py:8-13` 只放行 `localhost / 127.0.0.1 / app://renderer`，跨站请求一律 reject——**外部 Dify / AI 工作流目前无任何合法入口**。9 条都要从 0 搭：retrieval / context-pack / evidence-map / work-status / file-candidates / content-materials / writeback/draft / writeback/review-decision / writeback/task。

2. **没有 service token / 外部凭证体系**。全代码无 `api_token / service_credential / ai_caller / bearer` 任何鉴权字段。所有 `api_key` 都是出站给云端 LLM 用的（方向相反）。需要新建 `ai_service_credentials` 表 + 路由级 Depends 鉴权 + token 与 scope 绑定。

3. **状态机过短**。`proposal_records.status` 只有 `pending_review → approved / rejected`（`proposal_approval.py:178, 228`）。**缺 `draft / needs_review / needs_revision / ready_to_publish / published` 五个我们要求的状态**——grep 0 命中。AI 一旦生成就直接是 pending_review，没有"先入 draft 池"前置环节。

4. **没有独立的 `ai_drafts` 表**。AI 生成文本要么进 `answer_runs`（单次问答 retrieval_summary），要么进 `weekly_reviews / proposal_records` 等业务表。**没有统一的"AI 草稿入口表"承接 Dify 写回**。

5. **审计日志缺 AI 维度**。`SystemLogger` 是通用 JSONL，`approval_records` 只记审核动作，`execution_ticket_logs` 只记执行——**没有"哪个 AI 岗位 X 在时间 T 用资料 [a,b,c] 生成 draft Y、审核人 Z 处理"的链路**。需要新建 `ai_invocation_log (caller_id, role, request, response, retrieved_doc_ids, draft_id, client_id, latency_ms)` 与 drafts/审核关联。

6. **"只允许 ready + 非 private 素材"无强制点**。`visibility_scope` 字段虽存在于 documents/memory_facts/event_lines（`db.py:2334-2598`），但因为没有 AI Gateway 层，**没有"按 caller 视角裁决可见素材并拒绝越界"的中间件**。content-materials 接口必须在 Gateway 层加可见性过滤器，并把裁决记录写进审计日志，否则外部 AI 一旦联通就会越权。

7. **AI 调用的回写全是内部直调**。`record_imported_document_writeback / record_client_dna_writeback / record_meeting_publish_writeback / record_task_attachment_writeback / record_weekly_review_writeback`（`main.py:793-797`）全部是 Python 函数调用，**没有一条以 HTTP 路由暴露**。

## 跨层共性问题

排查中反复出现的几个系统性问题：

### A. "声明 ≠ 实现"
`work_status / diagnostic` 路径、`ImportRecord.scanned` 状态、`material_profile` 的"待定"档、`trust_signals` 表——枚举里写了，代码里 0 命中。这类"占位但未实现"在多处出现，是路径设计与代码实现的脱节。

### B. 两条平行管线
- 文件导入走 `imports + knowledge_jobs`，业务对象走 `data_center_ingest_events`——两条管线**不互通**。
- `data_center_sync_outbox`（生产者）与 `/knowledge/sync-to-cloud`（surrogate 旁路）——两条同步通道**不合流**。
- `workspace_query_router` 与 `query_router._rule_route_decision`——两个路由器**并存叠加**。
- `link_material_import_runs`——第三条独立入库管线。

### C. "写了不读"的经验沉淀
`workspaceAnswerExperience / trustSignals / boundaryNotes / missingContext` 都被写入 retrieval_summary_json，但下一次问答**不读回**。所谓"每次问答都能变成未来的经验数据"只有 `evidence_quality_annotations` 一条线真正闭环。

### D. CEO 默认上下文绕过权限
`DataCenterAccessContext` 只在唯一一处构造，其他调用传 None → CEO 默认上下文 → 绕过 scope 收口。权限是数据库层有但应用层没用上。

### E. 名义来源 vs 实际来源
`state_pool` / `master_index` / `event_line` / `customer DNA` 在 dataSources 清单里都出现，但实际**3 个是别名、1 个完全不是检索来源（DNA 是 prompt 拼贴）**。
