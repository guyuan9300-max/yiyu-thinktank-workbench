本文件不讨论理想产品，只回答：

1. 现在人实际会经过哪些高价值工作流

## 1. 当前模块真实存在的 ritual

当前“任务与日历”不是一个页面，而是 6 个并行 ritual 入口：

1. `协作收件箱`
   - 处理被派发/待确认任务
   - 更像 intake ritual
2. `任务列表`
   - 处理主执行面
   - 同时混入结构归属建议、AI 判断、编辑
3. `我的月历`
   - 处理日期承接、拖动重排、快速创建
   - 但仍偏时间容器，不是判断入口
4. `周复盘 collect`
   - 处理单项复盘采集
   - 已支持本条/本组状态、单条保存、结构化字段
5. `周复盘 document`
   - 处理生成文稿与再编辑
   - 更像报告阶段
6. `周判断 / 层级总结`
   - 处理个人、部门、机构层的判断、信号与动作
   - 已经在 collect 阶段就出现

### 当前已经存在的稳定 ritual

- 收件箱分诊 ritual
- 日历拖拽改期 ritual
- 单项复盘 ritual
- 生成整周文稿 ritual
- 部门/机构总结 roll-up ritual
- gap 处理 ritual
  - 上传材料
  - 立即澄清

### 当前还不稳定的 ritual

- “先把关键任务承接进日历”的每日节奏
- “会后行动项立刻收成任务”的节奏
- “遇到阻塞就发支持，而不是继续硬扛”的节奏
- manager 每周固定看哪几张卡、先点哪里、如何下动作的节奏

## 2. 现在真正能闭环的动作

不是所有页面元素都只是展示。当前能闭环的动作主要有：

1. 任务状态切换
   - `todo/doing/done/rejected`
2. 任务改期
   - 日历拖动
   - detail pane reschedule
3. 快速创建任务
   - 日历内 quick create
4. 审批类动作
   - calendar detail 中 `approve / return review`
5. 单项复盘保存
   - `WeeklyReviewStructuredFields.onSave`
6. 单项状态落库
   - 完成 / 延迟 / 取消
7. 事件线 gap 动作
   - `upload_docs`
   - `clarify_now`
8. Review action card 触发
   - `WeeklyReviewSummaryPanel -> HierarchyReportCard -> onTriggerAction`
9. 文稿保存 / 提交
   - collect → document → save/submit

### 当前“看起来像闭环但其实不够强”的动作

1. AI 洞察卡
   - 能看、能说，但多数情况下不能直接把判断变成任务/支持/会议动作
2. 周判断分析卡
   - 能提出风险和机会，但动作入口仍然不够前置
3. 日历标记与节日信息
   - 能看，但还没有和“是否需要重排本周行动”绑定起来
4. 结构归属建议
   - 能提示，但没有形成持续 adopt 的轻量 ritual

## 3. 当前最关键的 8 个高价值工作流

| # | 角色 | 工作流 | 当前真实入口 | 真正闭环动作 | 当前主要阻力 |
|---|---|---|---|---|---|
| 1 | 个人执行者 | 收任务 / 确认这周要做什么 | 协作收件箱、任务列表 | 接收、改状态、进入列表 | intake 和正式推进没有清晰分层，容易遗漏“先确认后排期” |
| 2 | 个人执行者 | 把关键任务承接进日历 | 任务列表、我的月历 | 拖到日期、改 dueDate、调整 duration | 日历是时间容器，不是判断容器；系统没有强提醒“关键任务尚未上历” |
| 3 | 个人执行者 | 会后快速形成行动项 | 会议相关任务、任务创建 | 创建任务、挂客户/事件线 | 会议到任务的显式桥不够前置，动作闭环更多靠人自觉 |
| 4 | 个人执行者 | 任务卡住后发起支持 | 复盘、支持字段、事件线判断 | 写支持需求、gap action、后续动作 | 当前更像写说明，不像真正发起 support workflow |
| 5 | 个人执行者 | 一周结束做单项复盘 | `reviewStage=collect` 下的 group 卡 | 单条保存、状态更新 | 单项复盘、分析卡、整周总结同时出现，心理负担偏大 |
| 6 | 个人执行者 | 基于系统判断决定下周先做什么 | 周判断分析卡、建议动作 | 改状态、排日历、后续任务 | 建议在页面里存在，但和“下周第一动作”连接不够短 |
| 7 | 部门负责人 | 看本周主线推进与阻塞热区 | `WeeklyReviewSummaryPanel` 部门视角 | drill-down、触发 action card | 个人、部门、机构三层都在一个面板，容易信息超量 |
| 8 | 机构管理者 | 看组织层风险、支持债务、是否需介入 | 机构总结 / executiveOrgReport | trigger action、下钻 | 首屏没有把“该我介入什么”压到最前，管理入口更像报告页 |

## 4. 当前交互阻力清单

这里不讨论“美观不美观”，只讨论阻碍使用和信任的地方。

### 4.1 让人不理解系统在干什么

1. `任务列表` 同时混入：
   - 执行卡
   - 结构归属建议
   - AI 背景卡
   - 编辑动作
   这让页面不像“工作台”，更像“混合调试区”。
2. `周复盘 collect` 阶段同时出现：
   - 单项采集
   - AI 分析
   - 层级总结
   - 后续动作
   对用户来说很难判断“现在到底是在录入、在判断，还是在汇报”。
3. `周复盘 document` 生成前后没有明显的 ritual 切换感。
   - 看起来像一键从采集跳到了长文编辑

### 4.2 让人不信任 AI 判断

1. 页面没有把“哪些信息已纳入判断、哪些未纳入”放在前面。
2. `WeeklyReviewAnalysisPanel` 已经有缺口和 completeness，但仍然放在分析区内部，不是首屏 gate。
3. 任务详情里的 `TaskOrgContextPanel` 仍偏“会说”，不是先给行动和证据入口。
4. 同一页面里同时出现：
   - 人工复盘文本
   - 系统识别
   - AI 判断卡
   但没有一层明确边界告诉用户：
   - 事实是什么
   - 判断是什么
   - 建议是什么

### 4.3 让人不知道下一步是什么

1. 单项复盘写完后，没有明确告诉用户下一步是：
   - 存草稿
   - 继续下一条
   - 生成整周
2. 周判断卡给出风险/机会后，动作入口还不够前置。
3. calendar view 很强，但和“这周最该挪哪件事”没有直接挂钩。
4. manager 在 summary panel 中看到结论后，不一定知道：
   - 应该触发任务
   - 发支持请求
   - 约会议
   - 还是继续观察

### 4.4 把录入、判断、行动混在一起

1. 复盘采集区和判断结果区没有强分层。
2. summary panel 在 collect 阶段就出现，打断“先收证据再判断”的心智。
3. 文稿模式仍然默认是长文编辑，而不是 action-first。

## 5. 当前最伤 adoption 的 10 个交互问题

1. **收件、执行、复盘、管理总结混在同一条主导航里，但没有强节奏切换。**
2. **周复盘 collect 阶段即出现总结和分析，用户容易在证据尚薄时就被拉去看报告。**
3. **任务列表页承担了太多职责，不像一个稳定的“执行台”。**
4. **日历虽然可拖拽、可改期，但没有变成“下周最重要动作承接台”。**
5. **单项复盘是高价值输入，但保存、状态、继续下一条之间的工作流仍然偏碎。**
6. **系统没有把“未纳入判断 / 降级输出”单独拉出来，信任边界不够清楚。**
7. **manager 视角缺少首屏排序：哪条主线该先看、哪条阻塞需要自己介入。**
8. **AI 卡片更像解释器，而不是直接促发动作的操作台。**
9. **会后行动项与任务承接桥不够前置，养不成稳定会议闭环习惯。**
10. **动作卡虽存在，但没有统一嵌进个人/leader/manager 三条主路径。**

## 6. 信任阻力清单

这些问题会直接导致用户觉得系统“会说，但未必可信”。

1. 没有首屏展示 `judgment readiness`
2. 没有首屏展示 `未纳入判断的证据/任务/会议`
3. 证据 drill-down 入口还不够前置
4. 任务卡里的判断摘要与周判断里的卡片，用户不容易理解是否同源
5. manager 看见的是“风险卡”，但不容易知道：
   - 这张卡来自哪些任务
   - 哪些是人工复盘
   - 哪些只是系统推断

## 7. 哪些地方让系统向用户索取过多

### 对个人执行者

1. 要维护任务状态
2. 要把任务排进日历
3. 要写单项复盘
4. 要生成整周复盘
5. 还要理解 AI 判断卡

这意味着系统把：
- 执行
- 记录
- 解释
- 提炼

都推给了同一个人，而且在同一窗口里发生。

### 对部门负责人

1. 要看部门总结
2. 要看机构总结
3. 要看 AI 判断
4. 要决定自己是否介入
5. 要自己把结论转成动作

但页面目前没有足够清楚地区分：
- 我必须处理的
- 我可以下钻的
- 我先不用管的

## 8. 信号到交互映射表

| 信号名 | 来源数据 | 触发条件 | 面向角色 | 应出现位置 | 表达方式 | 应触发动作 | 证据不足时降级 |
|---|---|---|---|---|---|---|---|
| 主线推进 | event line summary + review entries | 同一事件线本周有连续动作 | 个人、leader、管理者 | 首屏状态条 / 事件线卡 | 一句话主线 + 完整度 | 查看证据 / 排下周动作 | 只显示“本周发生了什么” |
| 最大阻塞 | blockerReason + supportNeeded + event line blocker | 重复 blocker 或 blocked status | 个人、leader | 首屏阻塞卡 | 阻塞句 + 谁受影响 | 发支持请求 / 指定负责人 | 只显示“阻塞待澄清” |
| 高风险任务 | overdue / low coverage / repeated slips | 状态或完整度触发 | 个人、leader | 列表顶层 / calendar signal | 红色风险条 | 改期 / 转支持 / 调整优先级 | 只显示逾期事实 |
| 会后未闭环 | meeting action item + open task | 会后任务未关闭 | 个人、leader | 会后信号卡 | 行动项未闭环 | 转任务 / 约跟进会议 | 仅显示“会后待跟进事项” |
| 日历承接缺口 | tasks without calendar slot | 关键任务未上历 | 个人 | 日历顶部状态条 | “关键任务未承接进日历” | 一键排期 | 只显示未排期数量 |
| 支持债务 | support requests + blocked tasks | 多条支持未回收 | leader、管理者 | summary 首屏 | 债务热区 | 指派 owner / 约同步 | 只显示支持请求数 |
| 审批瓶颈 | needsReview / approvalState pending | pending 累积 | leader、管理者 | 管理卡 | 审批链卡住 | 发审批动作 / 责任下钻 | 只显示待复核数量 |
| 负荷不均 | owner/task load distribution | 负荷集中 | leader、管理者 | summary 卡 | 人员负荷信号 | 重排任务 / 调整协作 | 只显示任务分布事实 |
| 角色偏离 | orgContext + task type | 角色与任务偏离 | leader、管理者 | 管理信号卡 | “谁在做不该由自己做的事” | 重分派 / 调整边界 | 只提示可能偏离 |
| 下周重点建议 | review analysis + plans + event line memory | 周判断完成 | 个人、leader | 周复盘底部动作区 | 1-3 项 next best actions | 转任务 / 排日历 / 指负责人 | 只输出“建议先补信息” |

## 9. 页面骨架改造建议

### 9.1 首屏最前面应该放什么

当前不该先看到长分析，也不该先看到大段文稿。  
首屏应该先看到：

1. 顶部判断状态条
   - 本周纳入判断的任务数
   - 未纳入判断数
   - 主线数
   - 高风险数
   - 支持债务数
2. 首屏 3 张优先卡
   - 现在最该做什么
   - 现在最该介入什么
   - 现在哪些判断只到 summary-only
3. 动作卡面板
   - 排入日历
   - 发支持请求
   - 转会议动作
   - 指向负责人

### 9.2 哪些内容应降到折叠层

1. 长段 AI 解释文
2. 大块 narrative 复盘正文
3. 细分 evidence weights
4. 过多管理层附属卡

### 9.3 顶部状态条应包含什么

1. 判断准备度
2. 未纳入判断计数
3. 高风险信号数
4. 支持债务
5. 日历承接缺口
6. 会后未闭环

### 9.4 “未纳入判断 / 降级处理”区如何呈现

必须单独有一个可见区，不可埋在分析面板内部：

- 哪些任务没纳入
- 哪些事件线只能 `summary_only`
- 缺哪些槽位
- 建议补什么
- 直接动作：
  - 补资料
  - 立即澄清
  - 延后判断

### 9.5 信号卡如何组织

按角色排序，而不是按数据来源排序：

- 个人：
  - 我现在先做什么
  - 哪个阻塞不能再拖
  - 哪件事必须上历
- 部门负责人：
  - 哪条线需要介入
  - 哪些支持债务在堆积
  - 谁过载 / 谁承接不足
- 管理者：
  - 哪条主线真推进
  - 哪些问题应自己介入
  - 哪些问题不该自己介入

### 9.6 证据抽屉如何进入

每张判断卡都应该有同一位置的“查看证据与下钻”入口，不要混乱分布。

### 9.7 动作卡如何闭环

每张卡至少应支持落到 5 类动作之一：

1. 转任务
2. 发支持请求
3. 转会议动作
4. 排入日历
5. 指向负责人

### 9.8 角色切换怎么做

不要把个人、部门、机构混在一张连续长页里。  
角色切换应变成显式视角切换：

- 我的判断
- 部门判断
- 机构判断

### 9.9 日历信号如何进入页面

日历不应只是一个独立 tab。  
它应该回流成首屏信号：

- 本周关键任务仍未上历
- 会后动作没有承接进日历
- 时间块与优先级不一致
- 高风险任务下周无时间承接

## 10. 对 manager adoption 最危险的当前流程

1. **collect 阶段就出现 summary panel**
   - manager 会被迫在“证据未收完时”看到结论
2. **个人、部门、机构总结放在同一组件内**
   - 容易造成“信息很多，但不知先看什么”
3. **动作卡不够前置**
   - 管理者看到结论，却还得自己想怎么落动作
4. **日历没有变成管理信号**
   - manager 看不到“关键任务是否真的承接了时间”
5. **页面更像 AI 报告页，而不是判断操作台**
   - 会削弱管理者信任和复用意愿

## 11. 本文件的收口判断

从人类工作流与 adoption 角度看，当前模块最需要修的，不是再给更多分析卡，而是：

**先把“收件 / 执行 / 排期 / 复盘 / 判断 / 动作”拆成更清晰的节奏层，并把动作闭环前置。**

因为只有这样，个人才知道今天该先做什么，leader 才知道自己该介入什么，manager 才会把它当作判断操作台，而不是又一张会说话的报告页。
