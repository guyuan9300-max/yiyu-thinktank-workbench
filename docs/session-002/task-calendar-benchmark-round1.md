# Session 002 · Task/Calendar Benchmark Round 1

## Scope

本轮不是做竞品抄作业，而是看成熟产品和成熟团队如何解决下面这些问题：

- 任务如何被快速收进系统
- 任务如何真正承接进日历
- 周计划 / 周复盘如何形成稳定 ritual
- 会后动作如何闭环
- manager 如何看信号而不是看流水账
- AI / 自动化提示如何不变成噪音

本轮选取的对象：

1. Asana
2. Jira
3. Linear
4. Sunsama
5. Reclaim
6. Fellow
7. 15Five
8. Range

## 1. 对标维度

| 维度 | 我们要学什么 |
|---|---|
| 任务捕捉方式 | 任务如何进入系统，不增加太多输入成本 |
| 日历承接方式 | 任务如何真正落到时间，而不是只挂 due date |
| 周计划 / 周复盘方式 | 如何形成固定节奏，而不是一周一次大报告 |
| 会后动作闭环 | 会议内容如何变成负责人与后续动作 |
| manager 信号呈现 | 管理者最先看到什么，不先看什么 |
| 角色分层 | 个人、leader、manager 是否看不同视角 |
| 证据透明度 | 为什么系统这么判断，能否下钻 |
| 动作闭环能力 | 看完是否能直接转任务 / 支持 / 会议 / 日历 |
| adoption / ritual 设计 | 如何形成习惯而不是一次性尝试 |
| 模板 / 节奏机制 | 如何把复杂流程变成重复可做的工作习惯 |

## 2. 对标对象与模式

| 对象 | 主要可借模式 | 适合借到益语的环节 | 不该直接照搬的地方 |
|---|---|---|---|
| Asana | Inbox、项目/组合视图、周逾期提醒、 recurring tasks | 协作收件箱、manager roll-up、例行节奏提醒 | 容易把益语拉回普通任务管理工具 |
| Jira | backlog / sprint / board 分层 | 将“收件池”和“本周承接面”分层 | 工程味太重，普通用户学习成本高 |
| Linear | custom views、triage、issue → initiative 视角切换 | 角色化默认视图、轻量 triage | 太偏 issue 管理，容易削弱人的工作语境 |
| Sunsama | 日/周 ritual、daily planning、shutdown、backlog groom | 个人日程承接、周节奏养成 | 不适合直接承载复杂团队判断链 |
| Reclaim | 任务时间自动承接、自动锁定未来时段 | 日历承接缺口、关键任务上历提醒 | 如果直接自动排满，会压过人的判断 |
| Fellow | meeting → action items → follow-up | 会后动作闭环、会议驱动任务 | 不适合把整个系统做成会议工具 |
| 15Five | 周 check-in、priorities、manager 周节奏 | 个人周总结、leader 周总结、重复 priority 机制 | 太偏 HR/check-in，会稀释任务推进语境 |
| Range | 团队 check-in 与 manager visibility | 团队周节奏与轻量更新 | 如果照搬，会让系统变成 standup 工具 |

## 3. 值得借的模式

### 3.1 Asana：收件和管理视角分层

可借模式：

- `Inbox` 作为协作输入入口
- `Calendar view` 明确区分有日期和无日期任务
- `Portfolio saved views` 让不同角色按风险/优先级快速切视角
- recurring task / overdue reminder 强化节奏

适配益语的方式：

- 保留“协作收件箱”，但把它做成真正的 triage 层
- 在任务页首屏加“未承接进日历 / 本周逾期 / 待复核”固定状态条
- 给 leader / manager 预置默认视角，不让他们和个人看同一排序

### 3.2 Jira：收件池和执行面分层

可借模式：

- backlog 和 sprint/board 分层
- 先决定本周承接什么，再看执行

适配益语的方式：

- 把“协作收件箱 / 候确认 / 本周承接 / 已上历”明确分层
- 不再让所有状态混在一个连续任务列表里

不该误抄：

- 不要把普通员工逼进工程 issue 心智
- 不要引入复杂字段和工程流程术语

### 3.3 Linear：默认视图和 triage 节奏

可借模式：

- custom views / favorited views
- triage 先收口，再分发到不同工作面

适配益语的方式：

- 给个人、部门负责人、机构管理者设置不同默认视角
- `program director` / manager 类角色应看到“需要介入什么”，不是“全部任务”

不该误抄：

- 不要让益语退化成 issue 列表

### 3.4 Sunsama：工作习惯先于复杂结构

可借模式：

- daily planning
- weekly ritual
- daily shutdown
- backlog grooming

适配益语的方式：

- 把“关键任务承接进日历”做成每日 ritual
- 把“周复盘 collect”做成短动作链，而不是一上来就长文

不该误抄：

- Sunsama 本身更像个人节奏工具，不能承载益语的 manager judgment 和证据链

### 3.5 Reclaim：时间承接是动作，不是装饰

可借模式：

- 任务进入 calendar 后有真实时间块
- 自动锁定可避免临时挤压

适配益语的方式：

- 高优先任务、会后动作、下周重点应有“未上历”提醒
- 任务判断系统必须能指出“关键动作没有时间承接”

不该误抄：

- 不要替用户把一切自动排满
- 益语要保留人的判断权，系统应提示而不是完全代做

### 3.6 Fellow：会议动作闭环

可借模式：

- meeting notes 不是结尾，action items 才是
- 跟进动作和下次会议直接关联

适配益语的方式：

- 会议纪要和 action item 必须更自然地桥接到任务与事件线
- 周判断首屏可增加“会后未闭环”信号卡

不该误抄：

- 不要把任务系统变成会议纪要系统

### 3.7 15Five：check-in 不是写报告，而是压 priorities

可借模式：

- priorities 用于“这周最重要的短期事项”
- 周期性 check-in 能形成节奏
- 已完成项可自动进入下一周期

适配益语的方式：

- 个人周总结不该继续拉长，而应围绕“本周推进 / 最大阻塞 / 下周重点”
- 下周重点建议应该能快速转成任务和日历动作

不该误抄：

- 不要把复盘做成 HR 问卷或文化打卡

### 3.8 Range：manager 看节奏和阻塞，不看流水账

可借模式：

- 团队 check-in 帮 manager 快速感知风险和状态

适配益语的方式：

- manager 首屏应看到：
  - 主线推进
  - 阻塞热区
  - 支持债务
  - 下周会放大的问题

不该误抄：

- 不要做成 standup feed
- 益语仍要保留事件线、证据链和动作闭环的深度

## 4. 不该误抄的成熟模式

### 4.1 普通任务软件路线

风险：

- 会把益语拉回：
  - 收任务
  - 改状态
  - 看列表

然后丢掉它真正独特的：
- 周判断
- 事件线
- 证据链
- 复盘信号

### 4.2 单纯仪表盘路线

风险：

- 看上去专业
- 实际不驱动个人和 manager 动作
- 管理者会看一次，但不会变成周节奏工具

### 4.3 长文 AI 报告页路线

风险：

- 会说得更多
- 但个人更难用，manager 更难信
- 也不会形成稳定 ritual

### 4.4 工程工具路线

风险：

- issue/board/sprint 过重
- 对顾问、负责人、普通成员学习成本太高

### 4.5 重流程管理后台路线

风险：

- 录入成本高
- adoption 低
- 复盘、支持、排期最终都回到人工维护

## 5. 对益语最有启发的成熟项目模式

除了产品本身，更重要的是成熟团队怎么让工具起作用。

### 5.1 周节奏必须稳定

成熟团队通常不是“想到才复盘”，而是固定 ritual：

- 每日收件 / 计划
- 会后 action closure
- 周末/周初 check-in
- manager 固定看团队风险

对益语的启发：

- 周复盘不是一张长文页面，而是一整周节奏的收束点

### 5.2 模板和 ritual 比更多字段更重要

成熟工具之所以有效，往往因为：

- 有固定节奏
- 有固定看板
- 有固定动作模板

而不是因为字段特别多。

对益语的启发：

- 要把“发支持请求”“会后承接”“下周重点上历”做成固定动作模板

### 5.3 manager adoption 依赖排序，而不是信息总量

成熟管理工具通常先告诉 manager：

- 你先看哪三件事
- 你现在介入哪一件

不是先给满屏信息。

对益语的启发：

- manager 首屏排序必须和个人完全不同

## 6. 第一轮对标结论

### 值得借的核心模式

1. 收件池和执行面分层
2. 关键任务必须承接进日历
3. 周节奏要靠轻 ritual，而不是长文
4. 会后动作必须显式闭环
5. manager 只看最值得介入的信号
6. 角色视图必须不同
7. 动作入口必须前置

### 不该误抄的核心模式

1. 不要做成普通任务软件
2. 不要做成单纯仪表盘
3. 不要做成长篇 AI 报告页
4. 不要做成工程团队专用 issue 工具
5. 不要把 adoption 建在更多字段和更多配置上

## 7. 对益语的第一轮适配建议

1. 任务页首屏先变成“判断状态条 + 动作区”，不是先变成更大分析页
2. 把“收件池 / 本周承接 / 已上历 / 周复盘”拆成更明确的节奏层
3. 给个人、leader、manager 做不同默认视角
4. 强化会后动作闭环
5. 强化“未上历关键任务”信号
6. 将周复盘压成短动作链，再决定是否展开长文文稿

## 8. 本文件收口判断

成熟产品真正值得借的，不是页面长相，而是：

**如何用更少的输入、稳定的节奏和更短的动作链，让任务、时间、会议和复盘真正连起来。**

益语如果要保持独特价值，就必须借：
- ritual
- role-specific views
- action closure

但不能误抄成：
- 普通任务软件
- 报表系统
- 长文 AI 展示页
