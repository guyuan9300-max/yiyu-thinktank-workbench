# Session 002 · Task/Calendar Synthesis

## Scope

本轮汇总了两条工作流：

- Workflow A
  - `product_manager`
  - `ux_researcher`
  - `interaction_designer`
- Workflow B
  - `competitive_intelligence_analyst`
  - `product_operations_manager`

同时纳入了两轮交叉质询。

## 1. 本 Session 的核心发现

### 1.1 当前模块不是能力不够，而是节奏混乱

当前“日历与任务判断系统”已经有很多真实能力：

- 协作收件
- 任务列表
- 日历拖拽改期
- 单项复盘
- 周判断分析
- 层级总结
- gap 处理
- action card

问题不在于没有功能，而在于：

**这些功能现在被压在同一条连续流程里，没有形成清晰的工作节奏。**

### 1.2 当前页面更像“AI 复盘报告页”，而不是“判断操作台”

原因不是 AI 卡片太多本身，而是：

- collect 阶段就叠加了分析与总结
- action 入口不够前置
- 未纳入判断 / 降级输出不可见
- calendar 没有成为判断链的一部分

### 1.3 当前模块最伤信任的地方不是 AI 会不会说，而是边界不清

用户当前不容易看明白：

- 哪些内容已经被纳入判断
- 哪些只是背景
- 哪些只是推断
- 哪些证据不足，只能 summary-only
- 这条建议到底应该怎么落动作

## 2. 当前模块真正的主问题

### 2.1 结构性问题

1. 收件、执行、排期、复盘、层级总结被压在一个连续模块里
2. judgment、evidence、action 没有在前台清晰分层
3. calendar 仍是时间容器，不是 judgment chain 的一部分
4. manager 和个人目前还容易共享同一套排序逻辑

### 2.2 页面问题

1. 首屏缺少判断状态条
2. 首屏缺少“未纳入判断 / 降级处理”区
3. collect 阶段就出现 summary panel，容易过早汇报化
4. 动作卡没有成为页面主结构

### 2.3 交互问题

1. 单项复盘、分析、整周生成之间的切换成本高
2. 会后动作和支持动作没有足够短的路径
3. 日历承接关键任务没有变成系统强信号
4. manager 看结论后，不够容易直接采取动作

### 2.4 信任问题

1. 缺少统一首屏 readiness 提示
2. 缺少证据抽屉前置
3. 缺少“这张卡为什么现在能判断 / 为什么现在不能判断”的解释

## 3. 两轮交叉质询

## Round 1 · 现状与工作流质询

### 由 `product_manager` 发起

对 Workflow B 的质询：

1. 外部成熟产品之所以有效，是因为它们做对了 workflow 还是做对了 UI？
2. 哪些模式是在解决“用户知道现在该做什么”，哪些只是让信息看起来更规整？
3. 如果借来了 backlog、calendar、check-in、portfolio，这些模式会不会把益语的判断链切碎？

### `competitive_intelligence_analyst` 的反问

1. 当前工作流组是否过度从现状页面出发，低估了成熟产品里“节奏先于页面”的重要性？
2. 如果不把收件池、日历承接、周 check-in 变成 ritual，再强的 judgment chain 是否也会停在展示层？

### `product_operations_manager` 的质疑

1. 当前页面里是否存在“系统想做很多，但没有一个稳定的周节奏让人持续回来使用”？
2. 如果没有明确的 ritual，leader 为什么会每周回来？

### Round 1 裁决

裁决：

- 不能只按“页面长什么样”讨论
- 必须先把任务系统拆成更清晰的节奏层：
  - 收件
  - 本周承接
  - 上历
  - 单项复盘
  - 周判断
  - action closure

## Round 2 · 外部模式适配性质询

### 由 `ux_researcher` 和 `interaction_designer` 发起

对 Workflow B 的质疑：

1. 哪些成熟产品只是因为用户群体更专业，而不是模式本身真的更好？
2. 哪些模式会让益语丢掉“事件线 + 周判断 + 证据链 + 动作闭环”这条独特路线？
3. 哪些产品之所以有效，是因为它们缩短动作链，而不是分析更复杂？

### 对 Workflow A 的反质疑

1. 你们是否还在用“当前已有组件”来解释问题，而没有真正指出哪些必须下沉、哪些必须前置？
2. 如果 collect 阶段还保留当前这种混合结构，是否会继续把用户推向“长文复盘心智”？

### Round 2 裁决

裁决：

- 不照搬成熟产品的页面长相
- 只借：
  - ritual
  - role-specific view
  - action closure
  - visible degrade
- 明确放弃：
  - 普通任务软件化
  - 长文 AI 报告页化
  - 工程 issue 工具化

## 4. 下一轮 P0 改造范围

### P0-1 首屏骨架重排

先把任务与日历首页收成：

1. 顶部判断状态条
2. 当前最该先做什么
3. 当前最该介入什么
4. 未纳入判断 / 降级处理
5. 动作卡面板

### P0-2 收件 / 承接 / 上历 分层

把：

- 协作收件
- 本周承接
- 已承接进日历

做成更清晰的节奏层，而不是都挤进任务列表。

### P0-3 单项复盘优先，长文后置

当前最该强化的是：

- 单条复盘可持续输入
- 单条复盘保存
- 单条复盘状态/支持/下一步闭环

长文文稿放到后面，不作为首要体验。

### P0-4 动作卡闭环前置

任何 judgment card 至少能落到：

1. 转任务
2. 发支持请求
3. 转会议动作
4. 排入日历
5. 指向负责人

### P0-5 证据与降级显式化

必须把：

- summary-only
- not-ready
- missing slots
- 未纳入判断

做成用户一眼可见的首屏区域。

## 5. 暂不处理项

1. 不先做高保真 redesign
2. 不先大改客户工作台
3. 不先扩写更长的 AI 叙事文稿
4. 不先增加大量新表单
5. 不先引入新的业务真源

## 7. 本 Session 的统一收口判断

**日历与任务判断系统当前最需要修的，不是“让 AI 说得更完整”，而是“把收件、执行、排期、复盘、判断、动作重新收成可持续的工作节奏，并把动作闭环前置”。因为只有先解决“节奏与动作断裂”，个人和管理者才能真正从这个系统获得启示和动作。**
