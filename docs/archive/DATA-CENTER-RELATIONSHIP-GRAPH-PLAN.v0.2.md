# 数据中心 · 关系网升级计划

> 状态：**v0.2 — Patch 后版本**
> 日期：2026-05-15
> 变更说明（相对 v0.1）：① 加三段流水线与上下游交接合同 / ② 明确产品定位为"所有业务判断的背景知识引擎" / ③ 用"高老师离职"端到端真实案例替换抽象演示 / ④ 加工作台线程进度跟踪表 / ⑤ "故事网"→"关系网"全文替换
> 用户原话定义的目标："让数据中心能读到每一份文档、完整理解每一份文档、检索时不遗漏内容，并把每个细节中的关键事实提炼出来，串成完整的逻辑描述系统 —— 发生了什么事、每个人扮演什么角色、每件事的价值和核心意义、它发生的时间段和关键里程碑、交付物是什么。"

---

## 0 · 一句话总结

**把数据中心从一座"文档仓库"升级成一张"客户关系网"** —— 不只是存了什么文档，而是能讲清楚每个客户在做什么、谁在做、做到哪一步了、为什么这件事重要 —— **作为所有上层业务模块（战略陪伴/成长中心/任务与日程）所有判断的背景知识**。

---

## 1 · 现在的问题

数据中心目前知道**每片树叶长什么样**（每份文档都存着），但**不知道这片森林是怎么长起来的**（这些文档说的是同一件事吗？谁负责？什么时候开始的？卡在哪儿？）。

举例：

| 问题 | 现在的回答（功能型） | 我们想要的回答（业务型） |
|---|---|---|
| 日慈这个客户现在卡在哪？ | 知识完整度 47%，建议补充文档 | 高老师上周离职导致品牌升级停摆；张真新加坡出差未谈成实质合作 |
| 詹瑶是谁？ | 一个名字 | 黔行秘书长 + 共同创始人 + 吴建林配偶 + 4 月刚从精神状态恢复回归 |
| 黔行 6 月工作坊要做什么？ | 找不到答案 / 散落在多份文档里 | 顾老师创新方案：用自研软件做工作坊，从绩效矛盾切入，不强调"AI"，备案豆包方案 |

第一行是数据中心目前能给的；第二行才是**超级员工**应该给的。第二行需要的不是更多文档，而是**把文档里的细节串成一张关系网**。

## 2 · 三段流水线（关键架构）

整个数据中心升级是三段流水线协同：

```
       上游                       中游（我）                   下游
   ┌─────────────┐          ┌──────────────────┐         ┌──────────────────┐
   │ 文档分解线程 │ ───────→ │ 关系网编织线程    │ ──────→ │ 业务判断模块      │
   │              │ 召回结果 │                   │ 判断背景│                   │
   │ R11 浅解构   │          │ 锁定关键细节      │         │ · 战略陪伴        │
   │ R11.x 精解构 │          │ 串成关系网        │         │ · 成长中心        │
   │ R12 召回FTS  │          │ 发现困惑          │         │ · 任务与日程     │
   │ R12 健康大盘 │          │ 前台问澄清        │         │   业务/任务/未来  │
   └─────────────┘          └──────────────────┘         │   判断           │
   "把每份文档解析好"        "把细节串成关系网"           │ · 客户工作台问答  │
                                                          │ · 周复盘          │
                                                          └──────────────────┘
                                                          "拿背景做出业务判断"
```

### 2.1 上下游交接合同

**上游给我什么**（消费工作台输出）：

| 我从工作台读 | 工作台何时给 | 我做什么 |
|---|---|---|
| 文档浅解构 16 个字段（人名/日期/金额/决策等） | 文档入库后异步几秒内 | 把人名进花名册、决策进事实卡片、金额进主线交付物 |
| 文档精解构（合同 9 字段、会议纪要字段、项目方案字段...） | R11.x 完成后批量回填 | 直接映射到关系网对应节点 |
| FTS 召回结果 | 实时查询时 | 用作澄清面板的"引文支撑" |
| 文档健康度（覆盖率/理解率/召回率） | 健康度大盘上线后 | 跟我的"关系网完整度"合并显示 |

**我给下游什么**（关系网作为背景输出）：

| 下游模块 | 它需要的背景 | 我提供的形式 |
|---|---|---|
| **战略陪伴** | 客户当前主线 / 卡点 / 风险 / 关键人物 / 矛盾 | 客户关系书 + 主线全景 + 澄清面板 |
| **成长中心** | operator 的能力证据 + 岗位锚点 | 从关系网投影出的人能力快照 |
| **任务与日程** | 任务关联到哪条主线 / 主线推进健康度 | 任务挂载到关系网主线 + 主线快照 |
| **客户工作台问答** | 当前客户的完整背景 | AI 问答时自动加载关系网 |
| **周复盘** | 上周新增承诺/决策/风险 + 主线变化 | 每周关系网快照 + 变化摘要 |

### 2.2 关键约定

- **工作台先于我跑一个 Phase** —— 它要先把"食材准备好"，我才能"编织关系网"
- **我不重复做工作台的事** —— 我不从原始文档抽字段，我**只从工作台抽好的字段里挑、归一、串接**
- **下游模块不重复做我的事** —— 它们不自己推理客户当前状态，**直接读我的关系网**

## 3 · 我的产品定位 + 评估标准

### 3.1 产品定位（用户原话定义）

> **"你做的东西就是所有判断的背景"**

准确翻译：

- 工作台模块 = **"问什么找什么"**（用户主动搜文档）
- 我 = **"AI 心里清楚整个客户的来龙去脉"**（背景知识引擎，被动驱动一切判断）
- 当战略陪伴说 "日慈品牌升级停摆了"，**它是从我这里读的背景**，不是它现场推理

### 3.2 我评估自己工作好不好的标准

❌ **不是**："关系网有多完整 / 我建了多少表 / 我抽了多少字段"
✅ **是**：**"下游模块拿到我的背景之后，能否给出更好的业务判断"**

具体到指标：

| 衡量维度 | 看哪里 |
|---|---|
| 战略陪伴的卡点描述是否变成业务语言 | 之前："AI 卡在没数据" → 之后："高老师离职导致品牌升级停摆" |
| 战略陪伴的澄清问题是否够针对性 | 是否达到我今天问日慈 10 问的水平（澄清质量评分 ≥ 80） |
| 客户工作台 AI 问答是否引用关系网背景 | 答案末尾是否带"基于：3 次会议 / 5 份文档 / 2 条主线" |
| 任务与日程的任务智能简报是否准确 | 简报是否能说出任务挂在哪条主线、跟谁有关 |
| 成长中心的能力评估是否有证据钻取 | 每个分数能否回到原始事实 |

**这套标准的意思**：关系网建好了不算赢，**下游模块因为关系网变好用了**才算赢。

## 4 · 关系网要回答的 4 件事（你定的目标）

| 你的要求 | 关系网里的承载体 |
|---|---|
| **发生了什么事** | 事件时序 + 事实证据 |
| **每个人扮演什么角色** | 人物档案 + 角色关系 |
| **每件事的价值和核心意义** | 业务主线 + 战略意义 |
| **时间段、关键里程碑、交付物** | 主线生命周期 + 节点 + 交付物 |

## 5 · 关系网的 4 层结构

关系网不是一张表能存下的东西，要分 4 层来构建：

```
       ┌─────────────────────────────────────────────────┐
   ④   │  关系书生成 = 「客户关系书」                     │
       │  自动生成的、可读的客户 DNA 文档                  │
       └─────────────────────────────────────────────────┘
                          ▲
       ┌─────────────────────────────────────────────────┐
   ③   │  业务主线 = 「项目台账」                         │
       │  每件正在做的事有起止时间、里程碑、交付物         │
       └─────────────────────────────────────────────────┘
                          ▲
       ┌─────────────────────────────────────────────────┐
   ②   │  事实证据 = 「事实卡片」                         │
       │  每个细节贴一张带语义标签的便签                   │
       └─────────────────────────────────────────────────┘
                          ▲
       ┌─────────────────────────────────────────────────┐
   ①   │  人物档案 = 「花名册」                           │
       │  机构内外所有人物的通讯录 + 关系图               │
       └─────────────────────────────────────────────────┘
                          ▲
       ┌─────────────────────────────────────────────────┐
       │  工作台供应：每份文档解析好的结构化字段           │
       └─────────────────────────────────────────────────┘
```

### 5.1 第一层 · 「花名册」（人物档案）

**它解决什么**：让关系网里每个人物节点是稳定的、唯一的。

**用日慈的例子**：

| 系统里看到的称呼 | 花名册里的同一个人 |
|---|---|
| "王强" | 王强（id: person_wang_qiang） |
| "强哥" | 同上 |
| "王老师" | 同上 |
| "王强老师" | 同上 |

**附带信息**：
- 当前职位：心灵魔法学院负责人 / 心盛实管
- 在岗状态：在职
- 关系：跟谁是配偶、跟谁是同事、跟谁是上下级
- 健康/精神状态信号
- 工作风格特征（"管得细 / 不被理解"这种 DNA）
- 是不是 AI 模拟角色（防止把测试角色当真）

**判断标准（什么算"花名册做好了"）**：
- 客户库里能查到所有真人物 + 他们的所有别名
- 同一个人的不同称呼能合并
- ASR 转写错误（"张瑶" 其实是 "詹瑶"）能识别并校正
- AI 模拟角色（如"庆华"）有清晰标记

---

### 5.2 第二层 · 「事实卡片」（事实证据）

**它解决什么**：让关系网里每条事实/每个事件是可解释的。

**当前问题**：所有 168 张证据卡片都打着同一个标签（"中性"），等于没标签。

**升级后**：每张卡片带 4 个标签：

| 标签维度 | 选项 | 例子 |
|---|---|---|
| **业务语义** | 承诺 / 决策 / 风险 / 事实 / 教训 / 进展 | 4/19 詹瑶承诺 5/1 前发更新价值观 → 标记"承诺" |
| **影响** | 推进 / 中性 / 阻塞 | 高老师离职 → 标记"阻塞" |
| **时间锚** | 业务时间（承诺日 / 发生日 / 失效日） | 2026/5/1（承诺日） |
| **置信度** | 来源（用户说/AI抽/多源交叉）+ 0-1 评分 | 用户说 → 0.95 / AI 抽 → 0.7 |

**判断标准**：
- 每张卡片都能说清"这是谁、什么时候、做了什么、属于哪种业务行为"
- 高置信度的事实可以直接用，低置信度的事实进入澄清队列
- 用户在战略陪伴里能一眼看到"这周新增的承诺 / 决策 / 风险"

---

### 5.3 第三层 · 「项目台账」（业务主线）

**它解决什么**：把散落的事实串成可追踪的"一条事情"。

**当前问题**：日慈下 5 条业务主线，颗粒度完全混乱 ——「日慈战略陪伴」（项目级）跟「跟笑雨老师核对教师项目进度」（一次性沟通级）被当作同种东西。

**升级后**：5 种颗粒度，每条主线明确属于哪种：

| 颗粒度 | 寿命 | 例子 |
|---|---|---|
| **项目级** | 1-3 年 | "日慈战略陪伴" / "大山里的音乐课堂" |
| **阶段级** | 1 季度 / 半年 | "日慈使命愿景价值观调整" / "黔行 6 月工作坊" |
| **承诺级** | 几周-几个月 | "5/1 前发更新价值观给詹瑶" |
| **一次约见级** | 一次性 | "4/19 河南面谈詹瑶" |
| **机制级** | 长期持续 | "建立重点客户周跟进节奏表" |

**每条主线带的信息**：
- 是哪种颗粒度（5 选 1）
- 起止时间 + 关键里程碑
- 责任人 + 协作人（从「花名册」引用）
- 交付物 + 验收标准
- 父主线 / 衍生主线（嵌套关系）
- 当前推进到哪一步
- **每个维度的澄清状态**：✓ 已确认 / ⚠️ 模糊 / ❌ 缺失

**判断标准**：
- 用户进入战略陪伴能看到一个客户当前**正在推进的所有主线**（按颗粒度分层）
- 每条主线能讲清楚"做到哪一步、卡在哪、下一步什么、距离交付日还有多久"
- 主线的卡点用**业务语言**描述（"高老师离职导致品牌升级停摆"），不是 AI 视角废话

---

### 5.4 第四层 · 「客户关系书」（关系书生成）

**它解决什么**：把上面三层串成可读的客户 DNA 文档（不是"AI 临时写的摘要"，而是"长期沉淀的资料大全"）。

**输出形态**：

每个客户自动生成一份"关系书"，包含 8 段：

1. **机构基本信息**（治理结构 / 决策机制）
2. **战略定位**（北极星 / 战略动因 / 落地状态）
3. **业务结构**（业务线占比 / 服务对象 / 项目谱系）
4. **关键人物**（花名册 + 角色 + 状态 + 风格）
5. **合作关系**（合同 / 服务期 / 续约状态）
6. **主线全景**（所有项目级 / 阶段级 / 承诺级主线的当前推进状态）
7. **风险信号**（资方流失 / 人员变动 / 战略不定数）
8. **未澄清矛盾点**（自动检测的矛盾 + 待澄清队列）

**判断标准**：
- 这份文档**可以直接给客户工作台用 / 给战略陪伴用 / 给周复盘用**
- 这份文档的每一句话**都能钻取到原始证据**
- 这份文档**会自动随新事件更新**，不是死的
- 这份文档**能驱动澄清面板**（缺什么字段就提什么问题）

## 6 · 5 个 Phase 的执行节奏

下面是从现在到完成的 5 步推进，每步说清楚：做什么、做完用户能看到什么、跟工作台怎么配合、什么算做完了。

---

### Phase 0 · 对齐定义（**我们正在做的事**）

**做什么**：把关系网的 4 个回答能力、4 层结构、跟工作台的协同边界全部定义清楚。本文档是 Phase 0 的产物。

**做完用户看到什么变化**：还看不到 UI 变化，但**整条改造路径已经清晰**，后面的工程不会跑偏。

**跟工作台配合**：工作台同期在做 R11.1（员工合同结构化）→ 即将做 R11.5（自动触发器）+ R11.2（会议纪要）+ R12（健康度大盘）+ FTS 召回。两边节奏对齐。

**完成标准**：
- 用户确认 4 层结构方向正确 ✅
- 用户确认 5 个 Phase 节奏（**v0.2 patch 完成，等用户最终拍板**）
- 拆出 Phase 1 的施工稿（这份文档之后再单独写）

---

### Phase 1 · 关系网骨架（先建第一层 + 第三层）

**做什么**：
- 建好「花名册」表（包含日慈 + 黔行所有真人物 + 他们的别名 + 关系 + 状态）
- 升级「项目台账」（把现有 41 条业务主线打上颗粒度标签 + 加生命周期字段）
- 清理测试数据（庆华 / 教室改造测试任务等标记为模拟/测试）

**做完用户看到什么变化**：
- 战略陪伴里点开日慈，能看到一份**真实的人物花名册**（10 来个真人物 + 别名 + 状态）—— 而不是只有"顾源源"和"本机用户"
- 战略陪伴里能看到日慈的 5 条主线**按颗粒度分层显示**（"项目级 1 条 + 阶段级 2 条 + 承诺级 1 条 + 一次约见级 1 条"）
- 测试数据从主视图里消失

**跟工作台配合**：消费工作台的 R11.1（员工合同字段 → 进花名册）+ R11.5（自动触发器，新文档实时进花名册）

**完成标准**：
- 日慈 + 黔行的所有真人物都进了花名册
- 日慈 + 黔行的所有业务主线都有颗粒度标签
- 用户看到的战略陪伴界面**测试数据消失** + **人物清晰** + **主线分层**

**预估时间**：3 周

---

### Phase 2 · 关系网血肉（第二层 · 事实证据）

**做什么**：
- 让每张事实卡片带上"业务语义 + 影响 + 时间锚 + 置信度"4 个标签
- 把工作台 R11.2 / R11.3 提取的结构化字段（会议/项目方案）摘到关系网的正确节点上
- 替换掉所有 AI 模板废话（"当前没有特别突出的阻塞..."这类）

**做完用户看到什么变化**：
- 战略陪伴里能看到"这周新增 N 条承诺 / M 条决策 / K 条风险"
- 每条主线的"卡点"用真实业务语言描述（"高老师离职 → 品牌升级停摆"）—— 不再有 AI 视角废话
- 战略陪伴的"事实澄清"模块开始有内容显示

**跟工作台配合**：工作台 R11.2（会议纪要）+ R11.3（项目方案）解构的字段**全部流入**关系网；FTS 召回**喂给澄清面板做引文**

**完成标准**：
- 日慈 + 黔行的 200+ 条事实卡片都有 4 个标签
- AI 模板废话从所有主线、所有判断、所有任务的 next_step 字段消失
- 用户进入战略陪伴能看到**业务语言写的卡点 + 推进信号**

**预估时间**：3-4 周

---

### Phase 3 · 关系网叙事（第四层 · 客户关系书 + 澄清面板）

## 9 · 端到端真实案例 · "高老师离职"走一遍三段流水线

用最近的真实事件，演示从文档进来到下游判断输出的完整流转：

### 9.1 事件背景

5 月初的某个工作日，张真给顾源源打电话，告知：
- 老高（高老师）跟章方同一天离职
- 起因：辞退小鱼成为导火索
- 老高躯体化严重，要去 600 号开药
- 章方有抑郁背景，写了辞职信
- 心盛由强哥接管
- 张真担心长期大额企业捐方可能因章方状态而流失

录音转文字后，文档进了数据中心。下面看三段流水线如何工作：

### 9.2 第一段 · 上游做的（工作台 R11.x）

```
输入：4/20 电话纪要文档（自由文本，500 字左右）
   ↓
工作台浅解构 + 精解构
   ↓
输出：结构化字段
  - 关键人物：高老师 / 老高 / 章方 / 小鱼 / 张真 / HR 顾问
  - 关键事件：
    · 辞退小鱼
    · 老高离职
    · 章方离职（辞职信已批）
    · 心盛由强哥接管
  - 关键日期：
    · 前天（4/18 左右）章方写辞职信
    · 明天（4/21）老高去 600 号
  - 关键金额：(无)
  - 关键决策：
    · 心盛归口变更
    · 章方辞职批准
  - 文档类型：phone_call_summary
  - 文档主旨：心盛团队人员变动同步
  - 200 字摘要：（略）
```

### 9.3 第二段 · 我做的（关系网编织）

```
读到上游输出的 6 个字段集 → 开始编织：

① 人物归一（花名册）：
   - "高老师" + "老高" → 归一到同一个 person（已是花名册里的人）
   - "章方" → 已存在
   - "小鱼" → 新人物，加入花名册
   - "强哥" → 已存在（确认是王强）
   - "HR 顾问" → 不是具名人物，不进花名册

② 状态变更（花名册）：
   - 高老师.employment_status: 在职 → 离职（status_changed_at: 5月初）
   - 高老师.health_signal: 躯体化 / 4/21 去 600 号
   - 章方.employment_status: 在职 → 离职
   - 章方.health_signal: 抑郁背景
   - 小鱼.employment_status: 被辞退

③ 关系变更：
   - 章方原是"小鱼上级" → 标记为"已交接"
   - 强哥（王强）roles 新增："心盛实管 / since: 2026-05"

④ 事实卡片（4 张）：
   - 卡片1：高老师离职 [业务语义=事实, 影响=阻塞, 时间锚=5月初, 置信度=0.95(用户口述)]

   - 卡片3：长期大额企业捐方可能流失 [业务语义=风险, 影响=阻塞, 时间锚=持续, 置信度=0.85]
   - 卡片4：章方辞职信批准 [业务语义=决策, 影响=中性, 时间锚=4月中, 置信度=0.95]

⑤ 主线影响传播：
   - 高老师原负责"机构介绍升级"主线 → 主线状态从 active → 阻塞
   - 高老师还负责"日慈品牌"（之前是 background fact）→ 影响"品牌升级"主线 → 主线状态也变阻塞
   - "心盛计划"主线 → responsible_person 从老高换成王强（带 since 时间戳）

⑥ 矛盾检测：
   - "心盛由强哥接管" vs 之前澄清答案"心盛由王强接管" → 检查别名表 → 强哥=王强 → 不是矛盾
   - "心盛由强哥接管" vs 用户之前提到"心盛由王老师管" → 检查别名表 → 王老师=王强 → 不是矛盾
   ✅ 无矛盾，相反这是"3 种说法 = 同一个人 = 王强"的强证据
```

### 9.4 第三段 · 下游消费（战略陪伴）

```

  • 心盛计划（项目级）: responsible_person 变更 老高 → 王强
    交接时间：2026-05-初

▍日慈本周事实卡片新增
  • 4 张：高老师离职 / 心盛交接 / 资方流失风险 / 章方辞职

▍澄清面板 AI 新问题
  (无新疑问 —— 因为这次纪要里说的事都自洽，"强哥=王老师=王强"已在花名册里归一)

战略陪伴**不需要自己推理这些** —— 它直接读关系网里我提供的 4 张卡片 + 主线状态变化。
```

### 9.5 这个例子说明的事
