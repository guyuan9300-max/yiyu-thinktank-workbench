## 0. 强制使用规则(顾源源 5/22 钦定)

A AI / B AI 在每轮目标检测或新计划制定之前,**必须**按下列顺序读三份文档:

1. **本文档** (`docs/V2.2_DATA_ASSET_INVENTORY.md`) — 知道仓库有什么资产
2. **`docs/V2.2_NORTH_STAR.md`** — 知道整体目标是什么
3. **`docs/V2.1_AI_COLLABORATION.md`** — 知道协作规则

读完后**才能**做:

4. 现状评估 / 偏差度评估
5. 提出修正方案
6. 执行新计划

**违反流程后果**: 像 5/22 之前一样,基于不完整资产视野设计 → 上下文衰减 → 偏差累积 → 大返工。

**触发点示例**:
- 用户说 "重新评估" / "新计划" / "你做错方向了" → 强制重读
- 单次 session 开始 → 强制重读
- 跨 docx 抽取 / 跨表设计 → 强制重读相关段落
- 上下文压缩通知收到后 → 强制重读

## 5. atomic_facts 应该 / 不应该做什么

### 应该做(N2 真正职责):

1. **LLM 跨结构提炼**: 从 v2_documents.markdown_content 抽出"现成表抓不到"的事实

## 6. 重新校正的 V2.2 设计原则(基于真实资产)

### 原则 1 · 现成表优先于 LLM 抽取
任何 NarrativeKernel 段实现,**先查 §4 映射表是否有现成表**;只有现成表为空 / 数据稀疏时,才用 atomic_facts 兜底。

### 原则 2 · atomic_facts 是"层"不是"源"
atomic_facts 提供 LLM 抽出的 fact + 5 维元数据,作为现成表的**补充和引用支撑**,不是替代。

### 原则 3 · 主时间线骨架走 activity_logs
顾源源讲的"时间串人物串文件"骨架,**主数据源是 activity_logs (26,677 行)** + event_line_activities (104 行视图);atomic_facts.time_anchor 是次要数据源。

### 原则 4 · IngestPipeline 4 主路径要绑现有数据源
| 路径 | 现成数据源 |
|---|---|
| workbench_file | v2_documents → DocumentLLMExtractor |
| **task_review** | **要绑** event_line_activities + tasks + weekly_reviews + meetings + agenda_items |
| internet_crawler | intelligence_candidate_items + intelligence_items |
| mobile_ai_chat | chat_messages + chat_threads(目前 N3 占位) |

当前 task_review 路径**有 normalizer 但没绑数据源** — 这就是顾源源说的"路径不通畅"。

### 原则 5 · 0 行表 ≠ 没用,可能是设计完整的空管道
14 张空表里 `key_decisions` (21 字段)、`org_events` (22 字段)、`event_line_state_changes` (19 字段)、`external_evidence_cards` (23 字段) 都是**完整设计但未启用**的管道。比新设计一张表更优先填这些。
